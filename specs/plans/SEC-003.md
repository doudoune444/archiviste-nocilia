# Plan — SEC-003 Security headers gateway : HSTS + couverture exhaustive

## Goal
Ajouter le 5ème header sécurité `Strict-Transport-Security` au router gateway et étendre les tests d'intégration pour garantir que les 5 headers couvrent toutes les réponses (2xx, 4xx, 5xx, 404 asset, 404 fallback, body-limit 400).

## Acceptance criteria recap
- AC-1 : Toute réponse gateway porte `Strict-Transport-Security: max-age=31536000; includeSubDomains; preload`.
- AC-2 : Les 4 headers UI-001 restent présents avec valeurs littérales exactes (CSP, X-Content-Type-Options, Referrer-Policy, X-Frame-Options).
- AC-3 : `GET /healthz` (succès ET échec amont) porte les 5 headers.
- AC-4 : `POST /v1/chat` body > 1 MiB → 400 porte les 5 headers.
- AC-5 : `GET /assets/inexistant.txt` (404 ServeDir) ET `GET /chemin/inexistant` (404 router) portent les 5 headers.
- AC-6 : `if_not_present` sémantique conservée, `get_all(name).iter().count() == 1` sur chaque header testé.
- AC-7 : Implémentation via `SetResponseHeaderLayer::if_not_present` dans le bloc existant `gateway/src/lib.rs`, aucun nouveau middleware, aucune nouvelle crate.
- AC-8 : Helper `assert_security_headers` étendu à 5 headers + constante `HSTS_VALUE`. Tests UI-001 (AC-6, AC-17) passent inchangés.
- AC-9 : Nouveaux tests : `/healthz` 200, `/healthz` workers-down, body-limit 400, asset 404, route 404 fallback.
- AC-10 : CI vert (`cargo fmt --check`, `cargo clippy -- -D warnings`, `cargo test`). Pas de modif `static/`, `handlers/`, `migrations/`.

## Risks / open questions
- **AC-3 ambiguïté "échec amont 5xx"** : le handler `healthz` actuel retourne 200 `"degraded"` sur workers-down (jamais 5xx). Interprétation : le test `sec003_healthz_workers_down_has_headers` couvre le chemin "workers indisponible" en assertant les headers sur cette réponse 200, conformément au comportement réel. Si l'humain veut un vrai 5xx, c'est un changement de handler hors-scope SEC-003 (proposer ticket séparé).
- **Ordre des layers Axum** : `.merge(chat_router).layer(security_headers)` — les layers racine s'appliquent **après** les layers internes (Axum 0.8 wrap outside-in). La 400 réécrite par `handle_body_limit_error` passe ensuite par les `SetResponseHeaderLayer` racine → headers ajoutés. Validation par le test AC-4 lui-même.
- **AC-5 "405-or-404"** : Axum 0.8 sans `fallback()` retourne 404 pour route inconnue (pas 405). Le test assert 404. Si la spec voulait 405 sur méthode mismatch (ex. `POST /healthz`), c'est un sous-cas séparé non listé → on ne le couvre pas.
- **Non-duplication** : aucun handler ni `ServeDir` actuel n'émet de header parmi les 5. Les tests AC-6 UI-001 existants passent déjà — ajouter l'assertion `count() == 1` ne casse rien. Si régression future (handler émettant un header), test AC-6 fail explicitement.
