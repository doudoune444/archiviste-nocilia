# OBS-006 — Endpoint dépendances workers + relais Mistral + clé `mistral` dans GET /v1/status

## Contexte

OBS-002 a livré `GET /v1/status` côté gateway avec un statut de santé agrégé sur trois dépendances directes (`postgres`, `gcs`, `workers`-reachability) et le widget `#health-widget`. La santé de **Mistral** (le LLM, atteignable uniquement depuis les workers, pas depuis la gateway) reste invisible. Ce ticket livre la seconde moitié de la slice santé : une route workers de dépendances qui rapporte l'état de Mistral et la vue interne des workers, son contrat OpenAPI, et l'ajout **additif** de la clé `mistral` au body `GET /v1/status` (la forme passe de 3 à 4 dépendances).

## Acceptance criteria

- AC-1 : Une route workers **`GET /health/dependencies`** est ajoutée sur le service workers ; elle retourne `200 OK`, `Content-Type: application/json; charset=utf-8`, body sanitisé de forme `{"mistral":{"status":"ok"|"down"},"workers":{"status":"ok"|"down"}}` où `workers` reflète la vue interne des workers sur eux-mêmes et `mistral` reflète la joignabilité de Mistral telle que probée par les workers. Aucun message, host, URL, clé API ni stack n'apparaît dans le body (sanitisation cohérente `security.md` §A05).
- AC-2 : La probe Mistral côté workers applique un timeout dur de **2 secondes** ; si la probe Mistral des workers est en timeout, erreur, ou réponse malformée (workers joignables mais incertains sur Mistral), `mistral.status="down"` (résolution OQ-5 : `down` sur incertitude, préserve l'enum fermé à 2 valeurs ; impossible de confirmer `ok` ⇒ `down`).
- AC-3 : Le contrat `specs/openapi/gateway-to-workers.yml` est mis à jour pour décrire `GET /health/dependencies` (méthode, path, schéma de réponse : `mistral` + `workers` en `status: ok|down`). Un run schemathesis sur le contrat modifié passe (CI gate `CLAUDE.md` §7). **Fichier source-vérité humain-only** : la modification est soumise à approbation humaine explicite au moment de l'implémentation (`CLAUDE.md` §Sources de vérité).
- AC-4 : La gateway, lors de la production d'un snapshot `GET /v1/status`, interroge `GET /health/dependencies` des workers (timeout 2 s, en parallèle des autres probes) et **ajoute** la clé `mistral` au body : le body devient `{"status":..,"dependencies":{"postgres":..,"gcs":..,"workers":..,"mistral":..},"checked_at":..}`. Les quatre clés `postgres`, `gcs`, `workers`, `mistral` sont **toujours** présentes (extension additive de OBS-002 D-1).
- AC-5 : `dependencies.mistral` conserve la forme `{"status":"ok"|"down","latency_ms":<int ≥ 0>}` (enum fermé à 2 valeurs, cohérent OBS-002 AC-2) ; `dependencies.mistral.status="ok"` seulement si les workers rapportent Mistral `ok` dans `GET /health/dependencies` (AC-1) ; si les workers sont injoignables OU rapportent Mistral `down`, alors `dependencies.mistral.status="down"` (résolution OQ-5).
- AC-6 : La probe `workers` côté gateway évolue (vs OBS-002 AC-12 simple joignabilité) pour consommer `GET /health/dependencies` : `dependencies.workers.status="ok"` si les workers répondent `2xx` à `GET /health/dependencies` dans le budget timeout, sinon `status="down"`. Si l'endpoint workers est injoignable, `dependencies.workers.status="down"` ET `dependencies.mistral.status="down"` (la vue Mistral transite par les workers, AC-5).
- AC-7 : L'agrégat racine `status` intègre désormais les quatre dépendances : `status="degraded"` si **au moins une** des quatre (`postgres`, `gcs`, `workers`, `mistral`) est `down`, `"ok"` ssi les quatre sont `ok` (extension de OBS-002 AC-3).
- AC-8 : Le `#health-widget` rend désormais l'état des **quatre** dépendances en `textContent` (style via `observability.css`, CSP-safe) ; la présence de la clé `mistral` dans le rendu est contractuelle (extension de OBS-002 AC-13). Aucune fuite d'erreur interne rendue.
- AC-9 : Le timeout 2 s, le parallélisme et la borne worst-case (AC-9/AC-8 de OBS-002) sont préservés : ajouter la probe workers→Mistral ne sérialise pas les probes ; un snapshot frais toutes dépendances down reste < 3 s.
- AC-10 : La route workers `GET /health/dependencies` ne porte aucune donnée de conversation, aucun contenu, aucune clé API Mistral, aucun `user_id` — uniquement deux statuts `ok|down` (cohérent `security.md` §A05/§A09).
- AC-11 : Test manuel auteur (procédure documentée `## Test manuel auteur`) coché par l'humain avant `Status: ready`.

## Non-goals

- Pas de re-livraison du handler gateway `GET /v1/status`, du cache TTL, ni du widget de base — livrés par OBS-002, ce ticket les étend additivement.
- Pas d'appel Mistral direct depuis la gateway — Mistral est atteint uniquement via les workers (frontière de confiance préservée).
- Pas de 3e état pour `mistral` (pas de `unknown`/`degraded` par dépendance) — enum fermé `ok|down`, incertitude ⇒ `down` (OQ-5 résolue).
- Pas de probe d'autres dépendances workers (Langfuse, pgvector, embeddings model) — seulement `mistral` + vue interne `workers` phase 1.
- Pas d'historique / série temporelle / alerting sur la santé Mistral — OBS-005.
- Pas de métrique OTel nouvelle ni d'uptime check GCP câblé — OBS-005.
- Pas de modification du widget usage OBS-001 ni de `GET /v1/stats`.

## Pre-conditions

- **OBS-002 mergé** : `GET /v1/status` gateway existant (3 deps), cache TTL 10 s, `#health-widget`, client workers gateway→workers en place, tests d'intégration de base. Ce ticket étend ces artefacts.
- Workers FastAPI opérationnels avec accès configuré à Mistral (probe Mistral AC-1/AC-2).
- **Approbation humaine explicite** de la modification de `specs/openapi/gateway-to-workers.yml` au moment de l'implémentation (source-vérité humain-only, AC-3).

## Failure modes

- Workers joignables mais probe Mistral en timeout/erreur → `mistral.status="down"` (OQ-5), workers `GET /health/dependencies` répond `2xx` donc `workers.status="ok"`, agrégat `degraded`, gateway `200 OK`.
- Service workers injoignable → `workers.status="down"` ET `mistral.status="down"` (AC-6), agrégat `degraded`, gateway `200 OK`.
- Réponse `GET /health/dependencies` malformée (schéma inattendu) → la gateway traite la dépendance concernée comme `down`, aucun détail dans le body gateway (sanitisation OBS-002 D-5).
- Mistral down rapporté par workers → `mistral.status="down"`, `workers.status="ok"`, agrégat `degraded`.

## Test manuel auteur

Procédure exécutée par l'humain avant `Status: ready` (AC-11).

1. **Stack** : `docker compose up -d`, vérifier gateway + workers + postgres + Mistral joignable.
2. **Route workers** : `curl -i http://localhost:8000/health/dependencies` → `200`, body `{"mistral":{"status":"ok"},"workers":{"status":"ok"}}`, aucune fuite (host/clé/message).
3. **Body gateway 4 deps** : `curl -i http://localhost:8080/v1/status` → `200`, body contient `dependencies.mistral` (4 clés au total), `checked_at` RFC3339.
4. **Widget** : ouvrir `/observability` → `#health-widget` affiche les 4 dépendances dont `mistral`.
5. **Mistral down** : simuler Mistral injoignable côté workers → `curl /v1/status` → `mistral.status="down"`, `workers.status="ok"`, agrégat `degraded`, `200`.
6. **Workers down** : `docker compose stop workers` → `workers.status="down"` ET `mistral.status="down"` (AC-6), agrégat `degraded`.
7. **Parallélisme** : toutes dépendances arrêtées → snapshot frais < 3 s (AC-9).
8. **Contrat** : `git diff specs/openapi/gateway-to-workers.yml` → ajout de `GET /health/dependencies` approuvé ; schemathesis vert.

Coller la checklist cochée des 8 étapes en commentaire de PR.

## Performance / SLO

- Ajout de la probe workers→Mistral ne dégrade pas le SLO OBS-002 : probes toujours en parallèle, worst-case ~2 s borné par le cap timeout (AC-9), pas la somme des 4.
- Timeout dur par probe (gateway→workers ET workers→Mistral) : 2 s, aligné `security.md` §A04.

## Security / trust boundary

- Frontière Gateway → Workers (consommation `/health/dependencies`) et Workers → Mistral (probe interne).
- **A05 Misconfiguration** : la route workers et la clé `mistral` du body gateway restreintes à l'enum `ok|down` ; aucune clé API Mistral, host, message d'erreur ni stack exposé (AC-1/AC-10). CSP UI-001 préservée (héritée OBS-002).
- **A04 Insecure Design** : timeout dur 2 s sur gateway→workers ET workers→Mistral, pas de retry non borné.
- **A09 Logging & Monitoring** : détail d'échec de la probe Mistral en log `warn` interne workers seulement, jamais dans le body. La clé API Mistral est un secret (`pydantic.SecretStr`, `secret-hygiene.md`) — jamais loggée.
- **A10 SSRF** : la cible Mistral côté workers est une URL fixe issue de la configuration (pas user-controlled) — pas de surface SSRF.
- **RAG-specific** : aucune donnée de conversation, aucun prompt, aucun contenu exposé — uniquement deux statuts de santé `ok|down`.

## Observability

- Log `tracing`/`logging` `warn` côté workers sur probe Mistral en échec (`event=health.mistral.down`) — sans clé API ni host brut.
- Log `tracing` `warn` côté gateway si `/health/dependencies` injoignable ou malformé.
- Pas de métrique OTel nouvelle ni d'alerte câblée phase 1 (OBS-005).
- Pas de tracing Langfuse (la probe Mistral est un health-check, pas un appel LLM de pipeline).

## Décisions

Arbitrages portés depuis OBS-002 + résolution OQ-4/OQ-5.

- **D-1 (héritée OBS-002) — Forme par-dépendance NOMMÉE, extension additive à 4 deps.** La clé `mistral` est ajoutée sans casser la forme 3-deps livrée par OBS-002 ; les 4 clés sont toujours présentes (AC-4/AC-7).
- **D-5 (héritée OBS-002) — Enum fermé `ok|down` + `latency_ms`.** `mistral` respecte la même forme sanitisée (AC-5).
- **D-8 (ex-OQ-4) — Route workers = `GET /health/dependencies`, schéma `{mistral, workers}` sanitisé `ok|down`.** Décrite dans `gateway-to-workers.yml` (source-vérité humain-only, approbation explicite à l'impl, AC-1/AC-3).
- **D-9 (ex-OQ-5) — `mistral="down"` sur incertitude.** Workers joignables mais probe Mistral en timeout/erreur ⇒ `mistral.status="down"` (impossible de confirmer `ok` ⇒ `down`), préserve l'enum fermé à 2 valeurs (AC-2/AC-5).
