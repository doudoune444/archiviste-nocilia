# OPS-005 — Ingestion `lore/` automatisée sur push `main` via Cloud Run Job

## Contexte

Aujourd'hui, après un merge de `lore/**` sur `main` (manuel ou via PR gdrive-sync ING-013), l'ingesteur ING-001 doit être déclenché à la main depuis un poste opérateur (cf. checklist PR gdrive-sync `.github/workflows/gdrive-sync.yml:141` et runbook §Ingestion DB après sync). Cet écart manuel laisse le corpus pgvector désynchronisé du contenu mergé tant qu'un humain n'a pas lancé la commande. Ce ticket ferme la boucle : tout changement `lore/**` sur `main` déclenche automatiquement l'ingestion contre la base Cloud SQL.

## Critères d'acceptation

- AC-1 : `.github/workflows/ingest-lore.yml` existe et déclare `on: push:` avec `branches: [main]` et `paths: ['lore/**']`, de sorte qu'un push sur `main` ne touchant aucun fichier sous `lore/` ne déclenche pas le workflow.
- AC-2 : Le workflow s'authentifie à GCP via `google-github-actions/auth@v2` en Workload Identity Federation (`workload_identity_provider: ${{ secrets.GCP_WIF_PROVIDER }}`), sans clé SA JSON commitée ni stockée en secret GitHub, cohérent avec `.github/workflows/deploy.yml:26`.
- AC-3 : Le workflow exécute le Cloud Run Job d'ingestion de manière synchrone via `gcloud run jobs execute <job> --wait --region <region>` et propage l'issue de l'exécution : le step échoue (et donc le run GitHub Actions échoue) si l'exécution du Job n'atteint pas l'état `Succeeded`.
- AC-4 : Le Cloud Run Job exécute l'ingesteur ING-001 (`python -m archiviste_workers.ingest --path lore/`) et termine en exit `0` quand `errors == 0`, exit `1` quand `errors >= 1` (au moins un fichier en erreur, les autres traités), conformément à ING-001 AC-17.
- AC-5 : Le Cloud Run Job s'exécute sous l'identité de service account `archiviste-runtime`, lit `MISTRAL_API_KEY` depuis GCP Secret Manager (référence secret, jamais valeur en clair dans la définition du Job), et se connecte à Cloud SQL via cloud-sql-proxy `--auto-iam-authn` (username = email SA, `roles/cloudsql.instanceUser`), cohérent avec le pattern OPS-003 / SEC-005.
- AC-6 : Le mapping exit code → état Cloud Run Job → verdict workflow est défini ainsi : Job exit `0` → execution `Succeeded` → workflow vert ; Job exit `1` (erreurs fichiers ING-001) → execution `Failed` → **workflow rouge** (signal opérateur qu'au moins un `.md` n'a pas été indexé) ; tout autre exit (init fatal, ex. exit `2` ING-001) → execution `Failed` → workflow rouge.
- AC-7 : Le log JSON `ingest.summary` (ING-001 AC-16) émis par le Job est récupérable dans Cloud Logging pour l'exécution déclenchée (filtrable par l'execution ID du Job), servant d'oracle d'observabilité du run.
- AC-8 : La section runbook `## §Ingestion DB après sync` (`docs/runbook.md:439`) est réécrite pour décrire le déclenchement automatique sur push `main`, et la checklist d'ingestion manuelle du PR body gdrive-sync (`.github/workflows/gdrive-sync.yml:141`) est mise à jour ou supprimée pour ne plus demander une étape manuelle désormais automatisée.

## Non-goals

- Pas de création/refonte de la **définition Terraform** du Cloud Run Job d'ingestion s'il n'existe pas encore : ce ticket suppose le Job provisionné (cf. Pre-conditions / Open questions). Si le Job n'existe pas, c'est un ticket INFRA-* pré-requis séparé.
- Pas d'ingestion locale one-shot depuis poste opérateur : le chemin `uv run python -m archiviste_workers.ingest` reste documenté en runbook comme procédure de secours, mais sort du périmètre de l'automatisation de ce ticket.
- Pas de modification du pipeline ING-001 lui-même (parse / chunk / embed / upsert) — réutilisé tel quel.
- Pas de schedule cron : déclenchement uniquement événementiel sur push `main` paths `lore/**` (un `workflow_dispatch` manuel est toléré comme commodité mais non requis par les AC).
- Pas de re-ingestion forcée / re-chunking sur changement de paramètres d'embedding (resté ticket futur ING-001 Non-goal).
- Pas de détection/suppression des documents orphelins (fichier `lore/` supprimé → ligne `documents` restante) — hors scope.
- Pas de notification externe (Slack / email) sur échec : le signal est le run GitHub Actions rouge + Cloud Logging.
- Pas de garde-fou concurrence avancé au-delà du `concurrency:` GitHub standard.

## Pre-conditions

- ING-001 mergé : `python -m archiviste_workers.ingest --path lore/` opérationnel, contrat exit code (0 / 1 / 2) et logs `ingest.start` / `ingest.document` / `ingest.summary` figés.
- Cloud Run Job d'ingestion **provisionné** dans le projet GCP (nom + région connus), exécutant l'image workers contenant le module `archiviste_workers.ingest`. Si absent → ticket INFRA-* pré-requis (cf. Open questions OQ-1).
- OPS-003 / SEC-005 en place : pattern cloud-sql-proxy `--auto-iam-authn` + SA `archiviste-runtime` (`roles/cloudsql.instanceUser`) opérationnel.
- `MISTRAL_API_KEY` présent dans GCP Secret Manager et accessible par `archiviste-runtime`.
- Workload Identity Federation configuré pour GitHub Actions (`secrets.GCP_WIF_PROVIDER`, `secrets.GCP_PROJECT_ID`), déjà utilisé par `deploy.yml`.
- Image workers déployée contenant le code ING-001 (sinon le Job ingère avec une version périmée — ordering vs `deploy.yml`, cf. OQ-2).

## Failure modes

- Job exit `1` (≥ 1 fichier ING-001 en erreur : frontmatter invalide, > 1 MiB, erreur DB partielle) → execution `Failed` → workflow **rouge** ; les fichiers valides ont été ingérés ; opérateur inspecte `ingest.summary` + `ingest.document` en erreur dans Cloud Logging.
- Job exit `2` ou crash init (connexion Postgres KO, modèle embedding non chargeable, secret Mistral absent) → execution `Failed` → workflow rouge ; aucun fichier traité.
- `gcloud run jobs execute --wait` time-out ou Job non trouvé (mauvais nom/région) → step `gcloud` échoue → workflow rouge ; message d'erreur `gcloud` propagé.
- Authentification WIF échoue (provider mal configuré, repo non autorisé) → step `auth` échoue avant `gcloud` → workflow rouge.
- cloud-sql-proxy ne devient pas prêt dans le Job → ING-001 sort en erreur init (exit `2`) → execution `Failed` → workflow rouge.
- Push `main` touchant `lore/**` ET déclenchant aussi `deploy.yml` en parallèle → risque d'ingestion contre une image workers pré-déploiement (ordering) → cf. OQ-2.

## Performance / SLO

- Pas de SLO dur. Le `duration_ms` du log `ingest.summary` reste l'indicateur. Le corpus MVP (~10-50 fichiers) garde le run sous quelques minutes ; le re-traitement complet à chaque push est acceptable car l'UPSERT idempotent skippe les fichiers `unchanged` (ING-001 AC-11).

## Security / trust boundary

- Conforme `secret-hygiene.md` : aucun secret commité ; `MISTRAL_API_KEY` via Secret Manager ; auth GCP via WIF (pas de clé SA JSON longue durée).
- Conforme `.claude/rules/security.md` §A09 : aucun secret loggué ; `ingest.summary` ne contient ni clé ni contenu de document.
- Frontière : le déclencheur est un push `main` (déjà soumis à la protection de branche + review humaine). L'ingesteur traite uniquement des `.md` versionnés, pas d'entrée réseau arbitraire (ING-001 normalise NFKC + cap 1 MiB).
- Le Job tourne en `archiviste-runtime` (moindre privilège : accès Cloud SQL + secret Mistral uniquement).

## Observability

- Cloud Logging : logs JSON ING-001 (`ingest.start`, `ingest.document`, `ingest.summary`) capturés pour chaque execution du Job, corrélables par execution ID.
- GitHub Actions : run vert/rouge = signal de premier niveau ; logs `gcloud run jobs execute --wait` visibles dans le step.
- Pas de métrique OpenTelemetry ni d'alerte GCP dédiée ajoutée par ce ticket.

## Open questions

- OQ-1 — Le Cloud Run Job d'ingestion existe-t-il déjà (Terraform `infra/`) avec son nom + région, ou faut-il le provisionner ? Si à créer, c'est un INFRA-* pré-requis séparé (sinon OPS-005 bascule en M et dépasse potentiellement le scope « workflow seul »). À trancher avant /plan.
- OQ-2 — Ordering vs `deploy.yml` : un push `lore/**` qui ne change pas le code workers déclenche l'ingestion sur l'image déjà déployée (OK). Mais un push mêlant code workers + `lore/**` pourrait ingérer avant que `deploy.yml` ait promu la nouvelle image. Faut-il chaîner ingest-lore après deploy (`workflow_run`), ou accepter l'ingestion sur l'image courante (les `.md` ne dépendent pas du code) ? Recommandation tacite : accepter, car le corpus ne dépend pas de la version workers — à confirmer.
- OQ-3 — Nom exact du Cloud Run Job + région à figer dans le workflow (dérive de OQ-1).
- OQ-4 — Concurrence : deux pushes `lore/**` rapprochés lancent deux exécutions Job. L'UPSERT idempotent ING-001 tolère-t-il deux runs concurrents sur les mêmes lignes (verrouillage transactionnel par fichier) ou faut-il `concurrency:` GitHub pour sérialiser ? À trancher au /plan.
- OQ-5 — Faut-il ingérer **tout** `lore/` à chaque push (idempotent, skippe les `unchanged`) ou seulement les fichiers du diff du push ? Le ticket suppose `--path lore/` complet (simple, robuste) ; restreindre au diff est une optimisation non requise.
