- **Intégration** (`#[sqlx::test]` Z3) :
  - AC-1 : JWT `author` → `GET /dashboard` 200 + `content-type: text/html; charset=utf-8` + body contient `<table id="tickets-table"`.
  - AC-3 : `GET /assets/dashboard.js` 200 + `Content-Type` JS ; `GET /assets/dashboard.css` 200 + `text/css; charset=utf-8`.
  - AC-4 : `GET /assets/random.txt` → 404 ; `GET /assets/../Cargo.toml` → 400/404 (régression UI-001).

## Risks / open questions
- **R1 (LOC)** : ~50 HTML + 100 JS + 40 CSS + 20 handler + 5 lib + 1 mod = **~216 prod**. Tests ~150. Sous 300 LOC OK.
- **R2 (dépendance PR1)** : `/v1/tickets` + `/v1/conversations/{id}/signed-url` doivent exister ; PR2 ouvre uniquement après merge PR1. Tests manuels AC-24 nécessitent PR1 deployé.
- **R3 (ServeDir vs handler gated)** : `/dashboard` HTML doit être gated `author_only` (AC-11), donc **handler custom** plutôt que `ServeFile` racine (un `ServeFile` ne porte pas d'extractor). Décision : `include_str!("../../static/dashboard.html")` + `Html<String>` pour conserver l'extractor `AuthUser`. Cohérent UI-001 OQ-1.
- **R4 (CSP `script-src 'self'`)** : `dashboard.js` chargé via `<script src="/assets/dashboard.js">` ; aucun inline. Compatible CSP UI-001 AC-6 byte-for-byte (AC-12 / D-7).
- **R5 (test manuel humain)** : AC-24 dépend de humain ; coordinateur attend checklist 16 étapes cochée dans PR avant `ready to ship`.
