# Plan — OBS-004 Widget qualité IA live + GET /v1/quality

## Goal
Expose le dernier run Ragas live de `eval_runs` via `GET /v1/quality` (public, lecture directe Postgres) et l'affiche dans un 3e widget `#quality-widget` indépendant sur `/observability`.

## Acceptance criteria recap
- AC-1 : `200`, `application/json; charset=utf-8`, body plat 6 clés exactes `{faithfulness, answer_relevancy, context_precision, context_recall, golden_set_version, finished_at}` quand ≥1 ligne live ; aucune autre clé.
- AC-2 : ligne servie = `SELECT … WHERE runner_mode='live' ORDER BY finished_at DESC LIMIT 1` (maximise `finished_at`).
- AC-3 : 4 métriques = nombres JSON égaux à la valeur `NUMERIC(5,4)` stockée, sans arrondi gateway.
- AC-4 : zéro ligne live → `200` body exact `{"status":"no_data"}` (ni 404 ni 503).
- AC-5 : anonyme, aucun guard auth ; test contractuel d'absence de guard.
- AC-6 : `finished_at` RFC3339/ISO-8601 UTC, parse valide.
- AC-7 : Postgres indispo (pool `None` ou requête échoue) → `503` body byte-for-byte `{"error":"upstream_unavailable","request_id":"<uuid>"}` via `ApiError::UpstreamUnavailable` ; aucun détail interne ; distinct du chemin `no_data`.
- AC-8 : `observability.js` fetch `/v1/quality` same-origin → `#quality-widget`, `textContent` (jamais `innerHTML`), 4 métriques + ligne fraîcheur, style via CSS. Libellés/formatage non figés.
- AC-9 : fetch indépendant de `/v1/stats` et `/v1/status` ; échec/`no_data` ne blanchit pas `#usage-widget`/`#health-widget`.
- AC-10 : `no_data` → `#quality-widget` affiche littéral `Aucune évaluation disponible.`.
- AC-11 : toute issue ≠ `200` → message d'indisponibilité littéral, distinct du message `no_data`.
- AC-12 : 5 headers de sécurité globaux présents byte-for-byte sur `GET /v1/quality` ; CSP inchangée.
- AC-13 : `observability.html` sans `<script>`/`<style>`/`style=`/`on*=` inline ; présence `<section id="quality-widget"`.
- AC-14 : `specs/openapi/gateway-to-workers.yml` non modifié.
- AC-15 : test manuel auteur coché avant `Status: ready`.

## Risks / open questions
- **R1 (TRANCHÉ — option a).** `gateway/Cargo.toml` ligne 24 : sqlx features = `["runtime-tokio-rustls","postgres","uuid","chrono"]` — ni `bigdecimal` ni `rust_decimal`. sqlx ne décode pas `NUMERIC` en l'état ; `f64` non supporté pour `NUMERIC` (et trahirait AC-3). Décision humaine : **(a) feature sqlx `bigdecimal` + crate `bigdecimal` (serde → nombre JSON, préserve `0.9234`)**. Rejetées : (b) `::float8` (fuzz binaire, AC-3 douteux), (c) `::text` re-parsé (fragile). Nouvelle dépendance → ADR léger requis (`CLAUDE.md` Garde-fous, `security.md` A06) ajouté en step 1.
- **R2 — sérialisation `bigdecimal` en nombre JSON.** `serde_json` sérialise `BigDecimal` en nombre (pas string) seulement si la feature `serde` de `bigdecimal` est activée ; à valider que `0.9234` ne devient pas `"0.9234"`. Test AC-3 (`assert_eq!(json["faithfulness"], 0.9234)`) doit échouer si string. Couvert par le test, mais conditionne le choix R1.
- **R3 — `specs/properties.md`.** Non lu (humain-only). À confirmer : aucune invariant property-based ne couvre `/v1/quality` (read-path trivial). Si une invariant existe, ajouter property test.
- **R4 — OQ-1/OQ-2 acceptance.** OQ-1 (`upstream_unavailable` confirmé, aligné `stats.rs`) et OQ-2 (forme de la ligne de fraîcheur non figée) : aucune ambiguïté bloquante, implémentation libre côté JS.
