# Plan — SEC-001 Auth tiers JWT (anonymous fingerprint / member signup / author)

## Goal
Livrer les 3 tiers `anonymous|member|author` de la vision via JWT EdDSA + sessions server-side hashées + fingerprint anonyme stable, en remplaçant le hardcoding `user_tier=anonymous` du gateway → workers. Découpé en **2 PRs sériels** (PR-a infra auth, PR-b signup/login/logout).

## Acceptance criteria recap
Voir `specs/acceptance/SEC-001.md` AC-1 → AC-20 (verbatim). Mapping PR :
- **PR-a** : AC-9, AC-10, AC-11, AC-12, AC-13, AC-14, AC-16, AC-18 (token/session/throttle non couverts), AC-19 (event `invalid_token`), AC-20 (migration colonnes nullable + sessions).
- **PR-b** : AC-1, AC-2, AC-3, AC-4, AC-5, AC-6, AC-7, AC-8, AC-15, AC-17, AC-18 (login/signup/logout events), AC-19 (events restants).
- Humain amende AC-20 en parallèle (D-H1) → schéma cible : `users(id UUID PK, email TEXT NULL, password_hash TEXT NULL, tier TEXT CHECK IN ('anonymous','member','author'), created_at)` + `CHECK (tier='anonymous' OR (email IS NOT NULL AND password_hash IS NOT NULL))` + index unique partiel `users_email_lower_uq ON users(LOWER(email)) WHERE email IS NOT NULL`. Sentinel `00000000-...` `anonymous` préservé (existe migration 0002).
- Humain amende oracle ajout AC fingerprint Cloudflare (D-H2) : header `CF-Connecting-IP` prioritaire si présent, sinon `ConnectInfo<SocketAddr>` (dev local).

## Test strategy

- **Intégration (majeur)** : `gateway/tests/auth_*.rs` couvrent chaque AC par status + body shape + ligne SQL + cookie attributes + log event nommé. Réutiliser fixture mock workers de `chat_test.rs`.
- **Contract** : décoder JWT en mode headers-only pour AC-5 ; schemathesis sur OpenAPI mis à jour (AC-14).
- **Property** : ajouter `INV-6` impl dans `gateway/tests/auth_properties.rs` (proptest) — toute requête sans JWT valide → 401, jamais 200 avec tier dégradé. PR-a livre le test (l'extractor existe).
- **Timing** : AC-2 / AC-6 informatif (warn si delta médian > 50 ms, pas hard-fail CI). D-3 confirmé.
- **Eval** : non concerné (auth hors pipeline RAG).

## Risks / open questions

- **U-1 (D-H1)** : amendement spec AC-20 par l'humain en parallèle. Plan reflète schéma cible (colonnes nullable + CHECK partiel + sentinel préservé). Si humain ne merge pas l'amendement avant PR-a, **stop**.
- **U-2 (D-H2)** : ajout AC explicite fingerprint Cloudflare (`CF-Connecting-IP` prio). Plan implémente la logique D-H2 telle quelle. Numérotation AC à arbitrer par humain (AC-10b ou AC-NEW).
- **U-3** : throttle login (AC-7) facilite un auto-DoS par un attaquant qui spamme l'email d'un user connu → user légitime locké 15 min. Mitigation V2 (SEC-002) via combinaison throttle email + IP + Cloudflare. Acceptable phase 1 (V1 beta privé).
- **U-4** : pré-condition ADR-0009 JWT/EdDSA (D-H4) rédigé **par l'humain** AVANT impl PR-a. Toute impl avant ADR mergé = stop.
- **U-5** : test AC-14 sous-cas (b) "requête avec JWT member" déplacé en PR-b (nécessite `sign()` pour générer un JWT de test) ; PR-a couvre AC-14 (a) anon uniquement + un JWT test signé par fixture statique (clé privée test injectée via `cfg(test)`).
- **U-6 (D-H3)** : trigger sub-split PR-a → PR-a1/PR-a2 = LOC prod (hors migration, hors tests) > 300 mesurée après step 10. Mesure via `tokei gateway/src/auth gateway/src/routes/me.rs gateway/src/handlers/chat.rs` (delta vs main). Documenter LOC observée dans la PR description.
- **U-7** : sentinel `00000000-...` `anonymous` reste référencé par `conversations.user_id` et `query_log.user_id` (migration 0002). AC-10 dérive un `user_id` UUIDv5 par fingerprint pour les requêtes anonymes — ces UUIDs n'existent **pas** comme ligne `users`. Conséquence : tout futur `INSERT conversations(user_id=<uuid5>)` violera la FK. Solution phase 1 : workers reçoivent `X-User-Id` informationnel uniquement (AC-14), tout `INSERT conversations` continue d'utiliser le sentinel pour les anonymes (ticket follow-up requis avant UI-001 persistance conversations anonymes).
- **U-8** : ADR-0006 du repo (= `gdrive-api-client`) existe déjà ; numérotation ADR JWT = **0009** (next free).
