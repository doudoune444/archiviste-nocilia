# GEN-005 — Mode 4 mystère + ACL post-retrieval filtering

## Contexte

La vision (`docs/vision.md` §25 ligne Mode 4, §41 « ACL contenu », ordre d'attaque #8 §167) impose un 4e mode de réponse déclenché quand l'intent est `in_domain` mais que tous les chunks pertinents portent un `access_tier` que le tier utilisateur n'a pas le droit de voir : la réponse doit être « mystérieuse sans révéler que l'info existe » et son **timing doit être identique aux autres modes** (anti side-channel ACL leak, conforme `.claude/rules/security.md` §A01 « Broken Access Control »). L'état actuel filtre `access_tier` côté SQL dans `retrieve/search.py:14-26` (`WHERE d.access_tier = ANY($2::text[])`), ce qui (a) contredit la vision §41 « filtrage post-retrieval » et (b) rend Mode 4 impossible à déclencher puisque le worker ne distingue pas « aucun chunk pertinent » de « chunks pertinents masqués par ACL ». Ce ticket déplace le filtrage ACL en post-retrieve dans le pipeline `generate`, livre la branche Mode 4 mystère, garantit le timing constant, et trace `mode='mystery'` dans `query_log` (la contrainte CHECK est déjà conforme cf. `migrations/0002_schema.sql:73`).

## Critères d'acceptation

- AC-1 : `POST /v1/generate` accepte le même body qu'en GEN-001/GEN-003 (`{query, conversation_id, user_id, user_tier, request_id}`) et retourne `200 OK` avec body `{answer, citations, mode, conversation_id, request_id, usage, retrieve_ms, llm_ms}` où `mode ∈ {"canon", "off_topic", "lore_gap", "mystery"}` ; aucun autre champ n'est ajouté ou retiré par rapport à GEN-003.
- AC-2 : `POST /v1/retrieve` retourne désormais **tous** les top-K chunks correspondant à la query (cosine ordering) **sans appliquer de filtre `access_tier` côté SQL** ; le champ `allowed_tiers` n'est plus envoyé en paramètre `$2` du `WHERE` (la clause est supprimée). Le schéma `RetrievedChunk` (`workers/src/archiviste_workers/retrieve/schemas.py`) ajoute un champ obligatoire `access_tier: Literal["public","members","author_only"]` lu depuis `documents.access_tier` via la JOIN existante.
- AC-3 : Le pipeline `generate/router.py` exécute le filtrage ACL **après** la réponse de `/v1/retrieve` via un module dédié `workers/src/archiviste_workers/services/acl.py` qui expose `filter_chunks_by_tier(chunks, user_tier) -> FilterResult` où `FilterResult` porte exactement les champs `(visible: list[Chunk], blocked_count: int)` et où le mapping des permissions est figé byte-for-byte :
  - `anonymous` → autorise `{"public"}`
  - `members` → autorise `{"public", "members"}`
  - `author_only` → autorise `{"public", "members", "author_only"}`
- AC-4 : Quand `intent='in_domain'` (cf. GEN-003) et que `filter_chunks_by_tier` retourne `visible=[]` ET `blocked_count ≥ 1` (toutes les chunks top-K ont été masquées par l'ACL), le handler entre en branche Mode 4 : il appelle le LLM de génération avec un prompt mystère dédié et retourne `mode='mystery'`, `citations=[]`.
- AC-5 : Quand `intent='in_domain'` et que `filter_chunks_by_tier` retourne `visible` non vide (au moins 1 chunk autorisé reste), le handler poursuit le pipeline Mode 1 canon GEN-001 inchangé en utilisant **uniquement** les chunks `visible` (les chunks bloqués sont silencieusement écartés, jamais injectés dans le prompt LLM, jamais cités, jamais loggés en clair) ; `mode='canon'`.
- AC-6 : Quand `intent='in_domain'` et que `/v1/retrieve` retourne `chunks=[]` (aucun match cosine, indépendant de l'ACL) ET donc `blocked_count=0`, le handler entre en branche Mode 3 `lore_gap` (livré par GEN-004) ou — si GEN-004 pas encore mergé — en branche Mode 1 canon avec le marqueur `<no_archives_found/>` existant (`generate/prompt.py:20`). **Ce ticket n'introduit pas Mode 3.**
- AC-7 : Le prompt mystère est exposé via une constante `MYSTERY_SYSTEM_PROMPT` figée byte-for-byte dans `workers/src/archiviste_workers/generate/prompt.py`, testable par assertion d'égalité stricte. Le system prompt instruit le LLM à : (a) répondre de manière claire et concise sans jeu de rôle ni mise en scène (continuité GEN-001 mode 1), en indiquant sobrement que les archives ne contiennent rien à partager sur ce sujet, (b) **ne jamais révéler l'existence d'information cachée** — clause de sécurité ACL non négociable (interdiction littérale dans le prompt : pas de « je ne peux pas vous le dire », pas de « ces archives sont scellées », pas de « vous n'avez pas accès »), (c) n'inventer aucun fait, (d) ne pas proposer de questions alternatives (différence avec Mode 2 off_topic), (e) répondre dans la langue de la question. Le ton mystérieux/poétique d'origine (brumes / silences / non-dits) est retiré pour cohérence de ton avec les autres modes ; la non-divulgation reste assurée par l'instruction (b) et par l'absence totale de chunk injecté (AC-8).
- AC-8 : Le builder `build_mystery_messages(query, suspected_injection)` dans `generate/prompt.py` retourne `[SystemMessage(MYSTERY_SYSTEM_PROMPT), HumanMessage(...)]` où le HumanMessage contient exclusivement la query préfixée par `[user query]: ` (ou `[user query, suspected injection]: ` si le filtre injection a matché, AC-13). **Aucun chunk lore n'est injecté dans le prompt mystère** — ni les chunks bloqués (sécurité), ni les chunks visibles (cohérence : si on est en mystère c'est qu'il n'y a pas de visible).
- AC-9 : Le champ `citations` de la réponse vaut toujours `[]` quand `mode='mystery'`. Le champ `retrieve_ms` reflète la durée réelle de l'appel `/v1/retrieve` (non zéro, à la différence de Mode 2 off_topic). Le champ `llm_ms` reflète la durée du seul appel LLM (génération du refus mystère).
- AC-10 : Le champ `usage` (`prompt_tokens`, `completion_tokens`, `cost_eur`) reflète le coût du seul appel LLM mystère, calculé via `extract_usage` + `pricing.compute_cost_eur` comme en Mode 1 canon.
- AC-11 : La conversation est persistée via ING-003 (`/v1/conversations/{id}/messages`) en mode mystère : exactement 2 POST séquentiels (rôle `user`=query, rôle `assistant`=answer mystère) avec le même contrat qu'en GEN-001 AC-14. Un échec ING-003 émet un log ALERT `conversation_log_failed` sans bloquer la réponse 200.
- AC-12 : Exactement une ligne `query_log` est insérée avec `(mode='mystery', intent='in_domain', status_code=200, latency_ms=<total wall clock>, prompt_tokens, completion_tokens, cost_eur)` ; le champ `query_text` est stocké tel quel (cf. GEN-001 — pas de redaction supplémentaire en mode mystère). Un échec INSERT émet un log ALERT `query_log_failed` sans bloquer la réponse 200.
- AC-13 : Le filtre prompt-injection (`detect_injection`, déjà livré GEN-001) est appliqué sur la query AVANT le routing ; si match, le préfixe `[user query, suspected injection]: ` est propagé dans le prompt mystère (comme dans canon) et le log warn `prompt_injection_suspected` est émis exactement une fois.
- AC-14 : Le type `Mode` dans `workers/src/archiviste_workers/generate/models.py` est étendu à `Mode = Literal["canon", "off_topic", "lore_gap", "mystery"]` ; aucun autre champ de schéma n'est changé. La compatibilité descendante est préservée : les réponses Mode 1 canon restent strictement identiques à GEN-001.
- AC-15 : **Timing-constant (critique sécurité)** : la latence p95 d'une requête Mode 4 mystère doit être à **≤ 100 ms** de la latence p95 d'une requête Mode 1 canon mesurée sur le même corpus et la même query. Concrètement : le handler en branche mystère NE court-circuite PAS le LLM (un appel LLM réel est toujours émis, jamais de réponse statique préfabriquée), NE court-circuite PAS la persistance conversation (2 POST ING-003), NE court-circuite PAS l'INSERT `query_log`. Tout chemin de code qui réduirait significativement la latence par rapport au canon est interdit.
- AC-16 : Quand `mode='mystery'`, si le LLM mystère timeout > 30 s → `504 Gateway Timeout` `{"error":"llm_timeout"}`, ligne `query_log` `status_code=504`, `mode='mystery'`, `intent='in_domain'`, `prompt_tokens=NULL`, `completion_tokens=NULL`, `cost_eur=NULL` (cohérent GEN-001 AC-22).
- AC-17 : Quand `mode='mystery'`, si le LLM mystère retourne 4xx/5xx provider → `502 Bad Gateway` `{"error":"llm_upstream"}`, ligne `query_log` `status_code=502`, `mode='mystery'`, `intent='in_domain'`, autres règles identiques AC-16.
- AC-18 : Le log INFO de fin de requête est étendu pour distinguer le mode mystère sans révéler l'existence de chunks bloqués au-delà du métier : `{"event":"generate","request_id":"...","conversation_id":"...","intent":"in_domain","mode":"mystery","status":200,"retrieve_ms":R,"llm_ms":L,"chunks":<top_k_total>,"citations":0,"blocked_count":<int>}`. Le champ `blocked_count` est ajouté uniquement quand `mode='mystery'` ou quand `mode='canon'` AVEC `blocked_count>0` (mixed visibility). Aucun log ne contient le texte des chunks bloqués, leur `source_path`, ni leur `chunk_id`.
- AC-19 : La couverture ACL « partial blocking » est testée : si top-K=5 retourne 3 chunks `public` + 2 chunks `members` et que `user_tier='anonymous'`, le handler poursuit en `mode='canon'` avec les 3 chunks `public` uniquement ; `blocked_count=2` dans le log structuré (AC-18) ; les chunks `members` n'apparaissent dans aucun output (prompt, response, citations, log message), aucune ligne `query_log` ne porte leur contenu.
- AC-20 : `services/acl.py` ne dépend ni de FastAPI, ni de httpx, ni d'asyncpg : c'est un module pur (fonction + dataclass). Test unitaire isolé sans I/O, mocking trivial. Grep statique : aucun `import` de `fastapi`/`httpx`/`asyncpg` dans ce fichier.

## Non-goals

- Pas d'UI ACL management auteur (V2 — vision §40).
- Pas d'ACL per-conversation : seul `documents.access_tier` est filtré, pas de `conversations.access_tier` ni `messages.access_tier`.
- Pas d'audit log dédié des événements `acl_deny` (sink alerting OBS-001 V2).
- Pas de modification du contrat workers `/v1/retrieve` au-delà de l'ajout du champ `access_tier` dans la réponse + suppression du filtre SQL (le paramètre d'entrée `user_tier` reste accepté mais devient **ignoré côté retrieve** ; documenté dans le code).
- Pas de modification de la gateway Rust (le forwarder `gateway/src/handlers/chat.rs` continue de propager `user_tier` sans changement, le worker arbitre).
- Pas de Mode 3 lore-gap : couvert par GEN-004 dédié.
- Pas de Mode 2 off-topic : couvert par GEN-003 dédié (pré-condition ; ce ticket assume `intent='in_domain'`).
- Pas de modification du vocabulaire `user_tier` côté API workers (reste `anonymous|members|author_only` cf. `retrieve/schemas.py:15-20`). SEC-001 introduit en parallèle le vocabulaire identitaire `anonymous|member|author` côté gateway et propage via `X-User-Tier` ; le mapping gateway → workers est **hors-scope GEN-005** et traité dans le plan SEC-001 ou un ticket d'alignement dédié.
- Pas de cache des décisions ACL (sécurité : un cache pourrait introduire un side-channel timing entre hits et misses).
- Pas de détection adversariale de chunks « presque visibles » (similarité cosine entre chunks bloqués et chunks visibles) — V2 si nécessaire.
- Pas de fine-tuning du prompt mystère par tier (un seul prompt pour les 3 tiers, le LLM ne sait pas quel tier interroge).
- Pas de mise à jour de `specs/golden_qa.jsonl` (vérification au plan : si entrées `mode:"mystery"` absentes, ouvrir ticket EVAL-002 dédié — hors-scope GEN-005).
- Pas de property test statistique poussé sur le timing-constant (>1000 échantillons + analyse statistique) : AC-15 se contente d'une mesure p95 sur 50 échantillons stub avec stubs LLM à durée fixe (cf. Test oracle AC-15). Durcissement statistique = OBS-001 V2.
- Pas de migration DB : `documents.access_tier` existe déjà (`migrations/0002_schema.sql:23-24`), `query_log.mode` accepte déjà `'mystery'` (`migrations/0002_schema.sql:73`), l'ingestion lit déjà `access_tier` du frontmatter (`workers/src/archiviste_workers/ingest/frontmatter.py:11-12,57-60`). **Aucun fichier SQL nouveau dans `migrations/`**.

## Failure modes

- `/v1/retrieve` 5xx ou timeout → `502 retrieve_failed` (comportement GEN-001 inchangé), aucune branche mystère évaluée.
- `/v1/retrieve` retourne un chunk avec `access_tier` hors `{"public","members","author_only"}` → log error `acl_unknown_tier` avec `chunk_id` (NOT le texte), chunk traité comme **bloqué** par défaut (fail-closed sécurité), `blocked_count` incrémenté. Aucune erreur retournée au client.
- `user_tier` reçu hors `{"anonymous","members","author_only"}` → déjà rejeté `422 invalid_user_tier` par `GenerateRequest` (GEN-001 AC validation), aucune branche mystère évaluée.
- LLM mystère timeout 30 s → `504 llm_timeout` (AC-16), ligne query_log partielle.
- LLM mystère 4xx/5xx → `502 llm_upstream` (AC-17), ligne query_log partielle.
- ING-003 4xx/5xx en mode mystère → réponse `200 OK`, log ALERT `conversation_log_failed` (cohérent AC-11).
- INSERT `query_log` échoue → réponse `200 OK`, log ALERT `query_log_failed` (cohérent AC-12).
- LLM mystère génère une réponse contenant explicitement « accès refusé », « tier insuffisant », « archive scellée », ou tout patron révélant l'ACL → **non détectable automatiquement phase 1** ; mitigation = prompt strict AC-7 instruction (d). Audit qualitatif via golden_qa + Ragas (EVAL-002 dédié, hors-scope).
- Chunks bloqués avec `access_tier='author_only'` retournés par retrieve à un user `anonymous` → jamais loggés en clair, jamais persistés dans la conversation, jamais transmis au LLM. Sécurité défense-en-profondeur (AC-19, AC-20).

## Performance / SLO

- Latence Mode 4 mystère round-trip : **p95 < 3 s** (vision §90, même budget que les autres modes).
- Overhead `filter_chunks_by_tier` : **< 1 ms** (opération in-memory sur ≤ 20 chunks, set membership).
- **Timing-constant AC-15** : `|p95(mystery) - p95(canon)| ≤ 100 ms` sur les stubs ; phase 1 = mesure indicative, hard-fail CI si delta > 200 ms (Decision D-3). Durcissement statistique = OBS-001 V2.
- Aucune gate CI bloquante au-delà du seuil 200 ms ci-dessus.

## Security / trust boundary

- **A01 Broken Access Control** (`security.md` §A01 + vision §167) : ACL contenu enforced post-retrieval (vision §41). Le filtrage ne peut être contourné par un client manipulant `user_tier` : le tier vient du JWT validé par la gateway (SEC-001, AC-14 de SEC-001) et est propagé via header `X-User-Tier` non manipulable côté client. **Pré-condition implicite** : le worker n'accepte `user_tier` que depuis la requête interne signée gateway → workers ; si SEC-001 n'est pas mergé, `user_tier='anonymous'` hardcodé (vision §73). Ce ticket ne change pas la confiance gateway → workers (réseau interne Cloud Run).
- **Side-channel timing attack** (vision §25, `security.md` §A01) : critère AC-15 timing-constant. Tout chemin de code qui rendrait Mode 4 mesurablement plus rapide ou plus lent que Mode 1 canon trahirait l'existence d'information cachée. Mitigations : appel LLM réel (jamais de réponse statique), même nombre de POST ING-003, même INSERT `query_log`, **pas de cache ACL** (un cache hit serait plus rapide qu'un miss).
- **Side-channel via réponse LLM** : le prompt mystère AC-7 instruit littéralement le LLM à ne pas révéler l'existence d'information cachée. Risque résiduel = le LLM ignore l'instruction (prompt injection via le système de retrieval poisoning, threat vision §41+ chunks adversariaux). Mitigation : aucun chunk bloqué n'atteint le prompt mystère (AC-8), donc surface d'injection nulle sur cette branche.
- **A05 Misconfiguration** : aucun log ne contient le contenu des chunks bloqués (AC-18, AC-19) — protection contre une fuite log → SIEM → analyste non habilité. Seul `blocked_count` (entier) est journalisé.
- **A09 Logging & Monitoring** : `query_log.query_text` stocké tel quel (cohérent GEN-001, base de données protégée par auth Cloud SQL). Pas de redaction supplémentaire en mode mystère (la query elle-même n'est pas secrète, c'est la disponibilité de la réponse qui est secrète).
- **Fail-closed sur tier inconnu** (Failure modes) : si un chunk arrive avec `access_tier` hors enum, il est considéré bloqué par défaut. Préfère faux mystère à fuite ACL.
- **RAG-specific « poisoned doc »** (`security.md` §RAG-specific) : ce ticket réduit légèrement la surface en mode mystère puisqu'aucun chunk n'atteint le prompt. En mode canon avec partial block, les chunks visibles atteignent le prompt comme avant (surface inchangée vs GEN-001).

## Observability

- Log INFO `event=generate` étendu avec `blocked_count` (AC-18) — utile pour mesurer le taux d'occurrence Mode 4 post-deploy.
- Log error nouveau : `acl_unknown_tier` (Failure modes) avec `chunk_id` (jamais `text`).
- Pas de métrique counter OTel dédiée `acl_blocked_total` phase 1 (OBS-001 V2 ; instrumentation triviale post-livraison).
- `query_log.mode='mystery'` permet requêtes analytiques (taux mystère par user_tier, par jour) sans modification schéma.
- Pas d'intégration Langfuse à ce stade (cohérent GEN-001, réservé OBS-*).
- Log warn `prompt_injection_suspected` propagé en mystère (cohérent GEN-001, AC-13).
- **Interdit** : logger le contenu (`text`) ou identifiant (`source_path`, `chunk_id`) des chunks bloqués. Seul `blocked_count` est observable.

## Decisions (auto-arbitrées vs vision.md + security.md, 2026-05-18)

- **D-1 — SEC-001 pré-condition tacite** : la vision (ordre d'attaque §167) place SEC-001 #7 avant GEN-005 #8 ; le ticket assume donc SEC-001 mergé avant le `/plan GEN-005`. Sans SEC-001, tous les users sont `anonymous` (vision §73), 100 % des questions touchant des chunks `members`/`author_only` partent en mystère, ce qui (a) masque les vrais bugs et (b) rend le test AC-19 partial-block infaisable end-to-end. Justification : respect strict de l'ordre vision §167.
- **D-2 — Retrieve devient ACL-agnostic** : le filtre SQL `WHERE d.access_tier = ANY($2)` est **supprimé** de `retrieve/search.py` (drift volontaire vs RET-001 livré). Justification vision §41 : « Filtrage post-retrieval ». L'ancien comportement (filtre SQL) (a) violait la vision et (b) rendait Mode 4 impossible. Le paramètre Python `allowed_tiers` de `search()` reste accepté pour compat ascendante du contrat de fonction mais est ignoré ; suppression complète = ticket de cleanup post-merge.
- **D-3 — Test timing-constant : informatif + garde-fou seuil élevé** : AC-15 mesure le delta p95 sur 50 échantillons stub. Hard-fail CI uniquement si `delta > 200 ms` (deux fois la cible « ≤ 100 ms ») pour absorber le bruit de runner. Warning sinon. Durcissement statistique (>1000 échantillons + analyse) reporté à OBS-001 V2. Justification : éviter les flakies CI tout en gardant un garde-fou contre une régression majeure.
- **D-4 — Fail-closed sur tier inconnu** (Failure modes) : préférer faux mystère à fuite ACL. Justification `.claude/rules/security.md` §A01 « Authorization checks happen in the handler ; deny by default ».
- **D-5 — Pas de cache ACL** (Non-goals) : un cache hit/miss serait observable en timing → side-channel. Justification vision §25 timing-constant.
- **D-6 — Vocabulaire `user_tier` côté workers non aligné avec SEC-001** : workers gardent `anonymous|members|author_only` (= valeurs `access_tier` réutilisées comme noms de tier user dans RET-001), gateway/SEC-001 utilisent `anonymous|member|author` (= identités). Mismatch documenté en Non-goal. Mapping `member`→`members`, `author`→`author_only` à câbler dans SEC-001 plan ou ticket d'alignement dédié. Justification : éviter scope creep ; corriger le vocabulaire RET-001 nécessite un ticket de refacto contractuel hors GEN-005.
- **D-7 — Aucun ajout golden_qa.jsonl** (Non-goals) : la vision n'impose pas d'entrées `mode:mystery` dans le golden set MVP. EVAL-002 dédié si Ragas révèle un besoin. Justification : sources humain-only (cf. CLAUDE.md « `specs/golden_qa.jsonl` source vérité humain »).
