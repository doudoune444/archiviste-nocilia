# Review GEN-002 — verdict REQUEST_CHANGES

## Summary

Forwarder gateway `POST /v1/chat → workers /v1/generate` est fonctionnellement correct : 21 tests passent, clippy/fmt clean, aucun `unwrap`/`unsafe`/forbidden pattern. Mais deux problèmes bloquants : (1) AC-13 (log JSON sans `query` brut) n'a **aucun test** — explicitement punté dans `chat_test.rs` lignes 484-490, en violation directe du plan step 9 et de la règle "test references AC explicitly" ; (2) AC-14 a un test gaming (`assert_eq!(ptr, ptr)` toujours vrai par construction). De plus, le diff prod fait ~315 LOC nettes (≈421 raw insertions) vs cap 300, sans split de ticket. Securité OWASP A01-A10 OK pour le scope phase-1 (no auth, internal docker).

## Findings

### CRITICAL (bloquant)

| File:line | Pattern | Evidence |
|---|---|---|
| `gateway/tests/chat_test.rs:484-490` | AC non couvert + test punté | Commentaire bloc affirme "Full log capture requires a custom tracing subscriber and is complex to wire... we verify AC-13 structurally" puis aucun test n'existe. Plan step 9 spécifiait explicitement « subscriber test custom + capture writer test → vert ». Le decision record R4 du plan ne couvre pas cet abandon. AC-13 du spec exige une assertion réelle de présence des clés et **absence de `query` brut**. La règle vertical-slice oblige un test par AC. |
| `gateway/tests/chat_test.rs:493-501` | Test gaming AC-14 | `let ptr1 = &raw const state.http; let ptr2 = &raw const state.http; assert_eq!(ptr1, ptr2);` — prendre l'adresse du même champ deux fois retourne toujours la même adresse, indépendamment de toute logique. Ce test passe par construction et ne vérifie **rien** sur le partage du `Client` à travers les requêtes ni l'unicité au boot. AC-14 demande "instancié une seule fois au boot, réutilisé pour toutes les requêtes". |

## Verdict

**REQUEST_CHANGES** — Bloquer le merge tant que :
1. (CRITICAL) Test AC-13 implémenté avec capture `tracing` réelle (subscriber writer JSON in-memory) asserting clés présentes + absence de `query` brut. Plan step 9 le prévoyait ; punter en review-time est inacceptable.
2. (CRITICAL) Test AC-14 réécrit pour vraiment vérifier le partage : par exemple deux requêtes consécutives via `oneshot` puis introspection du `pool_idle_timeout` ou comparaison `Arc::strong_count` ; ou simplement vérifier que le `state.http` survit à plusieurs appels concurrents sans recréation. Le test actuel est gaming pur.
3. (HIGH) Handler `chat` lit `RequestId` depuis extensions au lieu de regénérer (cohérence middleware ↔ handler).
4. (HIGH) LOC overrun adressé : soit split, soit dérogation explicite documentée dans body PR.

Les findings MED+LOW peuvent être traités en suivi si #1-#4 sont corrigés.
