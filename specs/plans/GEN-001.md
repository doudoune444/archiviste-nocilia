# Plan — GEN-001 Endpoint `POST /v1/generate` mode 1 canon

## Goal
Livrer end-to-end le pipeline workers `query → retrieve → prompt → LLM → parse citations → log` derrière un wrapper LLM config-driven (Mistral default), retourne `200 {answer, citations, mode:"canon", usage, retrieve_ms, llm_ms}` ou code d'erreur normalisé, persiste conversation (ING-003) + `query_log`.

## Acceptance criteria recap
Voir `specs/acceptance/GEN-001.md` AC-1..AC-25 (verbatim, non recopié pour respecter limite ≤ 100 lignes).

## Risks / open questions
- **R1 (BLOCKER potentiel)** : RET-001 n'est pas mergé sur `main` (aucun module `workers/src/archiviste_workers/retrieve/`). Le walking skeleton GEN-001 dépend de `POST /v1/retrieve` pour le test intégration nominal AC-1. Mitigation conforme `no-workaround.md` : tests intégration GEN-001 stubbent `/v1/retrieve` via httpx mock transport (AC-3 : « stub `/v1/retrieve` interne »). Si RET-001 merge avant GEN-001, ajouter un test e2e non-mocké en post-merge. Pas de patch autour. **Confirmer auprès humain l'ordre de merge prévu.**
- **R2 (LOC)** : effort estimé `L` par spec (OQ-8 humain accepte > 300). Estimation : ~80 (router) + 40 (llm wrapper) + 30 (prompt) + 25 (parser) + 20 (pricing) + 25 (models) + 30 (3 HTTP clients) + 20 (query_log) + 20 (lifespan delta) ≈ **290 LOC prod**. Tests ~350 LOC (non comptés vertical-slice). Marge serrée — si dérive > 300, signaler reviewer (pas de split car ticket déjà 1 vertical slice end-to-end).
- **R3** : `langchain-deepseek` et `langchain-google-genai` sont des deps neuves (security A06 — verrouillage `uv.lock`). Pas d'ADR requis (chacun < 1k LOC, pas de FFI), mais commit doit mentionner pin exact + `pip-audit` clean.
- **R4** : `query_log.cost_eur NUMERIC(10,6)` peut tronquer un `Decimal` calculé à partir de prix par MTok ; arrondir explicitement à 6 décimales avant INSERT (cohérent AC-16).
- **R5** : AC-14 séquentiel bloquant (OQ-3) ajoute jusqu'à ~200 ms à la latence perçue ; budget SLO p95 < 3 s tient (retrieve 500 + llm 2000 + ING×2 < 200 + overhead). Pas de gate CI mais à monitorer en EVAL-001.
- **R6** : `WORKERS_INTERNAL_BASE_URL` pointant sur soi-même (`http://localhost:8000`) sous Cloud Run = boucle réseau. Acceptable phase 1 (un seul service workers), mais à revisiter quand RET-* sera split en service distinct.
- **R7** : `conversation_id=null` (AC-2) : créer la conversation via `ConversationClient` AVANT le retrieve nécessite de générer le 1er message `user` côté ING-003 — or ING-003 attend un `MessageIn` complet. Décision plan : générer UUIDv4, lancer le pipeline, et au moment du log conversation post-LLM (AC-14) ING-003 créera la row à la 1re POST `user`. Le `conversation_id` est retourné dans la réponse même si la création row échoue (best-effort + log ALERT). Confirmer cohérence avec spec AC-2 « créer la ligne `conversations`...AVANT de répondre » : interprétation = créer **l'identifiant** avant retour, pas obligatoirement persister la row avant retrieve. À valider humain si lecture stricte.
