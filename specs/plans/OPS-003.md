# Plan — OPS-003 Deploy pipeline: prod DB migrations + boot-aware smoke gate

> **No acceptance file yet.** `specs/acceptance/OPS-003.md` is human-only and not
> authored. The "Acceptance criteria recap" below is derived from the incident
> brief and is provisional until a human writes the acceptance spec.

## Goal
Make prod deploy reproducibly apply Cloud SQL migrations and gate promote on a
smoke test that actually proves the canary workers reach their DB, so the
empty-DB + config-drift 502 incident cannot recur.

## Incident summary (root causes)
1. **Migrations never applied to prod Cloud SQL** — `deploy.yml` has no migration
   step; `migrations/run.sh` ran only against local CI postgres. Prod DB empty →
   `relation "chunks" does not exist` → retrieve 503 → generate 502 → gateway 502.
2. **Config drift** — live workers missing `CLOUD_SQL_IAM_AUTH=true` (terraform
   declares it at `cloud_run.tf:180`) → IAM token provider skipped → asyncpg
   `password=None` boot crash. Crashed revision had `--no-traffic`, never probed;
   gateway-canary smoke passed because it probes the LIVE workers, not the canary.

## Acceptance criteria recap (DERIVED FROM BRIEF — not a spec file)
- `deploy.yml` runs `migrations/run.sh` against prod Cloud SQL via Cloud SQL Auth
  Proxy impersonating `archiviste-runtime` SA, after canary deploy and **before**
  promote. Promote gated on migration success; migration failure → rollback +
  non-zero exit. Runner idempotent (skips applied `schema_version` rows).
- Terraform grants `gha-deploy` `roles/iam.serviceAccountTokenCreator` on
  `archiviste_runtime` SA (mirrors `iam.tf:65-69`).
- Workers DB-aware readiness probe (`/readyz`): `SELECT 1` short-timeout, reports
  `degraded` + HTTP 503 on DB loss. `/healthz` stays shallow liveness.
- Smoke step proves the workers **canary** boots + reaches DB before promote.
- Cloud Run probe behaviour reconciled: DB-failing health must not restart-loop.
- `docs/runbook.md` documents the one-time superuser bootstrap (GRANT CREATE on
  public + extensions), incident root causes, the new migration step. Incident
  logged in `docs/blockers.md`.

## Test strategy
- Unit (workers): `test_health.py` — `/readyz` `ok`+200 with a live fake pool;
  `degraded`+503 when `SELECT 1` raises/times out. Stub `app.state.db_pool` (no live
  DB). Keep existing shallow `/healthz` test green.
- Contract: no OpenAPI change unless human wants `/readyz` registered (human-only file).
- Pipeline: not CI-unit-testable; validated by a real `main` deploy run
  (human-observed). Migration idempotency already covered by
  `tests/migrations/run_tests.sh`.

## Implementation steps (ordered)
1. Terraform: add tokenCreator binding (`gha-deploy` → runtime SA). `terraform plan`
   clean. Confirm impersonator needs no DB-user grant (assumption 2).
2. Workers readiness (TDD): failing test first; then add `/readyz` probing
   `app.state.db_pool` with `SELECT 1` under a 2s timeout → `degraded` + HTTP 503 on
   failure. Keep `/healthz` shallow.
3. `deploy.yml` migrate step: after both canary deploys, before promote — install
   `cloud-sql-proxy`, start with
   `--impersonate-service-account=archiviste-runtime@flamme-496014.iam.gserviceaccount.com --auto-iam-authn`
   for `flamme-496014:europe-west9:archiviste-db`, set `DATABASE_URL` to the proxy
   endpoint, run `MIGRATIONS_DIR=./migrations migrations/run.sh`. Step id `migrate`.
4. Wire gating: promote steps already `if: success()`; extend the two rollback
   steps' `if:` to also fire on `steps.migrate.conclusion == 'failure'`.
5. `deploy.yml` smoke: prove workers-canary DB reachability pre-promote — see open
   question for chosen mechanism (recommend migrate-step-as-proof + post-promote
   `/readyz` poll).
6. Gateway: default no change (decision below).
7. `docs/runbook.md` + `docs/blockers.md` + `CHANGELOG.md`.

## Risks / open questions
- **LOC budget**: deploy.yml ~40 + iam.tf ~6 + health.py ~25 + tests ~30 ≈ 100 LOC
  (excl. docs ~60). Fits ≤300.
- **Cloud Run probes**: `cloud_run.tf` declares **no** `startup_probe`/
  `liveness_probe` (only implicit TCP startup probe on `container_port`). A DB-503
  health will **not** restart-loop today. Recommendation: DB probe on a separate
  `/readyz`, keep `/healthz` shallow so any future liveness probe never loops on
  transient DB blips.
- **Smoke-vs-live-workers** (core incident cause): confirmed — gateway `/healthz`
  probes `WORKERS_URL`, set in `cloud_run.tf:78-81` to the **live** `workers.uri`
  (not the canary tag URL). Workers ingress is ALL but invoker-gated to the runtime
  SA only; the GHA runner authenticates as `gha-deploy`, which is **not** bound
  `run.invoker` on workers → curling the workers canary from GHA likely 403s.
  Options for human:
  - (a) grant `gha-deploy` `run.invoker` on workers (new tf binding; widens surface);
  - (b) point a canary-tagged gateway env `WORKERS_URL` at the workers canary URL;
  - (c) **(architect recommends)** treat the migrate step (which connects to prod DB
    as the impersonated runtime SA) as DB-reachability proof, plus a post-promote
    `/readyz` poll as steady-state gate. Lowest added surface.
- **Impersonation DB path**: runtime SA already has `cloudsql.instanceUser`
  (`iam.tf:40-45`) and owns the schema (per manual hotfix); since `gha-deploy`
  impersonates it, `gha-deploy` itself should NOT need to be a Cloud SQL DB user —
  verify on first real run. `gha-deploy` has `cloudsql.client` already.
- **DATABASE_URL for run.sh under proxy**: `run.sh` uses `psql` (libpq). Confirm
  proxy listen mode (TCP loopback) and URL user = runtime SA IAM username
  (`.gserviceaccount.com` stripped, `@`→`%40`). With `--auto-iam-authn` the proxy
  injects the token; password empty. NOTE (verified during hotfix): GNU psql in a
  linux container handles the libpq URI fine; the GHA runner is ubuntu so this is OK.
- **Bootstrap is human-once, NOT in migrations**: GRANT CREATE on `public` +
  `CREATE EXTENSION pgcrypto/vector` need `cloudsqlsuperuser`; documented in runbook,
  never added to `migrations/*.sql`.

## Out of scope
- Re-coding the manual prod hotfixes (env-var update, traffic route, superuser GRANT,
  extension create, one-shot migrate) — documented, not re-coded.
- Splitting the runtime SA into per-service SAs (V2 SEC-001).
- Any migration SQL content change or new migration file.
- Automated down migrations (run.sh is forward-only).
- OpenAPI contract changes beyond optionally registering `/readyz`.
