## Security re-validation

- No secrets in code (SA private key fully gone, AC-1 + AC-10 + AC-11 confirmed).
- Sensitive types: bearer token wrapped in `SecretString`; `expose_secret()` only at `sign.rs:169,182` (`bearer_auth(...)` sink). No `Debug` impl on `CachedToken` reveals secret.
- SSRF surface: metadata URL constant (`http://metadata.google.internal` in prod), IAM URL templated from `sa_email` (config-trusted, not user input). Test-only `iam_base_url: Option<&str>` plumbed via `sign_get` signature, defaults to `None` from handler.
- JWT, CORS, CSP, HSTS, rate-limit — unchanged surface, not in scope of SEC-004.
- A09 logging: handler log site (`conversations.rs:97-103`) uses `reason_code = e.reason_code()` (one of 6 static strings) + `request_id`, `user_id`, `latency_ms`. No token, no signed_blob, no string_to_sign. Asserted via `assert!(!logs_contain("test-token"))` in 5 failure tests.
- HTTP client reuse safety: single `reqwest::Client` shared between metadata fetch (sends `Metadata-Flavor: Google` per request) and signBlob POST (sends `Authorization: Bearer …` + JSON per request). No default headers configured on the client builder (`token.rs:85-89`), so no Authorization leakage from a stale signBlob call to a metadata call. No cookie jar enabled by default in `reqwest::Client::builder()`. Connection pool sharing is correct (same TLS config, same timeouts).
- Cold-cache 401 path: handled (no retry, no extra metadata roundtrip). Maps to `SignBlob5xx` per current code; spec-text says it should map to `signblob_403` (R2-1 informational finding) but this is observability tuning, not security.
