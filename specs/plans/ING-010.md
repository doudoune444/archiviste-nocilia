# Plan — ING-010 Sync Google Drive → `lore/*.md` (core gdoc + png + state + CLI)

## Goal
CLI Python one-shot `uv run python scripts/gdrive_export.py --root-folder-id <id>` qui parcourt récursivement un dossier Drive (SA read-only), exporte chaque `gdoc` en `.md` (frontmatter merge user-managed/script-managed) et chaque `image/png` natif sous `lore/<drive_path>/`, persiste un état `scripts/.gdrive_state.json` (rename via `git mv`, archive sur suppression Drive, skip unchanged via md5).

## Acceptance criteria recap
- AC-1 : list récursive Drive (`gdoc` + `image/png`) via Drive v3.
- AC-2 : `gdoc` → `files.export(text/markdown)` → `lore/<drive_path>/<doc_slug>.md`.
- AC-3 : `image/png` → `files.get_media` → `lore/<drive_path>/<image_slug>.png` (pas de `.md`).
- AC-4 : chaque composant `<drive_path>` slugifié (`[a-z0-9-]+`).
- AC-5 : frontmatter strict {`title`, `tags`, `access_tier`, `source`, `source_id`, `drive_path`, `exported_at`, `archived`, `archived_at`}.
- AC-6 : `slugify` déterministe (NFKD → drop combining → lower → non-alnum→`-` → collapse → strip → cap 80, fallback `file-<id[:8]>`).
- AC-7 : collision path → suffixe `-<drive_file_id[:8]>` (ordre tri lexical sur `drive_file_id`).
- AC-8 : `scripts/.gdrive_state.json` JSON canonique trié indent 2, map `source_id → {local_path, drive_md5_checksum, last_exported_at, body_hash}`.
- AC-9 : rename Drive (gdoc et PNG) → `git mv` argv array, fallback `os.rename` log warn `git_mv_failed_fallback_rename` ; PNG rename log dédié `gdrive_sync.image_renamed`.
- AC-10 : suppression Drive → `archived: true` dans frontmatter `.md`, log seul pour PNG, jamais `rm`.
- AC-11 : `md5Checksum` (gdoc = md5 client-side de l'export Markdown ; PNG = `md5Checksum` Drive natif) inchangé → no-op.
- AC-12 : merge frontmatter — script-managed (`title`, `source`, `source_id`, `drive_path`, `exported_at`, `archived`, `archived_at`) écrasés ; user-managed (`tags` def `[]`, `access_tier` def `public`) préservés ; clés custom préservées.
- AC-13 : corps Markdown normalisé NFKC + strip C0 control sauf `\n\t` ; n/a aux PNG.
- AC-14 : fichier rendu > 1 MiB → skip, log error.
- AC-15 : creds via `GDRIVE_SA_KEY_JSON` (JSON inline) OU `GOOGLE_APPLICATION_CREDENTIALS` (path), aucun fallback ; fail-fast au boot.
- AC-16 : `--dry-run` → résolution complète, aucune écriture (fs ni état), logs `would_*`.
- AC-17 : log JSON `gdrive_sync.summary` final + `gdrive_sync.file` par fichier.
- AC-18 : exit code `0` si `errors == 0`, sinon `1`.
- AC-19 : log `gdrive_sync.start` au boot ; `gdrive_sync.first_run` si état absent (traité `{}`).
- AC-20 : Drive = source vérité corps ; drift local détecté via `body_hash` état → log `gdrive_sync.local_drift_overwritten` puis overwrite.

## Risks / open questions
- **`google-api-python-client` est une dep lourde** (>> 1k LOC, transitives `httplib2`, `googleapis-common-protos`, `uritemplate`). Selon `.claude/rules/security.md` A06 « New dep above 1k LOC or any FFI: requires ADR ». **Action** : ouvrir ADR léger `docs/adr/0006-google-api-python-client-for-gdrive-sync.md` AVANT merge, ou décision humaine documentée. Bloquer le merge sinon.
- **Localisation deps Python** : `scripts/` utilise le venv workers (deps ajoutées à `workers/pyproject.toml` en extras `gdrive`). Alternative `scripts/pyproject.toml` séparé = second venv à maintenir. Décision proposée : extras workers. À confirmer humain.
- **`workers/tests/` héberge des tests pour code dans `scripts/`** : entorse mineure à l'isolation. Alternative `scripts/tests/` + adaptation pyproject pytest config. Décision proposée : `workers/tests/test_gdrive_*.py` (réutilise infra pytest existante). À confirmer humain.
- **`mypy --strict` sur `scripts/`** : path absent du `[tool.mypy]` actuel ; ajouter `scripts/` au scope mypy ou marquer `# type: ignore` non autorisé (cf no-workaround.md). À ajouter dans `pyproject.toml`.
- **AC-9 `git mv` cross-platform Windows** : subprocess `git` argv-array OK Windows. Mais le runner local est Windows, le runner CI est Linux. Tests intégration doivent fonctionner aux deux (utiliser `Path` POSIX en logique, `subprocess` standard).
- **AC-11 md5 gdoc client-side** : Drive ne retourne pas de md5 natif pour gdoc, donc le script DOIT exporter d'abord puis hasher. Cela signifie que la détection « unchanged » exige toujours un export Drive (coût API). Conforme à la spec (« gdoc : md5 de l'export Markdown rendu, calculé client-side après export »). Pas un blocker, juste une note perf.
- **`scripts/.gdrive_state.json` versionné** : fichier modifié par chaque run ⇒ génère du churn git. Conforme spec mais à noter pour ING-012 (auto-PR).
