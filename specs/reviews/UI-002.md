## Findings HIGH

### HIGH-1 — Collapse 401/503 → 403 viole contrat SEC-001 et failure mode UI-002

`gateway/src/handlers/tickets.rs:93-95` et `gateway/src/handlers/conversations.rs:48-50` :

```rust
author: Result<RequireAuthor, crate::auth::extractor::AuthError>,
...
let author = author.map_err(|_| ApiError::AuthorRequired)?;
```

Le pattern `Result<Extractor, Error>` d'Axum livre TOUTES les variantes d'erreur de `RequireAuthor` au handler, et `map_err(|_| ApiError::AuthorRequired)` les écrase indistinctement en `403 author_required`. Les 4 variantes `AuthError` impactées :

- `AuthError::InvalidToken` (JWT absent / signature invalide / expiré / tier missing) — spec failure mode l.76 + SEC-001 AC-12 exigent `401 invalid_token` ; livré : `403 author_required`.
- `AuthError::SessionRevoked` — spec SEC-001 attend `401 session_revoked` ; livré : `403 author_required`.
- `AuthError::Upstream` (DB sessions indisponible) — spec SEC-001 et UI-002 failure mode "Postgres indisponible" attendent `503 upstream_unavailable` ; livré : `403 author_required`.
- `AuthError::AuthorRequired` (le seul cas correctement 403) — OK.

Conséquences :
1. Un auteur dont le JWT expire entre 2 clics (failure mode l.76) reçoit `403 author_required` au lieu de `401 invalid_token`. L'UI PR2 (`dashboard.js`) ne saura pas distinguer "ré-auth" de "tier insuffisant".
2. Une panne Postgres pendant validation de session est masquée derrière 403 → SLO observability brisée (alerting 503 non déclenché ; SEC-001 AC-19 `auth_failures_total{event="idor_attempt"}` faux positif).
3. La promesse "byte-for-byte AC-2 envelope" est respectée par hasard pour member/anonymous mais cassée pour les autres variantes.

Aucun test ne couvre JWT expiré / révoqué / DB-down sur les nouvelles routes (`grep expired|invalid_token` sur `test_dashboard_backend.rs` = 0 hits). Gaming pattern : `Result<RequireAuthor, AuthError>` est utilisé comme « catch-all author_required » au lieu de laisser `RequireAuthor` (sans `Result`) propager `AuthError::into_response()` natif.

**Remediation** : remplacer la signature handler par `RequireAuthor` direct (sans `Result`) ; laisser Axum renvoyer la `Rejection = AuthError` qui a déjà l'`IntoResponse` correct (extractor.rs:216-240 mappe chaque variante à son statut). Ajouter test JWT expiré sur `/v1/tickets` et `/v1/conversations/{id}/signed-url` → assert 401 `invalid_token`.

## Findings MED

### MED-1 — `request_id` absent des `tracing::info!` events (AC-23)

`gateway/src/handlers/tickets.rs:140-148` et `gateway/src/handlers/conversations.rs:104-110` :

Le commentaire de tickets.rs l.138 prétend "fields request_id, user_id, tier, latency_ms, count" mais la macro `tracing::info!` n'inclut PAS `request_id`. Idem `dashboard.conversation.signed_url`. Spec AC-23 et §Observability sont explicites : « champs `request_id`, `user_id`, `tier`, `latency_ms`, `count`|`conversation_id` ».

Pareil pour le warn `dashboard.signing_failed` (conversations.rs:90-94) : pas de `request_id` et pas de discriminant `reason ∈ {key_missing, iam_denied, secret_manager_unavailable}` (juste `%e` qui formate `"invalid RSA private key PEM: ..."` ou `"RSA signing failed"`).

**Remediation** : injecter via extension `RequestId` (déjà posée par `attach_request_id` middleware — cf `chat.rs:73`) — ajouter extractor + field au tracing event.

### MED-2 — AC-23 zéro couverture test (gaming par omission)

Aucun test ne capture le subscriber tracing ni n'assert :
1. émission des events nommés `dashboard.tickets.list` / `dashboard.conversation.signed_url` ;
2. champs obligatoires présents ;
3. absence littérale de `question`, `signed_url`, `gcs_uri` dans le payload sérialisé.

Spec Test oracle l.122 explicite cette exigence. Combiné à MED-1, on a un AC entièrement non vérifié et un drift silencieux. Pattern Gaming : « écrire la `tracing::info!` est suffisant » → non, AC-23 exige assertion sur le contenu.

**Remediation** : `tracing-subscriber` avec `with_test_writer` ou `tracing-test`, ou layer custom capturant `Event`s → assertion via `String` rendering.

### MED-3 — AC-12 testé uniquement sur 403, pas sur 200 OK ni 404

`ac12_tickets_security_headers_present` et `ac12_conversations_security_headers_present` n'exécutent que le chemin 403 (member token). Spec AC-12 et §Security/trust boundary exigent les 4 headers byte-for-byte sur les 4 routes "présents byte-for-byte sur `GET /v1/tickets`, `GET /v1/conversations/{id}/signed-url`" — donc aussi 200 OK auteur + 404 conversation_not_found. La couche `SetResponseHeaderLayer` est posée router-level → vraisemblablement OK en pratique, mais non testé.

**Remediation** : ajouter assertions headers dans `ac5_tickets_list_order_and_total` (200) et `ac10_nonexistent_conversation_returns_404` (404).

### MED-4 — Vertical-slice budget LOC dépassé (~480 LOC prod)

Plan §R1 anticipait ~312 LOC, "marge nulle". Mesure réelle (excluant `Cargo.lock`, tests, `specs/plans/`, `CHANGELOG.md`) : `tickets.rs 170 + conversations.rs 122 + sign.rs 93 + errors.rs 58 + config.rs +14 + lib.rs +12 + Cargo.toml +6 + gcs/mod 3 + handlers/mod 2 = ~480 LOC prod`. La règle `vertical-slice.md` fixe ≤ 300 LOC ; D-1 a déjà splitté en PR1+PR2, et PR1 seul excède la cible de ~60 %.

Cause probable : `errors.rs` (58 LOC, duplique partiellement la sérialisation d'`AuthError`) et `handlers/tickets.rs` (170 LOC vs 80 prévu — verbosité serde + `TicketRow` + `TicketItem` + `TicketsQuery` + `TicketsResponse` + `CountRow`).

**Remediation** : non bloquant à ce stade (PR déjà splittée, le code est exact spec) mais signaler à l'humain. Option future : factoriser `TicketRow == TicketItem` via `#[serde(transparent)]` ou `From`.

## Findings LOW

### LOW-1 — `gcs_uri` malformé → 503 au lieu de `internal`

`gateway/src/handlers/conversations.rs:63` : si `gcs_uri` en DB ne commence pas par `gs://<bucket>/`, `gcs_uri_to_object_path` renvoie `None` → `ApiError::UpstreamUnavailable` (503). C'est une corruption de données, pas un upstream indisponible. Plus correct : log `error!` + 500 `internal`. Cas-bord (post ING-003 régresse le format), non bloquant.

### LOW-2 — `sign.rs` n'encode pas le chemin d'objet (RFC3986)

`gateway/src/gcs/sign.rs:233` insère `object` brut dans la canonical request (`/{object}`) et dans l'URL finale (`https://{host}/{object}`). Si un futur `object` contient espaces / unicode / `?`, la signature casse. Phase 1, `object` provient de `gs://archiviste-conversations/conv/<uuid>.md` → safe par construction. À durcir si format `gcs_uri` évolue.

### LOW-3 — `ac10_non_uuid_path_returns_4xx` accepte 400 OR 404 OR 422

`test_dashboard_backend.rs:1276-1281` : assertion lâche `status == 400 || status == 404 || status == 422`. Spec AC-10 sub-case b exige byte-for-byte `400 invalid_request` avec enveloppe `{"error":"invalid_request","request_id":"<uuid>"}`. En l'état Axum extractor `Path<Uuid>` renvoie probablement 400 plain sans enveloppe — non testé. Risque drift PR2 qui consommera ces erreurs.
