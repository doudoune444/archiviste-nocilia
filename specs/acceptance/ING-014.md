# ING-014 — Extraction des images embarquées des Google Docs vers sidecar `<doc_slug>.images/`

## Contexte

ING-013 livre la sync `gdoc → .md` via `files.export(mimeType='text/markdown')`. L'export Markdown natif Drive représente les images embarquées par des URLs `lh*.googleusercontent.com` éphémères, non récupérables hors-ligne et non utilisables pour le rendu local ou l'ingestion future. Ce ticket rend ces images persistantes et stables côté repo : pour chaque `gdoc`, le script invoque l'API Docs, télécharge chaque image embarquée, l'écrit dans un dossier sidecar versionnable, et réécrit les références Markdown vers ces fichiers locaux.

## Critères d'acceptation

- AC-1 : Pour chaque fichier de MIME `application/vnd.google-apps.document`, le script invoque `docs.documents.get(documentId=<file_id>)` en complément de `files.export(mimeType='text/markdown')` afin de récupérer la map `inlineObjects` de la structure Docs.
- AC-2 : Chaque entrée `inlineObjects[*].inlineObjectProperties.embeddedObject.imageProperties.contentUri` est téléchargée via une requête HTTP `GET` authentifiée (même service account, header `Authorization: Bearer <token>`) avec timeout 30s.
- AC-3 : Chaque image téléchargée est écrite en `lore/<drive_path>/<doc_slug>.images/<md5_hex[:12]>.<ext>` où `<md5_hex>` est le MD5 du payload binaire et `<ext>` est dérivée du header HTTP `Content-Type` de la réponse via une map fixe `image/png→png, image/jpeg→jpg, image/gif→gif, image/webp→webp` ; tout `Content-Type` hors de cette map déclenche le mode placeholder (AC-7).
- AC-4 : Le corps Markdown post-`files.export` est réécrit avant écriture du `.md` : toute occurrence d'image inline est résolue par lookup sur l'`objectId` Docs associé et remplacée par `![<alt|"">](<doc_slug>.images/<md5_hex[:12]>.<ext>)` ; le chemin est relatif au `.md` produit (sibling), sans préfixe `/` ni `./`.
- AC-5 : Dédoublonnage intra-document : deux images du même `gdoc` partageant le même MD5 binaire produisent un seul fichier sur disque (le second `inlineObject` pointe vers la même cible). Le re-run sur Drive inchangé ne réécrit ni le binaire (`mtime` inchangé) ni le `.md` (cf AC-9 sur la signature).
- AC-6 : Markdown subset supporté en réécriture sans dégradation : paragraphes, titres H1–H6, gras, italique, liens externes, images inline, listes à puces et numérotées sur un seul niveau de profondeur. Toute construction Markdown hors de ce subset (tableaux, listes imbriquées, footnotes, blocs de code multi-lignes, citations) est conservée verbatim depuis l'export Drive ; un log warning unique `{"event":"gdrive_sync.md_subset_fallback","source_id":"...","construct":"<nom>"}` est émis la première fois qu'une telle construction est rencontrée dans le run pour ce `source_id`.
- AC-7 : Échec de téléchargement d'une image (HTTP non-2xx, timeout, `Content-Type` non mappable, écriture filesystem échouée) → l'image n'est PAS écrite, le Markdown reçoit un placeholder `![<alt|"image indisponible">](#image-failed-<objectId>)`, un log `{"event":"gdrive_sync.image_failed","source_id":"...","object_id":"<objectId>","reason":"<http_<status>|timeout|unsupported_content_type:<ct>|write_failed:<errno>>"}` est émis, et le compteur `images_failed` du summary est incrémenté de 1. Le traitement du document continue ; aucune exception n'interrompt la sync globale.
- AC-8 : Le `StateEntry` (cf ING-010 / ING-013) gagne un champ `images: dict[str, str]` mappant `md5_hex` (clé) → chemin POSIX relatif à `lore/` (valeur), pour les `.md` uniquement. Le champ est sérialisé dans `scripts/.gdrive_state.json` à la suite de `body_hash`. Pour les entrées PNG natives ING-013 le champ est l'objet vide `{}`. Un addendum à ING-010 AC-5 / ING-013 AC-5 est requis et tracé en « Amendements requis sur tickets adjacents ».
- AC-9 : Détection « contenu inchangé » du gdoc (cf ING-013 AC-11) est étendue : le statut `unchanged` est émis si et seulement si **(a)** `content_signature` (sha256 du corps post-`normalize_body`, **post-réécriture des références images** d'AC-4) est inchangé **ET** **(b)** le manifest `images` (set des `md5_hex` du run courant) est byte-identique au manifest persisté en état. Si l'un des deux change, le statut est `updated` et le `.md` est réécrit.
- AC-10 : Cleanup des orphelins : après écriture du `.md` (ou après confirmation `unchanged`), le script scanne `lore/<drive_path>/<doc_slug>.images/`, supprime tout fichier dont le préfixe de nom (avant le premier `.`) n'apparaît pas dans le manifest `images` du run courant, et émet pour chaque suppression un log `{"event":"gdrive_sync.image_orphan_removed","path":"<path>","source_id":"..."}`. Si le dossier sidecar devient vide après cleanup, il est supprimé.
- AC-11 : Cleanup d'archivage : si un `gdoc` passe en `archived: true` (cf ING-013 AC-8), le sidecar `<doc_slug>.images/` n'est **pas** supprimé (cohérence « jamais `rm` sur archived », ING-013 non-goals) ; le manifest `images` en état est conservé tel quel.
- AC-12 : Cleanup de rename : si un `gdoc` est renommé (cf ING-013 AC-6), le sidecar `<doc_slug>.images/` est renommé via `Path.rename(old_dir, new_dir)` dans la même opération que le `.md` ; un log `{"event":"gdrive_sync.images_dir_renamed","source_id":"...","old":"...","new":"..."}` est émis.
- AC-13 : Tous les `gdocs` invoquent Docs API uniformément, y compris ceux sans aucune image embarquée (un seul chemin de code, pas de fast-path conditionnel). Un `gdoc` sans `inlineObjects` produit un `.md` au corps inchangé vs `files.export` brut (modulo `normalize_body`) et un manifest `images` vide.
- AC-14 : `--dry-run` (cf ING-013 AC-12) couvre les images : aucun binaire image n'est téléchargé (les téléchargements sont skippés ou simulés via HEAD-only / mock), aucun fichier image n'est écrit, aucun orphelin n'est supprimé, aucun rename sidecar exécuté. Les compteurs summary `would_images_failed`, `would_images_written`, `would_orphans_removed` reflètent ce qui aurait été fait.
- AC-15 : Le log summary (`event=gdrive_sync.summary`, cf ING-013 AC-17) gagne trois champs : `images_written: int` (nb d'images binaires écrites sur disque dans le run), `images_failed: int` (cf AC-7), `orphans_removed: int` (cf AC-10). Préfixe `would_` en dry-run (cf AC-14).
- AC-16 : Cohérence ING-012 (auto-PR) : la règle de PR draft est étendue — la PR auto est créée en **draft** si `errors >= 1` **OU** `images_failed >= 1`. Un addendum à ING-012 AC-8 est requis (tracé en « Amendements requis »). Le body PR mentionne `images_failed` dans la ligne synthétique `Summary:`.
- AC-17 : Le frontmatter YAML écrit en tête du `.md` (cf ING-013 AC-4) reste byte-identique à ING-013 ; **aucun nouveau champ `images_dir`** n'est introduit. La convention path implicite « sidecar dir = `<basename(.md)>.images/` » suffit. Toute violation de cette convention par un consommateur futur est un bug de ce consommateur, pas une responsabilité ING-014.
- AC-18 : Tout chemin de fichier image produit est sous `lore/` et `Path.resolve().is_relative_to(lore_root)` (cohérence ING-010 AC-7 paths.py). Tout `md5_hex` est validé regex `^[0-9a-f]{12}$` avant insertion dans un chemin (défense path traversal).

## Non-goals

- Pas d'OCR des images.
- Pas de redimensionnement / compression / conversion de format (octets identiques au payload Drive).
- Pas d'extraction d'images depuis gsheet ou gslide (les converters ING-011 produisent leur propre Markdown ; ce ticket cible `gdoc` strict).
- Pas de gestion des images positionnées (`positionedObjects` Docs API) — seulement `inlineObjects`. Ticket futur.
- Pas de gestion des images de fond (page break, header/footer) — Docs API les expose ailleurs.
- Pas de retry sur 5xx du download image (phase 1 = log + placeholder + counter).
- Pas de cache cross-document : deux gdocs embarquant la même image binaire produisent deux fichiers dans deux sidecars distincts. Pas de `lore/_shared_images/`.
- Pas de support multi-niveau pour les listes imbriquées Markdown (cf AC-6).
- Pas de support tableaux Markdown (cf AC-6, conservés verbatim).
- Pas de réécriture des liens Drive (`https://docs.google.com/document/d/...`) dans le corps Markdown.
- Pas de validation MIME via inspection du payload (magic bytes) — confiance dans le `Content-Type` HTTP retourné par googleusercontent (cf threat-model si nécessaire).
- Pas de scope sur l'OAuth user (SA only, hérité ING-013).

## Pre-conditions

- **ING-013 mergé** : sync `gdoc → .md` opérationnelle, `scripts/.gdrive_state.json` versionné, schéma `StateEntry` existant.
- Service account ING-013 possède les scopes `https://www.googleapis.com/auth/drive.readonly` ET `https://www.googleapis.com/auth/documents.readonly` (le second est ajouté par ce ticket ; runbook à mettre à jour).
- API « Google Docs » activée sur le projet GCP (en plus de Drive).
- ING-012 mergé : workflow auto-PR existant à étendre (AC-16).

## Failure modes

- Scope `documents.readonly` manquant sur le SA → premier appel `documents.get` retourne 403 → exit non-zéro immédiat au boot du premier `gdoc` traité, message `gdrive docs scope missing: enable https://www.googleapis.com/auth/documents.readonly on the service account`.
- API Docs non activée sur le projet → 403 `accessNotConfigured` → même traitement.
- Download image HTTP 4xx/5xx → placeholder + counter (AC-7), traitement continué.
- Timeout 30s sur download image → placeholder + counter (AC-7), traitement continué.
- `Content-Type` hors map → placeholder + counter (AC-7), traitement continué.
- Écriture filesystem échoue (disque plein, permission) → placeholder + counter (AC-7), traitement continué.
- `inlineObjects` référencé dans le corps mais absent de la map Docs API (incohérence Drive) → placeholder + counter, log `reason: "object_id_unresolved"`.
- Document Docs API renvoie 404 alors que Drive list le voit → log error per-file standard ING-013 AC-16, exit final 1, `.md` non écrit.

## Performance / SLO

- Pas de SLO dur (cohérence ING-013).
- Pour un corpus auteur ~200 fichiers dont ~50% gdoc avec ~3 images en moyenne → ~300 GET supplémentaires par run. Aucun parallélisme phase 1 (sérialisé), timeout 30s par image. Borne pire-cas théorique d'un run : 200 × 30s = 100 min ; en pratique attendu < 15 min. Cap timeout job ING-012 = 30 min (AC-12 ING-012) reste applicable.
- Pas de retry phase 1 : un timeout → un placeholder, on continue.

## Security / trust boundary

- Cohérence threat-model + `.claude/rules/security.md` :
  - **A03 / A10 (SSRF)** : `contentUri` est résolu auprès de domaines `*.googleusercontent.com` contrôlés Google ; pas de fetch d'URL fournie par l'utilisateur final (l'humain auteur des Docs est implicitement trusted dans ce projet mono-auteur). Aucune validation CIDR additionnelle phase 1 — documenté en threat-model si nécessaire.
  - **A05** : binaires images persistés en `lore/` (repo public). L'auteur est responsable de ne pas embarquer d'image confidentielle dans un gdoc destiné à la sync.
  - **A08** : aucune désérialisation de format complexe — `open(path, 'wb').write(bytes)` brut, pas de parser image, pas de Pillow phase 1.
  - **Path traversal** : md5 hex validé regex (AC-18), nom fichier déterministe non-user-controlled, `Path.resolve().is_relative_to(lore_root)` asserté.
  - **Cap taille image** : aucune mention explicite phase 1. Le cap 1 MiB ING-013 AC-14 vise le `.md` post-normalisation ; les binaires images ne sont pas capés individuellement. **Cf Open questions OQ-1.**
- Cohérence `.claude/rules/secret-hygiene.md` : token OAuth du SA injecté dans header `Authorization` en mémoire process, jamais loggué, jamais persisté.

## Observability

- Log JSON image téléchargée : pas de log per-image en succès (volume) ; seulement comptage dans `images_written`.
- Log JSON image failed : `event=gdrive_sync.image_failed` (AC-7).
- Log JSON orphelin supprimé : `event=gdrive_sync.image_orphan_removed` (AC-10).
- Log JSON rename sidecar : `event=gdrive_sync.images_dir_renamed` (AC-12).
- Log JSON warning subset Markdown : `event=gdrive_sync.md_subset_fallback` (AC-6).
- Summary étendu : `images_written`, `images_failed`, `orphans_removed` (AC-15).

## Open questions
