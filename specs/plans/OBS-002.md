# Plan — OBS-002 gateway service-health endpoint `GET /v1/status` + santé widget

## Goal
Public anonymous `GET /v1/status` aggregating postgres/gcs/workers reachability into a `200`-always `ok|degraded` snapshot (10s cached, 2s-per-probe, parallel), surfaced as a `#health-widget` on `/observability`.

## Acceptance criteria recap
- AC-1 : body `{"status":"ok"|"degraded","dependencies":{"postgres":{..},"gcs":{..},"workers":{..}},"checked_at":"<rfc3339>"}` ; 3 deps toujours présentes, pas de `mistral`.
- AC-2 : chaque dep = exactement `{"status":"ok"|"down","latency_ms":<int≥0>}` ; enum fermé ; aucun host/url/message/SQL.
- AC-3 : racine `"ok"` ⇔ les 3 deps `"ok"` ; ≥1 down ⇒ `degraded`.
- AC-4 : `checked_at` RFC3339 UTC = instant de la mesure servie (pas l'instant HTTP si cache).
- AC-5 : `200 OK` même `degraded`.
- AC-6 : **jamais** de `5xx` ; chaque échec (erreur/timeout/non-2xx/`db_pool` absent) → `down`/`degraded` en `200`. Aucun chemin `503`.
- AC-7 : anonyme, hors sous-router auth, aucun guard ; test contractuel router.
- AC-8 : timeout dur 2 s/probe ; timeout → `down`, `latency_ms ≲ 2000`.
- AC-9 : 3 probes en parallèle ; all-down snapshot frais < 3 s (pas ~6 s).
- AC-10 : cache in-memory par-process TTL 10 s ; 2 appels <10 s = même `checked_at`, probes non relancées ; >10 s = `checked_at` distinct.
- AC-11 : probe en échec → `down` + log `warn` interne, aucun détail dans le body.
- AC-12 : probe `workers` = joignabilité seule de l'endpoint workers `/health` pré-existant ; `2xx`→`ok` sinon `down`.
- AC-13 : `observability.js` fetch `/v1/status` même-origine, rend `#health-widget` en `textContent` (statut global + 3 deps), style via CSS.
- AC-14 : issue non-200 → message littéral d'indisponibilité dans `#health-widget`.
- AC-15 : 5 headers sécurité router-wide présents byte-for-byte ; CSP non modifiée.
- AC-16 : `specs/openapi/gateway-to-workers.yml` non modifié.
- AC-17 : test manuel auteur coché avant `ready`.

## GCS helper design (locked decision 1)
Probe GCS = nouvelle fn libre dans `status.rs` (~15 LOC), **pas** de méthode sur `TokenProvider` :
```
async fn probe_gcs(state: &AppState) -> DependencyStatus
```
1. `let Ok((bearer, _)) = state.gcs_token_provider.get_or_refresh().await else { down }` (token échec → `down`).
2. `state.http.get(format!("https://storage.googleapis.com/storage/v1/b/{}/o?maxResults=1", state.config.gcs_bucket)).bearer_auth(bearer.expose_secret()).send()`.
3. `Ok(r) if r.status().is_success() => ok ; _ => down`. Prouve auth SA + bucket joignable sous `objectAdmin`.
Le timeout 2 s est appliqué par le wrapper `tokio::time::timeout` du handler (pas dans le helper). `get_or_refresh` retourne `(SecretString, bool)` — réutiliser l'instance existante (cache token interne déjà présent).

## Probe contract (les 3)
Chaque probe `async fn -> DependencyStatus { status: &'static str, latency_ms: u64 }` :
- postgres : `state.db_pool.as_ref()` → `None` ⇒ `down` (PAS d'erreur) ; sinon `sqlx::query("SELECT 1").execute(pool)` (runtime-typed, pas de macro → pas de `sqlx prepare`) → ok/down.
- gcs : helper ci-dessus.
- workers : MIRROIR de `health.rs::probe_workers` — `workers_id_token_provider.fetch_id_token()` puis `http.get("{workers_url}/health").bearer_auth(...)` ; échec/non-2xx ⇒ `down`.
Handler : `tokio::join!` sur les 3, chacun `tokio::time::timeout(PROBE_TIMEOUT, fut)` avec `const PROBE_TIMEOUT: Duration = Duration::from_secs(2);` ; timeout/err ⇒ `down`, `latency_ms = elapsed (≈2000)`, `tracing::warn!(event="status.probe.down", dep=...)` sans host/message.

## Cache design (D-3, mirror `token.rs`)
`CachedSnapshot { snapshot: StatusResponse (Clone), checked_at: DateTime<Utc> }`. `get_or_refresh` : read-lock fast-path (si `now - checked_at < 10s` → clone & serve, même `checked_at` ⇒ AC-4/AC-10) → write-lock double-check → exécute le closure de probes, stocke. `checked_at` vit DANS la valeur cachée. `const SNAPSHOT_TTL: Duration = Duration::from_secs(10);`.

- Property (AC-3, local) : table-driven sur les 8 combinaisons d'états de probes → `racine=="ok" ⇔ 3 ok`. **PAS** dans `specs/properties.md` (humain-only, aucun invariant status) → test local uniquement.
- Contract (AC-7) : `include_str!("../src/lib.rs")` — `/v1/status` dans `public_api`, hors `auth_router`/`dashboard_api`, aucun `RequireAuthor`/`RequireMember`.
- Schemathesis : N/A (AC-16, OpenAPI non touché).
- Eval : N/A (pas de chemin RAG).
- Injection mockito : GCS via `gcs_token_provider` base-url + `http` ciblant un mock googleapis ; workers via `new_with_id_token_provider` + mockito `/health`. AC-8/AC-9 = mock real-delay, pas de knob timeout en prod (locked decision 3).

## Risks / open questions
- **LOC** : estimation ~389 hors statique HTML/CSS, ~290 si on cible le code Rust+JS+tests. Le poste tests (~150) est le levier ; si >300 total, **trimmer** en déplaçant AC-4/AC-10 cache-timing + AC-8/AC-9 timing dans un second fichier de test n'aide pas le compteur. Proposition de trim : fusionner AC-1+AC-2+AC-5 en un seul test happy-path (db=None ⇒ tous `down` ⇒ `degraded`, vérifie forme+sanit+200 d'un coup) et AC-8 servant aussi d'AC-9 (un seul test all-timeout wall-clock). Reste ~290. **Si reviewer juge >300 net, signaler split statique en OBS-002b.**
- **GCS probe sans pool de mock** : `make_app_state` utilise `new_with_id_token_provider` (GCS provider = vrai metadata). Les tests GCS-`ok` nécessitent un ctor injectant un `gcs_token_provider` base-url-overridé (`new_with_token_provider` existe et est `#[cfg(test)]`). Sans ça, GCS sort `down` en test (acceptable pour les ACs forme/sanitisation, suffisant car aucun AC n'exige GCS=`ok` en intégration — seul le manuel auteur AC-17 le couvre).
- **AC-9 wall-clock flakiness** : seuil <3 s sur CI lente. Garder marge ; un seul tel test.
- **db_pool=None ⇒ postgres down** : confirme que la suite intégration verra `degraded` par défaut (db absente). Les ACs forme/200/sanitisation passent ; AC-3 "ok" branch couvert par le property test (stub), pas par l'intégration.
