# SEC-003 — Security headers gateway : ajout HSTS + couverture exhaustive

## Contexte

Vision §137 (Q3) et §162 ont sorti les headers sécurité d'INFRA-002 pour les livrer en ticket séparé ≤ 300 LOC. UI-001 a déjà câblé 4 des 5 headers requis par `.claude/rules/security.md` (A05) au niveau du router racine (`gateway/src/lib.rs` lignes 127-142) : `Content-Security-Policy`, `X-Content-Type-Options`, `Referrer-Policy`, `X-Frame-Options`. Il manque `Strict-Transport-Security` et la garantie que les 5 headers couvrent TOUTES les réponses (succès, erreur, body-limit 400, 404, 405, `/healthz`) — pas seulement les routes statiques + `POST /v1/chat` couvertes par UI-001 AC-6/AC-17. Ce ticket finalise le baseline A05.

## Critères d'acceptation

- AC-1 : Toute réponse HTTP émise par le router gateway (succès 2xx, redirection 3xx, erreur client 4xx, erreur serveur 5xx, sur n'importe quelle route existante `/`, `/assets/*`, `/healthz`, `/v1/chat`, ainsi que sur tout chemin inexistant retournant 404 ou 405) porte l'en-tête `Strict-Transport-Security` avec la valeur littérale exacte `max-age=31536000; includeSubDomains; preload`.
- AC-2 : Les 4 en-têtes déjà posés par UI-001 restent présents avec leurs valeurs littérales exactes sur l'ensemble des réponses couvertes par AC-1 :
  - `Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self'; img-src 'self' data:; object-src 'none'; frame-ancestors 'none'; base-uri 'none'; form-action 'self'`
  - `X-Content-Type-Options: nosniff`
  - `Referrer-Policy: strict-origin-when-cross-origin`
  - `X-Frame-Options: DENY`
- AC-3 : Sur `GET /healthz` (succès 200 ET échec amont workers retournant 5xx), les 5 en-têtes (HSTS + les 4 d'AC-2) sont présents avec les valeurs littérales exactes ci-dessus.
- AC-4 : Sur `POST /v1/chat` avec body > 1 MiB (déclenchant le `RequestBodyLimitLayer` GEN-002 / R2 réécrit en 400 par `handle_body_limit_error`), la réponse 400 porte les 5 en-têtes avec les valeurs littérales exactes ci-dessus.
- AC-5 : Sur `GET /assets/inexistant.txt` (404 servi par `ServeDir`) ET sur `GET /chemin/inexistant` (404 router fallback), les 5 en-têtes sont présents avec les valeurs littérales exactes ci-dessus.
- AC-6 : Si une réponse interne (handler, ServeDir, layer en aval) émet déjà l'un des 5 en-têtes, la valeur du handler est conservée (sémantique `if_not_present` du `SetResponseHeaderLayer`). Aucun en-tête n'est dupliqué : pour les 5 noms d'en-tête considérés, `Response::headers().get_all(name).iter().count() == 1` sur chaque réponse testée par AC-1 à AC-5.
- AC-7 : L'ajout du header `Strict-Transport-Security` est réalisé via un `SetResponseHeaderLayer::if_not_present` appliqué au router racine, dans le même bloc `.layer(...)` que les 4 headers existants de `gateway/src/lib.rs` (router-wide, ordre `attach_request_id` en haut, headers sécurité juste avant `TraceLayer`). Aucun nouveau middleware custom n'est introduit, aucune crate ajoutée au `Cargo.toml`.
- AC-8 : Les tests d'intégration de `gateway/tests/static_test.rs` (helper `assert_security_headers`) sont étendus pour assert les 5 en-têtes (au lieu de 4) avec leurs valeurs littérales. L'helper porte une constante `HSTS_VALUE = "max-age=31536000; includeSubDomains; preload"` utilisée dans toutes les assertions. Les tests existants AC-6 UI-001 (`ac6_root_has_security_headers`, `ac6_app_js_has_security_headers`, `ac6_styles_css_has_security_headers`) et `ac17_chat_endpoint_has_security_headers` continuent de passer.
- AC-9 : `gateway/tests/static_test.rs` ajoute au moins un test par cas non couvert par UI-001 : `GET /healthz` 200, `GET /healthz` 5xx (workers down — `workers_url` pointe sur port fermé), `POST /v1/chat` body > 1 MiB → 400, `GET /assets/inexistant.txt` 404, `GET /chemin/inexistant` 405-or-404. Chaque test appelle `assert_security_headers(&resp)` (qui vérifie les 5 headers + non-duplication AC-6).
- AC-10 : Aucune régression CI : `cargo fmt --check` + `cargo clippy -- -D warnings` + `cargo test` (job `contract`) verts. Pas de modification de `gateway/static/`, `gateway/src/handlers/`, ni de `migrations/`.

## Non-goals

- Pas de soumission au navigateur preload list `hstspreload.org` dans ce ticket (acte humain post-déploiement INFRA-002, hors-code).
- Pas de configuration TLS / certificat — Cloud Run + Cloudflare gèrent la terminaison TLS (INFRA-002, hors-scope).
- Pas de CSP nonce / `strict-dynamic` — V2 (vision §137 implicite : V1 = baseline statique UI-001).
- Pas de Turnstile / challenge bot — Cloudflare Bot Fight Mode V1 (vision Q19), Turnstile = V2.
- Pas de CORS allowlist — ticket séparé (cf vision Q non-tranché, hors A05 baseline).
- Pas de `Permissions-Policy` header — non listé dans `.claude/rules/security.md` A05, hors baseline.
- Pas de `Cross-Origin-Embedder-Policy` / `Cross-Origin-Opener-Policy` / `Cross-Origin-Resource-Policy` — non listés A05, V2 si SharedArrayBuffer un jour requis.
- Pas de retouche du CSP existant (CSP UI-001 est PLUS stricte que la baseline `.claude/rules/security.md` qui dit `default-src 'self'; object-src 'none'; frame-ancestors 'none'` ; on garde la version stricte UI-001 — pas de régression).
- Pas de retouche du Content-Type default (Axum + handlers existants posent déjà `application/json; charset=utf-8` sur les routes API et `text/html; charset=utf-8` via `ServeFile`).
- Pas de modification de `gateway/static/index.html` ni du handler `/healthz`.
- Pas de port de ces headers côté workers FastAPI — les workers sont internes (non exposés public), seul le gateway est en bordure.
- Pas de header `Server:` strippé / réécrit — V2 si fingerprinting devient un sujet.

## Failure modes

- Un handler aval émet explicitement un `Strict-Transport-Security` divergent → `if_not_present` conserve la valeur du handler, AC-1 échoue sur ce chemin. Couvert par AC-6 (non-duplication) et AC-9 (test 5xx `/healthz`). Mitigation : aucun handler actuel n'émet HSTS, donc le risque est de régression future détectée par les tests AC-9.
- `Strict-Transport-Security` ajouté côté navigateur sur un déploiement non-HTTPS → comportement spec HSTS : le header est ignoré sur connexion HTTP (RFC 6797 §8.1). Pas de breakage local `http://localhost:8080`, l'agent utilisateur ignore simplement HSTS. Aucun guard runtime nécessaire.
- Submitted to `preload` list mais déploiement HTTPS pas garanti sur sous-domaine → blocage navigateur sur sous-domaines non-HTTPS. Mitigation : la soumission preload est un acte humain hors-ticket (cf non-goals) ; SEC-003 livre uniquement le header.
- `tower-http` casse l'API `SetResponseHeaderLayer::if_not_present` (upgrade non-rétrocompatible) → CI clippy/build casse, blocker au sens `no-workaround.md`. Mitigation : `Cargo.lock` versionné, upgrade `tower-http` requiert un ticket dédié.
- Cloudflare ré-injecte `Strict-Transport-Security` côté edge avec une autre valeur → double header possible côté client. Mitigation : si observé post-INFRA-002, le ticket adjacent (Cloudflare rules) doit le désactiver edge-side ; SEC-003 reste source de vérité origine.

## Performance / SLO

- Overhead ajout d'un 5ème `SetResponseHeaderLayer` négligeable (constante statique, écriture O(1) par réponse). Vision §91 (`< 80 ms p95 à 500 users`) reste valable, pas de mesure formelle requise.
- Pas de mesure benchmark checked-in.

## Security / trust boundary

- A05 Security Misconfiguration : ce ticket clôt la baseline `.claude/rules/security.md` §A05 (HSTS + CSP + nosniff + Referrer-Policy + frame-ancestors via X-Frame-Options).
- A02 Cryptographic Failures : `Strict-Transport-Security: max-age=31536000; includeSubDomains; preload` force le navigateur à utiliser HTTPS pendant 1 an, inclut tous les sous-domaines, signale l'éligibilité preload list. Conforme `.claude/rules/security.md` §A02 ligne 14.
- Trust boundary : gateway = unique surface publique (workers internes). Headers posés origine, Cloudflare peut compléter edge-side (hors-ticket).
- Pas de secret manipulé, pas de log applicatif nouveau, pas de modification d'auth.
- Cohérence threat-model.md (SEC-001 / SEC-002 si existants — non-bloquants pour SEC-003).
