# ING-001 — Ingestion `lore/*.md` local (parse + chunk + embed + upsert idempotent)

## Contexte

FOUND-003 a posé les tables `documents` et `chunks` (embedding `vector(1024)`) mais aucun pipeline ne les remplit. Sans ingestion, RET-001 et GEN-001 ne peuvent pas démarrer : pas de corpus = pas de retrieval. Ce ticket livre le premier mécanisme d'alimentation du corpus, conforme à la phase MVP (vision §44-50) : fichiers Markdown locaux, frontmatter YAML, embeddings bge-m3, UPSERT idempotent invocable manuellement par l'auteur.

## Critères d'acceptation

- AC-1 : La commande `uv run python -m archiviste_workers.ingest --path lore/` parcourt récursivement tout fichier `*.md` sous le chemin fourni et tente l'ingestion de chacun.
- AC-2 : Chaque fichier ingéré voit son frontmatter YAML (délimité par `---` en tête) parsé en trois champs : `title` (str non-vide, requis), `tags` (list[str], défaut `[]`), `access_tier` (str ∈ {`public`,`members`,`author_only`}, défaut `public`).
- AC-3 : Un fichier sans frontmatter, ou sans `title`, ou avec `title` vide, ou avec `access_tier` hors ensemble est skippé : aucune écriture DB pour ce fichier, log JSON `{"event":"ingest","path":"...","status":"error","reason":"..."}` émis, traitement des autres fichiers continué.
- AC-4 : Le corps Markdown (frontmatter exclu) est normalisé `unicodedata.normalize('NFKC', text)` puis dépouillé des caractères de contrôle `\x00`-`\x1f` à l'exception de `\n` et `\t` avant chunking et hashing.
- AC-5 : Le `content_hash` persisté dans `documents.content_hash` est le SHA-256 hex du corps normalisé (post AC-4), frontmatter exclu.
- AC-6 : Le corps normalisé est chunké via `RecursiveCharacterTextSplitter` (langchain-text-splitters) avec `chunk_size=512` tokens mesurés via le tokenizer `BAAI/bge-m3`, `chunk_overlap=64`, séparateurs ordonnés `["\n\n","\n",". "," ",""]`.
- AC-7 : Chaque chunk persisté dans `chunks` porte `ord` égal à sa position 0-indexée dans la séquence émise par le splitter pour ce document, et le couple `(document_id, ord)` est unique (déjà garanti par schéma FOUND-003).
- AC-8 : Chaque chunk est embeddé via `BAAI/bge-m3` (sentence-transformers, self-host) en batches de taille `EMBED_BATCH_SIZE` (variable d'environnement, défaut documenté = 32) ; l'embedding inséré dans `chunks.embedding` est de dimension exactement 1024.
- AC-9 : Au premier run, le modèle `BAAI/bge-m3` est téléchargé dans le cache Hugging Face local (`~/.cache/huggingface/`) ; aux runs suivants, aucun téléchargement réseau du modèle n'a lieu (vérifié par absence de requête sortante après warm-up).
- AC-10 : Pour un fichier dont `source_path` n'existe pas dans `documents`, l'ingestion fait `INSERT documents` puis `INSERT chunks` ; le log JSON émis a `status: "inserted"` et `chunks: N`.
- AC-11 : Pour un fichier dont `source_path` existe dans `documents` avec un `content_hash` identique au calcul courant, l'ingestion ne fait aucune écriture (ni `documents`, ni `chunks`) ; le log JSON émis a `status: "skipped"` et `reason: "unchanged"`.
- AC-12 : Pour un fichier dont `source_path` existe dans `documents` avec un `content_hash` différent, l'ingestion fait dans une transaction unique : `DELETE FROM chunks WHERE document_id = $1` (CASCADE déjà posé), `UPDATE documents SET title=..., tags=..., access_tier=..., content_hash=..., updated_at=NOW() WHERE id=$1`, puis `INSERT chunks` nouvelle séquence ; le log JSON émis a `status: "updated"` et `chunks: N`.
- AC-13 : Toutes les écritures DB liées à un même fichier (insert ou update) s'exécutent dans une transaction unique : un échec en milieu de séquence ne laisse aucun chunk orphelin et `documents.updated_at` n'est pas avancé.
- AC-14 : `source_path` persisté est le chemin POSIX relatif à la racine du repo (ex. `lore/sample/archiviste.md`) ; un argument `--path` absolu hors du repo, ou un fichier dont le chemin résolu sort du repo, est rejeté avec exit non-zéro et message `path must be relative to repo root`.
- AC-15 : Tout fichier dont la taille sur disque dépasse 1 MiB est skippé sans lecture du contenu : log JSON `status: "error"`, `reason: "file exceeds 1 MiB cap"`.
- AC-16 : À la fin du run, un log JSON unique `{"event":"ingest.summary","total":N,"inserted":I,"updated":U,"skipped":S,"errors":E,"duration_ms":D}` est émis ; `total == inserted + updated + skipped + errors`. Les logs par fichier utilisent `event="ingest.document"` (namespacing).
- AC-17 : Le code de sortie du process est `0` si `errors == 0`, `1` si `errors >= 1` (les autres fichiers ayant été traités).
- AC-18 : Le répertoire `lore/sample/` est versionné et contient au moins 2 fichiers `.md` minimaux (frontmatter valide + corps non vide) suffisants pour une démo.
- AC-19 : Les tests d'intégration utilisent `workers/tests/fixtures/lore/` (isolé de `lore/sample/`) pour éviter les changements de seed démo qui invalideraient AC-11 (`unchanged → skipped`).
- AC-20 : Au démarrage du run, un log JSON unique `{"event":"ingest.start","embed_batch_size":N,"path":"...","cwd":"..."}` est émis pour traçabilité ops.

## Non-goals

- Pas d'ingestion d'images (phase 2, ING-002).
- Pas de sync GDrive (phase 3, ING-010).
- Pas de détection de fichier supprimé en `lore/` (orphan documents en DB) — ticket futur.
- Pas de re-chunking automatique sur changement de `chunk_size` / `chunk_overlap` — l'opérateur déclenche un force re-ingest manuel (mécanisme = ticket futur).
- Pas d'API HTTP, pas de UI, pas d'endpoint FastAPI — CLI one-shot uniquement.
- Pas de cron / daemon / watcher filesystem — invocation manuelle humaine.
- Pas de progress bar interactive — logs JSON seuls.
- Pas de parallélisme intra-fichier ni inter-fichier — fichiers traités séquentiellement (ticket perf futur).
- Pas de rejection embedding cosine ≥ 0.99 vs known-bad cluster — reportée à un ticket SEC-* dédié.
- Pas de fetch d'URL distante (vision §46, threat model W-E-1).

## Pre-conditions

- FOUND-002 mergé : `make migrate` opérationnel, Postgres + pgvector en docker-compose.
- FOUND-003 mergé : tables `documents` et `chunks` (vector(1024) + index HNSW cosine) appliquées par migration `0002_schema.sql`.
- Modèle `BAAI/bge-m3` téléchargeable depuis Hugging Face Hub au premier run (réseau sortant requis cette fois unique).

## Failure modes

- Frontmatter YAML manquant / malformé / `title` absent ou vide / `access_tier` hors ensemble → fichier skippé, log JSON `status: "error"`, `reason` explicite, autres fichiers traités, exit final `1`.
- Fichier > 1 MiB → skippé sans lecture du contenu, log JSON `status: "error"`, `reason: "file exceeds 1 MiB cap"`, exit final `1`.
- `--path` absolu hors repo, ou fichier résolu hors repo → exit non-zéro immédiat avec message `path must be relative to repo root`, aucun fichier traité.
- Échec connexion Postgres au démarrage → exit non-zéro avec message `database connection failed: <error>`, aucun fichier traité.
- Échec téléchargement modèle bge-m3 au premier run (réseau coupé) → exit non-zéro avec message `failed to load embedding model: <error>`, aucun fichier traité.
- Erreur SQL en milieu de transaction sur un fichier → transaction rollback (cf AC-13), log JSON `status: "error"`, `reason: <db error>`, autres fichiers traités, exit final `1`.
- Embedding produit avec dimension ≠ 1024 (improbable, mais défense en profondeur) → INSERT chunks lève l'erreur `vector` de FOUND-003 AC-11, transaction rollback, log JSON `status: "error"`, exit final `1`.

## Performance / SLO

- Pas de SLO dur cette itération (corpus MVP ~10-50 fichiers).
- Indicateur observable via `duration_ms` du log de summary, à mesurer comme baseline pour tickets perf ultérieurs.

## Security / trust boundary

- Conforme threat model W-T-1 : normalisation NFKC + strip control chars + cap taille fichier (1 MiB) sur tout document ingéré.
- Conforme threat model W-E-1 : aucune lecture d'URL réseau pendant l'ingestion (modèle bge-m3 = exception unique, téléchargement Hugging Face Hub one-shot premier run).
- Conforme `.claude/rules/security.md` "Embedding poisoning" : NFKC + strip control. Rejection cosine known-bad cluster = ticket SEC-* dédié, hors scope.
- `source_path` validé relatif au repo (pas de path traversal vers `/etc/passwd` ou hors corpus).
- Aucun secret dans le pipeline ; connexion DB lue depuis `DATABASE_URL` (FOUND-002).

## Observability

- Log JSON par doc traité : `{"event":"ingest","path":"...","status":"inserted|updated|skipped|error","chunks":N,"reason":"<si erreur ou skip>"}`.
- Log JSON final de summary : `{"event":"ingest_summary","total":N,"inserted":I,"updated":U,"skipped":S,"errors":E,"duration_ms":D}`.
- Pas de métrique OpenTelemetry exposée à ce stade (tickets OBS-* ultérieurs).
- Pas d'intégration Langfuse (ingestion ≠ chemin LLM de réponse).
