| `workers/src/archiviste_workers/main.py:35` | HIGH | broken-in-prod, masked by tests | Lifespan creates the asyncpg pool with raw `asyncpg.create_pool(...)`, **not** `db.create_pool(...)`. The pgvector codec (`pgvector.asyncpg.register_vector`) is therefore **never** installed on the pool used in production. The retrieve SQL binds `$1` as `vector` via `c.embedding <=> $1`, and `asyncpg` will fail to encode a Python `list[float]` as `vector` without the codec. Tests use `clean_db` -> `db.create_pool(...)` (with codec, see `workers/src/archiviste_workers/db.py:23-30` and `workers/tests/conftest.py:43-49`), so this never surfaces in CI. The plan's hypothesis #2 explicitly assumed the existing pool already had `register_vector` — that assumption is **false** for the lifespan-built pool (the only place that mattered was `ingest/cli.py`). | Switch `main.py:35` to use `archiviste_workers.db.create_pool(settings.database_url)`. Add a regression test that boots the real `lifespan(app)` against Postgres and asserts that `app.state.db_pool` can encode a 1024-dim list into `<=>` without raising. |
| `specs/openapi/gateway-to-workers.yml:171` | MED | spec breakage / out-of-scope side-effect | RET-001 renamed `components.schemas.Chunk` -> `RetrievedChunk` (sanctioned for `/v1/retrieve`), but `GenerateRequest.properties.contexts.items.$ref` (line 171) still points at `#/components/schemas/Chunk`, which no longer exists. Worth confirming intent with the spec author. | If spec wants stripped reaching the embedder + counted in `query_len`, change `return value` -> `return stripped` on `router.py:70`. The decision is reasonable (general beats specific) but contradicts the literal reading of AC-3/AC-5. | Either clarify the spec (AC-3/AC-5 win when the key is missing) and split the error path, or add an explicit note in `router.py` that AC-2 takes precedence over AC-3/AC-5 for missing keys. |

## Security

- **Secrets in code**: none detected.
- **SQL injection**: parameterised; ACL filter via `ANY($2::text[])`; embedding via `$1`. No `f-string` / `.format` into SQL. OK.
- **Trust boundary (W-S-1)**: `user_tier` accepted from gateway as documented in spec (out-of-scope for RET-001). OK.
- **Resource limits**: `query` <= 4096 bytes UTF-8 enforced (`MAX_QUERY_BYTES`); `top_k in [1,20]` enforced; SQL timeout 5.0s on `conn.fetch(..., timeout=...)`. Body cap (1 MiB per security.md A04) is **not enforced at the worker level** — relies on the upstream gateway. Spec marks worker rate-limit / body cap as out-of-scope. OK with caveat.
- **Logging redaction (AC-16 + A09)**: `query`, `text`, `embedding`, raw DB error strings never logged. Verified by `test_retrieve_logging.py` and by inspection of `router.py` (only `query_len`, `top_k`, `user_tier`, `results`, `embedding_ms`, `search_ms`, `status` are logged). OK.
- **A05 error responses**: 503 body is `{"error":"database_unavailable"}` with no DB error string; 400 body is `{"error":"<code>"}`. OK.
- **Prompt injection / output sanitization / SSRF**: not applicable to this slice (no LLM call, no user-supplied URL fetched). OK.
- **Forbidden patterns** (`security.md`): no `unwrap()`/`expect()` on user input (no equivalent reckless `dict[..]` access on user data); no `unsafe`; no `eval` / unsafe deserialization (`yaml.safe_load` in `test_retrieve_contract.py`); no `verify=False`; no CORS wildcard; no JWT misuse. OK.
