# ING-010 — Sync Google Drive → `lore/*.md` (core : gdoc + png + state + CLI)

## Contexte

La vision §52 fige la phase 3 d'ingestion : un dossier Google Drive auteur sert de source amont, exporté en `*.md` versionnés sous `lore/`. Sans ce ticket, l'auteur édite le Markdown à la main alors que son corpus de travail vit dans Drive ; toute édition Drive reste invisible du RAG. Ce ticket livre le **noyau** offline de l'export (gdoc + png + state file + frontmatter merge + auth service account + CLI manuel) ; les converters gsheet/gslide sont reportés à ING-011 et le workflow GitHub Actions à ING-012.

## Critères d'acceptation

- AC-1 : La commande `uv run python scripts/gdrive_export.py --root-folder-id <id>` parcourt récursivement le dossier Drive racine fourni (et ses sous-dossiers) via l'API Drive v3 et liste tous les fichiers de MIME `application/vnd.google-apps.document` et `image/png` accessibles au service account.
- AC-2 : Chaque `gdoc` (`mimeType=application/vnd.google-apps.document`) est exporté via `files.export(mimeType='text/markdown')` puis écrit en `lore/<drive_path>/<doc_slug>.md` où `<drive_path>` reproduit l'arborescence Drive relative à la racine et `<doc_slug>` est le titre Drive slugifié (cf AC-6).
- AC-3 : Chaque `image/png` natif Drive est téléchargé via `files.get_media(fileId=...)` et écrit en `lore/<drive_path>/<image_slug>.png` ; aucun `.md` n'est produit pour les PNG (binaire pur, pas de frontmatter). Les PNG ne sont pas embeddés ni indexés par ce ticket.
- AC-4 : Chaque composant de `<drive_path>` (nom de sous-dossier Drive) est slugifié via la fonction d'AC-6 ; l'arborescence locale produite est donc 100 % composée de noms `[a-z0-9-]+`.
- AC-5 : Le frontmatter YAML écrit en tête de chaque `.md` exporté contient strictement les champs : `title` (str = titre Drive), `tags` (list[str], défaut `[]`), `access_tier` (str, défaut `public`), `source` (str = `gdrive`), `source_id` (str = `<drive_file_id>`), `drive_path` (str = chemin POSIX Drive relatif à la racine, ex. `Lore/Personnages/`), `exported_at` (str ISO8601 UTC), `archived` (bool, défaut `false`), `archived_at` (str ISO8601 UTC ou absent).
- AC-6 : La fonction de slugification est déterministe et applique : `unicodedata.normalize('NFKD')` → drop combining marks → lower → remplacement de tout caractère hors `[a-z0-9]` par `-` → collapse runs de `-` → strip `-` aux extrémités → cap 80 chars. Si le slug résultant est vide, fallback `file-<drive_file_id[:8]>`.
- AC-7 : Une collision de path final (deux fichiers Drive distincts produisant le même chemin local après slug + extension) est résolue en suffixant `-<drive_file_id[:8]>` au stem du second, troisième, ... fichier rencontré (ordre déterministe = tri lexical sur `drive_file_id`) ; aucun fichier n'est silencieusement écrasé.
- AC-8 : Un fichier d'état persistant `scripts/.gdrive_state.json` (versionné dans le repo, JSON canonique trié, indent 2) maintient une map `source_id → {local_path, drive_md5_checksum, last_exported_at}`. Le script lit cet état au démarrage et le réécrit à la fin du run.
- AC-9 : Détection de **rename Drive** : si un `source_id` connu de l'état pointe vers un `local_path` différent du chemin courant calculé (changement de titre Drive ou déplacement entre dossiers), le script invoque `git mv <ancien_path> <nouveau_path>` (via `subprocess` avec argv array, jamais `shell=True`) avant de réécrire le contenu, préservant l'historique git. Si `git mv` échoue (ex. fichier non tracké), fallback `os.rename` avec log warn `git_mv_failed_fallback_rename`. Cette règle s'applique **identiquement** aux PNG (rename Drive du fichier `image/png` détecté via `drive_file_id` stable en état → `git mv` du PNG local) ; un événement spécifique `event=gdrive_sync.image_renamed` est loggé avec `old_path` / `new_path`.
- AC-10 : Détection de **suppression Drive** : si un `source_id` est présent dans `.gdrive_state.json` mais absent de la dernière liste Drive (fichier supprimé/déplacé hors racine/access perdu), le script ne supprime PAS le fichier local mais (a) pour un `.md` : pose/force `archived: true` et `archived_at: <ISO8601 UTC>` dans le frontmatter, conserve le corps ; (b) pour un PNG : aucun marqueur (binaire), simple log `event=gdrive_sync.file, status=archived` ; le `source_id` reste dans l'état avec `archived_at` figé.
- AC-11 : Détection de **contenu inchangé** : le script compare `md5Checksum` retourné par Drive (gdoc : md5 de l'export Markdown rendu, calculé client-side après export ; PNG : `md5Checksum` natif retourné par `files.get`) à la valeur en état. Identique ⇒ le fichier local n'est PAS réécrit (pas de churn `exported_at`), l'état n'est pas mis à jour pour ce fichier, status loggé `unchanged`.
- AC-12 : **Merge frontmatter sur ré-export** : lors d'un update (contenu Drive changé) d'un `.md` qui existe déjà localement, le script lit le frontmatter existant et applique la règle de merge :
  - Champs **script-managed** (toujours réécrits par le script, écrasent l'existant) : `title`, `source_id`, `exported_at`, `archived`, `archived_at`.
  - Champs **user-managed** (préservés tels quels si déjà présents dans le frontmatter local ; créés avec valeur par défaut UNIQUEMENT au premier export quand le fichier local n'existait pas) : `tags` (défaut `[]`), `access_tier` (défaut `public`).
  - Tout autre champ custom ajouté manuellement par l'humain (clé absente de la liste managed et user-managed ci-dessus) : préservé tel quel, jamais touché par le script.
  - Le champ `source` (= `gdrive`) et `drive_path` sont script-managed (réécrits).
- AC-13 : Le contenu Markdown écrit (corps post-frontmatter) est normalisé `unicodedata.normalize('NFKC', text)` puis dépouillé des caractères de contrôle `\x00`-`\x1f` à l'exception de `\n` et `\t` (cohérence ING-001 AC-4, threat-model W-T-1). Cette normalisation ne s'applique pas aux PNG (binaires).
- AC-14 : Tout fichier dont l'export rendu (gdoc Markdown post-export ; PNG binaire) dépasse 1 MiB est skippé : log JSON `status: "error"`, `reason: "exported size exceeds 1 MiB cap"`, état non mis à jour pour ce fichier, traitement des autres fichiers continué (cohérence ING-001 AC-15).
- AC-15 : Les credentials du service account sont lus exclusivement via la variable d'environnement `GDRIVE_SA_KEY_JSON` (contenu JSON inline) OU `GOOGLE_APPLICATION_CREDENTIALS` (path vers fichier) ; aucun fallback hardcodé, aucun chemin par défaut, fail-fast au boot avec message explicite si ni l'un ni l'autre n'est défini.
- AC-16 : Le script accepte un flag `--dry-run` qui exécute la résolution complète (list Drive, calcul des paths, diff vs état) et logge les actions qu'il aurait effectuées (`would_create | would_update | would_rename | would_archive | would_skip | would_unchanged`) sans écrire aucun fichier ni modifier l'état.
- AC-17 : À la fin du run, un log JSON unique `{"event":"gdrive_sync.summary","total":N,"created":C,"updated":U,"renamed":R,"archived":A,"unchanged":S,"errors":E,"duration_ms":D}` est émis sur stdout ; chaque fichier traité produit un log `{"event":"gdrive_sync.file","source_id":"...","local_path":"...","status":"created|updated|renamed|archived|unchanged|error","reason":"<si error>"}`.
- AC-18 : Le code de sortie du process est `0` si `errors == 0`, `1` si `errors >= 1` (les autres fichiers ayant été traités), cohérent ING-001 AC-17.
- AC-19 : Au démarrage du run, un log JSON unique `{"event":"gdrive_sync.start","root_folder_id":"...","dry_run":<bool>,"state_size":<N>}` est émis. Si l'état est absent (premier run), un log additionnel `{"event":"gdrive_sync.first_run"}` est émis et l'état est traité comme `{}` vide.
- AC-20 : **Drive = source de vérité unique pour le corps Markdown.** L'état persiste un `body_hash` (sha256 hex) du dernier corps généré côté script pour chaque `.md`. Lors d'un ré-export où Drive a changé (md5 différent), si le `body_hash` calculé sur le fichier local courant diffère du `body_hash` en état (= l'humain a édité localement), le script logge un warning `event=gdrive_sync.local_drift_overwritten` (avec `local_path`, `source_id`) puis overwrite le corps quand même avec le contenu Drive. Aucun mode « préserve user edit body » n'est offert ; les édits humains du corps doivent passer par Drive. Le merge de frontmatter (AC-12, clés user-managed et custom) reste appliqué.

- Pas de filtrage retrieval sur `archived: true` côté workers RET-* — ce ticket pose seulement le marqueur ; le filtre est un ticket RET-* dédié.
- Pas de suppression physique des fichiers `.md` ou `.png` archivés (jamais `rm` côté script).
- Pas de re-write de l'historique git (pas de `git mv` rétroactif sur fichiers déjà archivés).
- Pas de webhook Drive push notification — invocation manuelle CLI seule sur ce ticket.
- Pas de support OAuth user (seul service account avec scope dossier dédié).
- Pas d'auth API exposée par le script (offline tool).
- Pas d'intégration avec ING-001 / ingestion DB ; le script s'arrête après écriture des `.md` et `.png`. L'ingest run `python -m archiviste_workers.ingest --path lore/` est invoqué séparément.

- Pas de gestion des `application/vnd.google-apps.shortcut` — skippés silencieusement avec log info.
- Pas de gestion des fichiers natifs non-Google (PDF, DOCX uploadés dans Drive) — skippés avec log info, ticket futur.
- Pas de scope multi-racine (un seul `--root-folder-id` par run).
- Pas de support des Shared Drives (`supportsAllDrives`) phase 1 — `My Drive` du SA uniquement.
- Pas de retry de l'export Drive sur 5xx (failure → log error, fichier suivant) ; ticket OBS-* / résilience reporté.
- Pas de sync inverse (`lore/` → Drive). Drive est source amont unique.

## Failure modes

- `GDRIVE_SA_KEY_JSON` / `GOOGLE_APPLICATION_CREDENTIALS` absents au boot → exit non-zéro immédiat, message `gdrive credentials missing: set GDRIVE_SA_KEY_JSON or GOOGLE_APPLICATION_CREDENTIALS`, aucun fichier traité.
- `--root-folder-id` absent ou non résolvable (404 Drive) → exit non-zéro immédiat, message `root folder not found or inaccessible: <id>`.
- Service account non autorisé sur le dossier (403) → exit non-zéro immédiat, même format.
- Quota / rate-limit Drive API (429, 403 `userRateLimitExceeded`) → log error pour le fichier en cours, traitement continué, pas de retry phase 1, reflété dans `errors` du summary.
- Export Drive échoue avec 5xx → log error avec `reason: "drive_export_failed: <status>"`, fichier skippé, traitement continué, exit final `1`.
- Fichier exporté > 1 MiB → skippé, cf AC-14.
- Slug vide après normalisation → fallback `file-<drive_file_id[:8]>`, log warn, traitement continué (pas un error).
- Collision de path (cf AC-7) → suffixe `-<drive_file_id[:8]>` au second arrivant, log warn, traitement continué.
- `git mv` échoue (fichier non tracké) → fallback `os.rename`, log warn `git_mv_failed_fallback_rename`, traitement continué.
- État `.gdrive_state.json` corrompu / non-JSON → exit non-zéro immédiat, message `state file corrupted: <path>`, demande intervention humaine (pas de re-init silencieuse).
- État absent (premier run) → traité comme `{}` vide, tous les fichiers comptent comme `created`, log info `event=gdrive_sync.first_run` (cf AC-19).
- Frontmatter local existant illisible / YAML invalide lors d'un merge (AC-12) → log error `frontmatter_parse_failed`, fichier skippé sur cet update (préserve l'existant intact), exit final `1`.

## Security / trust boundary

- Outil offline / dev (vision §54). Pas de surface réseau exposée par ce ticket.
- Service account scopé read-only, sur dossier unique partagé manuellement (least-privilege).
- Secret SA jamais commité : `.env.example` commente la variable, `.gitignore` couvre `*-sa.json` (cf `.claude/rules/secret-hygiene.md`). Le hook `guard-secret-paths.sh` doit déjà couvrir.
- Cohérence threat-model W-T-1 : normalisation NFKC + strip control chars (AC-13) + cap 1 MiB (AC-14) sur tout contenu Markdown Drive avant écriture `lore/`.
- Path traversal : tous les paths produits sont sous `lore/`, tous les composants slugifiés (AC-4, AC-6). Aucune injection de `..` possible (slug regex `[a-z0-9-]`).
- `subprocess` `git mv` invoqué exclusivement avec argv array (pas de `shell=True`), arguments slugifiés, jamais d'injection shell possible (AC-9).
- PNG binaires : pas de normalisation, mais cap 1 MiB appliqué ; pas de désérialisation, simple `open(..., 'wb').write(bytes)`.

## Décisions résolues (humain, 2026-05-09 — round 2)

- **OQ-A — drift corps `.md` ↔ Drive (AC-20)** : Drive = source de vérité unique pour le corps. Le script overwrite TOUJOURS le body en cas de changement Drive. Détection du drift local via `body_hash` stocké en état → log warning `event=gdrive_sync.local_drift_overwritten` puis overwrite.
