# OBS-003 — Persistence Postgres des runs Ragas live (table eval_runs + flag --persist)

## Contexte

EVAL-001 a livré le runner Ragas (`eval/ragas_runner.py`) qui produit un artefact JSON par run mais ne persiste rien : aucune trace durable des métriques de qualité (faithfulness, answer_relevancy, context_precision, context_recall) interrogeable par la gateway. OBS-004 veut afficher la qualité IA « live » sur `/observability`, et a besoin d'une source SQL lisant le dernier run live — ce ticket débloque OBS-004. OBS-003 ajoute une table `eval_runs` (migration humaine) et un flag `--persist` (défaut off) au runner qui, en `--mode live` uniquement, écrit une ligne agrégée par run dans Postgres via requêtes paramétrées.

## Acceptance criteria

- AC-1 : La migration `migrations/0005_eval_runs.sql` crée la table `eval_runs` avec exactement les colonnes contractuelles : `id UUID PRIMARY KEY`, `git_sha TEXT NOT NULL`, `runner_mode TEXT NOT NULL CHECK (runner_mode = 'live')`, `golden_set_version TEXT NOT NULL`, `faithfulness NUMERIC(5,4) NOT NULL`, `answer_relevancy NUMERIC(5,4) NOT NULL`, `context_precision NUMERIC(5,4) NOT NULL`, `context_recall NUMERIC(5,4) NOT NULL`, `entries_total INT NOT NULL`, `entries_ok INT NOT NULL`, `entries_errors INT NOT NULL`, `started_at TIMESTAMPTZ NOT NULL`, `finished_at TIMESTAMPTZ NOT NULL`, `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()` (contrat de schéma figé, décision D-1 ; la migration est humain-only, cf Pre-conditions).
- AC-2 : La migration crée un index partiel `eval_runs_live_latest_idx ON eval_runs (finished_at DESC) WHERE runner_mode = 'live'`, shapé pour la lecture OBS-004 `SELECT ... WHERE runner_mode='live' ORDER BY finished_at DESC LIMIT 1` (décision D-1, OBS-004 read path).
- AC-3 : Le runner expose un flag `--persist` (booléen, store_true) ; **sans** ce flag, le runner ne touche jamais Postgres (`SELECT count(*) FROM eval_runs` inchangé avant/après un run sans `--persist`) et son comportement reste byte-identique à EVAL-001 (décision D-2, défaut off).
- AC-4 : Avec `--persist` ET `--mode live`, à la fin d'un run réussi, le runner insère **exactement une** ligne dans `eval_runs` dont les colonnes métriques (`faithfulness`, `answer_relevancy`, `context_precision`, `context_recall`) sont égales aux valeurs `metrics.*` du fichier de run EVAL-001 (AC-5), et `git_sha`/`started_at`/`finished_at`/`entries_total`/`entries_ok`/`entries_errors` égales aux champs homologues du run-file (`git_sha`, `started_at`, `finished_at`, `totals.entries`, `totals.ok`, `totals.errors`).
- AC-5 : Avec `--persist` ET `--mode offline`, le runner **n'insère aucune** ligne dans `eval_runs` (les métriques offline sont structurelles/nulles, non comparables ; non-persistence offline contractuelle, décision D-3) ; le runner émet un log `event=persist_skipped reason=offline_mode` et termine son exit code normalement (le skip de persist ne change jamais le verdict des gates EVAL-001).
- AC-6 : Le runner dérive `golden_set_version` = hash hexadécimal du contenu d'octets de `specs/golden_qa.jsonl` (SHA-256, lecture binaire du fichier) et l'écrit dans la colonne `golden_set_version` de la ligne insérée (AC-4) ; deux runs sur un `golden_qa.jsonl` byte-identique produisent le même `golden_set_version`, deux contenus distincts produisent des valeurs distinctes (décision D-4, OBS-004 compare uniquement des scores de même `golden_set_version`).
- AC-7 : La persistence est **append-only** : chaque persist insère une ligne neuve avec son propre `id` (UUID) et son propre `created_at` ; aucun `UPDATE`, aucun `UPSERT`, aucune déduplication ; deux persists consécutifs produisent deux lignes distinctes (`count(*)` augmente de 2) (décision D-5, append-only).
- AC-8 : L'INSERT est exécuté via requête **paramétrée** (placeholders liés, jamais de `f-string`/`%`/`format` du SQL avec des valeurs) ; aucune valeur n'est interpolée dans la chaîne SQL (security §A03 ; vérifiable par grep statique : le littéral SQL ne contient aucune f-string ni `.format(`).
- AC-9 : Aucune donnée de payload LLM (texte de réponse `answer`, `retrieved_contexts`, `citations`, question) n'est écrite dans `eval_runs` — seules les métriques agrégées et métadonnées de run le sont (décision D-6 ; cohérent EVAL-001 AC-5 qui exclut déjà les `answer` complets du baseline ; security §A09).
- AC-10 : `DATABASE_URL` reste lu depuis l'environnement et n'apparaît jamais dans le fichier de run, stdout, stderr, ni dans un message de log — y compris sur le chemin d'échec de persist (AC-11) ; aucune sous-chaîne de `DATABASE_URL` n'est sérialisée (EVAL-001 AC-16 étendu au write path).
- AC-11 : Si l'INSERT Postgres échoue (DB injoignable, contrainte violée, timeout), le runner **n'écrit ni ne corrompt** le fichier de run JSON (l'artefact EVAL-001 est produit et complet quel que soit le résultat de la persist), émet un log `warn event=persist_failed` **sans** détail de connexion brut ni `DATABASE_URL`, et termine avec exit code `4` (décision D-7, échec persist = exit non-zéro dédié, distinct des codes EVAL-001 1/2/3) ; aucun secret ni stack DB brute n'est exposé.
- AC-12 : L'écriture du fichier de run JSON (artefact EVAL-001) précède la tentative de persist : à l'instant où l'INSERT est tenté, le fichier `eval/runs/<timestamp>.json` est déjà écrit et fermé sur disque, garantissant qu'un échec de persist (AC-11) laisse un artefact intact et inspectable (décision D-7, ordre artefact-puis-persist).
- AC-13 : `specs/openapi/gateway-to-workers.yml` n'est **pas** modifié par ce ticket : la persist est un write direct runner→Postgres (le runner détient déjà `DATABASE_URL`), aucune route gateway/workers nouvelle, aucun changement de contrat REST (décision D-2).
- AC-14 : Test manuel auteur (procédure documentée `## Test manuel auteur`) coché par l'humain avant `Status: ready`.

## Non-goals

- Pas de lignes par-question (per-entry) dans `eval_runs` — une seule ligne agrégée par run ; le détail par question relève d'un ticket EVAL-* aval (décision D-3).
- Pas de persistence des runs `--mode offline` — métriques nulles/structurelles non persistées, colonnes métriques `NOT NULL` (décision D-3 ; non-persistence offline = AC-5).
- Pas de déduplication / upsert / idempotency key — append-only, chaque persist = une ligne neuve (décision D-5).
- Pas de panel qualité IA ni de lecture du dernier run — OBS-004 consomme `eval_runs` (ce ticket l'alimente seulement).
- Pas de nouvelle route gateway ni endpoint workers ni changement OpenAPI — write direct runner→Postgres (décision D-2).
- Pas de stockage du payload LLM (réponses, contextes, citations) — métriques agrégées uniquement (AC-9).
- Pas de migration `down` requise au-delà de la convention projet (le fichier `0005` suit le style des migrations existantes ; OBS-003 n'autorise pas l'agent à écrire le SQL — humain-only).
- Pas de rétention / purge / archivage des lignes `eval_runs` phase 1 — table append-only sans TTL (ticket futur si volume).
- Pas de breakdown_by_mode persisté ni de keyword_overlap_rate / context_recall_structural en DB — uniquement les 4 métriques Ragas racine (canon) du run-file (AC-4).
- Pas de changement du schéma du fichier de run EVAL-001 ni de `eval/baseline.json` (lecture seule des champs existants).
- Pas de persist en CI sur PR — le workflow PR EVAL-001 reste `--mode offline` (non persisté, AC-5) ; la persist est déclenchée manuellement / `workflow_dispatch` live avec `--persist` (décision D-2).

## Pre-conditions

- **EVAL-001 mergé** : `eval/ragas_runner.py` opérationnel, produit `eval/runs/<timestamp>.json` au schéma AC-5 (source des champs persistés AC-4), lit `DATABASE_URL`/`WORKERS_URL`/`LLM_API_KEY` depuis l'environnement (AC-16).
- **FOUND-003 / Postgres opérationnel** : instance Postgres 16 + extension `vector` joignable via `DATABASE_URL` (cible de l'INSERT).
- **`migrations/0005_eval_runs.sql` est humain-only** : `migrations/*.sql` est source de vérité (CLAUDE.md §Sources de vérité) — l'agent **ne rédige pas** le SQL ; il propose le contenu, l'humain l'écrit/valide avant impl. L'AC-1/AC-2 nomme le contrat (colonnes + index), pas l'implémentation SQL.
- `cargo sqlx prepare` n'est pas concerné (write côté Python runner, pas côté gateway Rust) ; côté Rust, la lecture `eval_runs` est introduite par OBS-004, pas ici.
- Driver Postgres Python disponible dans l'environnement du runner (le runner détient déjà `DATABASE_URL` ; si une dépendance driver manque → blocker `no-workaround.md`, ADR si lourde).

## Failure modes

- INSERT Postgres en échec (DB down, contrainte, timeout) → fichier de run JSON intact et complet (AC-12), log `warn event=persist_failed` sans `DATABASE_URL` ni stack DB brute (AC-10, AC-11), exit code `4`.
- `--persist` fourni en `--mode offline` → aucun INSERT, log `event=persist_skipped reason=offline_mode`, exit code inchangé (gouverné par les gates EVAL-001), aucune ligne `eval_runs` créée (AC-5).
- `--persist` absent → aucun accès Postgres, comportement EVAL-001 byte-identique (AC-3).
- `specs/golden_qa.jsonl` illisible au moment de dériver `golden_set_version` → le runner échoue avant l'INSERT (pas de ligne partielle), log explicite ; l'artefact de run reste celui produit par EVAL-001 (cf AC-12 — la dérivation du hash se fait dans l'étape persist, après écriture de l'artefact).
- Métriques `metrics.*` du run-file absentes/null en `--mode live` (cas anormal) → le runner n'insère pas une ligne avec NULL (colonnes `NOT NULL`) ; il loggue `persist_failed` et exit `4` plutôt que de violer la contrainte (cohérent `no-workaround.md`).

- `eval/ragas_runner.py` — ajout du flag `--persist` (AC-3), branche persist live-only après écriture de l'artefact (AC-12), dérivation `golden_set_version` SHA-256 (AC-6), mapping run-file → colonnes (AC-4), gestion d'échec exit `4` (AC-11).
- `eval/persist.py` (ou module dédié) — **nouveau** : fonction d'INSERT paramétré dans `eval_runs` (AC-8), pas de payload LLM (AC-9), redaction `DATABASE_URL` (AC-10).
- `eval/runs/` — artefact JSON écrit avant la persist (AC-12), inchangé en schéma.
- `eval/tests/` (ou `workers/tests/` selon emplacement runner) — tests d'intégration/unit persist (AC-3..AC-12).
- `.github/workflows/eval.yml` — **inchangé** : PR reste `--mode offline` non persisté ; `--persist` n'est jamais ajouté au job PR (décision D-2). (Câblage live `workflow_dispatch` + `--persist` : option humaine, pas requis par OBS-003.)

## Test manuel auteur

Procédure exécutée par l'humain avant `Status: ready` (AC-14).

1. **Migration** : appliquer `migrations/0005_eval_runs.sql` (humain-validé) sur la stack locale → `\d eval_runs` affiche les colonnes AC-1 + `\di eval_runs_live_latest_idx` présent.
2. **Sans persist** : `uv run python eval/ragas_runner.py --mode offline` (sans `--persist`) → `SELECT count(*) FROM eval_runs;` inchangé (0), sortie identique à EVAL-001.
3. **Offline + persist** : `... --mode offline --persist` → `count(*)` toujours 0, log `event=persist_skipped reason=offline_mode` visible (AC-5).
4. **Live + persist (nominal)** : `... --mode live --persist --baseline eval/baseline.json` (workers up) → run termine ; `SELECT * FROM eval_runs ORDER BY finished_at DESC LIMIT 1;` → 1 ligne, métriques == `metrics.*` du `eval/runs/<ts>.json` correspondant, `golden_set_version` non vide, `runner_mode='live'` (AC-4, AC-6).
5. **Append-only** : relancer l'étape 4 → 2e ligne, `id` distinct, `count(*)` = 2 (AC-7).
6. **golden_set_version** : `sha256sum specs/golden_qa.jsonl` → comparer à la valeur `golden_set_version` de la ligne insérée → égalité (AC-6).
7. **Échec persist** : `docker compose stop postgres`, relancer `... --mode live --persist` → exit code `4`, `eval/runs/<ts>.json` présent et JSON-valide, log `warn event=persist_failed` SANS `DATABASE_URL` ni stack SQL brute (AC-11, AC-12, AC-10).
8. **Read path OBS-004** : `EXPLAIN SELECT faithfulness, answer_relevancy, finished_at, golden_set_version FROM eval_runs WHERE runner_mode='live' ORDER BY finished_at DESC LIMIT 1;` → index partiel utilisé / plan attendu (AC-2).

## Performance / SLO

- INSERT `eval_runs` : 1 ligne par run, opération O(1), < 50 ms p95 (write trivial, hors chemin chaud) ; négligeable vs durée d'un run live Ragas (dominée par les appels LLM, EVAL-001 §Performance).
- Lecture OBS-004 (latest live) : servie par l'index partiel `eval_runs_live_latest_idx` (AC-2), < 10 ms attendu — dimensionnement assuré ici, exécution mesurée dans OBS-004.

## Security / trust boundary

- Frontière : Runner (process opérateur / CI manuel) → Postgres. Pas de surface Internet ajoutée (aucune route publique nouvelle).
- **A03 Injection** : INSERT via requête paramétrée uniquement, placeholders liés ; aucune interpolation de valeur dans le SQL (AC-8, `security.md` §A03 « pas de format! / f-string into SQL »).
- **A09 Logging & Monitoring** : `DATABASE_URL` lu en `pydantic.SecretStr` (ou équivalent redacted), jamais sérialisé dans run-file/stdout/stderr/logs, y compris sur `persist_failed` (AC-10, AC-11) ; aucune réponse LLM brute ni contexte stockés (AC-9).
- **A01 / données** : `eval_runs` ne contient aucune donnée per-utilisateur, aucun contenu de conversation, aucun `gcs_uri` — uniquement des métriques numériques agrégées et métadonnées de run (cohérent EVAL-001 AC-5 / baseline non-sensible).
- `migrations/0005_eval_runs.sql` humain-only : pas d'écriture agent sur `migrations/*` sans approbation (CLAUDE.md §Garde-fous, `secret-hygiene` n.a. — pas de secret dans la migration).
- Pas de fallback default sur `DATABASE_URL` (`os.getenv("DATABASE_URL", "default")` interdit, `secret-hygiene.md` §Production).

## Observability

- Log runner JSON structuré sur le chemin persist : `event=persist_ok` (champs `run_id`/`id` inséré, `golden_set_version`, `finished_at`) en succès ; `event=persist_skipped reason=offline_mode` en offline ; `event=persist_failed` (sans détail connexion brut) en échec (AC-5, AC-11).
- Aucune métrique OpenTelemetry nouvelle phase 1 — la table `eval_runs` + l'artefact de run sont la source de vérité.
- Pas de tracing Langfuse (hors pipeline LLM).

## Décisions

Arbitrages humains figés (round-1). Chaque décision verrouille une dimension du contrat.

- **D-1 — Schéma `eval_runs` per-run, métriques `NOT NULL`, index partiel live-latest.** Une ligne agrégée par run ; colonnes mirroring le run-file EVAL-001 ; `runner_mode TEXT CHECK = 'live'` (garde-fou conservé bien que live-only) ; métriques `NUMERIC(5,4) NOT NULL` (possible car live-only, D-3) ; index partiel `(finished_at DESC) WHERE runner_mode='live'` pour la lecture OBS-004 (AC-1, AC-2).
- **D-2 — Write direct runner→Postgres, pas de nouvel endpoint.** Le runner détient déjà `DATABASE_URL` ; aucune route gateway/workers, aucun changement `gateway-to-workers.yml`. Flag `--persist` défaut **off** : impact nul tant que non activé (AC-3, AC-13).
- **D-3 — Live-only, offline NON persisté.** Seuls les runs `--mode live` persistent ; les runs offline (métriques nulles/structurelles) ne sont jamais écrits (AC-5). Permet `NOT NULL` sur les métriques et garantit que le panel OBS-004 lit toujours des données réelles. Per-question rows hors périmètre (ticket EVAL-* aval).
- **D-4 — `golden_set_version` = SHA-256 du contenu d'octets de `specs/golden_qa.jsonl`.** Champ neuf introduit par OBS-003, dérivé dans l'étape persist du runner ; permet à OBS-004 de ne comparer que des scores de même version de golden set (AC-6).
- **D-5 — Append-only, pas de dedup/upsert.** Chaque persist insère une ligne neuve (`id` + `created_at` propres) ; OBS-004 lit le dernier run live via `finished_at DESC LIMIT 1` (AC-7).
- **D-6 — Pas de payload LLM en DB.** Seules métriques agrégées + métadonnées de run sont persistées ; aucune `answer`/`retrieved_contexts`/`citations`/`question` (AC-9, cohérent EVAL-001 AC-5 §A09).
- **D-7 — Échec persist : artefact-puis-persist + exit code `4`.** L'artefact JSON est écrit et fermé avant l'INSERT ; un échec persist laisse l'artefact intact (jamais corrompu), loggue `persist_failed` sans secret, et termine exit `4` (code dédié distinct des codes EVAL-001 1/2/3) (AC-11, AC-12).

## Open questions

- **OQ-1 — Driver Postgres Python du runner : déjà présent ou nouvelle dépendance ?** L'INSERT paramétré (AC-8) suppose un driver (p. ex. `psycopg`/`asyncpg`/`sqlalchemy`) dans l'environnement du runner. EVAL-001 lit `DATABASE_URL` mais ne l'utilise pas forcément pour écrire — à confirmer : (a) un driver est déjà installé/utilisé côté `eval/` ou `workers/`, OQ purement informative ; (b) sinon, l'ajout d'une dépendance peut requérir un ADR si « lourde » (CLAUDE.md §Garde-fous). Reco : réutiliser le driver déjà présent dans `workers/` (pgvector implique un driver Postgres Python en place) ; flaguer un ADR seulement si un nouveau driver distinct est introduit. À confirmer par l'humain / l'architect au pre-flight `/plan`.
- **OQ-2 — Code exit `4` : libre de collision avec EVAL-001 ?** EVAL-001 utilise exit codes `1` (gates/erreur-rate), `2` (schéma/flag), `3` (workers unreachable). OBS-003 réserve `4` pour échec persist (AC-11). Reco : `4` est libre et sémantiquement distinct — confirmer qu'aucun chemin EVAL-001 aval ne l'a réquisitionné depuis. Faible enjeu, purement allocation de code de sortie.
