# OPS-002 — Bump `jsonwebtoken` 9 → 10 (crypto backend `aws_lc_rs`)

## Contexte

`gateway/Cargo.toml:25` pin `jsonwebtoken = "9"`. La v10.0.0 introduit un breaking change : le choix d'un backend crypto explicite (`aws_lc_rs` ou `rust_crypto`) devient obligatoire via feature, et la v9 par défaut tirait `ring` transitivement. Dependabot a ouvert la PR #46 (`dependabot/cargo/gateway/jsonwebtoken-10.4.0`) ; ce ticket trace l'arbitrage humain pour la merger : choix du backend, validation que l'API utilisée (`verify` côté src, helpers `encode`/`Header::new` côté tests) compile et passe, et amendement minimal ADR-0002. Effet de bord souhaité : `jsonwebtoken` ne tire plus `ring` transitivement, ce qui débloque le merge de SEC-001 PR-b sans tirer une seconde chaîne crypto.

## Critères d'acceptation

- AC-1 : `gateway/Cargo.toml:25` déclare `jsonwebtoken = { version = "10", default-features = false, features = ["aws_lc_rs"] }`. La feature `rust_crypto` n'est PAS activée. Aucune autre feature `jsonwebtoken` n'est ajoutée par ce ticket. Le pin majeur est `"10"` (caret accepté, pas de pin `"=10.4.0"`).
- AC-2 : `cargo tree -p jsonwebtoken -e normal --target all` exécuté à la racine `gateway/` ne liste **aucune** occurrence de `ring` dans le sous-arbre de `jsonwebtoken`. La sortie remonte uniquement à `aws-lc-rs` (et son chaînage `aws-lc-sys`).
- AC-3 : `cd gateway && cargo build` réussit sans erreur de compilation sur `gateway/src/auth/jwt.rs` ni `gateway/tests/common/jwt_helpers.rs`. Les symboles publics de `jsonwebtoken` utilisés (`decode_header`, `DecodingKey::from_ed_pem`, `Validation::new`, `decode`, `EncodingKey::from_ed_pem`, `Header::new`, `encode`, `Algorithm::EdDSA`) résolvent contre l'API v10.
- AC-4 : `cd gateway && cargo test` exit 0 sur l'ensemble de la suite (intégration + unitaires). Les tests existants `auth_*` couvrant `verify()` (round-trip sign/verify, alg confusion, exp expirée, iss/aud mismatch) passent sans modification de leur logique d'assertion.
- AC-5 : `cd gateway && cargo clippy -- -D warnings` exit 0. Aucun nouveau lint introduit par le bump (ex : déprécation d'un symbole v9 utilisé).
- AC-6 : `docs/adr/0002-security-baseline-crates.md` est amendé inline dans la même PR : (a) ligne `jsonwebtoken` du tableau "Rust gateway" passe de `9+` à `10+ (feature aws_lc_rs)` avec rationale colonne mise à jour (`Pin alg via Algorithm enum, reject None. Backend crypto = aws_lc_rs (drops ring transitive dep).`) ; (b) ajout d'une entrée `## Forbidden` : `jsonwebtoken feature "rust_crypto" — utiliser aws_lc_rs (cohérence baseline crypto gateway).` ; (c) `Status` passe à `accepted (amended 2026-05-25)` et une ligne `Amendments:` listant la date + résumé `jsonwebtoken 9 → 10, backend aws_lc_rs` est ajoutée.

## Non-goals

- Pas de migration TLS rustls provider (`sqlx`, `reqwest`) sur `aws_lc_rs`. Suivi dans un ticket V2 dédié.
- Pas de retrait de la dépendance directe `ring = "0.17"` (`gateway/Cargo.toml:29`), confinée à `gateway/src/gcs/sign.rs`. Suivi dans un follow-up nommé `SEC-XXX — retirer ring de gcs::sign` (ID alloué hors OPS-002).
- Pas de modification de `gateway/src/` au-delà de ce qu'exige strictement la compilation contre l'API `jsonwebtoken` v10. Si v10 conserve la signature des symboles listés en AC-3, le diff `src/` est **0 LOC**. Si v10 casse un symbole, patch minimal autorisé (cf. note ci-dessous).
- Pas de modification des `dev-dependencies` (`ed25519-dalek`, `rsa`, `pkcs8`, etc.). Ces crates n'interagissent avec `jsonwebtoken` que via le format PEM produit/consommé.
- Pas d'install global de `CryptoProvider` rustls au boot. Hors scope (concerne la migration TLS, pas le bump JWT).
- Pas d'ajustement `deny.toml` (sauf cas allowlist licences `aws-lc-sys` si déjà bloquée — à valider au `/plan`, traité comme micro-ajustement borné).
- Pas de nouveau ADR. Amendement inline ADR-0002 uniquement (AC-6).
- Pas de mesure perf / régression overhead (les SLOs OPS-001 captent toute régression au prochain run).
- Pas de bascule des workers Python — `jsonwebtoken` est une crate Rust, Python utilise `python-jose`.

**Note in-scope** : si l'API v10 introduit un breaking sur un des symboles listés en AC-3 (ex : `Validation::new` change de signature, `Algorithm::EdDSA` renommé), un patch minimal côté `gateway/src/auth/jwt.rs` et/ou `gateway/tests/common/jwt_helpers.rs` est autorisé et reste in-scope. Le critère d'évaluation reste : « le ticket fait passer la suite de tests avec la v10 ». La migration TLS rustls reste exclue.

## Pre-conditions

- SEC-001 PR-a mergé : `gateway/src/auth/jwt.rs` ship `verify()` et est exercé par les tests d'intégration auth. Le bump prend effet sur du code shipped.
- SEC-001 PR-b en attente de merge : ce ticket est l'unblocker. Sans le bump, PR-b introduit `sign()` côté gateway et la dépendance transitive `ring` via `jsonwebtoken` v9 reste un signal faible sur la baseline crypto.
- Dependabot PR #46 (`dependabot/cargo/gateway/jsonwebtoken-10.4.0`) ouverte sur le repo et prête à recevoir des commits manuels (activation feature + amendement ADR par-dessus).
- `cargo deny check` opérationnel en CI (config existante). AC complémentaire `licenses` non-régression couverte implicitement par le gate CI standard.
- Aucun autre ticket en cours ne modifie `gateway/Cargo.toml` ligne 25 ni ADR-0002 (sinon merge conflict prévisible).

## Failure modes

- Une dépendance transitive de `jsonwebtoken` v10 + feature `aws_lc_rs` tire encore `ring` via un autre chemin (ex : un wrapper crypto interne) → AC-2 échoue → **stop** (rule `no-workaround.md`), log `docs/blockers.md`, ne PAS activer `rust_crypto` en mix.
- L'API v10 renomme/supprime un symbole utilisé (`Algorithm::EdDSA`, `DecodingKey::from_ed_pem`, `Validation::new`, etc.) → `cargo build` casse → patch minimal côté `src/auth/jwt.rs` + `tests/common/jwt_helpers.rs` autorisé (cf. note Non-goals). Si le patch dépasse ~20 LOC ou exige une refonte de l'extracteur auth → **stop**, log blocker, escalade humain (peut justifier un split ticket).
- `aws-lc-sys` (chaîné par `aws-lc-rs`) requiert `cmake` + `clang` sur le runner CI → build CI casse → **stop**, log blocker, escalade humain (option : ajustement image CI vs revert au pin v9 temporaire).
- `cargo deny licenses` casse car `aws-lc-sys` introduit une licence non-allowlistée → ajustement `deny.toml` allowlist documenté dans la PR avec justification écrite. Si l'allowlist nécessite l'ajout d'une licence non standard (ex : `OpenSSL`), validation humaine explicite avant merge.
- `cargo deny advisories` ouvre une CVE active sur `aws-lc-rs` / `aws-lc-sys` au moment du bump → **stop**, ne pas merger une crypto en CVE active, reporter le ticket jusqu'à patch upstream.
- Test flake sur `auth_*` post-bump (ex : sérialisation JSON des claims subtly différente) → **pas** de désactivation, identifier la cause (rule CLAUDE.md), corriger ou escalader.
- `cargo clippy` introduit un warning `deprecated` sur un symbole v10 utilisé → AC-5 échoue → patch minimal (renommage d'appel) in-scope.

## Performance / SLO

- Pas de SLO dédié OPS-002. Hypothèse : le bump n'introduit pas de régression mesurable sur l'overhead Rust gateway (vision.md §92, cible `< 80 ms p95 à 500 users`). Toute régression sera captée au prochain run OPS-001. Aucune mesure pré/post n'est exigée par ce ticket.

## Security / trust boundary

- Frontière inchangée. La gateway expose les mêmes routes, le même contrat JWT (`alg=EdDSA`, `iss=aud=archiviste-gateway`, leeway 60 s, `kid` requis). Le backend crypto remplacé impacte uniquement l'implémentation interne de la primitive Ed25519 sign/verify, pas la policy.
- Conforme `security.md` §A02 : algorithme pinned `EdDSA`, rejet de `alg=none` et alg hors allowlist préservés (cf. `gateway/src/auth/jwt.rs:69`).
- Conforme `secret-hygiene.md` : aucun secret commité, aucune clé manipulée par ce ticket.
- Aucun nouveau secret introduit. Aucune surface réseau étendue. Aucun handler modifié.
- Effet positif side : `jsonwebtoken` ne tire plus `ring` transitivement, réduit la surface d'audit crypto baseline gateway. `ring` reste autorisé en dépendance directe uniquement pour `gcs::sign` (suivi follow-up `SEC-XXX`).

## Decisions (Round 1, 2026-05-25)

- **D-1 (Q1) — Backend crypto `jsonwebtoken` = `aws_lc_rs`**. Maintenu par AWS, FIPS-ready, devenu défaut amont. Cohérent avec trajectoire baseline crypto gateway (évite `rust_crypto` qui mixerait des primitives pure-Rust de maturités hétérogènes). ADR-0002 amendé inline (AC-6).
- **D-2 (Q2) — Migration TLS rustls (`sqlx`, `reqwest`) HORS scope OPS-002**. Confinée à un ticket V2 séparé. Listée en Non-goal explicite pour cadrer la portée du PR Dependabot #46.
- **D-3 (Q3) — Retrait `ring` direct de `gcs::sign` HORS scope OPS-002**. Confiné à follow-up nommé `SEC-XXX — retirer ring de gcs::sign`. Listé en Non-goal et tracé dans l'amendement ADR-0002.
- **D-4 (Q4) — Aucun ajout côté `gateway/src/` au-delà du minimum requis par l'API v10**. Hypothèse de travail : 0 LOC source (l'API publique `decode_header`, `Validation::new`, `Algorithm::EdDSA`, `decode`, `encode`, `Header::new`, `DecodingKey::from_ed_pem`, `EncodingKey::from_ed_pem` est stable v9 → v10). Si breaking observé au build, patch minimal in-scope (cf. note Non-goals).
- **D-5 (Q5) — Oracles AC = CLI checks reproductibles** : `Cargo.toml` pin + feature, `cargo tree -p jsonwebtoken` exclude `ring`, `cargo build` + `cargo test` + `cargo clippy` clean, ADR-0002 amendé inline.
- **D-6 (Round 2) — OQ-1/2/3/5 déférées au /plan (résolution technique), OQ-4 placeholder SEC-XXX accepté.**
