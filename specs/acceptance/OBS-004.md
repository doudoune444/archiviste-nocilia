# OBS-004 — Widget qualité IA live + GET /v1/quality (dernier run Ragas eval_runs)

## Contexte

La vision EPIC OBS ouvre une surface d'observabilité publique pour la vitrine portfolio ; OBS-001 a livré la page `/observability` + widget usage (`GET /v1/stats`), OBS-002 le widget santé (`GET /v1/status`), et OBS-003 la table `eval_runs` qui persiste les métriques de qualité des runs Ragas live. Cette page ne montre pas encore la **qualité IA** du RAG : un visiteur ne voit pas si les réponses sont fidèles et pertinentes. Ce ticket livre la dernière slice OBS de la page : un widget qualité alimenté par un nouvel endpoint public anonyme `GET /v1/quality` qui lit le **dernier run live** de `eval_runs` (4 métriques Ragas + `golden_set_version`) et l'expose en JSON sanitisé.

## Acceptance criteria

- AC-1 : `GET /v1/quality` retourne `200 OK`, `Content-Type: application/json; charset=utf-8`, body JSON plat `{"faithfulness":<number>,"answer_relevancy":<number>,"context_precision":<number>,"context_recall":<number>,"golden_set_version":<string>,"finished_at":"<rfc3339>"}` quand au moins une ligne `runner_mode='live'` existe dans `eval_runs` ; les 6 clés sont **toujours** présentes dans ce cas et **aucune autre clé** n'est présente (décision D-1).
- AC-2 : La ligne servie par `GET /v1/quality` est le **dernier** run live, c.-à-d. `SELECT faithfulness, answer_relevancy, context_precision, context_recall, golden_set_version, finished_at FROM eval_runs WHERE runner_mode='live' ORDER BY finished_at DESC LIMIT 1` ; pour deux runs live de `finished_at` distincts `t1 < t2`, la réponse expose les valeurs de la ligne `t2` (propriété : la ligne servie maximise `finished_at` sur le sous-ensemble live).
- AC-3 : Les quatre champs métriques (`faithfulness`, `answer_relevancy`, `context_precision`, `context_recall`) sont sérialisés comme **nombres JSON** égaux à la valeur `NUMERIC(5,4)` stockée, **sans arrondi ni reformatage** côté gateway (décision D-5) ; pour une ligne de valeur stockée `0.9234`, le body expose `0.9234` (égalité numérique stricte avec la valeur DB).
- AC-4 : Quand la table `eval_runs` ne contient **aucune** ligne `runner_mode='live'` (table vide ou uniquement des lignes non-live — cas impossible vu le `CHECK` OBS-003 mais traité défensivement), `GET /v1/quality` retourne `200 OK`, body exactement `{"status":"no_data"}` (aucune clé métrique, aucun `finished_at`) ; ce cas n'est **ni** `404` **ni** `503` (décision D-2).
- AC-5 : `GET /v1/quality` est servi sans authentification (anonyme) ; il ne porte aucun guard `author_only`/`member_only` et n'apparaît pas dans le sous-router auth. Un test contractuel (inspection router) vérifie l'absence de guard d'auth sur ce handler (cohérent OBS-001 AC-6, OBS-002 AC-7).
- AC-6 : `finished_at` est sérialisé en RFC3339/ISO-8601 UTC, correspondant à la colonne `finished_at TIMESTAMPTZ` de la ligne servie (AC-2) ; il parse en RFC3339 valide.
- AC-7 : Quand Postgres est indisponible (pool `None` ou requête en échec) au moment de `GET /v1/quality`, la réponse est `503 Service Unavailable` body **sanitisé** byte-for-byte identique à OBS-001/OBS-002 : `{"error":"upstream_unavailable","request_id":"<uuid>"}` via `ApiError::UpstreamUnavailable` ; aucun détail interne (message SQL, host, nom de table, stack) n'apparaît dans le body (décision D-3, `security.md` §A05). Le chemin DB-down (`503`) et le chemin table-vide (`200 {"status":"no_data"}`, AC-4) sont **distincts et non confondus**.
- AC-8 : Au chargement de `/observability`, `observability.js` appelle `GET /v1/quality` (même-origine, `fetch` `credentials:'same-origin'`) et insère dans un point d'ancrage stable `<section id="quality-widget">` un rendu en `textContent` (jamais `innerHTML`, CSP-safe) affichant les **4 métriques** comme valeurs libellées + une ligne de fraîcheur dérivée de `finished_at` ; tout style passe par `observability.css` (aucun `style=` inline). Le libellé exact et le formatage JS des nombres ne sont **pas** figés byte-for-byte (souplesse implémentation, décision D-5) ; seuls l'usage `textContent` et la présence des 4 métriques sont contractuels.
- AC-9 : Le fetch `GET /v1/quality` est **indépendant** des fetchs OBS-001 (`/v1/stats`) et OBS-002 (`/v1/status`) : un échec ou un `no_data` de `/v1/quality` ne blanchit ni n'altère le rendu de `#usage-widget` ni de `#health-widget` (et réciproquement). Vérifiable : stub `/v1/quality` en échec → `#health-widget` et `#usage-widget` restent rendus normalement.
- AC-10 : Quand `GET /v1/quality` répond `200 {"status":"no_data"}` (AC-4), `observability.js` insère dans `#quality-widget` le message littéral `Aucune évaluation disponible.` (rendu `textContent`, décision D-2) ; aucune métrique n'est affichée.
- AC-11 : Pour toute issue de `GET /v1/quality` autre que `200 OK` (4xx, 5xx, network drop, timeout, JSON malformé), `observability.js` insère dans `#quality-widget` un message littéral d'indisponibilité (aligné sur le ton OBS-001 AC-10 `Données indisponibles.`) ; aucun branchement fin selon le code d'erreur. Ce message est distinct du message `no_data` (AC-10).
- AC-12 : Les 5 en-têtes de sécurité globaux posés router-wide (UI-001 AC-6 / SEC-003 AC-1 : `Content-Security-Policy`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, `X-Frame-Options: DENY`, `Strict-Transport-Security`) sont présents byte-for-byte sur `GET /v1/quality`. La valeur CSP n'est **pas** modifiée par ce ticket.
- AC-13 : `gateway/static/observability.html` après ce ticket ne contient toujours aucune balise `<script>`/`<style>` inline, aucun attribut `style="..."` ni gestionnaire `on*=` inline (cohérent OBS-001 AC-12, validable par grep statique) ; l'ajout de `<section id="quality-widget">` n'introduit aucun inline.
- AC-14 : `specs/openapi/gateway-to-workers.yml` n'est **pas** modifié par ce ticket : `GET /v1/quality` est un endpoint client↔gateway (lecture directe Postgres côté gateway, comme `/v1/stats`), aucune route workers nouvelle, aucun appel gateway→workers (décision D-1, cohérent OBS-001 AC-14).
- AC-15 : Test manuel auteur (procédure documentée `## Test manuel auteur`) coché par l'humain avant `Status: ready`.

## Non-goals

- Pas de modification de la table `eval_runs` ni de la migration `0005_eval_runs.sql` — lecture seule du dernier run live (OBS-003 alimente, OBS-004 consomme).
- Pas de réintroduction du write path Ragas (`--persist`) ni du runner — OBS-003.
- Pas d'historique / série temporelle / graphe d'évolution des métriques — affichage du **dernier** run live uniquement (un scalaire par métrique), pas de courbe.
- Pas d'exposition des métadonnées de run autres que les 6 champs contractuels (pas de `git_sha`, `started_at`, `entries_total`, `entries_ok`, `entries_errors`, `id`, `created_at` dans le body) — surface minimale.
- Pas d'arrondi / bucketisation / reformatage côté API — valeur `NUMERIC(5,4)` brute en nombre JSON (décision D-5 ; le formatage d'affichage est fait en JS, non contractuel).
- Pas de comparaison de versions de `golden_set_version` ni de filtrage par version côté gateway — la version est exposée telle quelle, l'interprétation est hors périmètre phase 1.
- Pas de cache HTTP applicatif ni cache gateway sur `GET /v1/quality` phase 1 — lecture indexée triviale à chaque appel (l'index partiel `eval_runs_live_latest_idx` OBS-003 AC-2 sert la requête).
- Pas de rafraîchissement auto / polling / SSE — `GET /v1/quality` appelé une fois au chargement (cohérent OBS-001/OBS-002).
- Pas de modification des widgets usage (`#usage-widget`, OBS-001) ni santé (`#health-widget`, OBS-002) ni de leurs endpoints `/v1/stats` / `/v1/status`.
- Pas de framework JS, pas de build step (ADR-0005 réaffirmé, cohérent OBS-001/OBS-002).
- Pas de rate-limit applicatif spécifique à `GET /v1/quality` (couverture Cloudflare front V1 + SEC-002 V2, cohérent OBS-001/OBS-002).
- Pas d'observabilité infra backend / alerting sur dégradation de qualité — OBS-005 / ticket futur.

## Failure modes

- Postgres indisponible (pool `None` ou requête en échec) au moment de `GET /v1/quality` → `503` body sanitisé `{"error":"upstream_unavailable","request_id":"<uuid>"}` via `ApiError::UpstreamUnavailable`, log `warn` ; pas de fallback ni de valeur par défaut (AC-7, cohérent `no-workaround.md` + `security.md` §A05).
- Table `eval_runs` sans ligne live (`LIMIT 1` retourne 0 ligne) → `200 OK` body `{"status":"no_data"}` (AC-4) ; `observability.js` affiche `Aucune évaluation disponible.` (AC-10).
- `GET /v1/quality` retourne `503` / network drop / JSON malformé → `observability.js` affiche le message d'indisponibilité (AC-11), `#usage-widget` et `#health-widget` non affectés (AC-9).
- Fichier statique manquant au déploiement → hors périmètre OBS-004 (couvert OBS-001 AC pour `/observability`) ; l'ancre `#quality-widget` absente serait détectée par un grep statique sur `observability.html`.

## Test manuel auteur

Procédure exécutée par l'humain avant `Status: ready` (AC-15).

1. **Stack** : `docker compose up -d`, vérifier gateway + workers + postgres `healthy` + migration `0005` appliquée (`\d eval_runs`).
2. **No data** : avec `eval_runs` vide → `curl -i http://localhost:8080/v1/quality` → `200`, body `{"status":"no_data"}`. Ouvrir `/observability` → `#quality-widget` affiche `Aucune évaluation disponible.` (AC-4, AC-10).
3. **Seed** : exécuter un run live persisté (`uv run python eval/ragas_runner.py --mode live --persist ...`, OBS-003) OU insérer une ligne live de test → `SELECT count(*) FROM eval_runs WHERE runner_mode='live'` ≥ 1.
4. **Nominal** : `curl -i http://localhost:8080/v1/quality` → `200`, body 6 clés, 4 métriques == valeurs de `SELECT ... ORDER BY finished_at DESC LIMIT 1`, `golden_set_version` non vide, `finished_at` RFC3339 (AC-1, AC-2, AC-3, AC-6).
5. **Widget** : ouvrir `http://localhost:8080/observability` → `#quality-widget` affiche les 4 métriques + ligne de fraîcheur. Network tab : `GET /v1/quality` 200 (AC-8).
6. **Latest** : insérer une 2e ligne live `finished_at` plus récent, valeurs distinctes → recharger → `/v1/quality` reflète la nouvelle ligne (AC-2).
7. **Isolation** : `docker compose stop postgres` → recharger `/observability` → `#quality-widget` affiche le message d'indisponibilité, MAIS `#usage-widget` et `#health-widget` restent rendus (degraded), pas blanchis (AC-9, AC-11) ; `curl -i /v1/quality` → `503` `upstream_unavailable`, aucun détail SQL (AC-7).
8. **En-têtes** : DevTools Network sur `GET /v1/quality` → les 5 headers de sécurité présents (AC-12).
9. **Anonyme** : `curl -i http://localhost:8080/v1/quality` (sans cookie) → `200` (AC-5).

## Security / trust boundary

- Frontière Internet → Gateway. Surface : 1 nouvel endpoint API public anonyme (`GET /v1/quality`) + 1 ancre HTML supplémentaire dans une page existante.
- **A01 Broken Access Control** : `/v1/quality` public anonyme par design (AC-5) ; n'expose aucune donnée per-utilisateur, uniquement des métriques agrégées de qualité du système (4 scores Ragas + `golden_set_version` + `finished_at`). Pas d'IDOR (aucun identifiant de ressource en path/query). Le body est volontairement réduit aux 6 champs (pas de `git_sha`, `id`, internes — surface minimale, Non-goals).
- **A03 Injection** : `SELECT` sans paramètre utilisateur (filtre constant `runner_mode='live'`), via `sqlx` macro/`query_as`, pas de `format!`.
- **A05 Misconfiguration** : CSP UI-001 strictement préservée (AC-12). Erreur `503` enveloppe sanitisée `{"error":"upstream_unavailable","request_id":"<uuid>"}`, jamais de message SQL/host/`eval_runs`/stack (AC-7). Le cas table-vide est un `200 {"status":"no_data"}` explicite, pas une fuite ni un `5xx` (AC-4).
- **A09 Logging & Monitoring** : log `tracing` structuré autorisé (`request_id`, `latency_ms`, présence/absence de données) ; aucune donnée de conversation ni payload LLM (les `answer`/`retrieved_contexts`/`citations` ne sont jamais dans `eval_runs`, OBS-003 AC-9). Chemin `503` loggé en `warn` sans détail SQL brut.
- **RAG-specific** : `eval_runs` ne contient aucun contenu de réponse ni contexte (OBS-003 AC-9) — l'exposition se limite à des scores numériques agrégés et un hash de version de golden set.
- Pas de secret dans `gateway/static/observability.{html,js,css}` (rappel `secret-hygiene.md`).

## Décisions

Arbitrages humains figés (round-1, 5 recos acceptées telles quelles). Chaque décision verrouille une dimension du contrat.

- **D-1 — Endpoint `GET /v1/quality`, body JSON plat 6 champs, read direct client↔gateway.** Body `{"faithfulness","answer_relevancy","context_precision","context_recall","golden_set_version","finished_at"}`. Lecture directe Postgres côté gateway (comme `/v1/stats`), aucun appel gateway→workers, `gateway-to-workers.yml` intact (AC-1, AC-14).
- **D-2 — Table vide (zéro ligne live) = `200 {"status":"no_data"}`, PAS `404`, PAS `503`.** Le widget rend le littéral `Aucune évaluation disponible.`. Le `no_data` est un état nominal (pas encore de run live persisté), distinct du chemin DB-down (AC-4, AC-10).
- **D-3 — DB-down = `503` réutilise byte-for-byte `ApiError::UpstreamUnavailable`.** Body `{"error":"upstream_unavailable","request_id":"<uuid>"}` identique à OBS-001 AC-7 / OBS-002. Asymétrie volontaire avec le cas `no_data` (D-2) : DB injoignable ≠ table vide (AC-7).
- **D-4 — Widget = 3e ancre `<section id="quality-widget">`, fetch indépendant, `textContent` only.** Fetché au chargement aux côtés de usage + santé ; un échec qualité ne blanchit pas les autres widgets (AC-9). 4 métriques en valeurs libellées + ligne fraîcheur de `finished_at`. Libellés/formatage non figés byte-for-byte ; seuls `textContent` + présence des 4 métriques sont contractuels (AC-8).
- **D-5 — Précision : API retourne la valeur `NUMERIC(5,4)` brute en nombre JSON (pas d'arrondi).** Le formatage d'affichage est fait en JS côté widget (non contractuel). La valeur exposée par l'API égale strictement la valeur stockée (AC-3).

## Open questions

- **OQ-1 — Chaîne d'erreur `503` : `upstream_unavailable` confirmée.** OBS-004 réutilise `ApiError::UpstreamUnavailable` qui sérialise `{"error":"upstream_unavailable","request_id":"..."}` (décision humaine D-3, vérifié dans `gateway/src/handlers/stats.rs` qui emploie le même variant). OBS-001 OQ-6 laissait ouvert `upstream_unavailable` vs `internal` ; OBS-004 s'aligne sur l'existant `stats.rs`/`status.rs` byte-for-byte — pas de nouvel arbitrage requis, mais à confirmer si OBS-001 OQ-6 tranche `internal` (alors OBS-004 doit suivre le même variant pour cohérence). Faible enjeu, purement nommage de contrat d'erreur partagé.
- **OQ-2 — `finished_at` : nom de champ confirmé, mais format de fraîcheur du widget.** Le body API expose `finished_at` brut RFC3339 (AC-6). La « ligne de fraîcheur » du widget (AC-8) peut être un rendu relatif (« il y a 3 h ») ou absolu (« 2026-06-08 14:00 UTC ») — non figé byte-for-byte (D-4). À confirmer par l'humain si une forme est préférée, sinon laissé à l'implémentation (souplesse assumée).
