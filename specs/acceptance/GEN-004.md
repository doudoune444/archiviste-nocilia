# GEN-004 — Mode 3 lore-gap : branche retrieval-faible + ticket auto avec dedup cosine

## Contexte

La vision (`docs/vision.md` §20-25 ligne Mode 3, §40 ticket lore-gap, ordre d'attaque #5) impose un troisième mode de réponse : quand le classifier `intent` retourne `in_domain` (cf GEN-003) mais que la meilleure similarité retournée par `/v1/retrieve` reste sous un seuil, le pipeline ne doit pas halluciner sur un canon vide. Il doit (a) répondre « noté pour archives » dans la persona Archiviste, (b) créer un ticket `lore_gap` lié à la conversation, (c) dédupliquer via cosine sur l'embedding de la question pour éviter la prolifération de tickets quasi-identiques (vision §40 : « `priority_score` incrémenté si question similaire détectée par cosine ≥ 0.85 »). Ce ticket livre la dernière branche productive du pipeline 4-modes avant Mode 4 mystère (GEN-005) et alimente la boucle feedback auteur via le dashboard UI-002.

## Critères d'acceptation

- AC-1 : `POST /v1/generate` accepte le même body que GEN-001/GEN-003 (`{query, conversation_id, user_id, user_tier, request_id}`) et retourne `200 OK` avec body `{answer, citations, mode, conversation_id, request_id, usage, retrieve_ms, llm_ms}` où `mode ∈ {"canon", "off_topic", "lore_gap"}` ; aucun champ ajouté ou retiré par rapport à GEN-003 AC-1.
- AC-2 : Après un classifier renvoyant `intent="in_domain"` (GEN-003 AC-2), le handler appelle `/v1/retrieve` (top_k=5 inchangé GEN-001) puis évalue `max_score = max(chunks[*].score, default=0.0)` ; si `max_score < LORE_GAP_THRESHOLD` la branche `lore_gap` est empruntée, sinon la branche `canon` GEN-001 reste empruntée. `LORE_GAP_THRESHOLD` est figé byte-for-byte à `0.45` dans une constante exportée (testable).
- AC-3 : Sur branche `lore_gap`, aucun appel au LLM de génération canon n'est effectué (le prompt canon avec chunks n'est jamais construit) ; un **second appel LLM** dédié génère la réponse « noté pour archives » via un system prompt figé byte-for-byte exporté sous `LORE_GAP_SYSTEM_PROMPT` (constante testable).
- AC-4 : `LORE_GAP_SYSTEM_PROMPT` contient au moins les instructions suivantes (assertion regex sur sous-chaînes clés) : identité « Archiviste », instruction « la question est in-domain mais les archives sont muettes / lacunaires », instruction « réponds de manière claire et concise, sans jeu de rôle ni mise en scène, et sans inventer », instruction « informe l'utilisateur que la question est notée pour enrichir les archives », instruction « réponds dans la langue de la question » (cohérent GEN-001 mode 1 — ton concis sans role-play / GEN-003 sur la dernière clause).
- AC-5 : Le user prompt envoyé au LLM lore_gap contient uniquement la question brute préfixée par `[user query]: ` (ou `[user query, suspected injection]: ` si AC-12 s'applique) ; aucun chunk lore n'est injecté (les chunks faibles ne sont pas pertinents, et la non-injection ferme la surface W-I-1 sur cette branche). Le builder est exporté sous `build_lore_gap_messages(query, suspected_injection)`.
- AC-6 : Quand `mode="lore_gap"`, le champ `citations` de la réponse vaut `[]` ; `retrieve_ms` reflète la latence réelle du `/v1/retrieve` exécuté (différence avec off_topic AC-9 où `retrieve_ms=0`) ; `llm_ms` agrège classifier + LLM lore_gap.
- AC-7 : Quand `mode="lore_gap"`, `usage.prompt_tokens` et `usage.completion_tokens` agrègent les **deux** appels LLM (classifier + générateur lore_gap) ; `usage.cost_eur` est la somme des deux coûts via `pricing.compute_cost_eur`. La latence du retrieve et l'embedding ticket (AC-9) ne comptent pas dans `usage`.
- AC-8 : Sur branche `lore_gap` réussie (200 OK), la conversation est persistée via ING-003 comme en GEN-001 AC-14 / GEN-003 AC-11 : exactement deux POST séquentiels `/v1/conversations/{id}/messages` (rôle `user` = `query`, rôle `assistant` = `answer`) ; un échec ING-003 émet un log ALERT `conversation_log_failed` sans bloquer la réponse 200.
- AC-9 : Sur branche `lore_gap` réussie (200 OK), exactement un appel `ticket_service.create_or_increment(conversation_id, question, request_id)` est effectué **après** la persistence ING-003 (l'ordre garantit que la FK `tickets.conversation_id` pointe vers une conversation déjà alimentée) et **avant** l'INSERT `query_log` ; un échec `ticket_service` émet un log ALERT `ticket_service_failed` sans bloquer la réponse 200 (la réponse au lecteur reste prioritaire sur le ticket).
- AC-10 : `ticket_service.create_or_increment` est implémenté côté workers (jamais gateway) et applique l'algorithme suivant dans une **unique transaction Postgres** :
  1. Calcule `question_embedding = embedder.encode_batch([question], batch_size=1)[0]` (dim 1024 conforme `EMBEDDING_DIM`).
  2. `SELECT id FROM tickets WHERE status='open' AND 1 - (question_embedding <=> $1) >= 0.85 ORDER BY question_embedding <=> $1 ASC LIMIT 1 FOR UPDATE`.
  3. Si une ligne match : `UPDATE tickets SET priority_score = priority_score + 1, updated_at = NOW() WHERE id = $matched` ; retourne `("incremented", matched_id, new_priority_score)`.
  4. Sinon : `INSERT INTO tickets (conversation_id, question, question_embedding, category, priority_score, status) VALUES ($1, $2, $3, 'uncategorized', 1, 'open') RETURNING id` ; retourne `("created", new_id, 1)`.
  Le seuil `TICKET_DEDUP_COSINE_THRESHOLD = 0.85` est figé byte-for-byte (constante exportée, conforme vision §40). La catégorie reste `'uncategorized'` (la classification fine est hors-scope, cf Non-goals).
- AC-11 : Exactement une ligne `query_log` est insérée par requête `lore_gap` réussie avec `(mode='lore_gap', intent='in_domain', status_code=200, latency_ms=<total wall clock>, prompt_tokens=<somme classifier + lore_gap>, completion_tokens=<somme>, cost_eur=<somme>)` ; un échec INSERT émet log ALERT `query_log_failed` sans bloquer la réponse 200 (cohérent GEN-001 AC-19, GEN-003 AC-12).
- AC-12 : Le pattern prompt-injection GEN-001 AC-20 / GEN-003 AC-17 reste appliqué AVANT classifier ; sur branche `lore_gap`, le préfixe `[user query, suspected injection]: ` est propagé au LLM lore_gap (AC-5) ; la question stockée dans `tickets.question` (AC-10) est la **query brute non préfixée**, identique à `request.query`, pour ne pas polluer le dashboard auteur avec des marqueurs internes.
- AC-13 : Si le LLM lore_gap timeout > 30 s → `504 Gateway Timeout` `{"error":"llm_timeout"}` ; **aucun ticket n'est créé** (`ticket_service` non appelé) ; ligne `query_log` insérée avec `status_code=504`, `mode='lore_gap'`, `intent='in_domain'`, `prompt_tokens` et `completion_tokens` reflètent uniquement le classifier (seul appel LLM comptabilisé). Cohérent GEN-003 AC-15.
- AC-14 : Si le LLM lore_gap retourne 4xx/5xx provider → `502 Bad Gateway` `{"error":"llm_upstream"}` ; aucun ticket n'est créé ; ligne `query_log` `status_code=502`, mêmes règles de comptage usage que AC-13. Cohérent GEN-003 AC-16.
- AC-15 : Si `embedder.encode_batch` lève (modèle non chargé, OOM, exception) pendant `ticket_service.create_or_increment` → log ALERT `ticket_service_failed` `{"event":"ticket_service_failed","request_id":"...","reason":"embed_failed"}`, **réponse 200 reste émise** au client avec la réponse lore_gap générée, ticket non créé, ligne `query_log` insérée nominalement (AC-11). La défaillance ticket ne bloque jamais le lecteur (AC-9 réaffirmé).
- AC-16 : Si la transaction Postgres de `ticket_service.create_or_increment` échoue (`asyncpg.PostgresError`, `TimeoutError`, `OSError`) → log ALERT `ticket_service_failed` `{"event":"ticket_service_failed","request_id":"...","reason":"db_failed"}`, **réponse 200 reste émise**, ligne `query_log` insérée nominalement (AC-11), aucun ticket créé / incrémenté (transaction rollback). Le timeout dur SQL de la transaction est figé à 5 s (constante exportée `TICKET_SQL_TIMEOUT_SECONDS = 5.0`, aligné `retrieve.schemas.SQL_TIMEOUT_SECONDS`).
- AC-17 : Si `conversation_id` est `None` dans la requête (création de conversation côté workers déjà gérée par GEN-001 / ING-003) **et** la branche choisie est `lore_gap`, la conversation est créée d'abord (logique ING-003 inchangée), puis le ticket référence la conversation nouvellement créée ; aucun ticket ne peut référencer une conversation inexistante (FK `tickets.conversation_id REFERENCES conversations(id) ON DELETE RESTRICT` déjà présente `migrations/0002_schema.sql:55`).
- AC-18 : Le log INFO de fin de requête (GEN-003 AC-20) est étendu pour inclure deux champs sur branche lore_gap : `top_score` (float, valeur de `max_score` calculée en AC-2, arrondie 4 décimales) et `ticket_action` (string `∈ {"created", "incremented", "skipped_error"}`). `chunks` reflète le nombre de chunks réellement retournés par retrieve (≥ 0, contrairement à off_topic où c'est 0) ; `citations` vaut `0` sur lore_gap (aucune citation parsée, AC-6).
- AC-19 : La branche `canon` GEN-001 reste intégralement inchangée du point de vue du client : la seule différence observable est que l'évaluation `max_score < 0.45` est désormais effectuée APRÈS retrieve ; quand `max_score >= 0.45`, le pipeline poursuit exactement comme avant (prompt build → LLM canon → parse citations → log).
- AC-20 : Une migration `migrations/0003_tickets_embedding.sql` ajoute exactement et exclusivement :
  - colonne `tickets.question_embedding vector(1024) NULL` (nullable pour rester compatible avec d'éventuels tickets pré-existants à la migration, bien qu'en pratique la table soit vide à ce stade) ;
  - index HNSW `tickets_question_embedding_idx ON tickets USING hnsw (question_embedding vector_cosine_ops) WHERE question_embedding IS NOT NULL` (partial index, évite les NULLs dans l'index) ;
  - index B-tree `tickets_conversation_id_idx ON tickets (conversation_id)` (lookup dashboard UI-002 + jointure FK) ;
  - index B-tree `tickets_status_priority_idx ON tickets (status, priority_score DESC, updated_at DESC)` (tri dashboard auteur par priorité décroissante au sein des `open`).
  Aucune modification de colonne existante (`id`, `conversation_id`, `question`, `category`, `priority_score`, `status`, `created_at`, `updated_at` figées par `0002_schema.sql:53-63`).
- AC-21 : Le schéma de `query_log.mode` accepte déjà `'lore_gap'` (cf `migrations/0002_schema.sql:73` contrainte CHECK existante) ; aucune modification de `query_log` n'est requise. Toute tentative d'INSERT `mode='lore_gap'` doit passer la contrainte sans amendement.
- AC-22 : Le pipeline workers ne fait **jamais** d'appel sortant pour Mode 3 vers une API externe autre que (a) le wrapper LLM `services/llm.py` (classifier + lore_gap) et (b) `embedder.encode_batch` local. Aucun fetch HTTP runtime au-delà des contrats internes `/v1/retrieve` et `/v1/conversations/{id}/messages` déjà couverts par GEN-001/GEN-003. Cohérent vision §46 (« le runtime lit uniquement des fichiers locaux »).

## Failure modes

- `query` vide / > 4 KiB / `user_id` ou `request_id` non-UUID / `user_tier` hors ensemble → comportement GEN-001/GEN-003 inchangé (400/422), aucun appel classifier, aucune branche lore_gap atteinte.
- Classifier timeout / 4xx / 5xx / unparseable → fail-open `in_domain` (GEN-003 AC-13/14/6) ; `retrieve` est alors appelé ; si `max_score < 0.45` → branche lore_gap normalement empruntée (pas de double fail-safe).
- Retrieve `/v1/retrieve` 5xx / timeout → comportement GEN-001 inchangé (`502`/`504`), pas de branche lore_gap déclenchée. `max_score` n'est jamais évalué sur retrieve en erreur.
- Retrieve retourne `chunks=[]` (corpus ACL-filtré vide pour ce user) → `max_score=0.0` < 0.45 → branche `lore_gap` empruntée (comportement souhaité : un utilisateur anonymous posant une question sur un doc author_only doit recevoir `lore_gap`, pas `canon` halluciné). **Note** : ceci diffère sémantiquement d'un Mode 4 mystère ACL-blocked (GEN-005), mais comme Mode 4 n'est pas livré, lore_gap couvre ce cas en V1.
- LLM lore_gap timeout > 30 s → `504 llm_timeout` (AC-13), ticket non créé.
- LLM lore_gap 4xx/5xx → `502 llm_upstream` (AC-14), ticket non créé.
- ING-003 4xx/5xx en mode lore_gap → réponse `200 OK`, log ALERT `conversation_log_failed` ; **AC-9 stipule que `ticket_service` est appelé APRÈS ING-003** : si ING-003 échoue, AC-17 (FK conversation existante) peut être violée si la conversation n'a pas été créée (cas `conversation_id=None`). **Arbitrage** : sur échec ING-003 dans lore_gap, `ticket_service` n'est PAS appelé (log ALERT `ticket_service_skipped_after_conversation_log_failed`), pour ne jamais émettre un INSERT FK-cassé. La réponse 200 est conservée.
- `embedder.encode_batch` échoue → log ALERT `ticket_service_failed reason=embed_failed`, ticket non créé, réponse 200 (AC-15).
- Transaction Postgres ticket échoue (timeout 5 s, conflict, OSError) → log ALERT `ticket_service_failed reason=db_failed`, ticket non créé (rollback), réponse 200 (AC-16).
- INSERT `query_log` échoue → log ALERT `query_log_failed`, réponse 200 conservée (cohérent GEN-001 / GEN-003).
- `query` matchant prompt-injection → préfixe `[user query, suspected injection]: ` propagé classifier + LLM lore_gap (AC-12) ; `tickets.question` reste la query brute non préfixée (AC-12 fin).

## Security / trust boundary

- Endpoint internal-only inchangé (cohérent GEN-001/GEN-003) ; le ticket_service est un module interne workers, pas une surface réseau nouvelle.
- Prompt injection : pattern détecté GEN-001/GEN-003 propagé en préfixe sur le LLM lore_gap (AC-12) ; aucun chunk lore n'est envoyé au LLM lore_gap (AC-5) — la surface W-I-1 « poisoned doc » est nulle sur cette branche, conforme `security.md` § « Prompt injection (input side) ».
- `tickets.question` stocke la query brute (AC-12 fin) — le dashboard auteur (UI-002) devra HTML-escape à la lecture, comme tout contenu user-derived (`security.md` § « Output sanitization »). Hors-scope GEN-004.
- Pas de bulk endpoint exposé sur `tickets` (Non-goal explicite) ; pas d'IDOR possible côté API publique car aucune route publique `tickets`.
- SQL injection : `ticket_service` utilise asyncpg paramétré (`$1`, `$2`, ...), aucun `format!` ni interpolation Python — conforme `security.md` A03.
- ACL contenu : `ticket_service` n'effectue **aucun** filtrage par `user_tier` ou `access_tier` (les tickets n'ont pas de `access_tier`, ils représentent une question utilisateur, pas un doc lore). Le dashboard tickets sera `author_only` côté UI-002 (vision §41).
- Logs : la `query` brute n'apparaît pas dans les logs ALERT `ticket_service_failed` (seul `request_id` et `reason`) ; cohérent `security.md` A09. La `query` est stockée en clair dans `tickets.question` car c'est la donnée produit pour l'auteur (équivalent au stockage de la conversation en GCS) — accepté par la vision §40.
- `LLM_API_KEY` reste `pydantic.SecretStr` partagé.
- Pas de PII attendu dans une question lore (univers de fiction publique) ; aucune scrub explicite phase 1 (cohérent GEN-001/GEN-003).
