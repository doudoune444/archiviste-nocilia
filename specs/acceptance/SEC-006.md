# SEC-006 — Gateway→workers ID-token signing + workers ingress=ALL

## Contexte

Le déploiement V1 manuel (post INFRA-002 / SEC-004 / SEC-005) produit un `502 upstream_error` sur `POST /v1/chat` : l'appel sortant de la gateway vers `workers /v1/generate` n'est pas authentifié, alors que Cloud Run impose une auth IAM sur le service workers (`INGRESS_TRAFFIC_INTERNAL_ONLY` + binding `roles/run.invoker` restreint au SA `archiviste-runtime`). Deux causes superposées : (a) la gateway ne joint pas l'header `Authorization: Bearer <id-token>` requis pour identifier l'appelant auprès du contrôle IAM de workers ; (b) le mode `INGRESS_TRAFFIC_INTERNAL_ONLY` rend le test du fix invisible / fragile selon le chemin réseau Cloud Run (le contrôle IAM reste de toute façon la vraie barrière de confiance). SEC-006 livre les deux moitiés du correctif en un seul vertical slice : signage ID-token côté gateway (réutilisant le pattern `auth_metadata::TokenProvider` SEC-004/005) et bascule workers vers `INGRESS_TRAFFIC_ALL`, IAM restant la frontière de confiance unique.

## Critères d'acceptation

- AC-1 : Un nouveau type `IdTokenProvider` est ajouté dans `gateway/src/auth_metadata/id_token.rs`, sibling de `TokenProvider`. L'audience est **capturée au constructeur** (`with_audience(audience: String)`), pas par appel — chaque instance est liée à exactement une audience et possède son propre cache. Il expose `fetch_id_token()` parameterless (cache + refresh-ahead + `Secret<String>` bearer interne) et stocke le bearer dans un `SecretString` (`secrecy` crate, conforme `security.md` §A09). Aucune fusion avec `TokenProvider` (OAuth access tokens) : `TokenProvider` reste strictement inchangé. Le cache interne est dédié, `RwLock<Option<CachedIdToken>>`, indépendant de toute autre cache. `grep -RIn 'IdTokenProvider' gateway/src` retourne ≥ 1 hit dans `auth_metadata/mod.rs` (réexport) et ≥ 1 hit dans `state.rs` (champ AppState).
- AC-2 : `IdTokenProvider::fetch_id_token` (parameterless, audience capturée au ctor) interroge le metadata server Cloud Run via `GET http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/identity?audience=<url-encoded audience>` avec header `Metadata-Flavor: Google` (même base URL injectable que `TokenProvider` pour mocking). La réponse — un JWT signé Google (corps texte brut, content-type `text/plain`) — est encapsulée dans un struct `CachedIdToken { bearer: SecretString, expires_at: DateTime<Utc> }`. Les timeouts client HTTP sont `connect_timeout = 2s` et `total_timeout = 5s` (mêmes constantes que `TokenProvider`, partagées ou dupliquées au choix `/plan`).
- AC-3 : Le calcul de `expires_at` parse le payload JWT (segment du milieu, base64url-decode sans signature verification) pour extraire le claim `exp` (epoch seconds UTC) et le convertit en `DateTime<Utc>`. Si le parsing échoue (segment manquant, base64 invalide, JSON invalide, claim `exp` absent ou non numérique), un fallback `Utc::now() + Duration::minutes(55)` est utilisé, ET un log `tracing` niveau `warn` `event="id_token.exp_parse_failed"` est émis (champs : `reason` ∈ `{"missing_segment","b64_decode","json_decode","missing_exp","exp_not_numeric"}`). Le bearer JWT lui-même n'est JAMAIS loggé. Aucune nouvelle crate ajoutée pour le parsing (utiliser `base64` + `serde_json` déjà présents dans `gateway/Cargo.toml`).
- AC-4 : La sémantique refresh-ahead réutilise la constante `REFRESH_AHEAD_SECS = 60` : le cache est considéré stale quand `now >= expires_at - 60s`. Le pattern read-lock fast-path + write-lock double-check de `TokenProvider::get_or_refresh` est reproduit à l'identique dans `IdTokenProvider::fetch_id_token`. Aucun retry-on-401 côté gateway : si workers répond 401/403, c'est traité comme un échec amont (cf AC-6) sans invalidation préventive (le cas nominal = ID-token frais valide ≥ 55 min).
- AC-5 : `gateway/src/state.rs` `AppState` expose un troisième champ `workers_id_token_provider: Arc<IdTokenProvider>` à côté des deux existants (`gcs_token_provider`, `sql_token_provider`). Pas de sous-struct composite. Le constructeur `AppState::new(...)` instancie `IdTokenProvider::with_audience(&config.workers_url)` (le `WORKERS_URL` lu de l'env devient l'audience JWT). Les constructeurs test `new_with_*` étendent la signature de manière minimale (audience fictive en test, par exemple `"http://test-workers"`).
- AC-6 : Le call site `forward_to_workers` (`gateway/src/handlers/chat.rs:155`) appelle `state.workers_id_token_provider.fetch_id_token().await` avant le `state.http.post(&url)`. Le bearer obtenu est attaché via `.bearer_auth(token.expose_secret())` sur la `RequestBuilder` reqwest. Si le fetch ID-token échoue (toute variante d'erreur), le handler retourne `503 upstream_unavailable` (enveloppe réutilisée du contrat SEC-001 inchangé : `{"error":"upstream_unavailable","request_id":"<uuid>"}`) — pas de nouveau code d'erreur public. Un log structuré `tracing` niveau `warn` `event="chat.id_token_failed"` est émis avec champ `reason_code` ∈ exactement `{"metadata_token_failed","timeout","network"}` (mapping : metadata server 4xx/5xx ou body non parseable → `metadata_token_failed` ; dépassement timeout 5s → `timeout` ; erreur transport DNS/TCP/TLS → `network`). Champs systématiques : `request_id`, `latency_ms`. Champs interdits : `id_token`, `bearer`, `audience` complète (seul un préfixe court si vraiment requis, à confirmer `/plan` — par défaut omis).
- AC-7 : Le header `Authorization: Bearer <id-token>` apparaît sur 100 % des requêtes sortantes `POST {workers_url}/v1/generate` émises par la gateway. Un test d'intégration (`gateway/tests/test_chat_workers_auth.rs` nouveau OU extension de `chat_test.rs`) vérifie ce comportement via un mock workers (mockito) qui inspecte le header `Authorization` reçu et fail si absent ou ne commençant pas par `Bearer `. Le test mocke aussi le metadata identity endpoint pour fournir un JWT-stub (header.payload.signature minimal avec `exp` dans 3600s).
- AC-8 : Le ticket SQL pool boot path (`gateway/src/lib.rs run()`) émet un fetch ID-token de chauffe AU BOOT (`fail_fast=true`) — option **rejetée explicitement phase 1** : le premier fetch est lazy au premier `POST /v1/chat`. Si le metadata server est down au boot mais up au premier chat, la gateway boot normalement et la première requête chat échoue 503 (failure mode connu, cohérent SEC-004 cache lazy). Aucun blocage boot ajouté. _AC vérifiable contractuellement : grep `rg 'IdTokenProvider.*fetch' gateway/src/lib.rs` retourne 0 ligne._
- AC-9 : `infra/terraform/cloud_run.tf` `google_cloud_run_v2_service.workers.ingress` passe de `"INGRESS_TRAFFIC_INTERNAL_ONLY"` à `"INGRESS_TRAFFIC_ALL"`. Le binding `google_cloud_run_v2_service_iam_member.workers_runtime_invoker` (role `roles/run.invoker`, member `serviceAccount:archiviste-runtime@<project>.iam.gserviceaccount.com`) est conservé tel quel — IAM reste la frontière de confiance. **Co-fixes V1 deploy bundlés** (orthogonaux au signage ID-token mais bloquants pour valider l'AC-11 post-deploy 403/200) : (a) `containers.ports { container_port = 8000 }` ajouté sur le service workers — Cloud Run par défaut probe TCP startup sur 8080 alors que uvicorn workers bind 8000 (`workers/Dockerfile CMD`), provoquant échec boot ; (b) deux env vars statiques injectées sur le service workers : `LLM_PROVIDER = "mistral"` et `LLM_MODEL = "mistral-small-latest"` — `LlmConfig.from_env` (`workers/src/archiviste_workers/services/llm.py:73`) rejette boot si absent ou hors allowlist `{mistral,anthropic,google,openai,deepseek}`. Valeurs alignées sur `.env.example`. Aucune secret value injectée ici (les API keys restent via Secret Manager, hors scope SEC-006).
- AC-10 : Un bloc Terraform `check` (Terraform >= 1.5) ajouté dans `infra/terraform/cloud_run.tf` (ou un fichier dédié `infra/terraform/checks.tf`) assert que la policy IAM du service workers ne contient AUCUN binding `allUsers` ni `allAuthenticatedUsers` sur le rôle `roles/run.invoker`. La vérif lit `data.google_cloud_run_v2_service_iam_policy.workers` (ou équivalent via `for_each` sur les bindings existants si le data source policy n'est pas dispo) et `assert` que `member != "allUsers" && member != "allAuthenticatedUsers"` pour tous les bindings du rôle invoker. `terraform plan` échoue (non-zéro) si la condition est fausse. Forme exacte du data source / `for_each` tranchée `/plan`.
- AC-11 : `docs/runbook.md` reçoit une nouvelle sous-section "Post-deploy smoke check — workers IAM ingress" décrivant la commande de validation manuelle : `curl -sw '%{http_code}\n' -o /dev/null https://<workers-url>/health` (sans `Authorization` header) DOIT retourner exactement `403`. Toute autre valeur (`200`, `404`, `502`) déclenche un incident — un `200` prouverait que workers est accessible non-authentifié (régression critique de la frontière IAM). La sous-section inclut aussi : (a) une commande gateway→workers chemin de bout en bout via gateway authentifié (`curl -H "Authorization: Bearer <user-jwt>" -X POST https://<gateway-url>/v1/chat ...`) qui doit retourner `200`, (b) un renvoi croisé à AC-10 (la check Terraform est la garantie pre-deploy, le curl 403 est la garantie post-deploy).
- AC-12 : `CHANGELOG.md` reçoit une entrée `### Security` sous `## [Unreleased]` : `SEC-006: gateway now signs outbound /v1/generate calls with a Google-issued ID token (audience=WORKERS_URL) via the Cloud Run metadata identity endpoint and reuses the auth_metadata cache pattern; workers ingress switched to INGRESS_TRAFFIC_ALL with IAM run.invoker still restricted to archiviste-runtime (Terraform check block + post-deploy 403 smoke); resolves V1 manual-deploy 502 upstream_error on POST /v1/chat.`

## Non-goals

- Pas de nouvelle erreur publique côté gateway (pas de `upstream_unauthenticated` ni équivalent) — toute défaillance ID-token ressort en `503 upstream_unavailable` opaque (réutilisation contrat SEC-001).
- Pas de binding IAM `allUsers` sur workers (explicitement vérifié par AC-10) — IAM reste la frontière de confiance ; `INGRESS_TRAFFIC_ALL` n'est pas un assouplissement de sécurité, juste de routing réseau.
- Pas de VPC connector ajouté ni envisagé (objet du non-goal : pas de chemin réseau privé entre gateway et workers — Cloud Run public IAM-gated suffit).
- Pas de fusion `TokenProvider` + `IdTokenProvider` ni d'extraction d'un helper `CachedSecret<T>` générique (flagué pour `/plan` mais pas dans le scope de ce ticket — règle clean-code "no premature abstraction", attendre une 4e cache).
- Pas de retry exponential backoff sur metadata identity fetch (un seul try ; échec = 503 chat ; cohérent SEC-004/005 phase 1).
- Pas de cache partagé entre les trois `*_token_provider` champs d'`AppState` (audiences/scopes différents, sémantique différente : un access token GCS/SQL n'authentifie pas auprès de Cloud Run IAM, et inversement un ID-token n'authentifie pas auprès de GCS/SQL).
- Pas de vérification signature JWT côté gateway (le parsing AC-3 lit `exp` sans validation crypto — la confiance vient de la connexion HTTPS au metadata server, pas du JWT lui-même).
- Pas de live test `--ignored` contre workers Cloud Run réel — vérification = curl runbook post-déploiement (AC-11).
- Pas de métrique counter dédiée `id_token_fetch_failures_total{reason_code}` phase 1 (instrumentation = log `tracing` AC-6, métrique → OBS-001 V2).
- Pas de propagation du `reason_code` ID-token vers le client HTTP (le client `/v1/chat` reçoit `upstream_unavailable` opaque).
- Pas de modification du contrat REST `POST /v1/chat` (path, request body, response shape, status codes inchangés).
- Pas de modification du contrat workers→gateway (workers continuent de répondre identiquement à `/v1/generate` ; la nouveauté est uniquement côté gateway sortant).
- Pas de fail-fast boot fetch ID-token (AC-8) — premier fetch lazy au premier chat.

## Failure modes

- Metadata identity endpoint indisponible / timeout / 5xx au premier `POST /v1/chat` → `503 upstream_unavailable` + log `event=chat.id_token_failed reason_code=metadata_token_failed` (ou `timeout` / `network` selon classification reqwest). Cache reste vide ; requête suivante retentera fetch. Aucun fallback.
- JWT renvoyé par le metadata server mais payload non parseable (mismatch contrat très improbable) → fallback `expires_at = now + 55min` (AC-3) + log `event=id_token.exp_parse_failed reason=<code>`. La requête courante réussit avec le bearer obtenu ; le cache contiendra un `expires_at` approximatif mais cohérent avec le TTL Google standard 1h.
- Workers répond 401/403 alors que gateway a joint un `Authorization: Bearer` valide → traité comme `502 upstream_error` ou `503 upstream_unavailable` selon le mapping existant `map_reqwest_error` / `handle_workers_response` de chat.rs (PAS modifié par SEC-006). Cause probable = audience JWT mismatch (gateway `audience=WORKERS_URL`, mais workers IAM attend une autre forme) — à investiguer en prod via log workers + gateway `request_id` corrélé.
- Workers répond 403 sur le binding IAM (SA gateway pas binder `roles/run.invoker` sur workers) → 502/503 côté `/v1/chat`. Mitigation = vérifier que `cloud_run.tf:229-235` est bien appliqué (déjà appliqué par INFRA-002 PR-e).
- Race cache ID-token : deux requêtes simultanées trouvent le cache stale, deux fetchs metadata parallèles → acceptable (idempotent, coût négligeable, sémantique `RwLock` = pas de blocage longue durée).
- Cache hit mais workers répond 401 (rotation ID-token Google, clock skew > 60s buffer) → 503 sur cette requête. La requête suivante (si après expiry réel) déclenchera refresh. Phase 1 : pas d'invalidation préventive cache sur 401 workers (AC-4).
- Terraform `check` block AC-10 détecte un binding `allUsers` (régression manuelle via `gcloud` ou un PR de drift) → `terraform plan` fail en CI, le déploiement est bloqué. Mitigation = retirer le binding offensif et rerun.
- Post-deploy curl `/health` sans auth retourne `200` au lieu de `403` → incident critique : workers exposé non-authentifié. Action immédiate = inspecter `terraform state` et re-vérifier l'ingress + le binding IAM.

## Performance / SLO

- `IdTokenProvider::fetch_id_token` cache-hit ≤ 1 ms wall-clock (lecture `RwLock` ~10 µs + comparaison `Instant`).
- Cache-miss : fetch metadata identity 5-20 ms sur Cloud Run (datacenter-local). Ajoute ≤ 25 ms p95 au premier `POST /v1/chat` après refresh-ahead, négligeable sur le SLO end-to-end p95 chat (~2-5 s dominé par LLM).
- Refresh metadata identity : ~1 fetch par 55 min par instance. Pour N=10 instances : ~11 fetchs/h. Limite metadata server = quasi infinie (endpoint local). Aucune contention attendue.
- p95 `POST /v1/chat` inchangé (overhead SEC-006 sub-ms cache-hit, ~20 ms cache-miss diluant dans le LLM call).

## Security / trust boundary

- A01 Broken Access Control : workers reste IAM-gated. SEC-006 ne crée aucun chemin d'accès non-authentifié — le passage `INTERNAL_ONLY` → `ALL` change uniquement le routing réseau public, le contrôle d'autorisation reste `roles/run.invoker` restreint au SA gateway. AC-10 (Terraform check) + AC-11 (curl 403 post-deploy) verrouillent cette invariance.
- A02 Cryptographic Failures : aucun matériel cryptographique nouveau côté gateway. Le JWT ID-token est produit par Google côté metadata server (preuve workload identity) ; la gateway ne possède aucune clé privée pour le forger.
- A07 Authentication Failures : auth gateway→workers utilise le mécanisme natif Cloud Run (ID-token Google-signé avec audience pinning). Pas de credential applicatif géré ; pas de token long-lived.
- A09 Logging : bearer ID-token JAMAIS loggé (AC-3, AC-6). Stocké dans `SecretString` (`secrecy` crate) qui redact en `Debug` / `Display`. Le claim `exp` peut être loggé (info publique, non sensible).
- A10 SSRF : URL metadata identity = constante fixe `http://metadata.google.internal/.../identity` ; l'`audience` est lue depuis `config.workers_url` (env trusted, pas d'input user). Aucun input utilisateur ne contrôle l'URL fetché ni la valeur d'audience.
- Trust boundary : nouvelle dépendance runtime sur l'endpoint metadata identity (déjà critique pour SEC-004/005 sur `/token`, donc pas d'élargissement). Indisponibilité = `503` sur `POST /v1/chat`, jamais fail-open.
- IAM principle of least privilege : aucun nouveau binding IAM ajouté par SEC-006 (le `roles/run.invoker` workers→gateway-SA est déjà en place via INFRA-002 PR-e). Aucun élargissement de scope.
- Ingress boundary : la bascule workers `INTERNAL_ONLY` → `ALL` est documentée comme "IAM is the trust boundary, not network ingress" — pattern standard Cloud Run pour les services privés (cf doc Google Cloud Run "Authenticating service-to-service"). AC-10 + AC-11 sont les deux garde-fous croisés.

## Observability
