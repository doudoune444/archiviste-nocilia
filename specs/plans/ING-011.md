**3 hypothèses clés**
1. **Path convention vs spec wording**: spec dit `scripts/gdrive/gsheet_renderer.py` mais la lib existante est `scripts/gdrive_export/`. Je place les renderers dans `scripts/gdrive_export/gsheet_renderer.py` et `scripts/gdrive_export/gslide_renderer.py` (cohérence package). De même `scripts/gdrive_export.py` (mentionné AC-1/14) = le package `gdrive_export/` (entrypoint `python -m gdrive_export`).
2. **Slugify extension non-breaking (AC-3)**: ajout d'une étape « map Unicode dashes → ASCII `-` » AVANT le `re.sub(r"[^a-z0-9]+", "-", ...)` actuel. Cette étape est idempotente (em-dash → `-` → déjà collapsed par la regex existante). Tous les tests existants de `test_slugify.py` (matrix + property idempotence/length/alphabet) doivent rester verts sans modification.
3. **DriveClient extension non-breaking (AC-1)**: `list_folder_recursive` étend `_SUPPORTED_MIMES` avec gsheet + gslide ; les fichiers gardent la même shape (`id, name, mimeType, parents, drive_path_components`) — pas de champ Sheets/Slides spécifique au listing. Les détails (tabs, slides) sont fetchés par appels dédiés (`spreadsheets.get`, `spreadsheets.values.get`, `presentations.get`) ajoutés au client. Scope SA additionnel détecté au boot via appel `spreadsheets.get(spreadsheetId="__probe__")` qui doit retourner 404 (not found) si le scope est présent, ou 403 (insufficient scope) sinon.

**Zones d'incertitude (risks détaillés plus bas)**
- AC-15 fail-fast: distinction 403 « insufficient scope » vs 403 « permission denied » sur l'ID probe — googleapiclient remonte `HttpError` avec `resp.reason` ; le message exact dépend de l'API. À valider via test mocké uniquement.
- AC-13 cap 1 MiB par onglet: le rendu Markdown peut dépasser 1 MiB après assemblage GFM mais avant écriture ; le cap doit être vérifié post-rendu mais avant `_write_*`, ET le `content_signature` ne doit pas être calculé sur un body skippé.
- AC-7 archived par-onglet: `_handle_archived` actuel itère sur les clés state absentes de `drive_ids` ; les onglets ont un `source_id` `<file_id>#<gid>` donc `drive_ids` (qui contient seulement les `file_id`) ne match jamais → faux positifs d'archive. La logique d'archive doit être étendue pour reconnaître les clés composées et vérifier l'existence du `file_id` parent + de l'onglet dans le classeur listé.

## Goal

Étendre `gdrive_export` pour exporter chaque onglet d'un Google Sheet vers un `.md` GFM et chaque Google Slides vers un `.md` structuré `## Slide N`, en réutilisant l'orchestration ING-013 (state, frontmatter, rename, archive, 1 MiB cap, normalize).

## Risks / open questions

- **AC-7 archive semantics pour onglets** : la logique actuelle `_handle_archived` compare `file_id ∈ drive_ids`. Pour onglets, archive ssi (a) `file_id` parent absent **OU** (b) `file_id` présent mais `gid` absent de tabs courantes. Cas (a) archive aussi tous les onglets siblings → cohérent avec « classeur supprimé ». À confirmer humain : si classeur entier supprimé, archive-t-on chaque onglet individuellement (N writes) ou un seul marqueur ? **Décision proposée**: archive chaque onglet (cohérent AC-7 « le `.md` local correspondant »).
- **AC-15 distinction 403 scope vs 403 permission** : googleapiclient remonte `HttpError(resp.status=403)` sans distinguer dans le code. Le message JSON Google contient `"reason": "insufficientPermissions"` vs `"reason": "forbidden"`. Parser `exc.error_details` ou `json.loads(exc.content)["error"]["errors"][0]["reason"]`. **Décision proposée**: parser `content` ; si échec parse, traiter comme scope manquant (fail-fast) — défensif.
- **Path nommage spec vs réalité** : spec mentionne `scripts/gdrive/gsheet_renderer.py` (singulier) ; package existant = `scripts/gdrive_export/`. Plan utilise `scripts/gdrive_export/gsheet_renderer.py`. Si humain veut le sous-package `gdrive/`, ré-arbitrer.
- **Sheets API quota** : `values.get` par onglet = N appels par classeur. Pour un classeur 50 onglets, 50 round-trips. Pas de batching dans plan (YAGNI). Si quota Sheets atteint (429), log error par onglet, exit 1 cohérent ING-013 AC-15.
- **AC-13 ordre des opérations** : rendu Markdown → cap 1 MiB check → sha256 → write. Si skip cap, **ne pas** mettre à jour le `StateEntry` de cet onglet (sinon prochain run le verrait « unchanged ») — cohérent avec ING-013 behavior actuel pour gdoc > 1 MiB (cf `_process_gdoc` L202-206 : early return sans new_state assignment).
