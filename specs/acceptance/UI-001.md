# UI-001 — Chat page MVP servie par la gateway (vanilla HTML+CSS+JS)

## Contexte

Le walking skeleton backend est complet (vision §107) : `POST /v1/chat` round-trip end-to-end mais reste curl-only. Avant `EVAL-001` (Ragas + golden_qa) il faut une UI publique pour tester le pipeline 4 modes manuellement et démontrer la vitrine portfolio. ADR-0005 tranche : page HTML+CSS+JS vanilla unique servie par la gateway Rust via `tower-http`, aucune build step, aucune dépendance JS tierce, `conversation_id` UUIDv4 généré côté client + `localStorage`.

## Critères d'acceptation

- AC-1 : `GET /` retourne `200 OK`, `Content-Type: text/html; charset=utf-8`, body contient le sélecteur stable `<form id="chat-form"` (point d'ancrage des tests d'intégration et de l'auteur du test manuel).
- AC-2 : `GET /assets/app.js` retourne `200 OK` avec un `Content-Type` JavaScript (`application/javascript` ou `text/javascript`, charset UTF-8).
- AC-3 : `GET /assets/styles.css` retourne `200 OK` avec `Content-Type: text/css` (charset UTF-8).
- AC-4 : Les actifs servis sont strictement limités à trois fichiers : `index.html` (servi sur `/`), `app.js` et `styles.css` (servis sous `/assets/`). Aucun autre chemin `/assets/*` n'expose de fichier supplémentaire.
- AC-5 : `GET /assets/../<n'importe-quoi>` (et toute variante de path traversal hors du dossier `assets/`) retourne `400` ou `404` ; aucun fichier hors du dossier statique exposé n'est lisible via la gateway.
- AC-6 : Toutes les réponses gateway (routes statiques `GET /`, `GET /assets/*` ET routes API existantes incluant `POST /v1/chat`) portent les en-têtes de réponse suivants, valeurs littérales :
  - `Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self'; img-src 'self' data:; object-src 'none'; frame-ancestors 'none'; base-uri 'none'; form-action 'self'`
  - `X-Content-Type-Options: nosniff`
  - `Referrer-Policy: strict-origin-when-cross-origin`
  - `X-Frame-Options: DENY`
- AC-7 : Aucune balise `<script>` inline, aucune balise `<style>` inline, aucun attribut `style="..."` ni gestionnaire `on*=` inline ne figure dans `index.html` ; tout JS est chargé via `<script src="/assets/app.js">` et tout CSS via `<link rel="stylesheet" href="/assets/styles.css">` (cohérent avec la CSP `script-src 'self'` / `style-src 'self'` de AC-6, validable par grep statique sur le fichier).
- AC-8 : Au chargement de la page, `app.js` lit la clé `archiviste.conversation_id` dans `localStorage` ; si la clé est absente, la valeur est laissée non-initialisée jusqu'au premier envoi (cf. AC-9). Le DOM ne réhydrate aucun message historique : un reload affiche une zone de conversation vide.
- AC-9 : Au premier clic sur "Envoyer" après chargement OU après un clic sur "Nouvelle conversation", `app.js` génère un `conversation_id` UUIDv4 via `crypto.randomUUID()` et l'écrit dans `localStorage` sous la clé `archiviste.conversation_id` AVANT d'émettre la requête `POST /v1/chat`.
- AC-10 : La soumission du formulaire (clic "Envoyer" ou submit clavier) envoie `POST /v1/chat` même-origine avec `Content-Type: application/json` et body `{"query": <texte saisi>, "conversation_id": <valeur localStorage>}`. La requête utilise `fetch` avec `credentials: 'same-origin'`. Aucun header `X-Request-Id` n'est envoyé par le client (cohérent GEN-002 AC-16, ignoré côté gateway).
- AC-11 : Avant l'envoi réseau, `app.js` insère dans la zone de conversation un nouvel `<article data-role="user">` contenant le texte de la query saisie (rendu en texte brut via `textContent`, jamais via `innerHTML`).
- AC-12 : À réception d'une réponse `200 OK` du gateway, `app.js` insère un `<article data-role="assistant">` contenant le champ `answer` du body workers en texte brut (rendu via `textContent`, pas de markdown rendering phase 1) ; les citations `[lore/x.md]` apparaissent telles quelles dans le texte. Aucun autre champ de la réponse (`citations`, `mode`, `usage`, `retrieve_ms`, `llm_ms`) n'est rendu visuellement phase 1.
- AC-13 : Pour toute issue de la requête autre que `200 OK` — incluant tout statut 4xx, tout statut 5xx, network drop, timeout, body JSON malformé — `app.js` insère un unique `<article data-role="error">` contenant le message littéral `L'archive ne répond pas. Réessayez.` ; aucun branchement spécifique selon le code d'erreur retourné par GEN-002 (`invalid_request`, `upstream_timeout`, `upstream_unavailable`, `upstream_error`, `internal`).
- AC-14 : Si la réponse (succès ou erreur HTTP) contient un header `X-Request-Id` ou un champ `request_id` dans le body JSON, sa valeur est rendue à l'intérieur de l'`<article data-role="error">` dans une balise `<small>req: <id></small>` (AC-13) ; absent → la balise `<small>` n'est pas insérée.
- AC-15 : Le bouton "Envoyer" est désactivé entre l'émission de la requête et la réception de sa réponse (succès ou erreur) ; il redevient cliquable après affichage du `<article data-role="user|assistant|error">` correspondant. La conversation peut continuer après une erreur — `conversation_id` n'est pas rotaté.
- AC-16 : Un bouton "Nouvelle conversation" est présent dans le DOM ; un clic supprime la clé `archiviste.conversation_id` de `localStorage`, vide entièrement la zone de conversation (`<article>` enlevés), et fait que le prochain envoi génère un nouvel UUIDv4 (cf. AC-9).
- AC-17 : Les middlewares posant les en-têtes de AC-6 sont appliqués globalement (router-level), pas seulement sur les routes statiques ; un test sur `POST /v1/chat` confirme que les 4 en-têtes sont présents.
- AC-18 : Aucune ligne de log applicatif n'est émise par le gateway pour les requêtes `GET /` et `GET /assets/*` (la `TraceLayer` existante suffit) ; les routes API conservent leur logging structuré antérieur (GEN-002 AC-13 inchangé).
- AC-19 : Un test manuel auteur est documenté dans le ticket sous `## Test manuel auteur` (procédure pas-à-pas reproduisible : démarrer la stack, ouvrir l'URL, envoyer une query, observer la réponse, observer le comportement d'erreur en coupant les workers, vérifier la persistence `localStorage`, vérifier le bouton "Nouvelle conversation"). L'exécution du test manuel par l'auteur fait partie de la définition de "ready to ship" pour ce ticket.

## Non-goals

- Pas de markdown rendering / pas de syntax highlighting / pas de linkification des citations `[lore/x.md]` — phase 1 plain text.
- Pas de streaming SSE / pas de typage progressif de la réponse — affichage atomique du body après round-trip complet.
- Pas de login, pas de signup, pas de cookie de session, pas de header `Authorization` — `user_tier="anonymous"` reste hardcodé gateway (vision §73, GEN-002 AC-3).
- Pas de page de login / pas de nav / pas de footer / pas de pagination — une seule page, un seul formulaire.
- Pas d'affichage des `citations` structurées, du `mode`, des `usage`, des `latency_ms` workers dans l'UI phase 1.
- Pas de switch d'erreur fin selon les 5 codes GEN-002 (`invalid_request` / `upstream_timeout` / `upstream_unavailable` / `upstream_error` / `internal`) — message générique unique.
- Pas de persistence de l'historique des messages côté client (seul `conversation_id` persisté). Reload → DOM vide.
- Pas de re-fetch d'un historique côté serveur au chargement (pas d'endpoint `GET /v1/conversations/{id}/messages` côté gateway — couvert plus tard par OPS-* dashboard auteur).
- Pas de Playwright, pas de Puppeteer, pas de browser headless test phase 1 — couverture par tests d'intégration Rust + test manuel auteur.
- Pas de framework JS tiers (React, Vue, htmx, Alpine, Stimulus) — ADR-0005 §Decision.
- Pas de build step, pas de bundler, pas de transpiler, pas de source maps — édition fichier → refresh navigateur.
- Pas de dépendance CDN (script tag externe) phase 1 — la CSP `script-src 'self'` interdit toute origine externe.
- Pas de header HSTS posé par la gateway — terminaison TLS chez Cloud Run / LB (sera porté par un ticket SEC/INFRA dédié).
- Pas de rate-limit applicatif sur les routes statiques — ticket SEC-* dédié couvrira l'ensemble du gateway.
- Pas de gzip / brotli / cache headers (`Cache-Control`, `ETag`) optimisé phase 1 — défauts `tower-http` acceptables, optimisation reportée si latence devient gênante.
- Pas d'i18n / pas de switch de langue — UI FR uniquement phase 1.
- Pas de favicon custom phase 1 (404 favicon toléré).
- Pas d'intégration analytics / métriques produit phase 1.
- Pas d'amendement du contrat `specs/openapi/gateway-to-workers.yml` (ce yml décrit gateway↔workers, pas client↔gateway).

## Pre-conditions

- GEN-002 mergé : `POST /v1/chat` opérationnel, contrat `{query, conversation_id}` accepté, réponse 200 byte-pour-byte du body workers, headers `X-Request-Id` posé, enveloppe d'erreur `{"error":"<code>","request_id":"<uuid>"}` figée.
- ADR-0005 accepté (`docs/adr/0005-frontend-vanilla-served-by-gateway.md`).
- `gateway/Cargo.toml` autorise l'ajout de la feature `tower-http::fs` (ou `tower-http::set-header`) sans déclencher un nouvel ADR au sens de `secret-hygiene.md` / clean-code "no new heavy dep" — `tower-http` est déjà dans le `Cargo.toml`.

## Failure modes

- Body workers JSON malformé (champ `answer` absent ou non-string) → `app.js` traite comme erreur, affiche `<article data-role="error">` AC-13. Pas de log côté front.
- `crypto.randomUUID()` non disponible (navigateur très ancien) → `app.js` n'a pas de fallback ; le test manuel auteur (AC-19) cible un navigateur moderne (Chromium/Firefox récent). Documenté en non-goal implicite ("phase 1 navigateurs modernes uniquement").
- `localStorage` indisponible (mode privé strict, navigateur exotique) → l'écriture de `archiviste.conversation_id` échoue silencieusement ; chaque envoi régénère un UUIDv4 (le serveur le tolère via GEN-001 AC-2). Pas de bannière utilisateur phase 1.
- Path traversal (`/assets/../*`, `/assets/%2e%2e/*`) → 400/404 servi par `tower-http` (AC-5).
- Fichier statique manquant au déploiement (`index.html` absent du conteneur) → `GET /` retourne 404 ; détecté en CI par les tests AC-1/AC-2/AC-3.
- Requête `POST /v1/chat` interceptée par un middleware tiers (proxy entreprise) modifiant les headers → l'UI affiche le message générique (AC-13). Hors périmètre.

## Touch points (informatif, non contraignant pour l'architect)

- `gateway/static/index.html` — page unique (form id `chat-form`, zone conversation, bouton "Nouvelle conversation"), aucun `<script>` / `<style>` inline (AC-7).
- `gateway/static/app.js` — fetch `/v1/chat`, gestion `localStorage`, insertion DOM via `textContent`.
- `gateway/static/styles.css` — mise en forme minimaliste de la page chat.
- `gateway/Cargo.toml` — activer la feature `tower-http` nécessaire pour servir des fichiers (`fs`) et/ou poser des headers (`set-header`).
- `gateway/src/lib.rs` — montage des routes `GET /` + `GET /assets/*` (servies depuis `gateway/static/`), montage du middleware d'en-têtes globaux (CSP + X-Content-Type-Options + Referrer-Policy + X-Frame-Options) appliqué à TOUT le router (chat + statiques).
- `gateway/tests/static_test.rs` — tests d'intégration nouveaux (AC-1, AC-2, AC-3, AC-5, AC-6 partiel, AC-17).
- `gateway/tests/chat_test.rs` — éventuelle assertion supplémentaire pour AC-17 (les 4 headers sont présents sur `POST /v1/chat`) si non déjà couvert.
- `Dockerfile` du gateway — `COPY gateway/static /app/static` (ou équivalent) pour que les fichiers soient présents dans l'image.
- `docker-compose.yml` — pas de changement attendu (la gateway expose déjà `8080`).
- `.env.example` — pas de changement attendu.

- AC-5 : intégration · `GET /assets/../Cargo.toml` (et variantes encodées) → 400 ou 404, body ne contient pas le contenu de `Cargo.toml`.

## Test manuel auteur

Procédure reproductible exécutée par l'humain comme part de la définition de "ready to ship" (AC-19).

1. **Démarrer la stack** : `docker compose up -d` ; vérifier `docker compose ps` que `gateway`, `workers`, `postgres` sont `healthy`.
2. **Ouvrir** `http://localhost:8080/` dans un navigateur Chromium ou Firefox récent.
3. **Vérifier les en-têtes** (DevTools Network, requête `GET /`) : `Content-Security-Policy`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, `X-Frame-Options: DENY` (AC-6).
4. **localStorage vide initial** : DevTools Application → Local Storage → vérifier `archiviste.conversation_id` absent.
5. **Envoyer une query canon** : saisir `Qui est l'Archiviste de Nocilia ?`, cliquer "Envoyer".
   - Pendant la requête : bouton désactivé (AC-15).
   - DOM : `<article data-role="user">` apparaît immédiatement (AC-11), puis `<article data-role="assistant">` avec la réponse workers en texte brut (AC-12), citations `[lore/...]` visibles tel quel.
   - localStorage : `archiviste.conversation_id` créé, valeur UUIDv4 (AC-9).
   - Network : requête `POST /v1/chat`, body `{"query":"...","conversation_id":"<uuid>"}`, `Content-Type: application/json` (AC-10).
6. **Reload de la page** : DOM vide, `archiviste.conversation_id` toujours présent (AC-8). Envoyer une 2e query → body envoyé contient le même `conversation_id` qu'à l'étape 5.
7. **Tester l'erreur réseau** : `docker compose stop workers` ; envoyer une query → `<article data-role="error">` avec message générique `L'archive ne répond pas. Réessayez.` + `<small>req: <uuid></small>` (AC-13, AC-14). Bouton "Envoyer" redevient cliquable (AC-15).
8. **Tester l'erreur 400 gateway** : redémarrer workers, envoyer une query vide (champ texte vide → l'UI doit toujours pouvoir soumettre, ou test alternatif via DevTools Console `fetch('/v1/chat', {method:'POST', headers:{'Content-Type':'application/json'}, body:'{}'})`) → même message générique unique (AC-13).
9. **Bouton "Nouvelle conversation"** : cliquer → DOM vidé (`<article>` retirés), `archiviste.conversation_id` retiré du `localStorage` (AC-16). Envoyer une nouvelle query → nouveau UUIDv4 différent du précédent.
10. **Test XSS basique** : saisir `<script>alert(1)</script>` → rendu en texte brut, aucune popup (AC-11 + CSP).
11. **Test path traversal** : `curl -i http://localhost:8080/assets/../Cargo.toml` → 400 ou 404 (AC-5).

Coller dans le commentaire de PR la liste cochée de chaque étape avec capture d'écran ou note succincte.

## Performance / SLO

- Pas de SLO chiffré phase 1. Indicatif : `GET /` < 50 ms p95 sur stack locale (fichier statique, pas de DB, pas d'upstream).
- Pas d'overhead mesurable sur `POST /v1/chat` p95 dû aux nouveaux headers (AC-6) ; vision §91 (`< 80 ms p95 à 500 users`) reste valable.
- Taille totale des actifs `index.html + app.js + styles.css` < 50 KiB phase 1 (cible souple ; vise zéro round-trip de chargement supplémentaire).

## Security / trust boundary

- Frontière Internet → Gateway (G-* du threat model). Surface étendue de routes API à routes statiques + UI.
- A05 Misconfiguration : CSP stricte (AC-6) interdit toute origine script/style externe, désactive `<object>` et `<frame>`, force `form-action 'self'`. Aucun `<script>` ni `<style>` inline (AC-7) — pas besoin de `'unsafe-inline'`.
- A03 Injection (XSS reflected/stored) : défense en profondeur — (a) CSP `script-src 'self'` (AC-6), (b) injection DOM via `textContent` jamais `innerHTML` (AC-11, AC-12), (c) le serveur (workers GEN-001 §Security) garantit déjà `answer` plain string sans HTML actif. Test AC-11 du test manuel auteur.
- A04 Insecure Design : `X-Frame-Options: DENY` + `frame-ancestors 'none'` interdisent le clickjacking. `Referrer-Policy: strict-origin-when-cross-origin` limite les fuites de Referer.
- A02 Cryptographic Failures : pas de HSTS posé par gateway (terminaison TLS amont Cloud Run / LB) — couvert hors ticket.
- Conversation persistence (security.md §RAG-specific) : le `conversation_id` est un UUIDv4 généré côté client, jamais path-traversal-able, conforme à la règle "Markdown filenames derived from UUID only" déjà respectée par ING-003.
- Aucun secret n'apparaît dans `gateway/static/*` (rappel `secret-hygiene.md` : pas de clé, pas de token, pas d'URL avec credentials).
- Path traversal `/assets/../*` rejeté par `tower-http` (AC-5).
- Le client ne peut pas influencer `user_id` ni `user_tier` (toujours hardcodés gateway, GEN-002 AC-3).

## Observability

- Aucun log applicatif spécifique aux routes statiques (AC-18). `TraceLayer` existante du gateway suffit pour observer `GET /` / `GET /assets/*`.
- Pas de métrique OpenTelemetry / pas de trace Langfuse phase 1.
- Pas d'alerte spécifique au-delà des codes HTTP émis.

## Effort estimate

S — 3 fichiers statiques courts (estimés < 200 LOC cumulés), 1 middleware d'en-têtes, 1 fichier de tests d'intégration, 0 changement workers, 0 changement DB. Vertical slice ≤ 300 LOC respecté (rule `vertical-slice.md`).
