# OPS-001 — Load tests 100 / 500 users : k6 scripts + rapport SLO V1

## Contexte

`docs/vision.md` (Objectifs ligne 16) revendique « test scaling crédible » comme trophée portfolio, et liste comme SLOs cibles (lignes 87-92) p95 chat round-trip < 3 s et overhead Rust gateway < 80 ms p95 à 500 users concurrents. À ce stade, aucune mesure n'étaye ces cibles sur le déploiement public `https://archiviste.nocilia.fr`. Ce ticket clôt l'ordre d'attaque V1 (vision.md ligne 168, #9) en produisant un harnais k6 reproductible aux paliers 100 et 500 users concurrents, et un rapport SLO checked-in qui sert d'artefact vitrine.

## Critères d'acceptation

- AC-1 : `scripts/load/k6/chat-round-trip.js` existe et expose deux scenarios k6 nommés exactement `chat_100_users` et `chat_500_users`, configurés via `scenarios:` (executor `constant-vus` ou `ramping-vus` documenté en commentaire d'en-tête), ciblant respectivement 100 et 500 VUs steady-state pendant ≥ 60 s chacun, avec ramp-up ≤ 30 s. Les scenarios sont sélectionnables individuellement via `--env SCENARIO=chat_100_users` ou `--env SCENARIO=chat_500_users` (option `executor: false` désactivant l'autre scenario via `env`).
- AC-2 : Le script k6 envoie sur chaque itération une requête `POST https://archiviste.nocilia.fr/v1/chat` avec body JSON `{"conversation_id": "<uuid v4 par VU>", "query": "<prompt issu d'une liste ≥ 10 prompts in-domain checked-in>"}`, `Content-Type: application/json`. Le champ `query` matche le contrat shipped UI-001 (cf. `gateway/src/handlers/chat.rs:115`). Le `conversation_id` est généré une fois par VU au `setup`/`default` (UUID v4) pour simuler une session unique par utilisateur. Les prompts sont chargés depuis `scripts/load/k6/prompts.json` (≥ 10 entrées, in-domain corpus, pas de PII, pas de prompt-injection).
- AC-3 : Le script déclare des `thresholds` k6 qui font **échouer** le run si l'un des SLOs est violé :
  - `http_req_duration{scenario:chat_100_users}: ['p(95)<3000']`
  - `http_req_duration{scenario:chat_500_users}: ['p(95)<3000']`
  - `http_req_failed: ['rate<0.01']` (proxy disponibilité — voir Failure modes pour la cible stricte « 0 erreur 5xx »)
  - Un threshold dédié sur la métrique custom `gateway_overhead_ms` (cf AC-4) : `['p(95)<80']` au scenario 500 users.
- AC-4 : Le script extrait l'overhead gateway depuis le response header `X-Gateway-Overhead-Ms` (entier en millisecondes, set par la gateway) et l'enregistre dans une métrique k6 custom `Trend` nommée `gateway_overhead_ms`, taggée par scenario. Si le header est absent sur une réponse 2xx, l'itération incrémente un compteur `gateway_overhead_header_missing` et le run émet un warning summary (pas un fail) — l'arbitrage header gateway vs SLO mesuré côté client uniquement est posé en Open questions.
- AC-5 : `scripts/load/README.md` documente : (a) pré-requis (k6 ≥ 0.50, IP source whitelist Cloudflare ou bypass — cf AC-7), (b) commande exacte par scenario (`k6 run --env SCENARIO=chat_100_users scripts/load/k6/chat-round-trip.js`), (c) budget Mistral estimé par run (ordre de grandeur €10-50, méthode de calcul = `VUs × itérations/min × durée × coût_par_req`), (d) procédure d'export du summary JSON (`--summary-export=summary.json`), (e) procédure de génération du rapport (cf AC-6).
- AC-6 : `docs/load-test-report-v1.md` existe en version commitée et contient, par scenario (100 / 500 users) :
  - Date du run, commit SHA gateway/workers déployé, région Cloud Run, taille instance.
  - Tableau métriques : `http_req_duration` p50 / p95 / p99, `http_reqs` total + rate, `http_req_failed` rate, `gateway_overhead_ms` p50 / p95.
  - Verdict SLO ligne par ligne : ✓ / ✗ vs cible (p95 < 3 s, overhead < 80 ms p95, 0 erreur 5xx).
  - Section « cold-start impact » : observation qualitative scale-to-zero (première vague de VUs après idle) — vision.md Q6 cold-start ~3 s gateway / ~5 s workers à confronter.
  - Section « budget réel » : coût Mistral observé (Mistral console export ou estimation post-run).
  - Section « follow-ups » : tickets ouverts si SLO non tenu.
- AC-7 : Le rapport `docs/load-test-report-v1.md` documente explicitement la méthode utilisée pour neutraliser la rule Cloudflare 100 req/min/IP (vision.md Q9) pendant le run : option choisie (whitelist IP source dans Cloudflare rule, OU bypass via worker GCP backbone, OU désactivation temporaire de la rule pendant la fenêtre de run avec timestamp début/fin loggué). L'option retenue est nommée et la trace de configuration (capture règle Cloudflare ou commande gcloud) est référencée.
- AC-8 : Tout run produisant un threshold k6 en échec (AC-3) bloque la promotion V1 « vitrine portfolio » : le rapport documente la cause racine (latence LLM upstream, overhead gateway, erreurs 5xx) et ouvre un ticket follow-up nommé (`OPS-002 <slug>` ou ticket de tuning pertinent). Le statut V1 reste « beta » tant que ce ticket follow-up n'est pas tranché.
- AC-9 : Pendant la fenêtre du run 500 users, Langfuse capture les traces LLM (config existante hors-ticket). Le rapport `docs/load-test-report-v1.md` inclut une capture ou un lien vers la vue Langfuse filtrée sur la fenêtre temporelle du run (timestamps début/fin) — preuve d'observabilité bout-en-bout, pas une mesure SLO.

## Non-goals

- Pas de stress tests > 500 users concurrents (V2).
- Pas de soak tests longue durée (> 1 h) (V2).
- Pas de chaos engineering / fault injection (V2).
- Pas de tuning autoscaling Cloud Run `min_instances` : on laisse scale-to-zero V1, on **documente** l'impact cold-start dans le rapport sans corriger.
- Pas de load test contre les workers en interne — endpoint public gateway uniquement (la gateway exerce déjà workers en cascade).
- Pas de GHA workflow `.github/workflows/load.yml` en V1 (run local depuis poste auteur OK ; CI nightly = V2).
- Pas de modification de code applicatif (gateway / workers / migrations) **sauf** l'ajout du header `X-Gateway-Overhead-Ms` si validé en Open questions (sinon ticket séparé).
- Pas de mise en cache CDN agressive Cloudflare pour `/v1/chat` (endpoint POST dynamique, hors caching).
- Pas d'eval Ragas pendant le run (couvert par EVAL-001 offline + hook live manuel Q16 vision.md).
- Pas de comparaison multi-régions (europe-west9 only V1).

## Failure modes

- Threshold k6 `http_req_duration p95 < 3000` violé à 100 users → run `exit 1`, rapport documente cause + ticket follow-up (AC-8). Promotion vitrine V1 bloquée.
- Threshold k6 `http_req_duration p95 < 3000` violé à 500 users mais OK à 100 users → run `exit 1`, rapport documente seuil de bascule observé, follow-up = tuning Cloud Run sizing ou autoscaling.
- Threshold `gateway_overhead_ms p95 < 80` violé à 500 users → run `exit 1`, follow-up = profiling gateway (`tower::trace` ou flamegraph), réduction de l'overhead (cache, hot path).
- `http_req_failed rate ≥ 0.01` (≥ 1 % erreurs) → run `exit 1`. Cible stricte « 0 erreur 5xx sur run » documentée dans le rapport : on compte explicitement les codes 5xx (séparés des 4xx attendus type 429 si bypass Cloudflare partiel) et toute occurrence 5xx → follow-up bloquant V1.
- Cloudflare rule rate-limit non bypassée → 429 massifs côté k6 → run invalide, à relancer avec config corrigée (AC-7).
- Budget Mistral épuisé mid-run (cap console atteint) → 5xx ou 429 upstream Mistral → run avorté, rapport documente l'incident, follow-up = ajustement cap ou réduction palier 500 users.
- Header `X-Gateway-Overhead-Ms` absent (gateway pas instrumentée) → métrique `gateway_overhead_ms` vide → AC-3 threshold gateway overhead non évaluable → décision en Open questions (instrumenter gateway dans OPS-001 ou ticket séparé pré-requis).
- Cold-start scale-to-zero observé sur première vague → premiers 1-5 % de requêtes p95 > 3 s → documentation dans rapport, **pas** considéré comme échec SLO si exclu de la fenêtre de mesure steady-state (warm-up ≥ 30 s exclu du calcul p95 via `summaryTrendStats` ou tag `warmup:true`).

## Performance / SLO

- Cible AC-3 / AC-6 :
  - `http_req_duration` p95 < 3000 ms aux paliers 100 et 500 users concurrents (vision.md SLO ligne 90).
  - `gateway_overhead_ms` p95 < 80 ms à 500 users (vision.md SLO ligne 92).
  - `http_req_failed` rate < 1 % (proxy disponibilité 99,0 % — non testable strictement en run court), avec exigence stricte additionnelle **0 réponse 5xx** sur l'ensemble du run (comptée séparément des 4xx attendus).
- Méthode de mesure :
  - Latence end-to-end : k6 builtin `http_req_duration` (mesure côté client, inclut DNS + TLS + TTFB + transfert).
  - Overhead gateway : header response `X-Gateway-Overhead-Ms` posé par middleware tower (en attente arbitrage Open questions). Alternative fallback : mesure indirecte = `http_req_duration - workers_internal_latency` via Langfuse (moins précis, documenté en annexe rapport).
- Fenêtre de mesure : steady-state uniquement, ramp-up ≤ 30 s exclu via tag k6 `warmup:true` ou `summaryTrendStats` filtré.
- Pas de benchmark continu / régression check ; mesure ponctuelle V1, V2 envisagera CI nightly.

## Security / trust boundary

- Cloudflare rate-limit rule 100 req/min/IP (vision.md Q9) : à bypasser pendant le run via une des options nommées en AC-7 (whitelist IP, bypass backbone GCP, désactivation temporaire avec fenêtre timestampée). Procédure documentée + trace conservée pour audit.
- IP source du run : depuis poste auteur (IP résidentielle) OU runner GCP éphémère. Si résidentielle, whitelist temporaire Cloudflare avec retrait post-run obligatoire (checklist runbook).
- Pas de secret applicatif manipulé par k6. Si auth tier `member` testée (post-SEC-001), le token de test est un compte dédié `load-test@`, non un compte réel — hors-scope V1 (SEC-001 pas pré-requis).
- Prompts envoyés : in-domain canon, pas de PII, pas de prompt-injection. La liste `prompts.json` est revue humainement avant commit.
- Cloud Run : exposition publique déjà acquise via INFRA-002 ; le load test n'élève pas la surface réseau.
- Conformité `.claude/rules/secret-hygiene.md` : aucun secret commité dans `scripts/load/`. Variables sensibles (URL prod, IP whitelist) passées en `--env` k6 ou env shell.

## Observability

- Langfuse : traces LLM capturées pendant le run (config existante, pas de modif). Capture / lien embarqué dans rapport (AC-9).
- k6 summary : export JSON via `--summary-export=summary.json`, archivé en annexe du rapport ou commit dans `scripts/load/runs/<date>-<scenario>.json` (option à trancher au plan, max 2 fichiers checked-in pour ne pas polluer le repo).
- Pas de métrique OpenTelemetry custom ajoutée côté gateway / workers (`X-Gateway-Overhead-Ms` mis à part, en Open questions).
- Pas d'alerte GCP / Cloud Logging dédiée load test (OBS-001 V2).

## Decisions (auto-arbitrées vs vision.md, 2026-05-18)

- **D-1 (ex OQ-1) — `X-Gateway-Overhead-Ms` IN-scope OPS-001** : middleware tower gateway (~30 LOC + test) pose le header. Vision §92 SLO impose mesure précise overhead < 80 ms p95 ; option Langfuse fallback rejetée (imprécise). Reste sous 300 LOC total.
- **D-2 (ex OQ-2) — Bypass Cloudflare = whitelist IP source dans la rule** existante. IP poste auteur ajoutée temporairement à la rule, retrait post-run obligatoire (checklist runbook). Option « désactivation temporaire de la rule » rejetée (risque sécu si oubli).
- **D-3 (ex OQ-3) — Cap Mistral plafond €30/run** (médiane fourchette €10-50). Cap console Mistral à ajuster avant run par auteur ; si run > €30 → run avorté.
- **D-4 (ex OQ-4) — Summary JSON commité dans `scripts/load/runs/<YYYY-MM-DD>-<scenario>.json`**. Reproductibilité offline, traçabilité git. Max 2 runs/scenario checked-in (rotation manuelle si dépassement).
- **D-5 (ex OQ-5) — GHA workflow `load.yml` = V2**. V1 = run local suffit pour rapport vitrine (vision ordre #9 = scripts + rapport seul). CI nightly = surcoût Mistral injustifié beta.
- **D-6 (ex OQ-6) — `prompts.json` liste indépendante** (découplage strict de `specs/golden_qa.jsonl`). Évite que les évolutions golden eval cassent les runs load test rétrospectifs.
