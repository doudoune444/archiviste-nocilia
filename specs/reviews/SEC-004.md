# Review — SEC-004

## Verdict

REQUEST_CHANGES

## Summary

Implementation is sound on the core path: IAM `signBlob` HTTP wiring (AC-2), `TokenProvider` with read-then-write-double-check cache (AC-3 race-safe), 6 `reason_code` variants (AC-5), 2s/5s hardcoded timeouts (AC-6), self-binding Terraform (AC-7), inline env block dropped from `cloud_run.tf` (AC-8 Option 1), no secrets in code, no `unwrap`/`expect` in `gateway/src/gcs/**`, no `unsafe`, SSRF surface clean (no user-controlled URL reaches metadata/IAM fetch), Debug/log discipline respected (`SignError`/`TokenError` are unit variants, the only log site is the handler and uses `reason_code` only).

But: **AC-9(f) coverage missing** — no test asserts `reason_code=timeout` on a real 5s timeout, only a `reason_code=network` on port-1 (ECONNREFUSED). The single existing "timeout-or-network" test conflates two distinct failure modes spec-mandated as separate `reason_code` values. This is a spec-violation finding (MED) that blocks `APPROVE`.

## SEC-005 reuse path

The brief flagged the SEC-005 reuse concern. `TokenProvider` constructor (`new()` + `with_base_url(metadata_base_url: String)`) accepts a base-URL injection but **hardcodes the scope** at the metadata endpoint level (`METADATA_TOKEN_PATH = "/computeMetadata/v1/instance/service-accounts/default/token"` returns the default-scoped access token, which on Cloud Run defaults to `https://www.googleapis.com/auth/cloud-platform` — already broad enough for both IAM `signBlob` and Cloud SQL IAM auth).

**No fundamental redesign needed for SEC-005**: the cached token is a default-SA OAuth access token usable for any IAM-authenticated Google API the SA has roles for. SEC-005 extension surface = (1) re-use the same `TokenProvider` from `AppState.token_provider` for the Cloud SQL Auth Proxy IAM token plumbing, (2) optionally add an explicit `scope` parameter to `with_base_url` if a narrower scope is desired (not required — default scope on metadata server already covers `cloudsql.instanceUser`). **Not blocking SEC-004**.

## Verdict justification

`REQUEST_CHANGES`. The signing path, security envelope, IAM Terraform binding, secret-hygiene posture, log discipline, and tests for 7/8 AC-9 sub-cases are correct and complete. The single blocker is AC-9(f) timeout coverage: the spec requires a dedicated `reason_code=timeout` assertion + `start.elapsed() < 6s` oracle (AC-6 worst-case latency check), and the current `sec004_network_error_logs_network` test conflates timeout and network into a `code == "network" || code == "timeout"` weakening. Add `sec004_signblob_timeout_logs_timeout` using mockito `with_chunked_body` + `tokio::time::sleep(6s)` (pattern already used in `overhead_header_test.rs:125/165`) and tighten the network test to `assert_eq!(code, "network")`. Once green, this PR is `APPROVE`-ready.
