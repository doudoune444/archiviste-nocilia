# ING-011 — Converters gsheet multi-onglets + gslide pour `gdrive_export.py`

## Contexte

ING-010 livre le noyau de sync Drive → `lore/*.md` (gdoc + png + state + frontmatter + CLI). La vision §52 prévoit aussi gsheet et gslide comme sources amont, mais ces deux MIME ont des sémantiques d'export non-triviales (un classeur ⇒ N onglets ⇒ N fichiers ; une présentation ⇒ structure slide explicite à reconstruire). Ce ticket étend l'orchestration ING-010 avec deux converters dédiés, en réutilisant l'auth, le state file, le frontmatter merge et la CLI déjà en place.

## Critères d'acceptation

- AC-1 : Le script `scripts/gdrive_export.py` (livré par ING-010) inclut désormais dans sa liste Drive (cf ING-010 AC-1) les MIMEs `application/vnd.google-apps.spreadsheet` et `application/vnd.google-apps.presentation` accessibles au service account.
- AC-2 : Chaque `gsheet` (`application/vnd.google-apps.spreadsheet`) génère **un fichier `.md` par onglet** (sheet/tab) ; chaque onglet est exporté via Sheets API v4 `spreadsheets.values.get` (range = nom de l'onglet) et rendu en table Markdown GFM (header = première ligne, séparateur `|---|`, échappement `|` → `\|` et `\n` → `<br>` dans les cellules).
- AC-3 : Le path d'un onglet gsheet est exactement `lore/<drive_path>/<sheet_slug>--<tab_slug>.md` (séparateur `--` entre titre du classeur et titre de l'onglet, tous deux slugifiés via la fonction d'ING-010 AC-6). Cette règle s'applique **aussi** aux classeurs à un seul onglet (pas de collapse en `<sheet_slug>.md`). La fonction de slugification d'ING-010 AC-6 est étendue (sans rupture de contrat) pour normaliser explicitement les dashes Unicode (`—` U+2014 em-dash, `–` U+2013 en-dash, `‒` U+2012, `−` U+2212) en ASCII `-` AVANT le collapse runs `-+ → -` ; le séparateur `--` est concaténé APRÈS slugify entre `sheet_slug` et `tab_slug`, donc un slug individuel ne peut jamais contenir `--` (collapse runs garantit ≤ 1 `-` consécutif intra-slug).
- AC-4 : Le `tab_slug` est dérivé du `title` de l'onglet ; en cas de collision intra-classeur (deux onglets slugifiant identique), suffixe `-<gid>` ajouté au tab_slug perdant l'unicité (ordre déterministe = ordre Drive `index`).
- AC-5 : Le `source_id` d'un onglet gsheet dans `.gdrive_state.json` est `<drive_file_id>#<gid_decimal>` où `<gid_decimal>` est l'entier numérique `gid` retourné par l'API Sheets, stringifié en base 10 (ex. premier onglet par défaut = `<drive_file_id>#0` ; onglet custom = `<drive_file_id>#1734218910`). Pas de hash, pas de préfixe `tab=` / `gid=`, pas de zéro-padding.
- AC-6 : Le frontmatter d'un `.md` d'onglet gsheet contient `title = "<titre classeur> — <titre onglet>"` (em-dash `—` U+2014, séparation lisible humain), tous les autres champs identiques au schéma ING-010 AC-5.
- AC-7 : Détection de **suppression d'onglet** : si un classeur existe toujours mais qu'un onglet précédemment exporté (clé d'état `<drive_file_id>#<tab_gid>`) n'est plus listé par `spreadsheets.get`, le `.md` local correspondant à cet onglet voit `archived: true` posé (mêmes règles que ING-010 AC-10) ; les autres onglets du classeur sont mis à jour normalement.
- AC-8 : Le `drive_md5_checksum` d'un onglet gsheet est le sha256 hex de la sérialisation Markdown produite (post-rendu GFM) — stocké en état pour AC-11 d'ING-010 (détection inchangé).
- AC-9 : Chaque `gslide` (`application/vnd.google-apps.presentation`) est exporté via Slides API v1 `presentations.get` (récupération structurée des slides, pas `files.export(text/plain)`) puis rendu en Markdown au format : titre = `<title>` du gslide, suivi pour chaque slide visible d'un header `## Slide <N>` (1-indexé). L'ordre des slides suit strictement celui retourné par `presentations.get` (= ordre du présentateur). À l'intérieur d'une slide, les `pageElements` de type `shape` sont triés par `transform.translateY` ASC puis `transform.translateX` ASC (top-to-bottom puis left-to-right) ; le texte de chaque shape (`shape.text.textElements[*].textRun.content`) est concaténé dans cet ordre, séparé par `\n\n`. La numérotation `## Slide <N>` est continue sur les slides visibles uniquement (cf AC-16) : si une slide est masquée en position 2 d'un deck de 5, la sortie produit `Slide 1, Slide 2, Slide 3, Slide 4` numérotation continue sans trou.
- AC-10 : Le path d'un gslide est `lore/<drive_path>/<deck_slug>.md` (un fichier par présentation, pas par slide). Le `source_id` est `<drive_file_id>` (sans suffixe).
- AC-11 : Les notes de speaker (`pageElements` du `slideProperties.notesPage`) sont rendues sous chaque `## Slide <N>` dans un blockquote `> **Notes**: <texte>` si non-vides ; absentes sinon.
- AC-12 : Tout texte extrait gsheet/gslide passe par la même normalisation NFKC + strip control chars qu'ING-010 AC-13 avant écriture.
- AC-13 : Le cap 1 MiB d'ING-010 AC-14 s'applique au rendu Markdown final (par fichier `.md` produit) : un onglet gsheet > 1 MiB en Markdown rendu est skippé individuellement (les autres onglets du classeur restent exportés).
- AC-14 : Les converters gsheet et gslide sont implémentés dans des modules dédiés (`scripts/gdrive/gsheet_renderer.py`, `scripts/gdrive/gslide_renderer.py`) et orchestrés depuis `scripts/gdrive_export.py` ; leur ajout ne casse pas les ACs d'ING-010 (gdoc + png inchangés).
- AC-15 : Le scope OAuth additionnel `https://www.googleapis.com/auth/spreadsheets.readonly` ET `https://www.googleapis.com/auth/presentations.readonly` est documenté dans `docs/runbook.md` (section onboarding service account) ; le script fail-fast au boot si le SA n'a pas les scopes requis (test = appel `spreadsheets.get` factice sur un ID bidon, capture du 403 « insufficient scope »).
- AC-16 : **Slides masquées** : toute slide dont `pageProperties.skipSlide == true` est skippée du rendu. Un log warning `event=gdrive_sync.slide_hidden_skipped` est émis avec `source_id`, `slide_index_original` (1-based dans le deck Drive complet). Les slides visibles sont renumérotées contigüment dans la sortie (cf AC-9).
- AC-17 : **HTML comment de traçabilité** : la première ligne du corps Markdown produit pour un `.md` issu de gsheet ou gslide (post-frontmatter, avant tout autre contenu) est exactement `<!-- gdrive: file=<drive_file_id> tab=<gid_decimal> -->\n\n` pour un onglet gsheet, et `<!-- gdrive: file=<drive_file_id> -->\n\n` pour un gslide. Permet le debug humain hors frontmatter. Note opérationnelle (informative, non bloquante) : ING-001 ingestion devra strip ce commentaire HTML avant embedding (cohérent avec normalize NFKC + strip control chars existants côté pipeline ingest) ; pas une dépendance dure de ce ticket.

## Non-goals

- Pas de rendu d'images embarquées dans gsheet (cell `=IMAGE()`) ou gslide (`pageElement` type `image`) — ticket futur.
- Pas de rendu des formules gsheet (l'export utilise `values.get` pas `formulas`, donc valeurs calculées seulement).
- Pas de rendu des animations / transitions / commentaires gslide.
- Pas de support des onglets gsheet protégés par mot de passe (rare, fail explicite si rencontré).
- Pas de support de la mise en forme cellule (couleur, gras, bordures) — Markdown brut.
- Pas de re-numérotation stable des slides si Drive renvoie un ordre instable (assumption : `presentations.get` renvoie les slides dans l'ordre déterministe du document).
- Pas de workflow GitHub Actions (reporté à ING-012).
- Pas de support Shared Drives (cohérence ING-010).

## Pre-conditions

- ING-010 mergé : noyau `gdrive_export.py` opérationnel (auth SA, state file, frontmatter merge, CLI manuel, slugification, collision, archive).
- Service account GCP autorisé en lecture sur le dossier Drive racine ET muni des scopes additionnels `https://www.googleapis.com/auth/spreadsheets.readonly` et `https://www.googleapis.com/auth/presentations.readonly` (configuration manuelle hors-script).

## Failure modes

- Scope SA manquant pour Sheets ou Slides API → 403 `insufficient scope` au boot ou au premier appel → exit non-zéro immédiat, message `gdrive scope missing: spreadsheets.readonly and presentations.readonly required`, aucun fichier traité.
- Onglet gsheet vide (zéro ligne) → `.md` créé avec frontmatter + corps `*(empty sheet)*`, status `created`/`updated` normal.
- Slide gslide sans textes (uniquement images) → `.md` rendu avec `## Slide N` suivi d'une ligne `*(no text content)*`.
- Onglet supprimé → archive (cf AC-7), pas une erreur.
- Onglet renommé → traité comme un onglet supprimé + un onglet créé (le `tab_gid` est l'identité stable, le slug change donc le path change, ING-010 AC-9 `git mv` s'applique).
- gsheet > 10 onglets : pas de cap, traité onglet par onglet ; cap 1 MiB par fichier `.md` rendu (AC-13).
- Quota Sheets / Slides API → mêmes règles qu'ING-010 (log error, file suivant, exit `1`).
- Rendu Markdown gsheet > 1 MiB pour un onglet → skip onglet, log error, autres onglets du classeur continués (AC-13).

## Décisions résolues (humain, 2026-05-09 — round 2)

- **OQ-D — em-dash vs séparateur `--` (AC-3)** : pipeline slugify étendu = NFKC normalize → drop combining marks → strip Unicode dashes (`—`, `–`, `‒`, `−`) remplacés par ASCII `-` → lower → non-alphanum → `-` → collapse runs `-+ → -` → strip extrémités → cap 80. Le séparateur `--` est concaténé APRÈS slugify entre `sheet_slug` et `tab_slug`, donc impossible qu'un `--` apparaisse dans un slug individuel. Em-dash dans le `title` frontmatter (AC-6) reste conservé (lisibilité humain).
- **OQ-E — ordre lecture slides (AC-9)** : ordre slides = ordre retourné par `slides.presentations.get` (= ordre du présentateur). Dans une slide, page elements triés par `transform.translateY` ASC puis `transform.translateX` ASC (top-to-bottom puis left-to-right). Texte concaténé dans cet ordre, séparé par `\n\n`. Pas de tri par z-index.
- **OQ-F — slides masquées (AC-16)** : skip si `pageProperties.skipSlide == true` ; log warning `event=gdrive_sync.slide_hidden_skipped` (avec slide index original) ; renumérotation `## Slide N` continue sur slides visibles uniquement (sans trou).
- **OQ-G — format gid (AC-5)** : numeric int décimal stringifié, sans préfixe ni padding. `source_id` = `<drive_file_id>#<gid_decimal>`. Premier onglet par défaut → `#0`. Cohérent avec parsing trivial via split sur `#`. Le suffixe collision `tab_slug` (AC-4) utilise donc aussi `-<gid_decimal>` (pas de hash).
- **OQ-H — HTML comment embed gid en body (AC-17)** : oui, première ligne body après frontmatter `<!-- gdrive: file=<drive_file_id> tab=<gid_decimal> -->` (gsheet) ou `<!-- gdrive: file=<drive_file_id> -->` (gslide). Note non-bloquante : ING-001 ingest devra strip ce commentaire HTML avant embedding (à traiter dans son pipeline normalize, pas une dépendance dure ING-011).
