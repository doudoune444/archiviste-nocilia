# Plan — ING-003 Conversation logger (Markdown GCS append + index Postgres)

## 1. Résumé

Endpoint FastAPI `POST /v1/conversations/{conversation_id}/messages` côté workers : valide payload + UUID, INSERT row `conversations` au premier message (avec header GCS via `ifGenerationMatch=0`), append bloc Markdown via cycle read-modify-write avec retries 50/200/800 ms, UPDATE counters DB best-effort, log structuré sans `content`. Service `gcs` (fake-gcs-server) ajouté à `docker-compose.yml` sous `profiles: ["tools"]`.

## 2. Hypothèses résolues

| # | Hypothèse | Décision |
|---|---|---|
| H1 | Conflit avec settings.py existant : champ `gcs_bucket_conversations` vs spec AC-12 qui exige `GCS_BUCKET` | Renommer le champ Settings en `gcs_bucket: str` (env var `GCS_BUCKET`, sans default → fail-fast pydantic-settings via `... = Field(...)`). Ajouter `gcs_emulator_host: str | None = None`. Champ `gcs_bucket_conversations` supprimé (jamais lu en prod, FOUND-002 ne le consomme nulle part). Mettre à jour `.env.example` (ligne 13) et `docker-compose.yml` env block workers. |
| H2 | Client GCS sync (`google-cloud-storage`) vs handler async FastAPI | Wrapper `GcsConversationStorage` exécute les appels via `asyncio.to_thread()` (pas de dépendance `gcloud-aio-storage` à ajouter — déjà dans deps). Acceptable en phase 1 (pas de SLO chiffré strict). |
| H3 | Pool DB asyncpg : créé par ce ticket ou hérité ? | Aucun pool n'existe encore (settings n'expose qu'un `database_url`). Créer le pool dans `lifespan` de `main.py` via `asyncpg.create_pool(dsn)` ; stocker dans `app.state.db_pool`. DSN dérivé de `DATABASE_URL` en remplaçant le préfixe `postgresql+asyncpg://` par `postgresql://` (asyncpg n'accepte pas le scheme SQLAlchemy). Helper privé `_asyncpg_dsn(url)`. |
| H4 | Conversion erreurs Postgres : où est gérée l'erreur `23503` (FK user_id) ? | Repository attrape `asyncpg.ForeignKeyViolationError` sur INSERT et lève une `UnknownUserError` custom ; router map → 422 `{"error":"unknown_user"}`. |
| H5 | Détection conflit `ifGenerationMatch=0` (objet déjà existant) | `google.api_core.exceptions.PreconditionFailed` (412) au moment du `blob.upload_from_string(..., if_generation_match=0)`. Le router map → 409 `{"error":"conversation_already_exists"}`. |
| H6 | Détection conflit `ifGenerationMatch=<G>` (concurrent write) | Même exception `PreconditionFailed` mais sur le chemin append (post-INSERT row). Wrapper retry 3× (backoff déterministe 50/200/800 ms, `asyncio.sleep`), re-read avant chaque retry. Épuisement → `ConcurrentWriteError` → 409 `{"error":"concurrent_write"}`. |
| H7 | Erreurs GCS non-récupérables vs récupérables | Récupérables = `PreconditionFailed` (gérées au-dessus). Non-récupérables = `google.api_core.exceptions.ServiceUnavailable`, `RetryError`, `requests.exceptions.Timeout`, `ConnectionError` → 503 `{"error":"storage_unavailable"}`. Pas de retry sur ces erreurs (seul le 412 a une sémantique de retry métier). |
| H8 | Validation timestamp ISO8601 | `datetime.fromisoformat()` (Python 3.12 supporte `Z` natif). Comparaison à `datetime.now(UTC)` (futur > 5 min) et à `conversations.created_at` (lu via `SELECT created_at FROM conversations WHERE id=$1` après s'être assuré que la row existe — pour le PREMIER message la row vient d'être créée donc on peut comparer au `created_at` retourné par `INSERT ... RETURNING created_at`). |
| H9 | Ordre INSERT row vs création GCS object pour le premier message | Row INSERT first (transaction commit), puis création objet GCS avec `ifGenerationMatch=0`. Si création GCS échoue → row reste orpheline (pas de rollback inverse possible). C'est acceptable : la prochaine tentative trouvera la row, sautera l'INSERT, et créera l'objet. Le `gcs_uri` est déterministe (`gs://{bucket}/{conversation_id}.md`). |
| H10 | Détection « première vs n-ième message » | `INSERT ... ON CONFLICT (id) DO NOTHING RETURNING id, created_at` : si la ligne est retournée → premier message (créer header GCS). Sinon → `SELECT created_at FROM conversations WHERE id=$1` puis append direct. |
| H11 | Validation `content` 16 KiB | `len(content.encode("utf-8")) > 16 * 1024` → 413 avant tout I/O. |
| H12 | Logging structuré : capture pour test AC-11 | `structlog.testing.capture_logs()` (déjà dispo via dep `structlog>=24.4`). |
| H13 | Test concurrence AC-15 | `asyncio.gather(post1, post2)` avec deux clients httpx en parallèle sur le worker. Asserter que les compteurs finaux sont cohérents (1 ou 2 selon retry). |
| H14 | INV-4 round-trip | Spec dit explicitement « hors-scope ce ticket » (Decisions résolues ligne 117). On ne crée PAS `test_conversation_format.py`. Le touch-point dans la spec est obsolète ; ce plan le retire. |
| H15 | OpenAPI gateway-to-workers | Endpoint internal-only entre workers et lui-même (consommé par GEN-* depuis le même service). Pas d'aller-retour gateway→workers. **Aucun fichier OpenAPI gateway-to-workers n'existe encore** (spec mentionnée mais non versionnée). Ce ticket ne touche PAS le contrat gateway→workers. |
| H16 | `conftest.py` partagé pour fixtures pool DB + bucket fake-gcs | Créer `workers/tests/conftest.py` (n'existe pas) avec fixtures `db_pool`, `gcs_storage`, `worker_client` (httpx ASGI). Marqueur `@pytest.mark.integration` requiert `GCS_EMULATOR_HOST` et `DATABASE_URL` set. |

## 9. Risques / questions ouvertes

- **R1** : Diff prod estimé ~310 LOC > 300. Mitigation : inliner mapping exception → HTTP dans router (cf §7). Si toujours > 300, signaler à reviewer pour décision de split/exception.
- **R2** : Test AC-13 (`docker compose config` parse profile `tools`) : exécuter en subprocess depuis pytest est fragile en CI Windows. Mitigation : test type `contract` séparé exécuté en bash via job CI, ou simple grep YAML dans le test Python (parse `docker-compose.yml` en pyyaml, assert `services.gcs.profiles == ["tools"]`). Préférence : grep YAML dans `test_conversation_integration.py::test_gcs_service_under_tools_profile`.
- **R3** : `fake-gcs-server` versions : `latest` non pinné contredit security A06 (« lock files committed »). Mitigation : pinner à `fsouza/fake-gcs-server:1.49` (dernière stable au 2026-05-05) dans `docker-compose.yml`. Spec dit `latest` mais on durcit ; documenter dans le commit.
- **R4** : `asyncio.to_thread` pour appels GCS sync : sous forte concurrence le pool de threads par défaut (CPU×5) peut saturer. Phase 1 acceptable (pas de SLO chiffré). Documenter dans `docs/runbook.md` si futur ticket OBS-* mesure latence.
- **R5** : Backoff 50/200/800 ms déterministe sans jitter (décision spec) : risque de thundering herd si N clients tapent la même conversation. Phase 1 acceptable (1 caller GEN-*). Notable mais hors scope.
