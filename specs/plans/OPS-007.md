# Plan — OPS-007 Correctifs runtime du canal corpus lore (régression OPS-006)

## Goal
Réparer la chaîne gdrive-sync → bucket → Job ingest en accordant à `gha-deploy` la permission `storage.buckets.get` et en injectant `GCS_BUCKET` dans le bloc env du Job `archiviste-ingest`.

## Acceptance criteria recap
- AC-1 : `gha-deploy` dispose, sur le seul bucket `archiviste-lore-corpus`, d'un rôle portant la permission `storage.buckets.get` ; `roles/storage.legacyBucketReader` est le rôle prédéfini de moindre privilège satisfaisant ce contrat. Binding scopé au seul bucket — aucune permission `storage.buckets.*` accordée au niveau projet.
- AC-2 : Le step `gcloud storage rsync lore/ gs://archiviste-lore-corpus` s'exécute sans erreur `does not have storage.buckets.get access` ; après un run gdrive-sync ayant produit ≥ 1 `.md`, le run est vert.
- AC-3 : Le bloc env du Job `archiviste-ingest` injecte `GCS_BUCKET=archiviste-conversations` ; `Settings()` (`ingest/cli.py:106`) se construit sans `ValidationError` et le Job ne sort plus en exit `1` au boot. `gcs_bucket` n'est jamais lu sur le chemin ingest (seul lecteur réel : `main.py:71`).
- AC-4 : Le binding lecture runtime reste `roles/storage.objectViewer` pour `archiviste-runtime` (inchangé) ; aucune permission `storage.buckets.get` à `archiviste-runtime`. Non-régression OPS-006 AC-3.
- AC-5 : Chaîne complète après run gdrive-sync ≥ 1 `.md` : (a) gdrive-sync vert, (b) `ingest-lore` vert, (c) `gcloud storage ls gs://archiviste-lore-corpus/**` liste les `.md`, (d) log JSON `ingest.summary` avec `total >= 1` filtrable par execution ID.
