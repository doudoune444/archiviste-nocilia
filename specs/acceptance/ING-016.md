# ING-016 — Chunker tokenizer swap (BGE-M3 → Mixtral-8x7B-v0.1)

## Status

ready

## Context

INFRA-002 a swappé l'embedder runtime BGE-M3 → `mistral-embed` mais a dû garder `transformers` en dépendance runtime parce que `workers/src/archiviste_workers/ingest/chunker.py:8,18` importe `transformers.AutoTokenizer` et appelle `AutoTokenizer.from_pretrained("BAAI/bge-m3")` pour dimensionner le LangChain text splitter. `transformers` (~52 MiB sur disque + RSS d'import lourd) est le driver mémoire qui fait OOM (signal 9) le Cloud Run ingest Job à 512Mi. Ce ticket retire ce dernier usage de `transformers` du graphe runtime ; il finit le travail amorcé par INFRA-002. Le corpus n'étant pas encore validé en prod ni eval-testé, ce ticket établit le tokenizer Mixtral-8x7B-v0.1 comme **point de départ** du chunking — il ne cherche pas la parité de découpage avec l'ancien split BGE-M3.

## Acceptance criteria

- AC-1 : `workers/src/archiviste_workers/ingest/chunker.py` n'importe plus `transformers` (aucune occurrence de `import transformers` ni `from transformers`).
- AC-2 : Le splitter mesure `chunk_size`/`chunk_overlap` via le tokenizer **Mixtral-8x7B-v0.1** (celui que `langchain-mistralai` utilise déjà pour le batch-sizing de tous les modèles Mistral, y compris `mistral-embed`) chargé avec la lib `tokenizers` (Rust, déjà présente dans le graphe via `langchain-mistralai`) ; aucune nouvelle dépendance runtime n'est ajoutée à `[project.dependencies]`.
- AC-3 : L'initialisation du chunker (chargement du tokenizer Mixtral-8x7B-v0.1) n'effectue aucune requête réseau sortante ; le tokenizer est chargé depuis un artefact `tokenizer.json` vendoré (committé dans le repo, Apache-2.0, baké dans l'image au build), pas de fetch HF au build ni au runtime, pas de HF token. (Vérifié par test : init du chunker avec réseau coupé / HTTP mock asserte zéro requête sortante.)
- AC-4 : Le chunker, utilisant le tokenizer Mixtral-8x7B-v0.1 vendoré via la lib `tokenizers`, découpe un document fixture en chunks respectant `chunk_size=512` / `chunk_overlap=64` mesurés en tokens Mixtral, avec l'ordre de séparateurs existant. (Test de correction pur ; aucune comparaison à un découpage BGE-M3 antérieur.)
- AC-5 : `workers/pyproject.toml` `[project.dependencies]` ne contient plus `transformers` (drop final du runtime) ; `transformers` n'apparaît dans aucune table de dépendances (ni runtime, ni dev, ni optional).
- AC-6 : Le bloc `[project.optional-dependencies] embedder-fallback` (qui ne servait que l'embedder self-host BGE-M3 mort, remplacé par mistral-embed en INFRA-002) est supprimé de `workers/pyproject.toml` ; `sentence-transformers` n'apparaît plus dans aucune table de dépendances (ni runtime, ni optional).
- AC-7 : L'image workers runtime perd ≥ 50 MB (mesure `docker images archiviste-workers --format '{{.Size}}'` avant/après build sur le même Dockerfile).
- AC-8 : Le gate CI Ragas standing (`eval/ragas_runner.py` sur `specs/golden_qa.jsonl`) continue de passer après le swap de tokenizer (le gate CI existant reste vert ; ce ticket n'introduit aucun nouveau seuil numérique ≥-baseline spécifique).

## Non-goals

- Pas de re-ingestion / re-chunk du corpus prod existant : le nouveau tokenizer ne s'applique qu'aux nouvelles ingestions. Le corpus déjà ingéré reste tel quel.
- Pas de tuning de `chunk_size` / `chunk_overlap` : les valeurs restent `512` / `64`. Seul le tokenizer qui les mesure change.
- Pas de parité de découpage avec l'ancien split BGE-M3 : le corpus n'étant ni validé en prod ni eval-testé, il n'existe pas de baseline BGE-M3 à protéger. Ce ticket part d'un découpage neuf sur le tokenizer Mistral et n'asserte aucune tolérance vs l'ancien split.
- Pas de suppression du fetch réseau côté **embedder** : `langchain-mistralai` (`MistralAIEmbeddings`) tente toujours, à l'init de l'embedder, un fetch du `tokenizer.json` Mixtral pour le batch-sizing (chemin léger, gated → fallback `DummyTokenizer` len-based). C'est un chemin séparé du chunker, hors-scope de ce ticket. Ce ticket ne garantit pas un ingest 100 % network-free — il retire uniquement le driver OOM `transformers`.
- Pas de migration vers un splitter character-based : le splitter reste token-based (`RecursiveCharacterTextSplitter` mesuré par tokenizer), seul le tokenizer change.
- Pas de ré-écriture de l'embedder (`embedder.py`) : il est déjà `mistral-embed` (INFRA-002), intouché ici.

## Supersedes

- **ING-001 « Decisions résolues » ligne 108** : « Tokenizer chunk_size : bge-m3 `AutoTokenizer.from_pretrained('BAAI/bge-m3')` confirmé » → caduque, remplacée par la décision tokenizer Mixtral-8x7B-v0.1 vendoré de ce ticket.
- (Pour mémoire, hors-périmètre de cette supersession) ING-001 AC-8/AC-9 (embedding bge-m3 sentence-transformers) étaient déjà superseded par INFRA-002 (swap vers `mistral-embed`), pas par ce ticket.
- L'édition effective de `specs/acceptance/ING-001.md` (source de vérité humain-only) est une étape humaine séparée. Ce ticket ne fait qu'enregistrer la supersession ; il ne modifie pas ING-001.md.

## Pre-conditions

- INFRA-002 mergé : embedder runtime = `mistral-embed`, BGE-M3 self-host embedder retiré, blocker `docs/blockers.md` 2026-05-18 ouvert.
- `langchain-mistralai` présent dans `[project.dependencies]` (apporte transitivement la lib `tokenizers` Rust).
- Artefact `tokenizer.json` Mixtral-8x7B-v0.1 (Apache-2.0, redistribution propre) obtenu hors-bande et committé dans le repo pour vendoring / baking au build.

## Failure modes

- Artefact `tokenizer.json` vendoré absent / chemin introuvable à l'init du chunker → `RuntimeError` explicite (`tokenizer artifact not found at <path>`) au boot ingest, exit non-zéro, aucune ingestion. (Pas de fallback réseau silencieux — cohérent `.claude/rules/no-workaround.md` et `secret-hygiene` no-network-fallback.)
- Artefact `tokenizer.json` corrompu / illisible par la lib `tokenizers` → erreur de chargement remontée explicitement au boot, exit non-zéro, aucune ingestion.
- Gate CI Ragas standing (AC-8) en échec après le swap → CI eval gate échoue, PR bloquée (le gate existant reste autorité ; pas de nouveau seuil ticket-spécifique).
- Gain image < 50 MB (AC-7) → la mesure avant/après ne passe pas le gate ; investigation requise (le ticket n'a pas atteint son objectif mémoire).

## Performance / SLO

- Objectif premier : faire tenir le Cloud Run ingest Job sous 512Mi (root cause OOM = import `transformers`). Pas de SLO latence dur ; le succès = Job qui ne reçoit plus signal 9 à l'ingestion.
- Init chunker network-free → latence d'init déterministe (pas de fetch HF au boot ni au build).

## Security / trust boundary

- Artefact tokenizer vendoré = donnée publique (tokenizer.json), pas un secret. Doit néanmoins passer `gitleaks` / scan (cohérent `secret-hygiene.md`) — c'est un JSON de vocabulaire, aucune entropie de secret attendue.
- Aucun fetch réseau ajouté à l'init du chunker (AC-3) — cohérent threat-model W-E-1 (pas de lecture d'URL distante pendant l'ingestion).
- Pas de `os.getenv(..., default)` ni fallback réseau silencieux (cohérent no-workaround + secret-hygiene).

## Observability

- Pas de nouvelle métrique / log event requis par ce ticket au-delà des events `ingest.*` déjà figés en ING-001 (décision humaine : YAGNI, pas de log de source tokenizer).
