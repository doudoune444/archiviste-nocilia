# Plan — GEN-002 Forwarder gateway `POST /v1/chat` → workers `/v1/generate`

## Goal
Poser un handler Axum minimaliste qui valide le body client, génère un `request_id`, forwarde vers workers `/v1/generate` avec timeouts différenciés, et retourne soit le passthrough byte-pour-byte soit une enveloppe d'erreur uniforme — tout en émettant un log JSON structuré par requête.

## Acceptance criteria recap
Voir `specs/acceptance/GEN-002.md` AC-1..AC-16 (verbatim, non recopié — limite ≤ 100 lignes).

## Risks / open questions
- **R1 (LOC)** : prod ≈ 130 (chat.rs) + 7 (lib/mod) + 10 (state) + 6 (config) ≈ **~155 LOC prod**. Tests ~330 (hors comptage). OK ≤ 300.
- **R2 (RequestBodyLimitLayer + enveloppe erreur)** : `tower-http` retourne 413 brut, pas notre enveloppe `{"error":"invalid_request",...}`. Solution : ajouter un `HandleErrorLayer` ou intercepter via extractor custom. À implémenter en step 11 ; si l'extractor `Json` rejette avec 413/400 avant le handler, on perd `request_id` (généré dans handler). **Décision** : générer `request_id` dans une layer/middleware en amont, le poser sur les extensions de request, et l'utiliser depuis le handler ET l'error-handler. ~15 LOC supplémentaires possibles.
- **R3 (timeout test AC-8)** : 35s en CI = inacceptable. Override via `Config.request_timeout_ms=200` en test. Production garde 35000. Documenter dans le rustdoc du champ.
- **R4 (proptest invariant non listé `specs/properties.md`)** : la table est humain-only. Le property test reste local (commenté `INV-GATEWAY-ERROR-ENVELOPE`, sans modifier le fichier). Si reviewer demande l'ajout au registre central → ticket suivi.
- **R5 (passthrough byte-pour-byte AC-1 vs cap 256 KiB AC-15)** : utiliser `response.bytes()` (pas `.text()` pour préserver bytes). Construire la réponse Axum via `Response::builder().header(content-type, "application/json; charset=utf-8").body(Body::from(bytes))` plutôt que `Json(value)` — évite re-sérialisation et préserve byte-équivalence.
- **R6 (header `X-Request-Id` côté workers)** : workers GEN-001 lit `request_id` depuis le **body**, pas le header. Le header `X-Request-Id` sortant (AC-4) est purement observationnel pour debug réseau. Confirmer : OK avec spec (AC-4 décrit le header sortant, pas ce que workers en fait).

## Decisions résolues (humain, 2026-05-07)

- **R2** — Middleware `request_id` en amont de `RequestBodyLimitLayer` : génère UUIDv4, pose sur `request.extensions_mut()`, lu par handler ET `HandleErrorLayer`. AC-7 body > 1 MiB → enveloppe `invalid_request` complète avec `request_id`. +15 LOC budgétés.
- **R3** — `Config.connect_timeout_ms` (défaut 5000) + `Config.request_timeout_ms` (défaut 35000) lus depuis env optionnels. Test AC-8 override via `request_timeout_ms=200`. Rustdoc documente l'usage test-only.
- **R4** — Property test AC-12 reste local au gateway, ID interne `INV-GATEWAY-ERROR-ENVELOPE`. Pas de modification `specs/properties.md` (humain-only). Reviewer notifié dans body PR si besoin promotion au registre central.
- **R5** — Réponse 200 construite via `Response::builder().header(content-type).body(Body::from(bytes))`. Pas de `Json(value)` (re-sérialisation casse byte-équivalence AC-1).
- **R6** — Header `X-Request-Id` sortant vers workers = observationnel (debug réseau). Workers GEN-001 lit `request_id` depuis body uniquement. Conforme AC-4.
