# ING-012 — Workflow GitHub Actions `gdrive-sync.yml` + auto-PR

## Contexte

ING-010 livre le noyau de sync Drive → `lore/*.md` invoqué manuellement (CLI). ING-011 ajoute les converters gsheet/gslide. Ce ticket automatise l'invocation : un workflow GitHub Actions hebdomadaire (cron lundi 03:00 UTC) + déclenchable manuellement, qui exécute le script et propose le diff sous forme de PR auto vers `main`. Sans ce ticket, l'auteur doit lancer le script à la main et ouvrir la PR ; le but est de réduire à zéro la friction de synchronisation.

## Critères d'acceptation

- AC-1 : Un workflow GitHub Actions `.github/workflows/gdrive-sync.yml` est livré avec deux triggers : `workflow_dispatch` (manuel, depuis l'onglet Actions ou `gh workflow run gdrive-sync.yml`) et `schedule: cron('0 3 * * 1')` (lundi 03:00 UTC, hebdomadaire).
- AC-2 : Le workflow s'exécute sur `runs-on: ubuntu-24.04` (pinné, pas `ubuntu-latest`), avec un seul job `sync` ; `permissions` minimales déclarées explicitement : `contents: write` (pour la branche), `pull-requests: write` (pour la PR), `actions: read`. Tous les autres permissions à `none`.
- AC-3 : Le workflow checkout le repo via `actions/checkout@<sha-pinned>` (commit SHA, pas tag flottant) avec `fetch-depth: 0` pour permettre `git mv` historique-aware (cf ING-010 AC-9). La version d'`actions/checkout` pinnée est documentée dans le commentaire YAML adjacent (numéro de release lisible).
- AC-4 : Le workflow installe `uv` via `astral-sh/setup-uv@<sha-pinned>` puis exécute `uv sync` (ou équivalent) pour résoudre les dépendances de `scripts/`.
- AC-5 : Le workflow exécute `uv run python scripts/gdrive_export.py --root-folder-id "${{ vars.GDRIVE_ROOT_FOLDER_ID }}"` avec la variable d'env `GDRIVE_SA_KEY_JSON` injectée depuis `secrets.GDRIVE_SA_KEY` au scope step uniquement (pas job, pas workflow). Le secret n'est jamais persisté en filesystem du runner.
- AC-6 : Si le diff git (`git status --porcelain` post-run sur `lore/` et `scripts/.gdrive_state.json`) est vide, le workflow termine en succès sans créer de PR (log step `no changes to sync`).
- AC-7 : Si le diff est non-vide, le workflow ouvre une PR via `peter-evans/create-pull-request@<sha-pinned-de-v6>` (commit SHA correspondant à la release v6.x, documenté en commentaire YAML adjacent), ciblant `main`, avec :
  - branche source : `chore/gdrive-sync-${{ github.run_id }}` (unique par run, jamais de collision).
  - title : `chore(lore): gdrive sync ${{ steps.date.outputs.iso }}` (date ISO `YYYY-MM-DD`).
  - body : contient le summary JSON du run (cf ING-010 AC-17), formatté en bloc code Markdown ; suivi d'un checklist humain « [ ] Reviewer skim diff », « [ ] Trigger ingestion `python -m archiviste_workers.ingest --path lore/` after merge ».
  - labels : `gdrive-sync`, `automation`.
  - commit message : `chore(lore): gdrive sync run ${{ github.run_id }}`.
- AC-8 : La PR est créée en **draft** si `errors >= 1` dans le summary du run, en **ready-for-review** si `errors == 0`.
- AC-9 : Le workflow upload un artifact `gdrive-sync-summary.json` (rétention 30 jours) contenant le summary JSON capturé sur stdout du run (un seul objet JSON, pas la totalité des per-file logs).
- AC-10 : Le workflow ne pousse jamais directement sur `main` ni sur une branche existante autre que `chore/gdrive-sync-<run_id>` ; aucun `force push` ; jamais de `merge --auto` ni `gh pr merge`.
- AC-11 : Le workflow définit `concurrency: { group: gdrive-sync, cancel-in-progress: false }` : deux runs simultanés (manuel + cron coïncident) sont sérialisés, jamais annulés.
- AC-12 : Le workflow définit un timeout au job (`timeout-minutes: 30`) ; tout run au-delà fail explicitement.
- AC-13 : Si `secrets.GDRIVE_SA_KEY` est absent (cas typique d'un fork ou config initiale), le workflow fail-fast en première step avec message explicite `secret GDRIVE_SA_KEY missing — see docs/runbook.md`, sans tenter le checkout ni l'install.
- AC-14 : Le workflow est validé localement par `actionlint` (pre-commit ou CI lint séparé) ; aucune warning `actionlint` non justifiée par `# actionlint-disable` commenté.
- AC-15 : Le repo `vars.GDRIVE_ROOT_FOLDER_ID` est documenté dans `docs/runbook.md` comme variable de repo (pas secret) à provisionner manuellement avant le premier run.
- AC-16 : Une section dédiée de `docs/runbook.md` décrit la procédure d'onboarding complète : (a) création SA GCP avec scopes `drive.readonly` (+ `spreadsheets.readonly` + `presentations.readonly` si ING-011 mergé), (b) partage du dossier Drive racine en lecture, (c) ajout du secret `GDRIVE_SA_KEY` (clé JSON SA inline) au repo, (d) ajout de la variable `GDRIVE_ROOT_FOLDER_ID`, (e) déclenchement manuel du premier run via `gh workflow run gdrive-sync.yml` puis review de la PR auto.

## Non-goals

- Pas de modification du script `scripts/gdrive_export.py` lui-même (ING-010 et ING-011 le livrent ; ING-012 l'invoque).
- Pas de merge automatique de la PR (humain review et merge).
- Pas de notification Slack / Discord / email (ticket OBS-* futur).
- Pas de cron ajustable via input du `workflow_dispatch` (cron fixe lundi 03:00 UTC).
- Pas de pinning de la version Python via `actions/setup-python` séparé : `astral-sh/setup-uv` gère l'installation Python en accord avec `pyproject.toml`.
- Pas de matrix build (un seul OS, une seule version Python).
- Pas de re-run automatique sur échec (humain réagit).
- Pas d'invocation post-merge de `python -m archiviste_workers.ingest --path lore/` ; cette étape est manuelle (checklist PR) ou ticket d'orchestration futur.

## Pre-conditions

- ING-010 mergé : `scripts/gdrive_export.py` opérationnel en CLI.
- ING-011 idéalement mergé : converters gsheet/gslide disponibles. Si non-mergé, le workflow fonctionne tout de même (gdoc + png seulement) ; le run.book mentionne les scopes additionnels conditionnels à ING-011.
- Secret GitHub `GDRIVE_SA_KEY` provisionné dans le repo (clé JSON service account complète, single line escaped).
- Variable GitHub `GDRIVE_ROOT_FOLDER_ID` provisionnée (string, non-secret, ID de dossier Drive).
- `peter-evans/create-pull-request@v6` non bloqué par la politique GitHub Actions de l'organisation (à vérifier au boot du ticket ; si bloqué → blocker `no-workaround`, ne pas substituer par un autre action sans ADR).

## Failure modes

- Secret `GDRIVE_SA_KEY` absent → fail-fast step 1, message exact `secret GDRIVE_SA_KEY missing — see docs/runbook.md`, exit code job non-zéro (cf AC-13).
- Variable `GDRIVE_ROOT_FOLDER_ID` absente → le script fail-fast avec son propre message (ING-010 AC-15 / failure mode), workflow exit non-zéro.
- Script exit code 1 (erreurs sur fichiers individuels) → la PR est tout de même créée en **draft** avec le summary embarqué (AC-8) ; le workflow exit succès (la PR draft = signal humain).
- Script exit code != 0,1 (crash inattendu) → la step échoue, pas de PR créée, artifact summary manquant, workflow exit non-zéro.
- `peter-evans/create-pull-request` échoue (rate-limit GitHub API, conflit branche) → step échoue, workflow exit non-zéro, pas de PR partielle.
- Quota Drive API (cf ING-010 failure modes) → reflété dans summary, PR draft (AC-8).
- Timeout 30 minutes dépassé → job killed par GitHub, PR potentiellement non créée, artifact non upload (AC-12).
- Run cron déclenché alors qu'un run manuel est en cours → sérialisé via `concurrency` (AC-11), pas annulé.

## Performance / SLO

- Pas de SLO dur. Run hebdo, durée attendue < 10 min pour < 200 fichiers (baseline ING-010).
- Timeout job 30 min hard cap (AC-12).

## Security / trust boundary

- Cohérence threat-model + `.claude/rules/secret-hygiene.md` : `secrets.GDRIVE_SA_KEY` injecté en env step uniquement, jamais persisté filesystem, jamais loggué (GitHub Actions masque automatiquement).
- Permissions GITHUB_TOKEN minimales explicites (AC-2) : pas de `contents: read` global qui leak le repo, pas de `id-token: write` (pas d'OIDC requis ici).
- Actions tierces pinnées par commit SHA (AC-3, AC-4, AC-7) : protection contre supply-chain attack via tag re-pointing (cohérence `.claude/rules/security.md` A08 « Software & Data Integrity »).
- `concurrency` (AC-11) sans `cancel-in-progress: true` : évite race condition sur `.gdrive_state.json` (deux runs simultanés écrasant l'état).
- PR créée par `peter-evans/create-pull-request` utilise le `GITHUB_TOKEN` standard ; jamais de PAT humain ; jamais de merge auto (AC-10).
- `fetch-depth: 0` (AC-3) : nécessaire pour `git mv` historique-aware ; pas de leak de secret car repo public-readable de toute façon (et SA key reste hors checkout).
- Pas d'execution de code provenant de la PR créée (le workflow `gdrive-sync.yml` ne tourne pas sur `pull_request` triggers, donc pas de risk de pull_request_target compromission).

- Pas de métrique externe (pas de push vers Langfuse / OpenTelemetry depuis CI ; ticket OBS-* futur).
