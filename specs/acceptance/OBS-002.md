# OBS-002 — Widget santé service + GET /v1/status (probes gateway Postgres/GCS/workers)

## Contexte

La vision EPIC OBS ouvre une surface d'observabilité publique pour la vitrine portfolio ; OBS-001 a livré la page `/observability`, le header nav et le widget d'usage (`GET /v1/stats`). Cette page ne dit rien de la **disponibilité** du service : un visiteur ne peut pas voir d'un coup d'œil si le RAG est opérationnel ou si une dépendance est tombée. Ce ticket livre la première moitié de la slice santé (découpée en OBS-002 gateway + OBS-006 workers/Mistral) : un widget de santé sur `/observability` alimenté par un nouvel endpoint public anonyme `GET /v1/status` qui agrège l'état des **trois dépendances que la gateway atteint directement** — `postgres`, `gcs`, `workers` (joignabilité du service workers depuis la gateway) — en un statut global `ok|degraded` sanitisé.

## Acceptance criteria

- AC-1 : `GET /v1/status` retourne `200 OK`, `Content-Type: application/json; charset=utf-8`, body JSON de forme `{"status":"ok"|"degraded","dependencies":{"postgres":{...},"gcs":{...},"workers":{...}},"checked_at":"<rfc3339>"}` ; les **trois** clés `postgres`, `gcs`, `workers` sont **toujours** présentes sous `dependencies`, et **aucune autre clé de dépendance** (en particulier pas de `mistral`) n'est présente (décision D-1, forme par-dépendance nommée à 3 deps).
- AC-2 : Chaque entrée de `dependencies.<dep>` a exactement la forme `{"status":"ok"|"down","latency_ms":<int ≥ 0>}` ; `status` est un **enum fermé** à deux valeurs (`"ok"` ou `"down"`) et ne contient jamais de message, host, URL, nom de base ni erreur SQL/HTTP (décision D-5, sanitisation). Aucune autre clé que `status` et `latency_ms` n'est présente par dépendance.
- AC-3 : `status` agrégé (racine) vaut `"degraded"` si **au moins une** des trois dépendances a `status="down"`, et `"ok"` si et seulement si les trois dépendances ont `status="ok"` (décision D-1, agrégat). Propriété : `status_racine == "ok"  ⇔  ∀ dep ∈ {postgres,gcs,workers}, dependencies[dep].status == "ok"`.
- AC-4 : `checked_at` est un timestamp RFC3339/ISO-8601 UTC correspondant au moment de production du snapshot (instant de la mesure servie, pas l'instant de la requête HTTP si servie depuis le cache, cf AC-9).
- AC-5 : `GET /v1/status` retourne **toujours `200 OK`** même quand `status="degraded"` (le statut vit dans le body, cohérent `GET /healthz` et OBS-001) ; aucun code `5xx` n'est retourné tant que le handler peut produire un snapshot (décision D-4).
- AC-6 : `GET /v1/status` ne retourne `503 Service Unavailable` **que** dans le cas où le handler ne peut produire aucun snapshot du tout (ni frais, ni en cache) ; ce body d'erreur est sanitisé `{"error":"upstream_unavailable","request_id":"<uuid>"}` et ne contient aucun détail interne (chaîne `error` figée à `"upstream_unavailable"`, cohérent OBS-001 AC-7, ex-OQ-2).
- AC-7 : `GET /v1/status` est servi sans authentification (anonyme) ; il ne porte aucun guard `author_only`/`member_only` et n'apparaît pas dans le sous-router auth. Un test contractuel (inspection router) vérifie l'absence de guard d'auth sur ce handler (cohérent OBS-001 AC-6).
- AC-8 : Chaque probe de dépendance applique un timeout dur de **2 secondes** ; une dépendance qui n'a pas répondu `ok` dans ce délai est exposée `status="down"` (un timeout est un `down`, décision D-4). `latency_ms` d'une dépendance en timeout reflète au plus ~2000 ms (le budget de timeout écoulé), jamais une valeur supérieure non bornée.
- AC-9 : Les trois probes sont exécutées **en parallèle** : la latence de bout-en-bout de production d'un snapshot frais est bornée par le timeout d'une seule probe (~2 s worst-case), pas par leur somme (~6 s). Mesurable : quand les trois dépendances sont down (toutes en timeout), le temps de calcul d'un snapshot frais reste < 3 s (marge sur le cap 2 s), pas ~6 s.
- AC-10 : Le résultat d'un snapshot est mis en cache **en mémoire, par process**, avec une TTL de **10 secondes** (décision D-3) ; deux appels `GET /v1/status` espacés de moins de 10 s servent le même snapshot (même `checked_at`) sans relancer les probes. Sur déploiement multi-réplica Cloud Run, le cache est par-instance (un visiteur peut voir des `checked_at` distincts selon l'instance servante) — acceptable phase 1 (décision D-3).
- AC-11 : Une dépendance dont la probe échoue (erreur de connexion, statut HTTP non-`2xx`, timeout, réponse malformée) est exposée `status="down"` ; le détail de l'échec (message, host, code interne, erreur SQL/HTTP) n'apparaît **jamais** dans le body et part uniquement dans un log `warn` interne (décision D-5, sanitisation ; cohérent `security.md` §A05/§A09).
- AC-12 : La probe `workers` reflète la **seule joignabilité du service workers depuis la gateway** : la gateway atteint un endpoint de liveness/healthz workers déjà existant (pas de nouvelle route workers dans ce ticket) ; si l'endpoint workers répond `2xx` dans le budget timeout, `dependencies.workers.status="ok"`, sinon `status="down"`. La vue des workers sur leurs propres dépendances (Mistral, etc.) n'est PAS consommée ici — elle relève de OBS-006.
- AC-13 : Au chargement de `/observability`, `observability.js` appelle `GET /v1/status` (même-origine, `fetch` `credentials:'same-origin'`) et insère dans un point d'ancrage stable `<section id="health-widget">` un rendu en `textContent` (jamais `innerHTML`) affichant le statut global et l'état par dépendance ; aucune fuite d'erreur interne n'est rendue (seuls `ok`/`down`/`degraded` et les `latency_ms` sont affichés). Tout style passe par `observability.css` (aucun `style=` inline, CSP-safe, ex-OQ-3). Le libellé exact n'est pas figé byte-for-byte en AC (souplesse implémentation) ; seuls l'usage `textContent`, la présence du statut global et la présence des trois dépendances sont contractuels.
- AC-14 : Pour toute issue de `GET /v1/status` autre que `200 OK` (5xx, network drop, timeout, JSON malformé), `observability.js` insère dans `#health-widget` un message littéral d'indisponibilité (aligné sur le ton OBS-001 AC-10 `Données indisponibles.`) ; aucun branchement fin selon le code d'erreur.
- AC-15 : Les 5 en-têtes de sécurité globaux posés router-wide (UI-001 AC-6 / SEC-003 AC-1 : `Content-Security-Policy`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, `X-Frame-Options: DENY`, `Strict-Transport-Security`) sont présents byte-for-byte sur `GET /v1/status`. La valeur CSP n'est PAS modifiée par ce ticket.
- AC-16 : `specs/openapi/gateway-to-workers.yml` n'est **pas** modifié par ce ticket : la probe `workers` cible un endpoint workers pré-existant (liveness/healthz), aucune route workers nouvelle, aucun changement de contrat (le changement de contrat relève de OBS-006).
- AC-17 : Test manuel auteur (procédure documentée `## Test manuel auteur`) coché par l'humain avant `Status: ready`.

## Non-goals

- Pas de clé `mistral` dans le body `GET /v1/status` — ajoutée additivement par OBS-006 (le body passe de 3 à 4 deps). Ce ticket n'expose jamais un Mistral fake-down.
- Pas de route workers nouvelle ni de relais Mistral via les workers — OBS-006 (la probe `workers` ici = simple joignabilité).
- Pas de modification du contrat `gateway-to-workers.yml` — OBS-006.
- Pas de page santé séparée — le widget santé s'ajoute à `/observability` existante (OBS-001), pas de nouvelle route HTML.
- Pas de score « qualité réponse » ni panel qualité IA (faithfulness / answer_relevancy) — OBS-004.
- Pas de persistence des résultats Ragas — OBS-003.
- Pas d'historique de santé / série temporelle / graphe d'uptime — affichage d'un snapshot instantané phase 1.
- Pas de rafraîchissement auto / polling / SSE sur le widget santé — `GET /v1/status` appelé une fois au chargement (cohérent OBS-001).
- Pas d'alerting / notification (email, Slack, PagerDuty) sur `status="degraded"` — observabilité infra backend OBS-005.
- Pas de métrique OTel nouvelle ni d'uptime check GCP câblé — OBS-005.
- Pas de probe d'autres dépendances que les trois nommées (Postgres, GCS, workers-reachability) phase 1 — pas de Langfuse, pas de pgvector dédié, pas de Cloud Run lui-même.
- Pas de cache partagé multi-réplica (Redis / DB) — cache in-memory par-process suffit phase 1 (décision D-3).
- Pas de message d'erreur détaillé par dépendance dans le body — enum fermé `ok|down` uniquement (décision D-5).
- Pas de `503` sur `status="degraded"` — `200` always, statut dans le body (décision D-4).
- Pas de framework JS, pas de build step (ADR-0005 réaffirmé, cohérent OBS-001).
- Pas de rate-limit applicatif spécifique à `GET /v1/status` (couverture Cloudflare front V1 + SEC-002 V2, cohérent OBS-001).
- Pas de modification du widget usage OBS-001 (`#usage-widget`) ni de `GET /v1/stats`.

- GCS bucket `archiviste-conversations` accessible via la SA runtime (probe GCS AC-1).

## Failure modes

- Une dépendance probe en échec (connexion refusée, HTTP non-2xx, réponse malformée) → `dependencies.<dep>.status="down"`, agrégat `status="degraded"`, `200 OK`, log `warn` sanitisé ; aucun détail interne dans le body (AC-2, AC-11).
- Une dépendance probe en timeout (> 2 s) → `dependencies.<dep>.status="down"`, `latency_ms ≈ 2000` borné, `200 OK` (AC-8).
- Service workers injoignable → `dependencies.workers.status="down"`, agrégat `degraded`, `200 OK` (AC-12).
- Handler incapable de produire aucun snapshot (ni frais ni caché) → `503` body sanitisé `{"error":"upstream_unavailable","request_id":"<uuid>"}` (AC-6).
- Les trois dépendances down (toutes en timeout) → snapshot frais calculé en < 3 s grâce au parallélisme (AC-9), `status="degraded"`, `200 OK`.

## Test manuel auteur

Procédure exécutée par l'humain avant `Status: ready` (AC-17).

1. **Stack** : `docker compose up -d`, vérifier gateway + workers + postgres `healthy`.
2. **Nominal** : `curl -i http://localhost:8080/v1/status` → `200`, body `status="ok"`, 3 dépendances `ok` (`postgres`, `gcs`, `workers`), aucune clé `mistral`, `checked_at` RFC3339, `latency_ms` plausibles.
3. **Widget** : ouvrir `http://localhost:8080/observability` → `#health-widget` affiche le statut global + l'état des 3 dépendances. Network tab : `GET /v1/status` 200.
4. **Cache** : deux `curl` consécutifs < 10 s → `checked_at` identique. Attendre > 10 s → `checked_at` change (AC-10).
5. **Dépendance down** : `docker compose stop postgres` → `curl -i /v1/status` → `200`, `dependencies.postgres.status="down"`, agrégat `degraded` ; aucun détail SQL dans le body (AC-2, AC-5, AC-11). Recharger `/observability` → widget reflète `degraded`.
6. **Workers down** : `docker compose stop workers` → `dependencies.workers.status="down"`, agrégat `degraded` (AC-12). (Aucune clé `mistral` n'apparaît — relève de OBS-006.)
7. **Parallélisme** : les 3 dépendances arrêtées → mesurer le temps de réponse d'un snapshot frais (vider le cache / attendre TTL) → < 3 s, pas ~6 s (AC-9).
8. **En-têtes** : DevTools Network sur `GET /v1/status` → les 5 headers de sécurité présents (AC-15).
9. **Anonyme** : `curl -i http://localhost:8080/v1/status` (sans cookie) → `200` (AC-7).

Coller la checklist cochée des 9 étapes en commentaire de PR.

## Performance / SLO

- `GET /v1/status` servi depuis le cache (TTL 10 s) : < 50 ms p95 stack locale (pas de probe relancée).
- `GET /v1/status` calcul d'un snapshot frais, cas nominal (dépendances saines) : borné par la probe la plus lente, cible < 500 ms p95 ; worst-case (dépendance en timeout) ~2 s borné par le cap timeout (AC-8), jamais ~6 s grâce au parallélisme (AC-9).
- Timeout dur par probe : 2 s (AC-8), aligné `security.md` §A04 « timeouts sur tout appel externe ».

## Security / trust boundary

- Frontière Internet → Gateway, plus Gateway → Workers (appel interne vers endpoint workers pré-existant, AC-12).
- **A01 Broken Access Control** : `/v1/status` public anonyme par design (AC-7) ; n'expose aucune donnée par-utilisateur, uniquement un statut agrégé de santé. Pas d'IDOR (aucun identifiant de ressource en path/query).
- **A05 Misconfiguration** : valeurs par dépendance restreintes à l'enum `ok|down` (AC-2) ; aucun host, URL, message d'erreur, nom de base ni stack n'est exposé (AC-11). Erreur `503` enveloppe sanitisée `{"error":"upstream_unavailable","request_id":"<uuid>"}` (AC-6). CSP UI-001 strictement préservée (AC-15).
- **A04 Insecure Design** : timeout dur 2 s par probe (AC-8), pas de boucle de retry non bornée ; cache TTL 10 s limite l'amplification de charge sur les dépendances (un pic de trafic `/v1/status` ne déclenche pas un pic de probes 1:1).
- **A09 Logging & Monitoring** : log `tracing` structuré autorisé (`request_id`, `latency_ms`, statut agrégé, statut par dépendance `ok|down`) ; le détail d'échec d'une probe va en `warn` interne SANS exposer host/message brut dans des champs qui fuiteraient (cohérent §A09).
- **A10 SSRF** : la probe workers cible une URL workers **fixe issue de la configuration** (pas d'URL fournie par l'utilisateur) — pas de surface SSRF.
- **RAG-specific** : aucune donnée de conversation, aucun contenu, aucun `gcs_uri`, aucun `user_id` exposé — uniquement un statut de santé agrégé.

## Observability

- Log `tracing` structuré `info` sur `GET /v1/status` : champs `request_id`, `latency_ms`, `status` agrégé, statut par dépendance (`ok|down`), indicateur cache hit/miss.
- Log `tracing` `warn` par dépendance en échec (`event=status.probe.down`, champ `dep`) — sans message/host brut dans le body, détail interne autorisé en log seulement.
- Pas de log applicatif sur le chemin statique `/observability` (TraceLayer existante suffit, OBS-001 AC).
- Pas de métrique OTel nouvelle ni d'alerte câblée phase 1 (OBS-005).
- Pas de tracing Langfuse (hors pipeline LLM).

## Décisions

Arbitrages humains figés (round-1, portés depuis OBS-002 + résolution OQ-2/OQ-3). Chaque décision verrouille une dimension du contrat.

- **D-1 — Forme par-dépendance NOMMÉE + agrégat, 3 deps.** Body `{"status":"ok"|"degraded","dependencies":{"postgres":..,"gcs":..,"workers":..},"checked_at":"<rfc3339>"}`. Agrégat = `degraded` si au moins une dépendance `down` (AC-1, AC-3). La clé `mistral` est ajoutée additivement par OBS-006 (la forme grandit de 3 à 4, jamais de fake-down dans l'intervalle).
- **D-3 — Cache TTL 10 s, in-memory par-process** (pattern `Arc` dans AppState, analogue au cache `TokenProvider`). Cache par-instance sur multi-réplica Cloud Run accepté phase 1 (AC-10).
- **D-4 — Timeout 2 s par probe, probes en PARALLÈLE, `200` always.** Worst-case ~2 s (pas 3×2 s) ; timeout ⇒ `down` ; `GET /v1/status` retourne `200` même `degraded` (statut dans le body, comme `healthz`) ; `503` réservé au cas « aucun snapshot productible » (AC-5, AC-6, AC-8, AC-9).
- **D-5 — Sanitisation : enum fermé `ok|down` par dépendance + `latency_ms` exposé.** Jamais de message/host/url/erreur SQL dans le body ; tout détail d'échec va en log `warn` interne. `latency_ms` exposé en plus — acceptable car ne fuite aucun détail interne, signal portfolio plus riche (AC-2, AC-11).
- **D-6 (ex-OQ-2) — Chaîne `error` du `503` = `"upstream_unavailable"`.** Cohérent OBS-001 AC-7 (la cause est une dépendance amont, le gateway lui-même fonctionne). Sanitisé : `{"error":"upstream_unavailable","request_id":"<uuid>"}` (AC-6).
- **D-7 (ex-OQ-3) — `#health-widget` = rendu texte simple, style via CSS.** Statut par dépendance rendu en `textContent`, toute mise en forme (couleur par statut) via `observability.css` (aucun `style=` inline, CSP-safe). Libellé exact non figé byte-for-byte ; usage `textContent` + présence du statut global + présence des 3 dépendances sont contractuels (AC-13).
