# ING-013 — Sync Google Drive : intégration API (auth, drive_client, orchestrator, CLI entrypoint)

## Contexte

ING-010 a livré les utilities pures (slugify, state, frontmatter_merge, paths, normalize, rename) sous `scripts/gdrive_export/`. Ce ticket complète la chaîne en ajoutant l'intégration Drive API v3 :

- auth service account (lecture creds via env vars),
- client Drive (list récursif, export gdoc, download PNG),
- orchestrateur sync (diff état ↔ Drive, dispatch créé/updated/renamed/archived/unchanged/error),
- entrypoint CLI `python -m gdrive_export --root-folder-id <id>` avec `--dry-run`,
- tests intégration end-to-end avec Drive API mockée (`pytest-httpserver`),
- runbook onboarding service account + dossier partagé,
- `.env.example` template.

Sans ce ticket, les utilities ING-010 n'ont pas de consommateur runtime.

## Critères d'acceptation

- AC-1 : La commande `cd scripts && uv run python -m gdrive_export --root-folder-id <id>` parcourt récursivement le dossier Drive racine fourni (et ses sous-dossiers) via l'API Drive v3 et liste tous les fichiers de MIME `application/vnd.google-apps.document` et `image/png` accessibles au service account.
- AC-2 : Chaque `gdoc` (`mimeType=application/vnd.google-apps.document`) est exporté via `files.export(mimeType='text/markdown')` puis écrit en `lore/<drive_path>/<doc_slug>.md` où `<drive_path>` reproduit l'arborescence Drive relative à la racine et `<doc_slug>` est le titre Drive slugifié (via `gdrive_export.slugify`).
- AC-3 : Chaque `image/png` natif Drive est téléchargé via `files.get_media(fileId=...)` et écrit en `lore/<drive_path>/<image_slug>.png` ; aucun `.md` n'est produit pour les PNG.
- AC-4 : Le frontmatter YAML écrit en tête de chaque `.md` exporté contient strictement les champs, **dans l'ordre figé suivant** : `title` (str = titre Drive), `source` (str = `gdrive`), `source_id` (str = `<drive_file_id>`), `drive_path` (str = chemin POSIX Drive relatif à la racine), `exported_at` (str ISO8601 UTC), `archived` (bool, défaut `false`), `archived_at` (str ISO8601 UTC ou absent), `tags` (list[str], défaut `[]`), `access_tier` (str, défaut `public`), puis les clés custom user-managed dans leur ordre d'insertion. La sérialisation YAML utilise `yaml.safe_dump(sort_keys=False)`. Le merge respecte les règles AC-6 d'ING-010.
- AC-5 : Le fichier d'état persistant `scripts/.gdrive_state.json` (JSON canonique trié, indent 2) maintient une map `source_id → StateEntry`. Lu au démarrage via `gdrive_export.state.load_state`, réécrit à la fin du run via `save_state`. Le fichier est **versionné dans git** : ING-013 retire `scripts/.gdrive_state.json` de `.gitignore`, commit initial avec contenu `{}`, et le diff inter-runs est inclus dans la PR auto-générée par ING-012. Les fichiers backup `*.bak` éventuels restent ignorés par `.gitignore`.
- AC-6 : Détection de **rename Drive** : si un `source_id` connu pointe vers un `local_path` différent du chemin courant calculé, le script invoque `gdrive_export.rename.rename_local_file(old, new)` (pure `Path.rename`, pas de subprocess) avant de réécrire le contenu. Identique pour gdoc et PNG. Log dédié `event=gdrive_sync.image_renamed` pour PNG, `event=gdrive_sync.file, status=renamed` pour gdoc.
- AC-7 : Les credentials du service account sont lus via les variables d'environnement `GDRIVE_SA_KEY_JSON` (contenu JSON inline) OU `GOOGLE_APPLICATION_CREDENTIALS` (path vers fichier) ; aucun fallback hardcodé, aucun chemin par défaut, fail-fast au boot avec message explicite si ni l'un ni l'autre n'est défini. Si **les deux** variables sont définies, `GDRIVE_SA_KEY_JSON` prend précédence et un log warning unique `{"event":"gdrive_sync.creds_conflict","resolved":"GDRIVE_SA_KEY_JSON"}` est émis au boot (pas fail-fast).
- AC-8 : Détection de **suppression Drive** : si un `source_id` est présent en état mais absent de la dernière liste Drive, le script ne supprime PAS le fichier local mais (a) pour un `.md` : pose/force `archived: true` et `archived_at: <ISO8601 UTC>` dans le frontmatter ; (b) pour un PNG : aucun marqueur, simple log `event=gdrive_sync.file, status=archived` ; le `source_id` reste en état avec `archived_at` figé. **Idempotence** : si l'entrée d'état porte déjà un `archived_at` non-null (déjà archivé sur un run précédent), le script n'ouvre pas le fichier, ne réécrit pas le frontmatter, ne touche pas au `mtime` ; le log per-file est émis avec `status=unchanged`.
- AC-9 : Détection de **collision path** : 2 fichiers Drive distincts produisant le même chemin local après slug + ext sont résolus via `gdrive_export.paths.resolve_local_path` (suffixe `-<drive_file_id[:8]>` au stem du second arrivant, ordre déterministe par tri lexical sur `drive_file_id`).
- AC-10 : Le contenu Markdown écrit (corps post-frontmatter) est passé à `gdrive_export.normalize.normalize_body` avant écriture (NFKC + strip control chars).
- AC-11 : Détection de **contenu inchangé** : le `StateEntry` persiste un champ `content_signature: str` formé d'un préfixe d'algorithme et d'un hex digest, calculé selon le type de fichier :
  - **gdoc** : `sha256:<hex>` du corps Markdown **post-`normalize_body`** (export Drive → NFKC + strip control chars → sha256). Le md5 natif Drive n'est pas utilisé pour les gdoc (l'export Markdown n'expose pas de md5 stable).
  - **PNG** : `md5:<hex>` = `md5Checksum` natif renvoyé par l'API Drive sur le fichier binaire.

  Comparaison `content_signature` calculée (run courant) vs valeur en état : identique ⇒ fichier local non réécrit (pas de churn `exported_at`), état non mis à jour pour ce fichier, status `unchanged`. **Note ING-010** : le champ `drive_md5_checksum: str` du `StateEntry` (ING-010 AC-5) doit être renommé `content_signature: str` ; un addendum à ING-010 AC-5 est requis et tracé en Open questions ci-dessous.
- AC-12 : Le script accepte un flag `--dry-run` qui exécute la résolution complète (list Drive, calcul des paths, diff vs état) sans écrire aucun fichier ni modifier l'état. Les logs JSON émis sont **identiques en structure** au run normal (`event=gdrive_sync.start`, `event=gdrive_sync.file`, `event=gdrive_sync.summary`) mais les valeurs `status` du per-file sont préfixées `would_` (`would_create | would_update | would_rename | would_archive | would_skip | would_unchanged | would_error`) et les compteurs du summary utilisent les mêmes préfixes (`would_created`, `would_updated`, `would_renamed`, `would_archived`, `would_unchanged`, `would_errors`). En dry-run, le **code de sortie est toujours `0`** indépendamment de `would_errors`.
- AC-13 : Au démarrage du run, le script émet **dans l'ordre** : (1) `{"event":"gdrive_sync.start","root_folder_id":"...","dry_run":<bool>,"state_size":<N>}`, puis (2) si et seulement si l'état est absent (premier run), `{"event":"gdrive_sync.first_run"}` ; dans ce cas `state_size=0` et l'état est traité comme `{}`.
- AC-14 : Tout fichier dont l'export rendu dépasse 1 MiB est skippé : log JSON `status: "error"`, `reason: "exported size exceeds 1 MiB cap"`, état non mis à jour, traitement des autres fichiers continué. Le cap est mesuré sur le **payload final post-traitement** : pour gdoc = corps Markdown **après `normalize_body`** (cohérence avec AC-11 qui hash le même payload) ; pour PNG = taille binaire téléchargée.
- AC-15 : Quota / rate-limit Drive API (429, 403 `userRateLimitExceeded`) ⇒ log error pour le fichier en cours, traitement continué, pas de retry phase 1, reflété dans `errors` du summary.
- AC-16 : Export Drive échoue avec 5xx ⇒ log error avec `reason: "drive_export_failed: <status>"`, fichier skippé, traitement continué, exit final `1`.
- AC-17 : À la fin du run, log JSON unique `{"event":"gdrive_sync.summary","total":N,"created":C,"updated":U,"renamed":R,"archived":A,"unchanged":S,"errors":E,"duration_ms":D}` est émis sur stdout ; chaque fichier traité produit un log `{"event":"gdrive_sync.file","source_id":"...","local_path":"...","status":"created|updated|renamed|archived|unchanged|error","reason":"<si error>"}`.
- AC-18 : Timeout HTTP par appel Drive fixé à 30s (cohérence security.md A04 « Timeouts on every external call »).
- AC-19 : Le code de sortie du process est `0` si `errors == 0`, `1` si `errors >= 1` (les autres fichiers ayant été traités), cohérent ING-001 AC-17.
- AC-20 : **Drive = source de vérité unique pour le corps Markdown.** L'état persiste un `body_hash` (sha256 hex via `gdrive_export.state.compute_body_hash`) du dernier corps généré côté script, **uniquement pour les fichiers `.md` (gdoc)**. Le champ est non-applicable / non-renseigné pour les PNG. Lors d'un ré-export d'un `.md` où Drive a changé (`content_signature` différent), si le `body_hash` calculé sur le fichier local courant diffère du `body_hash` en état (= l'humain a édité localement), le script logge un warning `event=gdrive_sync.local_drift_overwritten` (avec `local_path`, `source_id`) puis overwrite le corps quand même avec le contenu Drive. Aucun mode « préserve user edit body » n'est offert. Le merge de frontmatter (clés user-managed et custom) reste appliqué.
- AC-21 : `docs/runbook.md` contient une section « Sync Google Drive (ING-013) » : création SA, partage dossier, env vars, invocation manuelle, dry-run.
- AC-22 : `.env.example` racine contient (commentés) : `GDRIVE_SA_KEY_JSON=`, `GDRIVE_ROOT_FOLDER_ID=`, `GOOGLE_APPLICATION_CREDENTIALS=`. Aucune valeur réelle.

## Failure modes

- `GDRIVE_SA_KEY_JSON` / `GOOGLE_APPLICATION_CREDENTIALS` absents au boot → exit non-zéro immédiat, message `gdrive credentials missing: set GDRIVE_SA_KEY_JSON or GOOGLE_APPLICATION_CREDENTIALS`.
- `--root-folder-id` absent ou non résolvable (404 Drive) → exit non-zéro immédiat, message `root folder not found or inaccessible: <id>`.
- Service account non autorisé sur le dossier (403) → exit non-zéro immédiat.
- État `.gdrive_state.json` corrompu / non-JSON → `StateCorruptedError` propagé, exit non-zéro immédiat, message `state file corrupted: <path>`.
- Frontmatter local invalide → `FrontmatterMergeError` ⇒ log error, fichier skippé, exit final `1`.
- Slug vide → fallback `file-<id[:8]>` (cf ING-010 AC-2), traitement continué.
- Collision path → suffixe `-<id[:8]>` (cf ING-010 AC-7), traitement continué.

## Security / trust boundary

- Outil offline / dev (vision §54). Pas de surface réseau exposée.
- Service account scopé read-only, dossier unique partagé manuellement (least-privilege).
- Secret SA jamais commité : `.env.example` commente la variable, `.gitignore` couvre `*-sa.json` (déjà actif), hook `guard-secret-paths.sh`.
- Cohérence threat-model W-T-1 : normalisation NFKC + strip control chars (AC-10) + cap 1 MiB (AC-14).
- Path traversal : tous les paths produits sous `lore/`, composants slugifiés via ING-010 utilities (AC-7 paths.py asserte `is_relative_to(root)`).
- **Aucun `subprocess` dans le code de production** (override OQ-N) : élimine toute injection shell par construction.
- PNG binaires : pas de normalisation, mais cap 1 MiB ; pas de désérialisation, simple `open(..., 'wb').write(bytes)`.
- Lockfile `scripts/uv.lock` séparé du venv `workers/` (cf ADR-0006) → blast radius zéro runtime.

## Décisions résolues (round 2)

- **OQ-1 — conflit creds** : `GDRIVE_SA_KEY_JSON` précède `GOOGLE_APPLICATION_CREDENTIALS`. Log warning `gdrive_sync.creds_conflict`, pas fail-fast. (AC-7.)
- **OQ-2 — ordre clés frontmatter** : ordre figé `title, source, source_id, drive_path, exported_at, archived, archived_at, tags, access_tier, <custom>` ; `yaml.safe_dump(sort_keys=False)`. (AC-4.)
- **OQ-3 — content signature gdoc** : sha256 hex préfixé `sha256:` sur corps **post-`normalize_body`**, le champ `drive_md5_checksum` d'ING-010 AC-5 est renommé `content_signature`. (AC-11, AC-14.)
- **OQ-4 — dry-run sémantique** : statuts `would_*`, summary `would_*`, exit code toujours `0`. (AC-12.)
- **OQ-5 — versioning de state.json** : versionné en git, retiré de `.gitignore` en ING-013, commit initial `{}`, diff inclus dans PR auto ING-012, backup `.bak` reste ignoré. (AC-5.)
- **OQ-6 — idempotence archive** : si `archived_at` non-null en état, aucun toucher au fichier local ni au `mtime`. (AC-8.)
- **OQ-7 — ordre logs boot** : `start` puis `first_run`. (AC-13.)
- **OQ-8 — body_hash scope** : `.md` uniquement, non-applicable PNG. (AC-20.)
- **OQ-9 — cap 1 MiB** : appliqué post-`normalize_body` côté gdoc, cohérent avec AC-11. (AC-14.)
