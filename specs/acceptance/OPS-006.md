# OPS-006 — Canal corpus privé `gs://archiviste-lore-corpus` (gdrive-sync → bucket → ingest)

## Contexte

OPS-005 a livré le Cloud Run Job `archiviste-ingest` + le workflow qui le déclenche, mais le Job sort en exit `2` : l'image workers ne contient pas le corpus `lore/` (gardé hors du repo public par `.gitignore /lore/*`, cf `docs/runbook.md:452`, `docs/blockers.md:190`) et le CLI `find_repo_root` (`cli.py:59-65`) ne trouve aucune racine `.git/`. Le corpus narratif ne peut donc pas transiter par git public ; il lui faut un canal privé. Ce ticket ferme la boucle de bout en bout : gdrive-sync publie les `.md` dans le bucket privé `gs://archiviste-lore-corpus`, et le Job d'ingestion télécharge ce bucket vers un dossier local puis ingère ce dossier via une racine explicite, de sorte que tout changement Drive synchronisé finisse indexé dans pgvector sans étape manuelle.

## Critères d'acceptation

- AC-1 : Une ressource Terraform `google_storage_bucket` nommée `archiviste-lore-corpus` (région `europe-west9`, uniform bucket-level access, aucun ACL public) est définie dans `infra/terraform/`, conforme à `security.md` §RAG "Conversation persistence" (pas d'ACL public, accès SA scopé).
- AC-2 : Un binding IAM `roles/storage.objectAdmin` scopé sur le seul bucket `archiviste-lore-corpus` est accordé à l'identité GitHub Actions `gha-deploy` (celle utilisée par WIF dans `deploy.yml`), sans création d'un 3e service account ni clé JSON.
- AC-3 : Le service account runtime du Job `archiviste-ingest` (`archiviste-runtime`) dispose d'un accès lecture (`roles/storage.objectViewer`) scopé sur le seul bucket `archiviste-lore-corpus`, suffisant pour télécharger (list + read) les objets du bucket.
- AC-4 : Le workflow `gdrive-sync.yml` rsync les `.md` synchronisés vers `gs://archiviste-lore-corpus` ; après un run gdrive-sync ayant produit ≥ 1 `.md`, `gcloud storage ls gs://archiviste-lore-corpus/**` liste ces `.md`. Le rsync s'exécute **sans** `--delete-unmatched-destination-objects` (additif seulement). Ce step rsync **remplace** l'ouverture de la PR git `lore/` de gdrive-sync (le corpus ne transite plus que par le bucket).
- AC-5 : Le Job `archiviste-ingest`, avant d'invoquer l'ingesteur, télécharge tous les objets de `gs://archiviste-lore-corpus` vers un dossier local `<ingest_root>/lore/` (ex. `/tmp/lore/lore/`) via la lib Python `google-cloud-storage`, puis exécute le CLI avec `--root <ingest_root>` et `--path <ingest_root>/lore/` ; aucune racine `.git/` n'est requise et le Job ne sort plus en exit `2` `find_repo_root`.
- AC-6 : Le CLI `archiviste_workers.ingest` accepte un argument `--root <dir>` (chemin d'un dossier existant) qui remplace `find_repo_root` : la base de validation de `--path` et l'ancre de calcul de `source_path` deviennent `--root` au lieu du répertoire `.git/` walké. `find_repo_root` (`cli.py:59-65`) est retiré ; un `--path` hors de `--root` est toujours rejeté avec exit non-zéro et message `path must be relative to repo root` (contrat ING-001 AC-14 préservé, ancre redéfinie en `--root`).
- AC-7 : Sur un corpus bucket contenant ≥ 1 `.md`, le Job termine en exit `0` et émet le log JSON `ingest.summary` (ING-001 AC-16) avec `total >= 1`, récupérable dans Cloud Logging par execution ID.
- AC-8 : Le déclencheur du workflow `ingest-lore.yml` passe de `on: push: paths: ['lore/**']` à `on: workflow_run: { workflows: [gdrive-sync], types: [completed], branches: [main] }` avec garde `if: github.event.workflow_run.conclusion == 'success'`, en conservant `workflow_dispatch` ; de sorte que l'ingestion se déclenche après chaque run gdrive-sync réussi, et non plus sur un push `lore/**` (qui n'existe plus puisque le corpus ne transite plus par git).
- AC-9 : Sur un corpus bucket vide (0 `.md`), le Job termine en exit `0` (no-op idempotent) avec `ingest.summary` `total: 0` ; un bucket vide n'est pas un échec (aligné idempotence ING-001 AC-11).

## Non-goals

- Pas de suppression des documents orphelins dans le bucket ni dans pgvector (rsync additif, AC-4) : un `.md` supprimé côté Drive laisse un objet bucket et une ligne `documents` résiduels. Dette reportée, alignée OPS-005 Non-goal "documents orphelins".
- Pas de modification du pipeline ING-001 (parse / chunk / embed / upsert) — réutilisé tel quel. Seul le contrat CLI (`--root` vs `find_repo_root`) est amendé (cf Amendements).
- Pas de 3e service account ni de clé SA JSON : réutilisation de `gha-deploy` (WIF) et `archiviste-runtime` (Job).
- Pas de réinstallation de `git` dans `workers.Dockerfile` : le téléchargement passe par `google-cloud-storage` (déjà dans l'image, cf Pre-conditions) et l'ingestion n'a plus besoin de racine `.git/`.
- Pas de publication des fixtures `lore/sample/` dans le bucket : elles restent repo-only (versionnées, tests locaux). Le corpus bucket = uniquement le contenu Drive.
- Pas de schedule cron pour l'ingestion : déclenchement uniquement sur `workflow_run` gdrive-sync réussi + `workflow_dispatch` manuel.
- Pas de versionnement / lifecycle / rétention configurés sur le bucket au-delà des défauts GCS (ticket futur si besoin).
- Pas de chiffrement applicatif des `.md` dans le bucket (repose sur le chiffrement at-rest GCS par défaut, cf `security.md` §A02).
- Pas de re-chunking forcé sur changement de paramètres d'embedding (resté ING-001 Non-goal).
- Pas d'ingestion restreinte au diff : download complet + full scan `--path lore/`, idempotence assurée par le `content_hash` ING-001 (AC-11).
- Pas de chaînage avec `deploy.yml` : l'ingestion tourne sur `workers:latest` courant (cf OPS-005 Non-goals, cas limite documenté en commentaire workflow).

Côté Job, **aucun `gcloud` n'est requis** : le download passe par `google-cloud-storage`, **déjà présent** dans `workers/pyproject.toml` (`google-cloud-storage>=2.18`, ligne 30 ; override mypy déjà déclaré ligne 113) → zéro nouvelle dep, pas d'ADR.
- Image `workers:latest` poussée et contenant le module `archiviste_workers.ingest`.

## Failure modes

- Bucket `archiviste-lore-corpus` inexistant ou inaccessible côté gdrive-sync → step rsync gdrive-sync échoue → run gdrive-sync rouge → `ingest-lore` ne se déclenche pas (garde `conclusion == 'success'`).
- `archiviste-runtime` sans droit lecture sur le bucket → le download `google-cloud-storage` dans le Job lève une erreur d'autorisation → Job exit non-zéro → execution `Failed` → run `ingest-lore` rouge.
- `--root` pointant sur un dossier inexistant → exit non-zéro immédiat, message explicite, aucun fichier traité.
- `--path` hors de `--root` → exit non-zéro, message `path must be relative to repo root` (ING-001 AC-14, ancre `--root`).
- Corpus bucket vide (0 `.md`) → full scan sur 0 fichier → `ingest.summary` `total: 0`, exit `0` (AC-9, no-op idempotent).
- ≥ 1 `.md` en erreur ING-001 (frontmatter invalide, > 1 MiB) → Job exit `1` → execution `Failed` → run `ingest-lore` rouge (signal opérateur), les `.md` valides ayant été ingérés (OPS-005 AC-7).
- gdrive-sync réussit mais ne produit aucun nouveau `.md` (run "no changes") → `workflow_run` se déclenche quand même (conclusion `success`) → ingestion full scan sur le bucket inchangé → tous `unchanged`, exit `0` (idempotence ING-001 AC-11).

## Performance / SLO

- Pas de SLO dur. Le `duration_ms` de `ingest.summary` reste l'indicateur ; le download ajoute un coût marginal (corpus MVP ~10-50 fichiers). Re-traitement complet acceptable car UPSERT idempotent skippe les `unchanged` (ING-001 AC-11).

## Security / trust boundary

- Bucket `archiviste-lore-corpus` : uniform bucket-level access, aucun ACL public, conforme `security.md` §RAG "Conversation persistence" (transposé au corpus lore).
- Least-privilege : `gha-deploy` write (objectAdmin) scopé au seul bucket ; `archiviste-runtime` read-only (objectViewer) scopé au seul bucket. Aucune clé SA JSON, auth via WIF (gha) et identité Cloud Run (Job).
- Conforme `secret-hygiene.md` : aucun secret commité ; aucun contenu `.md` ni clé loggué (`ingest.summary` ne porte que des compteurs).
- Frontière : le corpus `.md` reste hors repo public (`.gitignore /lore/*`) ; il transite uniquement par le bucket privé. L'ingesteur normalise NFKC + cap 1 MiB (ING-001), pas d'entrée réseau arbitraire.
- Dossier de download `<ingest_root>/lore/` : éphémère, dans le filesystem du conteneur du Job, détruit à la fin de l'execution ; aucun risque de fuite du corpus dans une couche d'image versionnée.

## Observability

- Cloud Logging : logs JSON ING-001 (`ingest.start`, `ingest.document`, `ingest.summary`) capturés par execution du Job, corrélables par execution ID.
- GitHub Actions : run gdrive-sync vert/rouge (maillon a) + run `ingest-lore` vert/rouge (maillon b) = signaux de premier niveau.
- Pas de métrique OpenTelemetry ni d'alerte GCP dédiée ajoutée par ce ticket.
