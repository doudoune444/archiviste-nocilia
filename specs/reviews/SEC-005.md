## Security

Mapped against `.claude/rules/security.md`. No HIT.

- **Secrets in code**: none. Mock token literals (`test-sql-token`, `cached-token`, `fresh-token`, `stale-token`) in tests only — these are fixture strings, not real credentials.
- **Sensitive types**: bearer wrapped in `SecretString` in `auth_metadata/token.rs:77` (`bearer: SecretString`). Pool injection at `lib.rs:571` uses `expose_secret()` once at boot to feed `PgConnectOptions::password`. Background refresher at `lib.rs:78` exposes once per refresh to call `opts.password(...)`. Acceptable — both are immediately consumed by sqlx, not logged.
- **SQL injection**: no new SQL. Pool config only.
- **SSRF (A10)**: metadata URL is hardcoded `http://metadata.google.internal` (`token.rs:101` and `:112`). No user input flows in. Injectable base URL is gated to construction paths (`with_base_url`) used only by tests. Pre-existing pattern from SEC-004, unchanged threat surface.
- **JWT misuse**: not touched.
- **CORS / CSP / HSTS**: not touched.
- **Rate limit**: pool-init code path is boot-only, not a public route.
- **Unbounded input**: metadata response parsed via `serde_json` into a typed struct (`MetadataTokenResponse` with two fields); reqwest applies the 5 s total / 2 s connect timeout. No body cap, but the metadata server response is tiny and the timeout caps damage.
- **Trust boundary**: no header trust delegation introduced; gateway computes the SQL token itself.
- **Embedding poisoning / prompt injection / LLM output XSS / unsafe deserialization**: not in scope.

Pre-existing `docs/blockers.md` 2026-05-29 SEC-004 Windows-only `network` vs `timeout` test failure remains open — unrelated to this PR.
