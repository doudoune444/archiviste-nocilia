# UI-002 — Dashboard auteur tickets lore-gap (liste + signed URL conversation)

## Contexte

La vision §10 charge l'auteur de consommer les tickets lore-gap pour combler le canon ; §40 décrit la table `tickets`, §39 stipule que « le dashboard auteur ouvre le `.md` via signed URL », et l'ordre d'attaque #6 place ce ticket juste après GEN-004 (qui crée la table et alimente les tickets) et juste avant SEC-001 (qui livre le tier `author`). Sans cette UI les tickets s'accumulent en base sans surface auteur exploitable, et la boucle feedback corpus → enrichissement reste cassée. Ce ticket livre une page HTML+CSS+JS vanilla supplémentaire servie par la gateway (cohérent ADR-0005) qui liste les tickets `open` triés par priorité décroissante puis date, expose un lien vers le `.md` GCS de chaque conversation via une signed URL courte, et gate la page + ses endpoints sur le tier `author` (SEC-001 AC-16).

## Critères d'acceptation

- AC-1 : `GET /dashboard` retourne `200 OK` avec `Content-Type: text/html; charset=utf-8` quand le requester porte un JWT valide de tier `author` (cf. SEC-001 AC-11 marker `author_only` posé sur l'extractor `AuthUser`) ; le body contient le sélecteur stable `<table id="tickets-table"` (point d'ancrage des tests d'intégration et du test manuel auteur).
- AC-2 : `GET /dashboard` retourne `403 Forbidden` `{"error":"author_required","request_id":"<uuid>"}` quand le requester porte un JWT `member` OU est anonyme (cohérent SEC-001 AC-16, byte-for-byte le même body d'erreur).
- AC-3 : `GET /assets/dashboard.js` et `GET /assets/dashboard.css` retournent `200 OK` avec `Content-Type` respectif `application/javascript` ou `text/javascript` (UTF-8) et `text/css; charset=utf-8` ; ces deux assets sont publics (servis sans auth, cohérent UI-001 AC-4 qui sert déjà `/assets/*` sans gate — les fichiers ne contiennent aucun secret ni donnée auteur).
- AC-4 : Les assets statiques servis sous `/assets/*` après ce ticket sont exactement cinq : `app.js`, `styles.css` (UI-001), `dashboard.js`, `dashboard.css` (UI-002), `favicon.ico` toléré 404 (UI-001 non-goal inchangé). Toute autre route `/assets/<x>` retourne 404. Le test path-traversal `/assets/../*` reste rejeté 400/404 (UI-001 AC-5 inchangé).
- AC-5 : `GET /v1/tickets` accepte les query params optionnels `limit` (int, défaut 50, max 200) et `offset` (int, défaut 0, min 0) ; retourne `200 OK` avec body JSON `{"items":[{"id":"<uuid>","conversation_id":"<uuid>","question":"<string>","category":"<string>","priority_score":<int>,"status":"open","created_at":"<rfc3339>","updated_at":"<rfc3339>"}],"total":<int>,"limit":<int>,"offset":<int>}` quand le requester est `author`. La liste contient exactement les tickets `status='open'` triés par `priority_score DESC, created_at DESC` (clé secondaire stable cohérent vision §40 + GEN-004 AC-20 index `tickets_status_priority_idx`).
- AC-6 : `GET /v1/tickets` retourne `403 author_required` byte-for-byte AC-2 si le requester n'est pas `author` (même enveloppe d'erreur que AC-2).
- AC-7 : `GET /v1/tickets?limit=0` ou `limit=-1` ou `limit=201` ou `offset=-1` ou tout paramètre non-entier retourne `400 Bad Request` `{"error":"invalid_request","request_id":"<uuid>"}` (cohérent enveloppe GEN-002).
- AC-8 : `GET /v1/conversations/{id}/signed-url` accepte un `id` UUIDv4 dans le path ; retourne `200 OK` `{"signed_url":"<https-url>","expires_at":"<rfc3339>","conversation_id":"<uuid>"}` quand le requester est `author` ET la conversation existe (ligne `conversations` non-NULL avec ce `id`). La signed URL pointe vers l'objet GCS dont le path est `conversations.gcs_uri` de la ligne SQL correspondante.
- AC-9 : La signed URL retournée par AC-8 a une TTL exacte de **5 minutes** (`expires_at = now() + 300 secondes`, tolérance ±2 s sur l'horloge serveur). La méthode HTTP signée est `GET` exclusivement (lecture seule, conforme `security.md` §RAG-specific « conversation persistence : objectAdmin scope, signed read URL »).
- AC-10 : `GET /v1/conversations/{id}/signed-url` retourne `404 Not Found` `{"error":"conversation_not_found","request_id":"<uuid>"}` si aucune ligne `conversations` n'a cet `id`. Retourne `400 invalid_request` si `id` n'est pas un UUIDv4 valide. Retourne `403 author_required` (byte-for-byte AC-2) si le requester n'est pas `author`.
- AC-11 : Les trois routes nouvelles (`GET /dashboard`, `GET /v1/tickets`, `GET /v1/conversations/{id}/signed-url`) sont **toutes** gated `author_only` via l'extractor `AuthUser` posé par SEC-001 ; un test contractuel (grep AST ou inspection router) vérifie l'absence de marker `#[public]` sur ces handlers et la présence d'un guard `tier='author'`. Cohérent SEC-001 AC-11.
- AC-12 : Les headers de sécurité globaux posés par UI-001 AC-6 (CSP, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, `X-Frame-Options: DENY`) sont présents byte-for-byte sur `GET /dashboard`, `GET /assets/dashboard.{js,css}`, `GET /v1/tickets`, `GET /v1/conversations/{id}/signed-url`. La valeur CSP n'est PAS modifiée par ce ticket : `default-src 'self'; script-src 'self'; style-src 'self'; img-src 'self' data:; object-src 'none'; frame-ancestors 'none'; base-uri 'none'; form-action 'self'`. Les signed URLs GCS sont rendues comme `<a href="...">` mais ne sont jamais auto-chargées (ni `<iframe>` ni `<img src>` ni `<script src>`), donc aucune extension `connect-src` n'est requise.
- AC-13 : `gateway/static/dashboard.html` ne contient aucune balise `<script>` inline, aucune balise `<style>` inline, aucun attribut `style="..."`, aucun gestionnaire `on*=` inline (cohérent UI-001 AC-7) ; tout JS est chargé via `<script src="/assets/dashboard.js">`, tout CSS via `<link rel="stylesheet" href="/assets/dashboard.css">`. Test validable par grep statique.
- AC-14 : Au chargement, `dashboard.js` appelle `GET /v1/tickets` (même-origine, `fetch` avec `credentials: 'same-origin'` pour propager le cookie `archiviste_session` SEC-001 AC-4) ; insère dans `<tbody id="tickets-tbody">` une `<tr>` par ticket avec colonnes (dans cet ordre) : `priority_score`, `category`, `question` (rendue en `textContent`, **jamais `innerHTML`**), `created_at` (rendu RFC3339 ISO 8601 tel quel — pas de formatage localisé phase 1), un bouton `<button data-action="open-conversation" data-conversation-id="<uuid>">Ouvrir conversation</button>`.
- AC-15 : Un clic sur un bouton `<button data-action="open-conversation">` appelle `GET /v1/conversations/<uuid>/signed-url` ; à réception 200, `dashboard.js` ouvre la signed URL via `window.open(signed_url, '_blank', 'noopener,noreferrer')` (attributs `noopener` + `noreferrer` obligatoires pour empêcher la nouvelle fenêtre d'accéder au `window.opener` et de leaker le Referer vers GCS, cohérent `security.md` §A05 Misconfiguration).
- AC-16 : Pour toute réponse non-200 sur `GET /v1/tickets` ou `GET /v1/conversations/<id>/signed-url` (4xx, 5xx, network drop, timeout, JSON malformé), `dashboard.js` insère dans `<div id="dashboard-error" role="alert">` le message littéral `Erreur de chargement. Réessayez.` (cohérent ton UI-001 AC-13). Si la réponse contient un header `X-Request-Id` ou un champ `request_id` JSON, une `<small>req: <id></small>` est ajoutée au div d'erreur (cohérent UI-001 AC-14). Aucun branchement fin selon le code (`author_required`, `invalid_request`, `conversation_not_found`) phase 1.
- AC-17 : Si la liste retournée par `GET /v1/tickets` est vide (`items=[]`, `total=0`), `dashboard.js` insère dans `<tbody id="tickets-tbody">` une `<tr><td colspan="5">Aucun ticket ouvert.</td></tr>` (message littéral, pas de skeleton ni de spinner phase 1).
- AC-18 : Pagination phase 1 = "Charger plus" basique : si `total > offset + items.length`, un bouton `<button id="load-more">Charger plus</button>` est rendu sous le tableau ; un clic incrémente l'offset de `limit` (50) et concatène les résultats sans recharger les précédents. Atteindre `offset + items.length >= total` retire le bouton du DOM. Aucune URL-state, aucun historique navigateur, aucun deep-link `?offset=` phase 1.
- AC-19 : `gateway/src/lib.rs` route les 3 endpoints nouveaux + sert `/dashboard` via `tower-http::services::ServeFile` (cohérent UI-001 décision OQ-1) ; le middleware d'en-têtes de sécurité globaux (UI-001 AC-17) couvre les nouvelles routes sans modification (vérifié par AC-12). La limite de body 1 MiB de GEN-002 reste scopée au sous-router `/v1/chat`, sans régression sur les nouvelles routes GET (body absent).
- AC-20 : `gateway/src/handlers/tickets.rs` (ou module équivalent) lit la base Postgres directement via `sqlx::query!` paramétré contre la table `tickets` (cf migration `0002_schema.sql:53-63` + index `0003_tickets_embedding.sql` GEN-004 AC-20) ; aucun reverse-proxy vers les workers n'est introduit pour cette lecture (workers n'expose pas d'endpoint tickets — `tickets` est lu côté gateway comme tout autre artefact Postgres lu par la gateway dans les tickets futurs). La requête SQL est : `SELECT id, conversation_id, question, category, priority_score, status, created_at, updated_at FROM tickets WHERE status='open' ORDER BY priority_score DESC, created_at DESC LIMIT $1 OFFSET $2` + `SELECT count(*) FROM tickets WHERE status='open'` pour `total`.
- AC-21 : `gateway/src/handlers/conversations.rs` (ou module équivalent) signe l'URL GCS côté gateway via l'API V4 signed URL (`google-cloud-storage` crate ou équivalent Rust ; choix précis arbitré au plan) en utilisant la SA runtime `archiviste-runtime@` (vision §123 `storage.objectAdmin` bucket-scoped). La méthode signée est `GET`, la TTL `300 s` (AC-9). La clé de signature n'est jamais loggée (typage `secrecy::SecretString` côté loader si applicable).
- AC-22 : `specs/openapi/gateway-to-workers.yml` n'est **pas** modifié par ce ticket : aucune route workers nouvelle, aucun changement de contrat workers. Les nouveaux endpoints `/v1/tickets` et `/v1/conversations/{id}/signed-url` sont exclusivement gateway → client (pas gateway → workers) et ne participent donc pas à ce contrat (cohérent UI-001 non-goal qui exclut déjà ce yml du périmètre client↔gateway).
- AC-23 : Aucune ligne de log applicatif `INFO` n'est émise par la gateway pour `GET /dashboard`, `GET /assets/dashboard.{js,css}` (la `TraceLayer` existante suffit, cohérent UI-001 AC-18) ; en revanche, `GET /v1/tickets` et `GET /v1/conversations/{id}/signed-url` émettent une ligne `tracing` structurée niveau `info` `event=dashboard.tickets.list` ou `event=dashboard.conversation.signed_url` avec champs `request_id`, `user_id`, `tier="author"`, `latency_ms`, `count` (pour tickets) OU `conversation_id` (pour signed-url) — **jamais** `question`, `signed_url`, `gcs_uri` en clair (cohérent `security.md` §A09 et SEC-001 AC-18).
- AC-24 : Test manuel auteur (procédure documentée dans `## Test manuel auteur`) cochée par l'humain avant `Status: ready to ship` au sens UI-001 AC-19 (workflow §6 `CLAUDE.md`).

## Non-goals

- Pas d'édition / resolve / dismiss ticket workflow (`PATCH /v1/tickets/{id}` reporté V2, déjà non-goal explicite GEN-004).
- Pas de notification email / webhook / Slack sur nouveau ticket (vision Q20, V2).
- Pas de pagination cursor avancée (token-based, `?after=<ticket_id>`) — offset/limit basique suffit phase 1 (AC-5, AC-18).
- Pas de recherche / filtre / facettes côté UI (`?q=`, `?category=`, `?status=`) — V2.
- Pas d'export CSV / JSON depuis la UI — V2.
- Pas de mobile responsive avancé : CSS basique flex/grid, breakpoint unique ≥ 768px suffit phase 1.
- Pas de multi-author (vision §42 décrit `author` comme tier singulier ; SEC-001 AC-15 confirme seed unique). Phase 1 = 1 auteur exclusif.
- Pas de markdown rendering du champ `question` (rendu `textContent` plain, cohérent UI-001 AC-11/AC-12).
- Pas d'affichage de l'historique des changements du `priority_score` (pas de table d'audit dédiée).
- Pas de bulk action UI (sélection multiple → résoudre N tickets).
- Pas de classification fine de `category` côté UI (GEN-004 fige à `'uncategorized'`, ré-arbitré post-NLP).
- Pas de Playwright / Puppeteer / browser headless tests phase 1 (couverture tests d'intégration Rust + test manuel auteur, cohérent UI-001).
- Pas de framework JS (React/Vue/htmx/Alpine) — ADR-0005 §Decision réaffirmé.
- Pas de build step / bundler / transpiler / source maps — édition fichier → refresh navigateur, cohérent ADR-0005.
- Pas de dépendance CDN externe (CSP `script-src 'self'` interdit, AC-12).
- Pas de i18n / pas de switch FR/EN — UI FR uniquement (cohérent UI-001).
- Pas d'intégration analytics produit phase 1.
- Pas de rate-limit applicatif spécifique aux nouvelles routes (Cloudflare front V1 + SEC-002 V2 couvrent l'ensemble).
- Pas de cache HTTP des signed URLs côté client ni serveur — chaque clic regénère une URL TTL 5 min (AC-9).
- Pas de support pour les conversations non-author_only (vision §10 — l'auteur a accès à toutes les conversations, pas de filtre par tier de doc lore ; vision §41 ACL `access_tier` ne s'applique qu'aux docs lore, pas aux conversations).
- Pas de page de profil auteur, pas de stats agrégées (`nb tickets open`, `priority_score moyen`) phase 1 — V2.
- Pas d'écran d'erreur dédié pour `403 author_required` côté UI : si un member tente `GET /dashboard`, le navigateur affiche le body JSON brut (acceptable car aucun member légitime n'est censé connaître l'URL `/dashboard` phase 1).

## Pre-conditions

- **GEN-004 mergé** : table `tickets` alimentée par le pipeline lore_gap ; migration `0003_tickets_embedding.sql` appliquée incluant index `tickets_status_priority_idx` ON `(status, priority_score DESC, updated_at DESC)` qui sert le tri d'AC-5 (note divergence mineure : l'index porte sur `updated_at` en clé tertiaire alors qu'AC-5 demande `created_at` en clé secondaire ; à `priority_score` égal, l'ordre `created_at DESC` est garanti car phase 1 `created_at ≈ updated_at` sur tickets nouvellement créés et l'optimiseur Postgres peut servir le SELECT ; documenté en Decisions D-3 ci-dessous).
- **SEC-001 mergé** : extractor `AuthUser` opérationnel, JWT EdDSA vérifié, sessions server-side, marker `author_only` posé sur les handlers (SEC-001 AC-11, AC-16). Au moins un user `tier='author'` seedé en base via `scripts/seed_author.sh` (SEC-001 D-5).
- **INFRA-002 mergé** : SA runtime `archiviste-runtime@` avec `storage.objectAdmin` sur bucket `archiviste-conversations` (vision §123) ; le runtime peut signer des URLs V4 GCS via la méthode d'impersonation IAM ou via clé privée SA (choix arbitré au `/plan`, cohérent `secret-hygiene.md` : aucune clé JSON ne doit être commitée).
- **UI-001 mergé** : `tower-http::services::ServeDir`/`ServeFile` opérationnel, middleware d'en-têtes de sécurité globaux (CSP + 3 autres) posé router-level (UI-001 AC-17), répertoire `gateway/static/` existant, route `/` + `/assets/*` opérationnelles.
- **ING-003 mergé** : la colonne `conversations.gcs_uri` est renseignée pour toute conversation persistée ; les conversations alimentées par GEN-004 (`mode='lore_gap'`) sont donc accessibles via ce `gcs_uri`.
- `gateway/Cargo.toml` peut accueillir une dépendance pour la signature GCS V4 (`google-cloud-storage` crate, ou `gcp_auth` + signature manuelle, ou crate équivalente). **Note** : si la dépendance dépasse le seuil "heavy dep" du `CLAUDE.md`, un ADR `docs/adr/NNNN-gcs-signed-url-rust.md` doit être ajouté en pré-condition (arbitré au `/plan` ; surface signature V4 est petite, possible sans crate lourde via `ring` ou `openssl` déjà transitifs).
- `cargo sqlx prepare` exécuté après lecture des nouvelles requêtes SQL (`SELECT` sur `tickets`, `SELECT gcs_uri FROM conversations WHERE id=$1`) — rule `vertical-slice.md` §Order.

## Failure modes

- JWT `author` valide mais SA runtime ne peut pas signer (clé privée absente, IAM révoqué, Secret Manager indisponible) → `GET /v1/conversations/{id}/signed-url` retourne `503 Service Unavailable` `{"error":"upstream_unavailable","request_id":"<uuid>"}`, log `warn` `dashboard.signing_failed` avec `reason ∈ {key_missing, iam_denied, secret_manager_unavailable}` ; pas de fallback (cohérent `no-workaround.md` + `secret-hygiene.md` "no fallback default").
- Postgres indisponible au moment de `GET /v1/tickets` → `503 upstream_unavailable` (cohérent enveloppe GEN-002), latence ≤ 5 s timeout DB.
- Postgres indisponible au moment de `GET /v1/conversations/{id}/signed-url` lookup `gcs_uri` → `503 upstream_unavailable` même formulation.
- Conversation existe en base mais l'objet GCS sous `gcs_uri` est manquant (incident GCS, lifecycle 30j vision §117 ayant expiré) → la signed URL est tout de même retournée 200 (gateway ne vérifie pas l'existence de l'objet à la signature) ; le clic auteur ouvre alors une page GCS `404 NoSuchKey`. **Arbitrage** : phase 1 acceptable car (a) coût d'un HEAD GCS additionnel par signed URL non justifié, (b) le lifecycle 30j est explicite vision §117 (Q12), (c) l'auteur comprend le contexte. Tracé en Decisions D-4.
- Ticket sans conversation associée (FK violation impossible vu `ON DELETE RESTRICT` migration `0002_schema.sql:55`) → cas inatteignable ; pas de défense supplémentaire.
- JWT `author` expiré entre le chargement de `/dashboard` et un clic ultérieur sur "Ouvrir conversation" → `GET /v1/conversations/{id}/signed-url` retourne `401 invalid_token` (SEC-001 AC-12) → `dashboard.js` affiche le message générique AC-16 ; l'auteur doit recharger `/dashboard` et se ré-authentifier (re-login = SEC-001).
- Limit/offset hors bornes (AC-7) → 400 `invalid_request`, pas de log applicatif côté server au-delà de la trace HTTP standard.
- Signed URL valide mais le réseau de l'auteur intercepte (proxy entreprise qui modifie HTTPS) → erreur navigateur, hors périmètre.
- Path traversal `GET /v1/conversations/../../etc/passwd/signed-url` → routing Axum rejette 404 avant même d'atteindre le handler (le `{id}` extractor ne match que des UUIDs après AC-10 validation). `GET /v1/conversations/<uuid>/../<other>/signed-url` également rejeté par routing.

## Test oracle

- AC-1 : intégration · `test_dashboard_returns_html_for_author` — fixture JWT `author` valide → `GET /dashboard`, assert 200 + `content-type` débute par `text/html` + body contient `<table id="tickets-table"`.
- AC-2 : intégration · `test_dashboard_403_for_member_and_anonymous` — 2 sous-cas : JWT `member` valide → 403 `author_required` byte-for-byte ; pas de JWT (anonyme) → 403 `author_required`.
- AC-3 : intégration · `test_dashboard_assets_served` — `GET /assets/dashboard.js` 200 JS, `GET /assets/dashboard.css` 200 CSS.
- AC-4 : intégration · `GET /assets/random.txt` → 404 ; `GET /assets/../Cargo.toml` → 400/404 (path traversal regression).
- AC-5 : intégration · DB fixture insère 3 tickets `open` priorités 5/3/3 (les deux `priority_score=3` avec `created_at` différents) + 1 ticket `resolved` priorité 99 + 1 ticket `dismissed` priorité 50. JWT `author` → `GET /v1/tickets` → 200, `items` length 3, ordre strict (priorité 5 d'abord ; entre les deux 3 le plus récent `created_at` d'abord), `total=3`, `limit=50`, `offset=0`. Les tickets `resolved`/`dismissed` ne figurent jamais.
- AC-6 : intégration · `test_tickets_403_for_non_author` — JWT `member` → 403 `author_required` ; pas de JWT → 403.
- AC-7 : intégration · table de cas (`limit=0`, `limit=-1`, `limit=201`, `limit=abc`, `offset=-1`, `offset=foo`) → assertion 400 `invalid_request` pour chaque.
- AC-8 : intégration · DB fixture insère 1 conversation `gcs_uri='gs://archiviste-conversations/conv/<id>.md'` + JWT `author` → `GET /v1/conversations/<id>/signed-url` → 200, body contient `signed_url` (https URL commençant par `https://storage.googleapis.com/` ou `https://storage.cloud.google.com/`), `expires_at` RFC3339, `conversation_id` exact.
- AC-9 : unit + intégration · capture `expires_at` retourné, assert `expires_at - now() ∈ [298, 302]` secondes (tolérance ±2 s). Parse query string de `signed_url`, assert `X-Goog-Expires=300` (paramètre V4 standard). Assert `X-Goog-SignedHeaders` ne contient pas de méthode non-GET. Test unitaire du module `gateway/src/gcs/sign.rs` : `sign_get("conv/abc.md")` produit une URL dont la TTL est `SIGNED_URL_TTL_SECONDS=300` byte-for-byte.
- AC-10 : intégration · 3 sous-cas : (a) `GET /v1/conversations/00000000-0000-4000-8000-000000000000/signed-url` (UUID valide non existant) → 404 `conversation_not_found` ; (b) `GET /v1/conversations/not-a-uuid/signed-url` → 400 `invalid_request` ; (c) JWT `member` → 403 `author_required`.

## Performance / SLO

- `GET /dashboard` (HTML statique) : **< 50 ms p95** sur stack locale (fichier statique, pas de DB, pas d'upstream — cohérent UI-001).
- `GET /v1/tickets?limit=50` : **< 80 ms p95** (1 SELECT indexé sur `tickets_status_priority_idx` GEN-004 AC-20 + 1 SELECT count). Vision §92 budget `< 80 ms p95 à 500 users` global gateway respecté.
- `GET /v1/conversations/{id}/signed-url` : **< 150 ms p95** (1 SELECT `conversations.gcs_uri` indexé sur PK + 1 appel signature V4 GCS local OU 1 appel IAM `signBlob` ≤ 100 ms). Si signature via IAM impersonation et latence > 200 ms p95 mesurée post-deploy, ré-arbitrage au `/plan` vers signature locale via clé privée SA.
- Taille `dashboard.html + dashboard.js + dashboard.css` : **< 30 KiB** cumulé (cible souple, cohérent UI-001).
- Le rendu UI de 50 tickets : < 100 ms côté navigateur (insertion 50 `<tr>` via `textContent`, négligeable).

## Security / trust boundary

- Frontière Internet → Gateway. Surface : 1 nouvelle page HTML + 2 assets + 2 endpoints API. Tous gated `author_only` (AC-11) sauf les 2 assets statiques publics (AC-3, justifié : pas de secret dans le JS, pas de données auteur exposées par les fichiers).
- **A01 Broken Access Control** : 3 routes nouvelles déclarent explicitement `author_only` (AC-11, SEC-001 AC-11 marker). Aucune IDOR sur `GET /v1/conversations/{id}/signed-url` car (a) le tier `author` a accès à toutes les conversations par design (vision §10 « auteur consomme tickets lore-gap » + §39 « ouvre `.md` via signed URL » sans restriction par user_id), donc l'absence de check `conversations.user_id = author.id` est volontaire et documentée. Si SEC-001 introduit un jour un second auteur, ce raisonnement doit être ré-arbitré (cohérent SEC-001 D-1 split + author phase 1 = singulier).
- **A02 Cryptographic Failures** : signature V4 GCS = HMAC-SHA256 standard Google ; clé SA jamais loggée (cf AC-21) ; TLS 1.2+ enforced par GCS lui-même côté lien storage.googleapis.com. Si la signature locale exige une clé privée SA en JSON, elle est chargée depuis Secret Manager via env var et typée `secrecy::SecretString`, jamais dans le filesystem du container (cohérent `secret-hygiene.md`).
- **A03 Injection** : SQL via `sqlx::query!` macros (compile-checked) (AC-20). Pas de `format!` dans les SELECT. Le `{id}` du path est extrait via Axum extractor UUID-typé, pas de chaîne brute concaténée à la requête.
- **A04 Insecure Design** : `noopener,noreferrer` sur `window.open` (AC-15) empêche `window.opener` leak vers GCS et empêche le Referer leak du `/dashboard` vers `storage.googleapis.com`. Body size cap des routes : routes GET sans body, donc pas applicable.
- **A05 Misconfiguration** : CSP UI-001 AC-6 strictement préservée sur les nouvelles routes (AC-12). Aucun `connect-src` étendu (signed URL ouverte via `window.open`, pas via `fetch` depuis la page). Erreurs : enveloppe `{"error":"<code>","request_id":"<uuid>"}` cohérente, jamais de stack trace, jamais de `gcs_uri` brut en réponse 4xx/5xx.
- **A07 Authentication Failures** : tier check fait par l'extractor `AuthUser` SEC-001 ; l'expiration JWT 7j SEC-001 AC-5 vaut aussi pour ces routes ; pas de bypass.
- **A09 Logging & Monitoring** : `tracing` events nommés (AC-23). Interdits absolus dans les logs : `question` brute (PII auteur), `signed_url` complet (token signé), `gcs_uri` (path objet). Seuls `request_id`, `user_id`, `tier`, `latency_ms`, `count`, `conversation_id` autorisés. La métrique `auth_failures_total{event="idor_attempt"}` SEC-001 AC-19 est implicitement alimentée par les 403 sur ces routes (un member tentant `/dashboard` = `event=idor_attempt`).
- **A10 SSRF** : non applicable (aucun fetch d'URL utilisateur côté serveur — la signed URL est consommée par le navigateur client, pas par la gateway).
- **Conversation persistence** (`security.md` §RAG-specific) : signed URL en méthode GET stricte (AC-9), TTL court 5 min (AC-9), audience-restricted implicite via la clé SA `archiviste-runtime@`, bucket uniform bucket-level access vision §117. Pas d'écriture exposée. Pas de path-traversal sur `conversation_id` car UUID validé.
- **RAG-specific Output sanitization** : le `question` rendu côté UI passe par `textContent`, défense en profondeur avec CSP `script-src 'self'`. Toute future migration vers markdown rendering exigera un sanitizer (DOMPurify ou équivalent ; non requis phase 1).
- Pas de secret dans `gateway/static/dashboard.{html,js,css}` (rappel `secret-hygiene.md`).

## Decisions (auto-arbitrées vs vision.md + sources non-négociables, 2026-05-18)

- **D-1 — Split en 2 PRs, 1 spec UI-002 unique** : effort estimé ~430-530 LOC dépasse `vertical-slice.md` ≤ 300 LOC. Découpage proposé (architect tranche au `/plan`) :
  - **PR1 (UI-002a)** — Backend uniquement : `gateway/src/handlers/tickets.rs` + `gateway/src/handlers/conversations.rs` + `gateway/src/gcs/sign.rs` + tests intégration des 2 endpoints API. Pas de HTML/CSS/JS. Couvre AC-5 à AC-12 (sauf AC-1, AC-3, AC-13 à AC-18). ~250 LOC. **GEN-005 Mode 4 ACL n'est pas débloqué par cette PR** mais le pattern `author_only` est démontré.
  - **PR2 (UI-002b)** — Frontend : `gateway/static/dashboard.html` + `dashboard.js` + `dashboard.css` + route `GET /dashboard` gated + tests intégration page rendue. Consomme PR1 mergé. Couvre AC-1, AC-3, AC-4, AC-13 à AC-18. ~200 LOC.
  - Ordre imposé : PR1 → PR2. SEC-001 doit être mergé avant PR1.
- **D-2 — `tickets` lu directement par la gateway, pas reverse-proxy workers** : la vision §40 et GEN-004 ne créent pas d'endpoint workers `/v1/tickets`. La gateway lit déjà directement Postgres pour SEC-001 (sessions, users) ; étendre à `tickets` est cohérent SRP (`clean-code.md`). Workers ne deviennent pas une dépendance pour le dashboard auteur (pas d'appel cross-service runtime). Évite une route workers nouvelle (non listée dans `specs/openapi/gateway-to-workers.yml`) et évite une latence aller-retour superflue.
- **D-3 — Tri secondaire `created_at DESC` (AC-5) vs index `updated_at DESC` (GEN-004 AC-20)** : l'index GEN-004 a `(status, priority_score DESC, updated_at DESC)`. La spec UI-002 demande `created_at DESC` en clé secondaire. **Arbitrage conservateur** : phase 1, `created_at ≈ updated_at` sur la quasi-totalité des tickets (un ticket n'est `UPDATE`-é que sur dedup cosine ≥ 0.85 incrémentant `priority_score`, et son tri primaire `priority_score` change alors). L'optimiseur Postgres peut servir le SELECT via l'index sur les 2 premières clés + tri en mémoire sur la 3e (acceptable phase 1, cardinalités faibles). **Si performance dégradée post-deploy** (à vérifier OBS-001 V2), 2 options : (a) ajouter un index `(status, priority_score DESC, created_at DESC)` en ticket dédié, (b) amender AC-5 pour aligner sur `updated_at`. Pas de changement migration ce ticket — cohérent `vertical-slice.md` "out-of-scope refactor → new ticket".
- **D-4 — Signed URL sans HEAD GCS préalable** : pas de vérification d'existence de l'objet GCS avant signature (cf Failure modes). Coût +1 appel GCS = +50-100 ms p95 non justifié pour un cas-bord (lifecycle 30j vision §117). L'auteur comprend si GCS retourne 404 sur clic. À ré-arbitrer si retours UX.
- **D-5 — TTL 5 minutes byte-for-byte** : vision §39 ne fixe pas de TTL ; `security.md` §RAG-specific dit « signed URL = audience-restricted ». 5 minutes = fenêtre suffisante pour cliquer + lire sans rester signée trop longtemps (réduit fenêtre de leak si URL copiée). Constante exportée `SIGNED_URL_TTL_SECONDS = 300` (AC-9, AC-21), testable byte-for-byte.
- **D-6 — Pas de filtre `tickets.user_id = author.id`** : décision documentée Security §A01. Phase 1 author tier = singulier (SEC-001 D-5 seed unique). Aucun risque IDOR car aucun second author. À ré-arbitrer si phase 2 introduit multi-author.
- **D-7 — Pas d'extension CSP `connect-src`** : signed URL ouverte via `window.open` (AC-15) côté navigateur, pas via `fetch` depuis la page. Aucune origine externe atteinte depuis le JS de la page. CSP `default-src 'self'` reste suffisante (AC-12).
- **D-8 — `dashboard.js` ne consomme pas `archiviste.conversation_id` localStorage** : `conversation_id` localStorage est un état de UI-001 chat (cohérent vision §74). Le dashboard auteur n'écrit ni ne lit cette clé. Les `conversation_id` du dashboard viennent exclusivement des réponses API `/v1/tickets`. Cohérent UI-001 AC-8.
- **D-9 — Pagination "Charger plus" basique** : pas d'URL-state `?offset=` ni de deep-link (cohérent Non-goals). Phase 1 simplification volontaire ; multi-tab dashboard pas de cas d'usage auteur singulier.
- **D-10 — Crate signature GCS V4 vs implémentation manuelle** : décision déléguée au `/plan` architect. **Préférence** : implémentation manuelle via `ring` (déjà transitif via `rustls`) pour éviter une crate lourde (`google-cloud-storage` tire `tonic` + `prost` + ~100 deps transitives, justifiant un ADR au sens `CLAUDE.md`). Si implémentation manuelle dépasse 100 LOC, ajout d'une crate spécialisée (`gcp_auth` ou équivalent) acceptable avec ADR `docs/adr/NNNN-gcs-signed-url-rust.md` mergé en pré-requis PR1.
- **D-11 — Aucune modification de `specs/openapi/gateway-to-workers.yml`** : AC-22 explicit. Les nouveaux endpoints sont client↔gateway, hors périmètre de ce yml. Cohérent UI-001 non-goal "Pas d'amendement du contrat gateway↔workers".
- **D-12 — Pas d'ADR nouvelle obligatoire pour la page dashboard elle-même** : ADR-0005 §Triggers de réévaluation #1 mentionne « OPS-001 dashboard auteur déclenchera très probablement cette migration » vers un framework. **Arbitrage** : phase 1, le dashboard reste 1 page additionnelle vanilla (2 pages totales : `/` + `/dashboard`), JS reste < 500 LOC (ADR-0005 trigger #2), pas de nouveau composant réutilisable. Conditions de réévaluation ADR-0005 non encore franchies. À ré-arbitrer si UI-003 / OPS-001 ajoute une 3e page ou des composants partagés.
