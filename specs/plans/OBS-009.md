# Plan — OBS-009 Cloud Run Job d'eval live one-shot vers Cloud SQL prod

## Goal
Provisionner en Terraform un Cloud Run Job `archiviste-eval` qui, exécuté manuellement une fois, lance le runner Ragas live (OBS-008 image) contre les workers prod, persiste exactement une ligne `live` dans `eval_runs`, et fait passer `GET /v1/quality` de `no_data` à une mesure réelle.

## Acceptance criteria recap
- AC-1 : `gcloud run jobs execute archiviste-eval --region=<region>` se termine exit code `0` (run live complet jusqu'au persist inclus).
- AC-2 : `SELECT count(*) FROM eval_runs WHERE runner_mode='live'` augmente d'exactement 1.
- AC-3 : ligne insérée porte `runner_mode='live'` et `finished_at` non nul du run courant.
- AC-4 : `GET /v1/quality` retourne `200`, corps ≠ `no_data`, `finished_at` == celui de la ligne insérée.
- AC-5 : Job lit golden depuis `gs://archiviste-lore-corpus/golden/golden_qa.jsonl` ; sha256 == sha256 du `specs/golden_qa.jsonl` local opérateur au moment de la publication.
- AC-6 : connexion Cloud SQL prod `archiviste-db` en IAM authn sous `archiviste-runtime` (volume `cloudsql` + `DATABASE_URL` IAM-auth + `INSTANCE_CONNECTION_NAME`), miroir Job ingest, zéro mot de passe en clair.
- AC-7 : Job défini en Terraform avec `containers.command` figeant `python -m eval.ragas_runner --mode live --persist --set <golden_path> --output <tmp_path>` ; aucune exécution GHA.
- AC-8 : aucun nouveau rôle ni binding IAM (objectViewer + cloudsql.client + cloudsql.instanceUser existants réutilisés).

## OQ-3 resolved — golden fetch strategy
Le runner (`eval/loader.py:55` `path.open()`) ne lit **que** le filesystem local — pas de `gs://` natif. L'image eval (`infra/docker/eval.Dockerfile`) est `python:3.12-slim`, deps = `google-auth`/`httpx`/`psycopg2`/`pydantic`/`structlog`/ragas — **ni `gcloud` CLI, ni `google-cloud-storage`** (`download_corpus.py` vit dans l'image workers, pas eval). `ENTRYPOINT` = `python -m eval.ragas_runner`.
→ **Décision** : `command` = bash deux-temps, miroir du Job ingest, fetch via `google-auth` + `httpx` (deps déjà présentes) vers l'API JSON GCS, PAS `gcloud`/`gsutil` (absents). Le `command` override l'ENTRYPOINT. Fetch écrit `/tmp/golden_qa.jsonl` (tmpfs writable), puis `--set /tmp/golden_qa.jsonl --output /tmp/run.json`.
→ **Sous-contrainte** : l'image n'a pas `bash` garanti ni un one-liner google-auth trivial. Voir Risk R1 — à valider avant impl. Si fetch httpx+google-auth s'avère non trivial inline, c'est un blocker (no-workaround) → remonter, pas bricoler.

## Risks / open questions
- **R1 (bloquant potentiel)** : l'image eval n'a ni `gcloud`/`gsutil` ni `google-cloud-storage`. Le fetch golden doit se faire en bash inline avec `google-auth`+`httpx` (API JSON GCS + token metadata). Si non trivial / non robuste inline → blocker `no-workaround` : remonter (options : ajouter `google-cloud-storage` à l'image = touche OBS-008 hors scope, OU ajouter step fetch). NE PAS modifier `eval/` ni l'image en douce. À trancher avant impl.
- **R2** : `bash` présent dans `python:3.12-slim` ? Debian slim fournit `/bin/bash`. À confirmer (sinon `sh -c`).
- **R3** : `--workers-url` doit cibler workers prod (ingress=internal, auth IAM). `workers_runtime_invoker` (`cloud_run.tf:287`) donne déjà `run.invoker` à `archiviste-runtime` → OIDC OBS-007 OK, zéro nouveau binding. Confirmé, pas un gap.
- **R4** : env exacts requis par le judge Mistral / generate client côté runner non énumérés ici (lus à l'impl, pas devinés). Clé API = secret existant `MISTRAL_API_KEY` ; nom de var injectée à confirmer contre `eval/clients.py`.
- **R5 (humain-only)** : `specs/golden_qa.jsonl` gitignored — la publication GCS est une action opérateur manuelle, pas un step agent/CI. AC-5 = vérif opérateur, pas invariant CI.
