# OBS-002 — Widget santé service + GET /v1/status (probes Mistral/GCS/Postgres/workers)

## Contexte

La vision EPIC OBS ouvre une surface d'observabilité publique pour la vitrine portfolio ; OBS-001 a livré la page `/observability`, le header nav et le widget d'usage (`GET /v1/stats`). Cette page ne dit rien de la **disponibilité** du service : un visiteur ne peut pas voir d'un coup d'œil si le RAG est opérationnel ou si une dépendance est tombée. Ce ticket livre la deuxième slice de l'EPIC : un widget de santé sur `/observability` alimenté par un nouvel endpoint public anonyme `GET /v1/status` qui agrège l'état de quatre dépendances (Mistral, GCS, Postgres, workers) en un statut global `ok|degraded` sanitisé.

## Acceptance criteria

- AC-1 : `GET /v1/status` retourne `200 OK`, `Content-Type: application/json; charset=utf-8`, body JSON de forme `{"status":"ok"|"degraded","dependencies":{"mistral":{...},"gcs":{...},"postgres":{...},"workers":{...}},"checked_at":"<rfc3339>"}` ; les quatre clés `mistral`, `gcs`, `postgres`, `workers` sont **toujours** présentes sous `dependencies` (décision D-1, forme par-dépendance nommée).
- AC-2 : Chaque entrée de `dependencies.<dep>` a exactement la forme `{"status":"ok"|"down","latency_ms":<int ≥ 0>}` ; `status` est un **enum fermé** à deux valeurs (`"ok"` ou `"down"`) et ne contient jamais de message, host, URL, nom de base ni erreur SQL/HTTP (décision D-5, sanitisation). Aucune autre clé que `status` et `latency_ms` n'est présente par dépendance.
- AC-3 : `status` agrégé (racine) vaut `"degraded"` si **au moins une** des quatre dépendances a `status="down"`, et `"ok"` si et seulement si les quatre dépendances ont `status="ok"` (décision D-1, agrégat). Propriété : `status_racine == "ok"  ⇔  ∀ dep, dependencies[dep].status == "ok"`.
- AC-4 : `checked_at` est un timestamp RFC3339/ISO-8601 UTC correspondant au moment de production du snapshot (instant de la mesure servie, pas l'instant de la requête HTTP si servie depuis le cache, cf AC-9).
- AC-5 : `GET /v1/status` retourne **toujours `200 OK`** même quand `status="degraded"` (le statut vit dans le body, cohérent `GET /healthz` et OBS-001) ; aucun code `5xx` n'est retourné tant que le handler peut produire un snapshot (décision D-4).
- AC-6 : `GET /v1/status` ne retourne `503 Service Unavailable` **que** dans le cas où le handler ne peut produire aucun snapshot du tout (ni frais, ni en cache) ; ce body d'erreur est sanitisé `{"error":"<error_code>","request_id":"<uuid>"}` et ne contient aucun détail interne (cf OQ-2 pour figer `<error_code>`).
- AC-7 : `GET /v1/status` est servi sans authentification (anonyme) ; il ne porte aucun guard `author_only`/`member_only` et n'apparaît pas dans le sous-router auth. Un test contractuel (inspection router) vérifie l'absence de guard d'auth sur ce handler (cohérent OBS-001 AC-6).
- AC-8 : Chaque probe de dépendance applique un timeout dur de **2 secondes** ; une dépendance qui n'a pas répondu `ok` dans ce délai est exposée `status="down"` (un timeout est un `down`, décision D-4). `latency_ms` d'une dépendance en timeout reflète au plus ~2000 ms (le budget de timeout écoulé), jamais une valeur supérieure non bornée.
- AC-9 : Les quatre probes sont exécutées **en parallèle** : la latence de bout-en-bout de production d'un snapshot frais est bornée par le timeout d'une seule probe (~2 s worst-case), pas par leur somme (~8 s). Mesurable : quand toutes les dépendances sont down (toutes en timeout), le temps de calcul d'un snapshot frais reste < 3 s (marge sur le cap 2 s), pas ~8 s.
- AC-10 : Le résultat d'un snapshot est mis en cache **en mémoire, par process**, avec une TTL de **10 secondes** (décision D-3) ; deux appels `GET /v1/status` espacés de moins de 10 s servent le même snapshot (même `checked_at`) sans relancer les probes. Sur déploiement multi-réplica Cloud Run, le cache est par-instance (un visiteur peut voir des `checked_at` distincts selon l'instance servante) — acceptable phase 1 (décision D-3).
- AC-11 : Une dépendance dont la probe échoue (erreur de connexion, statut HTTP non-`2xx`, timeout, réponse malformée) est exposée `status="down"` ; le détail de l'échec (message, host, code interne, erreur SQL/HTTP) n'apparaît **jamais** dans le body et part uniquement dans un log `warn` interne (décision D-5, sanitisation ; cohérent `security.md` §A05/§A09).
- AC-12 : La probe `workers` reflète la **joignabilité des workers depuis la gateway** ET la vue des workers sur leurs propres dépendances : la gateway interroge un endpoint de dépendances exposé par les workers (route workers nouvelle, cf AC-15) qui rapporte l'état de Mistral et la vue interne des workers ; la gateway combine cette réponse avec ses probes directes Postgres + GCS + joignabilité-workers. Si l'endpoint workers est injoignable, `dependencies.workers.status="down"`.
- AC-13 : La probe `mistral` reflète l'état de Mistral **tel que rapporté par les workers** (la gateway n'appelle pas Mistral directement) : `dependencies.mistral.status="ok"` seulement si les workers rapportent Mistral atteignable dans leur réponse de dépendances (AC-15) ; si les workers sont injoignables, `mistral` est exposé `status="down"` (impossible de confirmer `ok` sans la vue workers).
- AC-14 : Au chargement de `/observability`, `observability.js` appelle `GET /v1/status` (même-origine, `fetch` `credentials:'same-origin'`) et insère dans un point d'ancrage stable `<section id="health-widget">` un rendu en `textContent` (jamais `innerHTML`) affichant le statut global et l'état par dépendance ; aucune fuite d'erreur interne n'est rendue (seuls `ok`/`down`/`degraded` et les `latency_ms` sont affichés). Le libellé exact n'est pas figé byte-for-byte en AC (souplesse implémentation, cf OQ-3) ; seuls l'usage `textContent`, la présence du statut global et la présence des quatre dépendances sont contractuels.
- AC-15 : Une route workers nouvelle (endpoint de dépendances workers) est ajoutée ET le contrat `specs/openapi/gateway-to-workers.yml` est mis à jour pour la décrire (méthode, path, schéma de réponse : état Mistral + vue interne workers, valeurs sanitisées `ok`/`down`). Un run schemathesis sur le contrat modifié passe (CI gate `CLAUDE.md` §7). NB : cet AC est la rupture vs OBS-001 (qui n'ouvrait aucune route workers) et vs UI-002 AC-22 — c'est ici un changement de contrat workers assumé.
- AC-16 : Pour toute issue de `GET /v1/status` autre que `200 OK` (5xx, network drop, timeout, JSON malformé), `observability.js` insère dans `#health-widget` un message littéral d'indisponibilité (aligné sur le ton OBS-001 AC-10 `Données indisponibles.`) ; aucun branchement fin selon le code d'erreur.
- AC-17 : Les 5 en-têtes de sécurité globaux posés router-wide (UI-001 AC-6 / SEC-003 AC-1 : `Content-Security-Policy`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, `X-Frame-Options: DENY`, `Strict-Transport-Security`) sont présents byte-for-byte sur `GET /v1/status`. La valeur CSP n'est PAS modifiée par ce ticket.
- AC-18 : Test manuel auteur (procédure documentée `## Test manuel auteur`) coché par l'humain avant `Status: ready`.

## Failure modes

- Une dépendance probe en échec (connexion refusée, HTTP non-2xx, réponse malformée) → `dependencies.<dep>.status="down"`, agrégat `status="degraded"`, `200 OK`, log `warn` sanitisé ; aucun détail interne dans le body (AC-2, AC-11).
- Une dépendance probe en timeout (> 2 s) → `dependencies.<dep>.status="down"`, `latency_ms ≈ 2000` borné, `200 OK` (AC-8).
- Workers injoignables → `dependencies.workers.status="down"` ET `dependencies.mistral.status="down"` (la vue Mistral transite par les workers, AC-13), agrégat `degraded`, `200 OK`.
- Handler incapable de produire aucun snapshot (ni frais ni caché) → `503` body sanitisé `{"error":"<error_code>","request_id":"<uuid>"}` (AC-6, code exact à figer OQ-2).
- Toutes dépendances down (toutes en timeout) → snapshot frais calculé en < 3 s grâce au parallélisme (AC-9), `status="degraded"`, `200 OK`.

## Test manuel auteur

Procédure exécutée par l'humain avant `Status: ready` (AC-18).

1. **Stack** : `docker compose up -d`, vérifier gateway + workers + postgres `healthy`.
2. **Nominal** : `curl -i http://localhost:8080/v1/status` → `200`, body `status="ok"`, 4 dépendances `ok`, `checked_at` RFC3339, `latency_ms` plausibles.
3. **Widget** : ouvrir `http://localhost:8080/observability` → `#health-widget` affiche le statut global + l'état des 4 dépendances. Network tab : `GET /v1/status` 200.
4. **Cache** : deux `curl` consécutifs < 10 s → `checked_at` identique. Attendre > 10 s → `checked_at` change (AC-10).
5. **Dépendance down** : `docker compose stop postgres` → `curl -i /v1/status` → `200`, `dependencies.postgres.status="down"`, agrégat `degraded` ; aucun détail SQL dans le body (AC-2, AC-5, AC-11). Recharger `/observability` → widget reflète `degraded`.

7. **Parallélisme** : toutes dépendances arrêtées → mesurer le temps de réponse d'un snapshot frais (vider le cache / attendre TTL) → < 3 s, pas ~8 s (AC-9).
8. **En-têtes** : DevTools Network sur `GET /v1/status` → les 5 headers de sécurité présents (AC-17).
9. **Anonyme** : `curl -i http://localhost:8080/v1/status` (sans cookie) → `200` (AC-7).

## Performance / SLO

- `GET /v1/status` servi depuis le cache (TTL 10 s) : < 50 ms p95 stack locale (pas de probe relancée).
- `GET /v1/status` calcul d'un snapshot frais, cas nominal (dépendances saines) : borné par la probe la plus lente, cible < 500 ms p95 ; worst-case (dépendance en timeout) ~2 s borné par le cap timeout (AC-8), jamais ~8 s grâce au parallélisme (AC-9).
- Timeout dur par probe : 2 s (AC-8), aligné `security.md` §A04 « timeouts sur tout appel externe ».

## Security / trust boundary

- Frontière Internet → Gateway, plus Gateway → Workers (appel interne nouveau, AC-12).
- **A01 Broken Access Control** : `/v1/status` public anonyme par design (AC-7) ; n'expose aucune donnée par-utilisateur, uniquement un statut agrégé de santé. Pas d'IDOR (aucun identifiant de ressource en path/query).
- **A05 Misconfiguration** : valeurs par dépendance restreintes à l'enum `ok|down` (AC-2) ; aucun host, URL, message d'erreur, nom de base ni stack n'est exposé (AC-11). Erreur `503` enveloppe sanitisée (AC-6). CSP UI-001 strictement préservée (AC-17).
- **A04 Insecure Design** : timeout dur 2 s par probe (AC-8), pas de boucle de retry non bornée ; cache TTL 10 s limite l'amplification de charge sur les dépendances (un pic de trafic `/v1/status` ne déclenche pas un pic de probes 1:1).
- **A09 Logging & Monitoring** : log `tracing` structuré autorisé (`request_id`, `latency_ms`, statut agrégé, statut par dépendance `ok|down`) ; le détail d'échec d'une probe va en `warn` interne SANS exposer host/message brut dans des champs qui fuiteraient (cohérent §A09). Pas de log applicatif sur le chemin statique `/observability`.
- **A10 SSRF** : la probe workers cible une URL workers **fixe issue de la configuration** (pas d'URL fournie par l'utilisateur) — pas de surface SSRF (le requester ne contrôle aucune cible de probe).
- **RAG-specific** : aucune donnée de conversation, aucun contenu, aucun `gcs_uri`, aucun `user_id` exposé — uniquement un statut de santé agrégé.

## Observability

- Log `tracing` `warn` par dépendance en échec (`event=status.probe.down`, champ `dep`) — sans message/host brut dans le body, détail interne autorisé en log seulement (cf OQ-4).
- Pas de log applicatif sur le chemin statique `/observability` (TraceLayer existante suffit, OBS-001 AC).

## Décisions

Arbitrages humains figés (round-1). Chaque décision verrouille une dimension du contrat.

- **D-1 — Forme par-dépendance NOMMÉE + agrégat.** Body `{"status":"ok"|"degraded","dependencies":{"mistral":..,"gcs":..,"postgres":..,"workers":..},"checked_at":"<rfc3339>"}`. Agrégat = `degraded` si au moins une dépendance `down` (AC-1, AC-3).
- **D-2 — Scope = Mistral INCLUS + disponibilité workers.** La gateway interroge un endpoint de dépendances workers (les workers rapportent Mistral + leur vue interne) ; la gateway ajoute ses probes directes Postgres + GCS + joignabilité-workers. Implique une route workers nouvelle + changement de contrat OpenAPI (AC-12, AC-13, AC-15). Conséquence slice : probable dépassement 300 LOC → OQ-1 (découpe non décidée ici, renvoyée à l'architect).
- **D-3 — Cache TTL 10 s, in-memory par-process** (pattern `Arc` dans AppState, analogue au cache `TokenProvider`). Cache par-instance sur multi-réplica Cloud Run accepté phase 1 (AC-10).
- **D-4 — Timeout 2 s par probe, probes en PARALLÈLE, `200` always.** Worst-case ~2 s (pas 4×2 s) ; timeout ⇒ `down` ; `GET /v1/status` retourne `200` même `degraded` (statut dans le body, comme `healthz`) ; `503` réservé au cas « aucun snapshot productible » (AC-5, AC-6, AC-8, AC-9).
- **D-5 — Sanitisation : enum fermé `ok|down` par dépendance + `latency_ms` exposé.** Jamais de message/host/url/erreur SQL dans le body ; tout détail d'échec va en log `warn` interne. `latency_ms` (nombre de timing) est exposé en plus — acceptable car ne fuite aucun détail interne, signal portfolio plus riche (AC-2, AC-11).
