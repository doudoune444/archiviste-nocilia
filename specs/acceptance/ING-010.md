# ING-010 — Sync Google Drive → `lore/*.md` (export hebdo gdoc/gsheet/gslide)

## Contexte

La vision §52 fige la phase 3 d'ingestion : un dossier Google Drive auteur sert de source amont, exporté périodiquement en `*.md` versionnés sous `lore/`. Sans ce ticket, l'auteur doit éditer le Markdown à la main alors que son corpus de travail vit dans Drive ; toute édition Drive reste invisible du RAG. Ce ticket livre l'export offline (outil dev, pas du code applicatif gateway/workers) qui aligne `lore/` sur l'arborescence Drive et propose les changements en PR pour revue humaine.

## Critères d'acceptation

- AC-1 : La commande `uv run python scripts/gdrive_export.py --root-folder-id <id>` parcourt récursivement le dossier Drive racine fourni (et ses sous-dossiers) via l'API Drive v3 et liste tous les fichiers de MIME `application/vnd.google-apps.{document,spreadsheet,presentation}` accessibles au service account.
- AC-2 : Chaque `gdoc` (`mimeType=application/vnd.google-apps.document`) est exporté via `files.export(mimeType='text/markdown')` puis écrit en `lore/<drive_path>/<doc_slug>.md` où `<drive_path>` reproduit l'arborescence Drive relative à la racine et `<doc_slug>` est le titre Drive slugifié (cf AC-7).
- AC-3 : Chaque `gslide` (`mimeType=application/vnd.google-apps.presentation`) est exporté via `files.export(mimeType='text/plain')` puis écrit en `lore/<drive_path>/<deck_slug>.md` ; le contenu plain text est ré-emballé tel quel sous le frontmatter (pas de reconstruction de structure slide phase 1).
- AC-4 : Chaque `gsheet` (`mimeType=application/vnd.google-apps.spreadsheet`) génère **un fichier `.md` par onglet** (sheet/tab), pas un seul fichier par classeur ; chaque onglet est exporté via Sheets API v4 `spreadsheets.values.get` (range = nom de l'onglet) et rendu en table Markdown GFM (header = première ligne).
- AC-5 : Le path d'un onglet gsheet est exactement `lore/<drive_path>/<sheet_slug>--<tab_slug>.md` (séparateur `--` entre titre du classeur et titre de l'onglet, tous deux slugifiés). Le `tab_slug` est dérivé du `title` de l'onglet ; en cas de collision intra-classeur (deux onglets slugifiant identique), suffixe `-<gid>` ajouté au tab_slug perdant l'unicité.
- AC-6 : Le frontmatter YAML écrit en tête de chaque `.md` exporté contient strictement les champs : `title` (str = titre Drive ou `<titre classeur> — <titre onglet>` pour gsheet), `tags` (list[str], défaut `[]`), `access_tier` (str, défaut `public`), `source` (str = `gdrive`), `source_id` (str, cf AC-9), `drive_path` (str = chemin POSIX Drive relatif à la racine, ex. `Lore/Personnages/`), `exported_at` (str ISO8601 UTC), `archived` (bool, défaut `false`).
- AC-7 : La fonction de slugification est déterministe et applique : `unicodedata.normalize('NFKD')` → drop combining marks → lower → remplacement de tout caractère hors `[a-z0-9]` par `-` → collapse runs de `-` → strip `-` aux extrémités → cap 80 chars. Si le slug résultant est vide, fallback `file-<drive_file_id[:8]>`.
- AC-8 : Une collision de path final (deux fichiers Drive distincts produisant le même chemin local après slug + extension) est résolue en suffixant `-<drive_file_id[:8]>` au stem du second, troisième, ... fichier rencontré (ordre déterministe = tri lexical sur `drive_file_id`) ; aucun fichier n'est silencieusement écrasé.
- AC-9 : Le champ `source_id` du frontmatter est :
  - `gdoc` / `gslide` : `<drive_file_id>` (ex. `1aB2cD3eF4...`).
  - `gsheet` (chaque onglet) : `<drive_file_id>#<tab_gid>` (ex. `1aB2cD3eF4...#1734218910`) — le `#<tab_gid>` distingue les onglets d'un même classeur.
- AC-10 : Un fichier d'état persistant `scripts/.gdrive_state.json` (versionné dans le repo, JSON canonique trié) maintient une map `source_id → {local_path, drive_md5_checksum, last_exported_at}`. Le script lit cet état au démarrage et le réécrit à la fin du run.
- AC-11 : Détection de **rename Drive** : si un `source_id` connu de l'état pointe vers un `local_path` différent du chemin courant calculé (changement de titre Drive ou déplacement entre dossiers), le script `git mv <ancien_path> <nouveau_path>` (via `subprocess` `git mv`, pas écrit-puis-supprimé) avant de réécrire le contenu, préservant l'historique git. Si `git mv` échoue (ex. fichier non tracké), fallback `os.rename`.
- AC-12 : Détection de **suppression Drive** : si un `source_id` est présent dans `.gdrive_state.json` mais absent de la dernière liste Drive (fichier supprimé/déplacé hors racine/access perdu), le script ne supprime PAS le `.md` local mais ajoute/force `archived: true` dans son frontmatter et conserve le contenu. Le `source_id` reste dans l'état avec un flag `archived_at` (ISO8601 UTC).
- AC-13 : Détection de **suppression d'onglet gsheet** : si un classeur existe toujours mais qu'un onglet précédemment exporté (clé d'état `<drive_file_id>#<tab_gid>`) n'est plus listé par `spreadsheets.get`, le `.md` local correspondant à cet onglet voit `archived: true` posé (mêmes règles qu'AC-12) ; les autres onglets du classeur sont mis à jour normalement.
- AC-14 : Détection de **contenu inchangé** : si `drive_md5_checksum` (gdoc/gslide : md5 de l'export rendu ; gsheet onglet : sha256 de la sérialisation Markdown produite) est identique à la valeur en état, le fichier local n'est PAS réécrit (pas de churn `exported_at`), et l'état n'est pas mis à jour pour ce fichier.
- AC-15 : Le contenu Markdown écrit (corps post-frontmatter) est normalisé `unicodedata.normalize('NFKC', text)` puis dépouillé des caractères de contrôle `\x00`-`\x1f` à l'exception de `\n` et `\t` (cohérence ING-001 AC-4, threat-model W-T-1).
- AC-16 : Tout fichier dont l'export rendu dépasse 1 MiB est skippé : log JSON `status: "error"`, `reason: "exported size exceeds 1 MiB cap"`, état non mis à jour pour ce fichier, traitement des autres fichiers continué (cohérence ING-001 AC-15).
- AC-17 : Les credentials du service account sont lus exclusivement via la variable d'environnement `GDRIVE_SA_KEY_JSON` (contenu JSON inline) OU `GOOGLE_APPLICATION_CREDENTIALS` (path vers fichier) ; aucun fallback hardcodé, aucun chemin par défaut, fail-fast au boot avec message explicite si ni l'un ni l'autre n'est défini.
- AC-18 : Le script accepte un flag `--dry-run` qui exécute la résolution complète (list Drive, calcul des paths, diff vs état) et logge les actions qu'il aurait effectuées (`would_create | would_update | would_rename | would_archive | would_skip`) sans écrire aucun fichier ni modifier l'état.
- AC-19 : Un workflow GitHub Actions `.github/workflows/gdrive-sync.yml` est livré avec deux triggers : `workflow_dispatch` (manuel) et `schedule: cron('0 3 * * 1')` (lundi 03:00 UTC). Le workflow checkout main, installe `uv`, exécute le script avec `secrets.GDRIVE_SA_KEY` injecté en `GDRIVE_SA_KEY_JSON`, et si le diff git est non vide, ouvre une PR via `peter-evans/create-pull-request@v6` ciblant `main`, branche `chore/gdrive-sync-<run_id>`.
- AC-20 : Le workflow upload un artifact `gdrive-sync-summary.json` (rétention 30 jours) contenant le summary JSON du run (cf AC-22).
- AC-21 : Le workflow ne pousse jamais directement sur `main` ; la PR créée est en draft si `errors >= 1` dans le summary, prête-à-review sinon.
- AC-22 : À la fin du run, un log JSON unique `{"event":"gdrive_sync.summary","total":N,"created":C,"updated":U,"renamed":R,"archived":A,"unchanged":S,"errors":E,"duration_ms":D}` est émis ; chaque fichier traité produit un log `{"event":"gdrive_sync.file","source_id":"...","local_path":"...","status":"created|updated|renamed|archived|unchanged|error","reason":"<si error>"}`.
- AC-23 : Le code de sortie du process est `0` si `errors == 0`, `1` si `errors >= 1` (les autres fichiers ayant été traités), cohérent ING-001 AC-17.

## Non-goals

- Pas d'export d'images (PNG inclus dans gdoc, fichiers `image/*` au sens Drive) — reportée à un ticket ING-* ultérieur.
- Pas de filtrage retrieval sur `archived: true` côté workers RET-* — ce ticket pose seulement le marqueur ; le filtre est un ticket RET-* dédié explicite.
- Pas de suppression physique des fichiers `.md` archivés (jamais `rm` côté script).
- Pas de re-write de l'historique git (pas de `git mv` rétroactif sur fichiers déjà archivés).
- Pas de webhook Drive push notification — polling hebdo seul.
- Pas de support OAuth user (seul service account avec scope dossier dédié).
- Pas d'auth API exposée par le script (offline tool).
- Pas d'intégration avec ING-001 / ingestion DB ; le script s'arrête après écriture des `.md`. L'ingest run `python -m archiviste_workers.ingest --path lore/` est invoqué séparément (humain ou ticket futur d'orchestration).
- Pas de re-chunking automatique côté DB ; ING-001 détecte les changements via `content_hash` post-merge.
- Pas de gestion de fichiers `application/vnd.google-apps.shortcut` — skippés silencieusement avec log info.
- Pas de gestion des fichiers natifs non-Google (PDF, DOCX uploadés dans Drive) — skippés avec log info, ticket futur.
- Pas de scope multi-racine (un seul `--root-folder-id` par run).
- Pas de support des Shared Drives (`supportsAllDrives`) phase 1 — `My Drive` du SA uniquement, sauf si trivial à activer (à confirmer plan).
- Pas de retry de l'export Drive sur 5xx (failure → log error, fichier suivant) ; ticket OBS-* / résilience reporté.
- Pas de sync inverse (`lore/` → Drive). Drive est source amont unique.

## Pre-conditions

- ING-001 mergé : structure `lore/`, frontmatter YAML, conventions slug-friendly côté ingest. Ce ticket réutilise les mêmes invariants frontmatter (`title`, `tags`, `access_tier`).
- Service account GCP créé avec scope `https://www.googleapis.com/auth/drive.readonly` ET `https://www.googleapis.com/auth/spreadsheets.readonly`, partagé en lecture sur le dossier Drive racine de l'auteur (configuration manuelle hors-script).
- Secret GitHub `GDRIVE_SA_KEY` provisionné dans le repo (clé JSON service account complète).
- `peter-evans/create-pull-request@v6` non bloqué par la politique GitHub Actions de l'organisation (à vérifier).
- Variable d'env `GDRIVE_ROOT_FOLDER_ID` (ou flag CLI `--root-folder-id`) fournie au workflow.

## Failure modes

- `GDRIVE_SA_KEY_JSON` / `GOOGLE_APPLICATION_CREDENTIALS` absents au boot → exit non-zéro immédiat, message `gdrive credentials missing: set GDRIVE_SA_KEY_JSON or GOOGLE_APPLICATION_CREDENTIALS`, aucun fichier traité.
- `--root-folder-id` absent ou non résolvable (404 Drive) → exit non-zéro immédiat, message `root folder not found or inaccessible: <id>`.
- Service account non autorisé sur le dossier (403) → exit non-zéro immédiat, même format que ci-dessus.
- Quota / rate-limit Drive API (429, 403 `userRateLimitExceeded`) → log error pour le fichier en cours, traitement continué, pas de retry phase 1, reflété dans `errors` du summary.
- Export Drive échoue avec 5xx → log error avec `reason: "drive_export_failed: <status>"`, fichier skippé, traitement continué, exit final `1`.
- Fichier exporté > 1 MiB → skippé, cf AC-16.
- Slug vide après normalisation → fallback `file-<drive_file_id[:8]>`, log warn, traitement continué (pas un error).
- Collision de path (cf AC-8) → suffixe `-<drive_file_id[:8]>` au second arrivant, log warn, traitement continué.
- `git mv` échoue (fichier non tracké) → fallback `os.rename`, log warn `git_mv_failed_fallback_rename`, traitement continué.
- État `.gdrive_state.json` corrompu / non-JSON → exit non-zéro immédiat, message `state file corrupted: <path>`, demande intervention humaine (pas de re-init silencieuse).
- État absent (premier run) → traité comme `{}` vide, tous les fichiers comptent comme `created`, log info `event=gdrive_sync.first_run`.

## Security / trust boundary

- Outil offline / dev (vision §54 : « pas du code applicatif. Pas d'amendement threat-model requis »). Pas de surface réseau exposée par ce ticket.
- Service account scopé read-only, sur dossier unique partagé manuellement (least-privilege, threat-model alignement).
- Secret SA jamais commité : `.env.example` commente la variable, `.gitignore` couvre `*-sa.json` (cf `.claude/rules/secret-hygiene.md`). Le hook `guard-secret-paths.sh` doit déjà couvrir.
- En CI, `secrets.GDRIVE_SA_KEY` injecté en variable d'env de step uniquement, jamais persisté en filesystem du runner.
- Cohérence threat-model W-T-1 : normalisation NFKC + strip control chars (AC-15) + cap 1 MiB (AC-16) sur tout contenu Drive avant écriture `lore/`.
- Cohérence threat-model A03 « Embedding poisoning (ingestion side) » : la défense reste au niveau ingest ING-001 (AC-4 cohérent). Ce ticket ne court-circuite pas la normalisation aval.
- Path traversal : tous les paths produits sont sous `lore/`, tous les composants slugifiés (AC-7). Aucune injection de `..` possible (slug regex `[a-z0-9-]`).
- L'auto-PR du workflow est créée par un GitHub App / `GITHUB_TOKEN` standard, jamais merge auto ; humain review et merge.

## Open questions

- OQ-1 : Convention de séparateur `--` entre `<sheet_slug>` et `<tab_slug>` (AC-5) — alternatives possibles : `__`, sous-dossier `<sheet_slug>/<tab_slug>.md`, ou suffixe `.<tab_slug>.md`. Reco proposée = `--` (lisible, pas de collision filesystem, pas de répertoire fantôme si le classeur n'a qu'un onglet). Confirmer ou substituer.
- OQ-2 : Que faire d'un classeur gsheet à **1 seul onglet** ? Reco = appliquer la même règle (AC-5) `<sheet_slug>--<tab_slug>.md` pour cohérence ; alternative = collapse en `<sheet_slug>.md` sans suffixe tab. Trancher.
- OQ-3 : Le `tab_gid` Drive est un entier décimal non borné. Est-il acceptable dans `source_id` tel quel (`<file_id>#1734218910`) ou faut-il un préfixe (`#tab=` / `#gid=`) ? Reco = `#<gid>` sans préfixe (compact, parsable trivialement).
- OQ-4 : Comportement attendu si l'auteur édite manuellement un `.md` exporté (drift Drive ↔ local) entre deux runs → le run suivant écrasera l'édition locale. Faut-il détecter et logger un warning (`local_diverged_from_drive_md5`) avant overwrite, ou écraser silencieusement ? Reco = warn + overwrite (Drive = source de vérité, vision §52).
- OQ-5 : Shared Drives (`supportsAllDrives=true`) supportés phase 1 ou hors scope ? Reco = hors scope (My Drive du SA seulement) — flagger explicitement Non-goal si confirmé.
- OQ-6 : Frontmatter `tags` source — est-ce extrait de quelque part dans Drive (description fichier, custom property) ou toujours `[]` à l'export, charge à l'humain de l'enrichir manuellement post-PR ? Reco = `[]` par défaut (vision §52 ne précise pas), enrichissement humain via PR review.
- OQ-7 : Frontmatter `access_tier` source — pareil, `public` par défaut systématiquement ou lecture d'une description / property Drive ? Reco = `public` par défaut, surcharge humaine via PR review.
- OQ-8 : Si l'humain modifie manuellement le frontmatter d'un `.md` exporté (ex. passe `access_tier: members`) puis Drive change le contenu, le script doit-il préserver le frontmatter humain ou le réécraser ? Reco = **préserver** les champs humain-éditables (`tags`, `access_tier`) en les ré-injectant ; réécrire les champs script-managed (`title`, `source_id`, `drive_path`, `exported_at`, `archived`). Mécanisme à figer.
- OQ-9 : Stockage de `.gdrive_state.json` — sous `scripts/` (versionné repo, simple) ou GCS (durable, mais introduit une dépendance cloud côté CI) ? Reco = `scripts/` versionné. Volume attendu < 100 KiB.
- OQ-10 : Quel format pour `<drive_path>` quand un dossier Drive contient des espaces / caractères spéciaux ? Reco = chaque composant de path slugifié (AC-7) → arborescence `lore/lore/personnages/...`. Confirmer (ou conserver les noms Drive bruts si filesystem-safe).
- OQ-11 : Effort L vs split en 3 sous-tickets (cf Effort estimate). Le plan tranchera, mais préférence humain à exprimer maintenant si forte.
- OQ-12 : Le workflow `.github/workflows/gdrive-sync.yml` doit-il aussi exécuter `python -m archiviste_workers.ingest --path lore/` post-merge de la PR (pour repeupler Postgres) ou cette étape reste-t-elle hors scope (autre workflow / déclenchement humain) ? Reco = hors scope ce ticket (séparation des responsabilités : ce ticket aligne `lore/`, ING-001 est invoqué séparément).
