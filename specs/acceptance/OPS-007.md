# OPS-007 — Correctifs runtime du canal corpus lore (régression OPS-006)

## Contexte

OPS-006 a livré le canal corpus privé `gs://archiviste-lore-corpus` (gdrive-sync → bucket → Job d'ingestion), mais deux défauts runtime, tous deux dans des fichiers livrés par OPS-006, cassent la chaîne de bout en bout. Le step `gcloud storage rsync` de gdrive-sync échoue car `gha-deploy` (`storage.tf:16-20`, `roles/storage.objectAdmin`) ne porte pas `storage.buckets.get`, requis pour stat le bucket destination. Et le Job `archiviste-ingest` sort en exit `1` au boot car `Settings()` (`ingest/cli.py:106`) lève une `ValidationError` pydantic : `settings.py:28` déclare `gcs_bucket: str = Field(...)` (requis, sans défaut) alors que le bloc env du Job (`cloud_run_job.tf:64-106`) n'injecte pas `GCS_BUCKET`.

## Critères d'acceptation

- AC-1 : `gha-deploy` (l'identité GitHub Actions utilisée par gdrive-sync) dispose, sur le seul bucket `archiviste-lore-corpus`, d'un rôle portant la permission `storage.buckets.get` ; `roles/storage.legacyBucketReader` est le rôle prédéfini de moindre privilège satisfaisant ce contrat (un rôle custom restreint à `storage.buckets.get` le satisferait aussi). Le binding est scopé au seul bucket — aucune permission `storage.buckets.*` accordée au niveau projet.
- AC-2 : Le step `gcloud storage rsync lore/ gs://archiviste-lore-corpus` de `gdrive-sync.yml` s'exécute sans erreur `does not have storage.buckets.get access` ; après un run gdrive-sync ayant produit ≥ 1 `.md`, le run GitHub Actions gdrive-sync est vert.
- AC-3 : Le bloc env du Job `archiviste-ingest` (`cloud_run_job.tf`) injecte `GCS_BUCKET=archiviste-conversations` ; `Settings()` (`ingest/cli.py:106`) se construit sans `ValidationError` et le Job ne sort plus en exit `1` au boot pour cause de champ requis manquant. Note : `gcs_bucket` n'est jamais **lu** sur le chemin d'ingestion — il ne satisfait que le contrôle de champ requis pydantic à la construction de `Settings()` (son seul lecteur réel est `main.py:71`, chemin de persistance des conversations, hors Job ingest) ; toute valeur valide ferait booter le Job, `archiviste-conversations` est choisi car c'est le bucket conversations canonique déjà existant.
- AC-4 : Le binding lecture runtime reste `roles/storage.objectViewer` pour `archiviste-runtime` sur `archiviste-lore-corpus` (inchangé) ; aucune permission `storage.buckets.get` n'est accordée à `archiviste-runtime`. Le download du Job (`list_blobs` + `download_to_filename`) n'exige que `storage.objects.list` + `storage.objects.get`, déjà couverts par `objectViewer`. Non-régression : OPS-006 AC-3 préservé.
- AC-5 : Sur la chaîne complète, après un run gdrive-sync ayant publié ≥ 1 `.md` : (a) le run gdrive-sync est vert, (b) le run `ingest-lore` est vert, (c) `gcloud storage ls gs://archiviste-lore-corpus/**` liste les `.md`, (d) le Job émet le log JSON `ingest.summary` (ING-001 AC-16) avec `total >= 1`, filtrable par execution ID dans Cloud Logging (oracle mirroir OPS-006 AC-7).

## Non-goals

- Pas de modification du pipeline ING-001 (parse / chunk / embed / upsert) ni du contrat CLI `--root` (OPS-006 AC-6) — aucun changement de code applicatif, ce ticket est un correctif IAM + env.
- Pas de migration DB, pas de nouvelle dépendance, pas de nouvel ADR.
- Pas de persistance du state-file gdrive-sync (ING-013) — hors scope.
- Pas de correction du commentaire obsolète `gdrive-sync.yml:15` ni de retrait du `contents: write` inutilisé — dette cosmétique reportée.
- Pas d'amendement des sources de vérité humain-only (OPS-006 OQ-B : `ING-001.md`, `ING-013.md`) — hors scope.
- Pas de suppression du champ `gcs_bucket: str = Field(...)` ni refactor de `settings.py` pour le rendre optionnel sur le chemin ingest — le correctif est l'injection de l'env, pas la modification du modèle de settings (qui toucherait le chemin conversations).
- Pas de correction des 3 erreurs content-level résiduelles du pull Drive (28 total, 25 créés, 3 en erreur) — séparé et mineur.

## Failure modes

- `gha-deploy` toujours sans `storage.buckets.get` → step rsync gdrive-sync échoue `... does not have storage.buckets.get access` → run gdrive-sync rouge → `ingest-lore` ne se déclenche pas (garde `conclusion == 'success'`, OPS-006 AC-8).
- `GCS_BUCKET` toujours absent du bloc env du Job → `Settings()` lève `ValidationError` à `cli.py:106` → Job exit `1` au boot → execution `Failed` → run `ingest-lore` rouge, `ingest.summary` jamais émis.
- `GCS_BUCKET` injecté avec une valeur invalide (chaîne vide) → `Settings()` construit (champ présent) mais aucun impact ingest car `gcs_bucket` non lu sur ce chemin ; non couvert ici (valeur fixée `archiviste-conversations`).
- `storage.buckets.get` accordé par erreur à `archiviste-runtime` (sur-privilège) → viole AC-4 / least-privilege OPS-006 → à rejeter en revue Terraform.

- AC-2 : run réel · après `apply`, un run gdrive-sync produisant ≥ 1 `.md` est vert, sans erreur `storage.buckets.get` dans les logs du step rsync.

- AC-4 : contract (Terraform) + revue manuelle · le plan montre le binding `archiviste-runtime` toujours en `roles/storage.objectViewer` et aucun binding accordant `storage.buckets.get` à `archiviste-runtime`.
- AC-5 : run réel (chaîne complète) · gdrive-sync vert → `ingest-lore` vert → `gcloud storage ls gs://archiviste-lore-corpus/**` liste les `.md` → requête Cloud Logging filtrée par execution ID retourne `ingest.summary` avec `total >= 1`.

## Security / trust boundary

- Least-privilege préservé : `gha-deploy` gagne uniquement `storage.buckets.get` (via `legacyBucketReader`) scopé au seul bucket, en plus de son `objectAdmin` existant ; aucun privilège projet-wide. `archiviste-runtime` reste read-only `objectViewer` (AC-4).
- Conforme `secret-hygiene.md` : `GCS_BUCKET=archiviste-conversations` est un nom de bucket, pas un secret ; injecté en clair comme les autres env non-sensibles du Job (`INGEST_ROOT`, `LORE_CORPUS_BUCKET`). `MISTRAL_API_KEY` reste en `secret_key_ref` (inchangé).
- Aucun nouveau secret, aucune clé SA JSON, auth GCP toujours via WIF (gha) et identité Cloud Run (Job).
