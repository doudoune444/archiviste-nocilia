# Plan — GEN-003 Mode 2 off-topic : classifier intent + branche refus poli

## Goal
Greffer une étape `intent` (LLM zero-shot binaire) en tête de `/v1/generate` ; si `off_topic`, court-circuiter retrieve + canon LLM et générer un refus poli in-world via un second appel LLM ; tracer `intent` dans `query_log` ; rester strictement compatible GEN-001 sur la branche `in_domain`.

## Pre-flight
- **Fichiers/dirs lus** : `specs/acceptance/GEN-003.md`, `specs/plans/GEN-001.md`, `specs/plans/GEN-002.md`, `docs/vision.md` (§20-25, §32, §69, §89), `workers/src/archiviste_workers/generate/{router,prompt,models,injection_filter,pricing}.py`, `workers/src/archiviste_workers/services/{llm,query_log}.py`, `migrations/0002_schema.sql` (lignes 65-79), `specs/golden_qa.jsonl` (GQA-035..038), `workers/tests/test_generate_*.py`, `.claude/rules/{clean-code,vertical-slice,no-workaround,security,secret-hygiene}.md`, `CLAUDE.md`.
- **3 hypothèses clés** :
  1. `LlmClient.invoke()` accepte un argument optionnel `timeout_s: float | None = None` (override 5 s classifier ; défaut conserve 30 s canon). Modification interne `services/llm.py` : `asyncio.wait_for(..., timeout=timeout_s or LLM_TIMEOUT_S)`. Cohérent AC-13 et touch-point spec.
  2. `query_log.intent` est binde dans une seule colonne TEXT (déjà présente, ligne 71 migration). Aucune contrainte CHECK sur `intent`. On élargit `QueryLogRow` avec `intent: str | None` et on remplace le `NULL` hardcodé du SQL par `$N`.
  3. Le générateur de refus off_topic réutilise `LlmClient.invoke(messages, timeout_s=30)` (même provider, même client, prompt distinct via `build_off_topic_messages`). Pas de second client, pas de retry. Mesure usage via `extract_usage(ai, provider)` identique canon.
- **Zones d'incertitude** :
  - (a) `Mode = Literal["canon"]` est utilisé par `GenerateResponse.mode` ET par `QueryLogRow.mode: str | None`. Élargir le `Literal` peut casser un `model_validate_strict` ailleurs — pas trouvé d'autre occurrence mais à vérifier en step 1.
  - (b) Comptage `llm_ms` quand classifier fail-open canon : la spec ne tranche pas. Choix proposé : `llm_ms` = somme `classifier_ms + canon_llm_ms` même en fail-open (AC-9 ne s'applique qu'en mode off_topic ; AC-19 demande pipeline canon inchangé observable côté client — `llm_ms` étant un nombre, ajouter `classifier_ms` ~5s fail-open serait observable). Décision retenue : `llm_ms` côté canon = uniquement durée LLM canon ; durée classifier non comptée dans `llm_ms` (cohérent « fail-open vers canon » = canon strictement inchangé). À confirmer humain.
  - (c) `usage`/`cost_eur` en fail-open canon : la spec dit « pipeline canon GEN-001 intégralement inchangé » (AC-19). Donc `usage.prompt_tokens` reflète **uniquement** l'appel canon, pas le classifier. Mêmes règles pour `cost_eur`. Cohérent (b).

## Risks / open questions
- **R1 (LOC)** : ~70 (intent.py) + 25 (prompt delta) + 110 (router refactor) + 4 (query_log delta) + 3 (llm delta) + 1 (models) ≈ **~215 LOC prod**. Tests ~310 LOC (non comptés). Marge OK ≤ 300.
- **R2 (drift D1 timeout 5 s classifier)** : non spécifié par vision. Confirmer 5 s acceptable ou descendre à 2 s (Mistral Small ~80 tok/s → 2 mots binaire < 200 ms typique, 5 s laisse marge cold-start).
- **R3 (drift D2 fail-open canon)** : choix conservateur retenu (AC-13/14). Coût marginal en cas pannes classifier répétées ; mitigation = monitoring `query_log` filtré `intent IS NULL` post-deploy (out-of-scope, OBS-*).
- **R4 (drift D3 génération refus via LLM)** : alternative templatée plus pas chère mais ton in-world Archiviste perdu. Choix LLM retenu (AC-7). À ré-évaluer si coût/latence devient un problème.
- **R5 (zone d'incertitude b/c)** : `llm_ms` et `usage` en fail-open canon = canon-only, pas de cumul classifier. Confirmer humain avant impl step 6.
- **R6 (TimeoutError → LlmTimeoutError)** : `asyncio.wait_for` lève `TimeoutError` (Py 3.11+). `services/llm.py` map déjà → `LlmTimeoutError`. Override `timeout_s=5` produit le même `LlmTimeoutError` ; `classify_intent` doit le catch séparément pour distinguer du timeout canon.
