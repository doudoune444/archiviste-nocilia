# Plan — ING-014 Extraction images embarquées gdoc → sidecar `<doc_slug>.images/`

## Goal
Pour chaque `gdoc`, invoquer Docs API en sus de `files.export`, télécharger chaque `inlineObject` image, appliquer pipeline compression Pillow (resize Lanczos 2048px → JPEG q=85 conditionnel → cap 5 MiB), persister `lore/<drive_path>/<doc_slug>.images/<md5_post[:12]>.<ext>` avec md5 et ext **post-compression**, réécrire le Markdown vers ces chemins relatifs, gérer dédup intra-doc / cleanup orphelins / rename / archive / dry-run, étendre `StateEntry.images` + signature `unchanged` + summary, et étendre la condition draft PR du workflow ING-012.

## Acceptance criteria recap
Voir `specs/acceptance/ING-014.md` AC-1..AC-18 (verbatim, non recopié — > 100 lignes). Couvre Docs API call uniforme (AC-1, AC-13), download authentifié 30s (AC-2), pipeline compression 3 étages + naming md5 post (AC-3/3a/3b/3c), réécriture md subset (AC-4/6), dédup intra-doc (AC-5), placeholders + counter `images_failed` (AC-7), `StateEntry.images` (AC-8), signature unchanged étendue (AC-9), cleanup orphelins (AC-10), archive no-op (AC-11), rename sidecar (AC-12), dry-run (AC-14), summary +3 champs (AC-15), PR draft étendu (AC-16), frontmatter inchangé (AC-17), path safety md5 regex (AC-18).

## Risks / open questions
- **OQ-2 — signature `objectId` dans markdown export Drive** : la spec demande investigation pré-plan. Hypothèse : l'export `text/markdown` Drive rend les `inlineObjects` sous une URL `lh*.googleusercontent.com/...` qui ne contient PAS l'`objectId` Docs directement → remapping `objectId ↔ position markdown` doit s'appuyer sur l'**ordre d'apparition** (n-ème image markdown ↔ n-ème inlineObject par traversal Docs API). **Investigation 1 doc réel obligatoire avant impl steps 5-6.** Si l'ordre Docs API traversal ne matche pas l'ordre markdown : fallback = reconstruire le markdown depuis la structure Docs directement → rescope significatif, alerter humain. **Non-bloquant pour ce plan, bloquant pour l'impl.**
- **LOC budget** : spec estime ~320 LOC src + ~160 LOC tests, dépasse plafond 300 LOC. Préférence humaine = ne pas splitter (cohérence AC-3 ↔ AC-9 sur md5 post-compression). Acter le dépassement (~320 LOC) en commit message ; pas de split ING-014-bis. Reviewer humain arbitre.
- **`StateEntry.images` ordre JSON** : `save_state` trie alpha → l'ordre lexical place `images` après `body_hash` (mécaniquement conforme à AC-8 « à la suite de body_hash »). Aucune logique custom requise.
- **`PIL.Image.MAX_IMAGE_PIXELS` global** : configuration au module-load de `images.py`. Effet de bord sur le process ; acceptable pour ce CLI offline. Si jamais workers/ consomme Pillow plus tard → re-évaluer (ADR-0007 amendment trigger).
- **`download_image` SSRF** : la spec acte que les URLs `lh*.googleusercontent.com` sont contrôlées Google (pas d'input user) → pas de validation CIDR. Documenté en Security spec L120. Conformité `.claude/rules/security.md` A10 acceptée par spec.
- **Pillow `DecompressionBombError`** : doit être catché et traité comme `ImageUndecodableError` (placeholder AC-7 `reason="undecodable"`). Vérifier le mapping exception dans tests.
- **Scope Docs API au boot** : la spec ING-014 demande fail-fast 403 « documents.readonly missing » au premier `documents.get`. Le scope est déjà ajouté côté `auth.py` (`_SCOPES`) ? **À vérifier en impl** : oui, `_SCOPES` actuel inclut `drive.readonly`, `spreadsheets.readonly`, `presentations.readonly`. **Manque `documents.readonly`** — l'ajouter dans `auth.py:_SCOPES`. Étendre `verify_extra_scopes` ? Spec demande fail-fast au premier appel `documents.get` (Failure modes L59), donc soit étendre `verify_extra_scopes` (probe Docs API), soit laisser le 403 naturel remonter (DriveApiError propagé). Décision plan : **étendre `verify_extra_scopes`** pour cohérence avec scopes Sheets/Slides ; ajouter assertion test.
- **Amendements specs adjacents** : ING-010 AC-5 / ING-012 AC-8 / ING-013 AC-11 nécessitent edit des `specs/acceptance/*.md` (humain-only). Plan **propose** les amendements, n'édite pas — humain les acte en pre-flight ou au moment du commit (cf CLAUDE.md « ne jamais modifier `specs/` sans approbation humaine »).
