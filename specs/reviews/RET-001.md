## Re-review of prior findings

### [HIGH] `main.py:35` — pgvector codec missing on lifespan pool — **CLOSED**

- `workers/src/archiviste_workers/main.py:33` now calls `from archiviste_workers.db import create_pool` and `pool = await create_pool(settings.database_url)`.
- `db.create_pool` (line 23-30 of `db.py`) wires `init=_init_connection`, where `_init_connection` calls `register_vector(conn)`. So every connection acquired from the prod pool now has the `vector` codec installed.
- The raw `asyncpg.create_pool` import was removed; `_asyncpg_dsn` helper deleted (URL normalisation is now centralized in `db.normalize_database_url`). One source of truth.
- **Regression test** `workers/tests/test_main_lifespan.py:27`:
  - Uses `async with lifespan(app):` against the real `archiviste_workers.main.lifespan` — i.e. the production code path, **not** a direct call to `db.create_pool`. Non-tautological.
  - Reads `app.state.db_pool` (the field the production retrieve handler consumes at `router.py:157`) and runs `await pool.fetchval("SELECT $1::vector", [0.0]*1024)`.
  - Asserts the round-trip equals the bind list. This locks in BOTH that the pool is created AND that the pgvector codec is installed for `vector` encoding/decoding.
  - Test ran (not skipped) in this environment: `tests/test_main_lifespan.py::test_lifespan_pool_encodes_pgvector PASSED [100%]`.
- The `pytest.skip` fallback inside the test catches `(OSError, ConnectionError, RuntimeError)` only — it doesn't swallow `asyncpg.DataError` or `TypeError`, which is what would surface if the codec was missing. So the test fails loudly on regression.

### [MED] `gateway-to-workers.yml:171` — dangling `Chunk` $ref — **CLOSED**

- `specs/openapi/gateway-to-workers.yml:172` now reads `$ref: '#/components/schemas/RetrievedChunk'` (line 171 carries a comment marking the rationale and a forward note to GEN-001).
- `grep -n Chunk specs/openapi/gateway-to-workers.yml` returns: lines 118, 126 (`RetrievedChunk:` schema definition), 171 (comment), 172 ($ref to `RetrievedChunk`). No remaining reference to `#/components/schemas/Chunk`.
- `redocly lint` reports 8 errors / 9 warnings — **none of them about `Chunk`**. The two `Can't resolve $ref` errors point at `#/components/schemas/Error` (used by the unused `BadRequest` / `RateLimited` response components), which is a pre-existing structural issue tracked outside RET-001.

### [MED] `retrieve/router.py:151` — blocking encode in async handler — **CLOSED**

- `workers/src/archiviste_workers/retrieve/router.py:154`: `[vector] = await asyncio.to_thread(embedder.encode_batch, [query], 1)`.
- Timing window correctness: `embedding_started = time.perf_counter()` is captured **before** `await asyncio.to_thread(...)` (line 151) and `log_fields["embedding_ms"] = _elapsed_ms(embedding_started)` is computed **after** the awaited call returns (line 155). So `embedding_ms` includes thread dispatch + encode + thread resume — i.e. the total wall-clock window the spec intends. Not a wall-clock-after-thread-resume artifact.
- AC-15 still holds (`embedding_ms >= 0`); integration test `test_happy_path_returns_chunk_shape` and `test_empty_corpus_returns_empty_chunks` continue to assert `>= 0` and pass against real Postgres + real bge-m3.
