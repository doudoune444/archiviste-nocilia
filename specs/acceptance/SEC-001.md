# SEC-001 — Auth tiers JWT (anonymous fingerprint / member signup / author)

## Contexte

Phase MVP : `user_tier="anonymous"` est hardcodé gateway → workers (vision §73, GEN-002 AC-3). Le ship beta INFRA-002 expose l'app sans authentification. Trois jalons produit exigent désormais une identité réelle : Mode 4 mystère + ACL (GEN-005) doit distinguer `members` vs `author_only` (vision §40), le dashboard auteur tickets (UI-002) doit être protégé, et le journal conversations doit attribuer un `user_id` stable. SEC-001 introduit les 3 tiers de la vision (`anonymous` / `member` / `author`), un JWT signé asymétrique côté gateway, des sessions server-side hashées révoquables et le throttling login conforme `security.md` §A02/A07.

## Critères d'acceptation

- AC-1 : `POST /v1/auth/signup` accepte `{"email": <string>, "password": <string>}` (`Content-Type: application/json`), crée une ligne dans la table `users` avec `tier="member"`, `password_hash` calculé via **argon2id** (paramètres exacts `m=19456 KiB, t=2, p=1`, sel de 16 octets `OsRng`), retourne `201 Created` avec body `{"user_id":"<uuid>","tier":"member"}`, et **ne pose pas** de cookie de session (signup ≠ login implicite).
- AC-2 : `POST /v1/auth/signup` retourne `409 Conflict` avec body `{"error":"email_taken","request_id":"<uuid>"}` si l'email (normalisé : `trim` + `toLowerCase`) existe déjà. Aucune révélation par timing : même latence ±50 ms p95 que le cas succès (la comparaison effectue un hash argon2id factice sur un sel constant si la ligne existe).
- AC-3 : `POST /v1/auth/signup` retourne `400 Bad Request` `{"error":"invalid_request","request_id":"<uuid>"}` si `email` ne matche pas la regex `^[^\s@]+@[^\s@]+\.[^\s@]+$` (longueur ≤ 254), OU si `password` < 12 caractères, > 128 caractères, ou contient `\0`.
- AC-4 : `POST /v1/auth/login` accepte `{"email": <string>, "password": <string>}`, vérifie le hash argon2id (timing-safe), crée une ligne dans la table `sessions` (`id` UUIDv4, `user_id` FK, `token_hash` = argon2id du token brut, `created_at`, `expires_at` = `created_at + 7 jours`, `revoked_at` NULL), retourne `200 OK` avec body `{"access_token":"<jwt>","token_type":"Bearer","expires_in":604800}` ET pose un cookie `archiviste_session=<jwt>; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=604800`.
- AC-5 : Le JWT émis par `/v1/auth/login` est signé en `alg=EdDSA` (Ed25519). Header obligatoire : `{"alg":"EdDSA","typ":"JWT","kid":"<key-id>"}`. Payload : `{"sub":"<user_uuid>","tier":"member"|"author","sid":"<session_uuid>","iat":<unix>,"exp":<iat+604800>,"iss":"archiviste-gateway","aud":"archiviste-gateway"}`. Toute valeur `alg` autre que `EdDSA` (`none`, `HS256`, `RS256`, autres) est rejetée à la vérification (cf. AC-12). La clé privée est chargée depuis `JWT_ED25519_PRIVATE_KEY_PEM` (Secret Manager via env var), jamais loggée, typée `secrecy::SecretString`.
- AC-6 : `POST /v1/auth/login` retourne `401 Unauthorized` `{"error":"invalid_credentials","request_id":"<uuid>"}` pour email inexistant OU password invalide — message identique dans les deux cas, latence identique ±50 ms p95 (hash argon2id systématique même si l'utilisateur n'existe pas, via un hash de référence constant).
- AC-7 : `POST /v1/auth/login` applique un throttling : après **5 échecs** (`401 invalid_credentials`) sur le même `email` (normalisé) dans une fenêtre de **15 minutes**, les tentatives suivantes pour ce même email retournent `429 Too Many Requests` `{"error":"login_throttled","request_id":"<uuid>","retry_after_seconds":<int>}` avec header `Retry-After: <int>` ; le throttle expire 15 min après le 5e échec. Un login réussi remet le compteur à zéro.
- AC-8 : `POST /v1/auth/logout` (authentifié, cf. AC-11) marque la session courante (`sid` du JWT) comme révoquée en base : `UPDATE sessions SET revoked_at=NOW() WHERE id=$sid`. Retourne `204 No Content` et pose `Set-Cookie: archiviste_session=; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=0`. Toute requête ultérieure portant ce JWT est rejetée `401 session_revoked` (cf. AC-13) — la vérification du `sid` server-side est obligatoire (pas de logout JWT-only, conforme `security.md` §A07).
- AC-9 : `GET /v1/me` retourne `200 OK` `{"user_id":"<uuid>","tier":"anonymous"|"member"|"author","fingerprint":"<sha256>"|null}` pour toute requête (publique). Pour `anonymous`, `user_id` est dérivé du fingerprint (cf. AC-10), `fingerprint` est le hash SHA-256 hex (64 chars) ; pour `member`/`author`, `fingerprint` est `null`.
- AC-10 : Pour toute requête entrante sans JWT valide, la gateway calcule un `fingerprint` = `SHA-256(<ip_v4_ou_v6>||"|"||<user_agent_string>||"|"||<cookie_anon_id>)`, où `<cookie_anon_id>` est lu depuis le cookie `archiviste_anon` (UUIDv4) ; si le cookie est absent, la gateway génère un UUIDv4 (`rand::rngs::OsRng`) et pose `Set-Cookie: archiviste_anon=<uuid>; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=31536000`. Le `user_id` pour `anonymous` est un UUIDv5 déterministe (`namespace = NIL`, `name = fingerprint_hex`). Aucune persistance en base pour les anonymes (pas de ligne `users`).
- AC-11 : Chaque route gateway (existante : `GET /`, `GET /assets/*`, `POST /v1/chat`, et nouvelles `POST /v1/auth/signup`, `POST /v1/auth/login`, `POST /v1/auth/logout`, `GET /v1/me`) déclare explicitement son exigence d'auth via un extractor `AuthUser` (résolution obligatoire — 401 si auth requise et absente) OU un marker `#[public]` (route ouverte aux `anonymous`). Une grep statique sur le code router montre que **chaque** handler porte exactement un des deux. Routes publiques phase 1 : `GET /`, `GET /assets/*`, `POST /v1/chat`, `POST /v1/auth/signup`, `POST /v1/auth/login`, `GET /v1/me`. Routes authentifiées (tier ≥ `member`) : `POST /v1/auth/logout`.
- AC-12 : Le vérificateur JWT (gateway) rejette avec `401 Unauthorized` `{"error":"invalid_token","request_id":"<uuid>"}` tout JWT dont : `alg` ≠ `EdDSA`, signature invalide, `exp` dépassé, `iat` futur (> now + 60s), `iss` ≠ `archiviste-gateway`, `aud` ≠ `archiviste-gateway`, `kid` inconnu, ou structure JWT malformée. Aucune trace stack / champ JSON ne distingue ces cas dans la réponse (un seul code `invalid_token`).
- AC-13 : Un JWT structurellement valide dont `sid` correspond à une `sessions.revoked_at IS NOT NULL` OU `sessions.expires_at < NOW()` OU `sessions.id` introuvable est rejeté `401 Unauthorized` `{"error":"session_revoked","request_id":"<uuid>"}`. La vérification frappe la table `sessions` à chaque requête authentifiée (pas de cache phase 1).
- AC-14 : La gateway propage le tier vers les workers via le header `X-User-Tier: anonymous|member|author` sur **toutes** les requêtes sortantes vers workers (le hardcoding `anonymous` actuel disparaît). Le header `X-User-Id: <uuid>` est ajouté en parallèle. `specs/openapi/gateway-to-workers.yml` est mis à jour pour décrire ces deux headers comme `required: true` sur les opérations existantes (`/v1/generate`, futures opérations workers).
- AC-15 : Le tier `author` est attribué exclusivement via un seed SQL (migration ou script `scripts/seed_author.sql`) qui `INSERT INTO users (email, password_hash, tier) VALUES ($email, $hash, 'author')` où `$email` est l'adresse auteur et `$hash` est un argon2id calculé hors-ligne par l'humain. **Aucun endpoint runtime ne permet de promouvoir un user en `author`**. La contrainte SQL `CHECK (tier IN ('member','author'))` interdit toute autre valeur ; une contrainte applicative supplémentaire au point de signup force `tier='member'`.
- AC-16 : Toute route dont l'extractor `AuthUser` exige `tier='author'` (phase 1 : aucune dans ce ticket, mais le mécanisme est livré et testé) retourne `403 Forbidden` `{"error":"author_required","request_id":"<uuid>"}` si le tier du JWT est `member` ou si la requête est anonyme. Test contractuel : un handler de test gated `author_only` couvre ce cas dans `gateway/tests/test_auth.rs`.
- AC-17 : Les routes `POST /v1/auth/{signup,login,logout}` rejettent `415 Unsupported Media Type` si `Content-Type` ≠ `application/json`, et `413 Payload Too Large` si body > 4 KiB (limite spécifique aux routes auth, plus stricte que la limite générale `POST /v1/chat` = 1 MiB de GEN-002).
- AC-18 : Tout événement d'auth émet une entrée `tracing` structurée niveau `info` (succès : `auth.signup.ok`, `auth.login.ok`, `auth.logout.ok`) ou `warn` (échec : `auth.signup.email_taken`, `auth.login.invalid_credentials`, `auth.login.throttled`, `auth.token.invalid`, `auth.session.revoked`) avec champs `request_id`, `event`, `email_sha256` (jamais l'email en clair), `tier`, `latency_ms`, `ip_sha256`. **Aucun** champ ne contient l'email en clair, le password (hashé ou non), le token brut, ni le `token_hash`.
- AC-19 : Une métrique compteur `auth_failures_total{event="invalid_credentials"|"throttled"|"invalid_token"|"idor_attempt"}` est exposée par la gateway (OTel ou équivalent prêt pour OBS-001) ; cohérent avec `security.md` §A09 ("Failed auth + IDOR attempts go to alerting channel"). L'instrumentation est posée même si l'alerte aval n'est pas câblée phase 1 (OBS-001 V2 branchera la sink).
- AC-20 : Migration `migrations/0003_users_auth.up.sql` (a) `ALTER TABLE users ADD COLUMN email TEXT`, `ADD COLUMN password_hash TEXT` — les deux nullable, car la ligne sentinel `00000000-0000-0000-0000-000000000000` `tier='anonymous'` créée par migration 0002 est préservée (référencée par FK `query_log.user_id` et `conversations.user_id`) ; (b) `ALTER TABLE users ADD CONSTRAINT users_auth_consistency CHECK (tier = 'anonymous' OR (email IS NOT NULL AND password_hash IS NOT NULL))` ; (c) `CREATE UNIQUE INDEX users_email_lower_uq ON users (LOWER(email)) WHERE email IS NOT NULL` ; (d) `CREATE TABLE sessions (id UUID PRIMARY KEY, user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE, token_hash TEXT NOT NULL, created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), expires_at TIMESTAMPTZ NOT NULL, revoked_at TIMESTAMPTZ)` ; (e) `CREATE INDEX sessions_user_id_idx ON sessions(user_id)`. La contrainte CHECK existante migration 0002 `tier IN ('anonymous','member','author')` reste inchangée (le tier `'anonymous'` reste valide pour le sentinel). Fichier down `migrations/0003_users_auth.down.sql` drop `sessions`, drop l'index unique email, drop la contrainte `users_auth_consistency`, drop colonnes `email` + `password_hash` (sentinel préservé).
- AC-21 : Le calcul fingerprint anonyme (cf. AC-10 composante `<ip_v4_ou_v6>`) lit l'IP cliente avec priorité stricte : (a) header `CF-Connecting-IP` si présent → utilise la valeur brute du header (V1 beta : Cloudflare = seul ingress public, vision Q1) ; (b) sinon fallback `ConnectInfo<SocketAddr>` Axum (dev local sans front Cloudflare). Les headers `X-Forwarded-For` et `X-Real-IP` sont ignorés phase 1 (spoofing trivial sans Cloudflare). Aucun rejet si `CF-Connecting-IP` malformé ou IP invalide : la valeur est utilisée telle quelle (fingerprint reste déterministe ; un attaquant connaît déjà sa propre IP, le risque résiduel est nul).

## Non-goals

- Pas d'OAuth providers (GitHub / Google / autres) — V2.
- Pas de flow de vérification email (lien magique, double opt-in) — ticket séparé si scope dépasse 300 LOC.
- Pas de reset password / forgot password — V2.
- Pas de MFA concrète (TOTP / WebAuthn / SMS) phase 1. La protection des routes `author_only` se limite au tier check (AC-16). Un ADR ultérieur (`docs/adr/NNNN-mfa-author.md`) couvrira la MFA effective.
- Pas de rotation automatique de la clé Ed25519 phase 1 (un seul `kid` actif, rotation manuelle via redéploiement). Mécanisme de multi-`kid` reporté à un ticket SEC-* dédié.
- Pas de refresh token / pas de short-lived access + long-lived refresh — un seul JWT 7 jours phase 1.
- Pas d'invalidation globale "logout all devices" exposée à l'utilisateur — endpoint reporté.
- Pas de page de signup / login frontend (UI auth phase 1 = `curl` ou DevTools ; UI-XXX ticket dédié).
- Pas de dashboard auteur tickets dans ce ticket (UI-002, qui consommera `tier='author'`).
- Pas de migration des conversations anonymes existantes vers un `user_id` member après signup (pas de "claim" d'historique phase 1).
- Pas de rate-limit applicatif global (`tower_governor`) — couvert par Cloudflare V1 (vision Q9) + SEC-002 V2. Seul le throttle `/v1/auth/login` (AC-7) est applicatif.
- Pas de support `Authorization: Bearer <jwt>` ET cookie simultanément avec priorité — phase 1 le cookie suffit (frontend same-origin) ; le header `Authorization` est accepté en parallèle mais le cookie reste prioritaire si les deux sont présents et divergent.
- Pas de chiffrement du `token_hash` au-delà de l'argon2id (pas de pepper, pas de HSM phase 1).
- Pas de tests de charge spécifiques auth phase 1 (OPS-001 couvrira l'ensemble).
- Pas de tests fuzz / property-based phase 1 (couverture par tests d'intégration sur les chemins nommés AC-1 → AC-20).
- Pas d'audit log dédié table `audit_logs` — les `tracing` structurés (AC-18) suffisent phase 1, table dédiée = OBS-001 / SEC-* V2.

## Pre-conditions

- INFRA-002 mergé : Cloud SQL Postgres 16 + extension `vector` opérationnel, Secret Manager câblé via env vars Cloud Run.
- GEN-002 mergé : `POST /v1/chat` opérationnel, contrat `X-Request-Id` posé, enveloppe d'erreur `{"error":"<code>","request_id":"<uuid>"}` figée (réutilisée par toutes les nouvelles routes auth).
- ADR à créer en amont de `/plan SEC-001` : `docs/adr/NNNN-jwt-eddsa-and-server-side-sessions.md` (choix EdDSA vs RS256, choix sessions server-side hashées vs JWT stateless, paramètres argon2id) — exigé par `CLAUDE.md` "no ADR = no heavy dep" car `jsonwebtoken` (Rust) ou équivalent et `argon2` crate sont des deps lourdes nouvelles.
- Clé Ed25519 générée hors-ligne par l'humain (`ssh-keygen -t ed25519` ou équivalent OpenSSL), partie privée stockée dans Secret Manager `MISTRAL_API_KEY`-style (nouveau secret `JWT_ED25519_PRIVATE_KEY_PEM`), partie publique stockée dans `JWT_ED25519_PUBLIC_KEY_PEM` (utilisé par les workers si vérification distribuée future, phase 1 = gateway uniquement). Procédure documentée dans `docs/runbook.md` ou ADR.
- `gateway/Cargo.toml` peut accueillir les nouvelles deps : `jsonwebtoken` ou `jose`-compatible EdDSA, `argon2`, `secrecy` (déjà), `uuid` (déjà), `time`/`chrono` (déjà). Listées dans l'ADR.
- `scripts/seed_author.sql` ou équivalent rédigé par l'humain ; l'agent ne génère pas le hash argon2id du password auteur (rule `secret-hygiene.md`).
- `sqlx-cli` opérationnel et `cargo sqlx prepare` exécutable post-migration (rule `vertical-slice.md` §Order).

## Failure modes

- Clé privée Ed25519 absente / mal formatée au boot gateway → la gateway refuse de démarrer avec un log `fatal: jwt_private_key_load_failed` (aucun fallback, aucun "default key for dev"). Cohérent `secret-hygiene.md` "no fallback default in code".
- Postgres indisponible au moment d'un `/v1/auth/login` → `503 Service Unavailable` `{"error":"upstream_unavailable","request_id":"<uuid>"}`, latence ≤ timeout DB (5s).
- Postgres indisponible au moment d'une vérif `sid` (AC-13) sur route authentifiée → `503 Service Unavailable` `{"error":"upstream_unavailable","request_id":"<uuid>"}`. Pas de bypass "fail-open" — la rule `no-workaround.md` impose stop.
- Collision UUIDv4 sur `users.id` ou `sessions.id` (probabilité ~0) → l'INSERT échoue sur la contrainte PK, la gateway renvoie `500 internal`, log `warn`. Pas de retry phase 1.
- Cookie `archiviste_anon` falsifié (UUID rejouée par un attaquant pour usurper un anonyme) → impact limité : l'anonyme n'a aucune donnée privée associée (pas de persistance). Documenté comme acceptable phase 1, MFA réelle = author tier uniquement.
- JWT volé via XSS (mitigé par CSP UI-001 + cookie HttpOnly) → impact = session compromise jusqu'à `expires_at` (7j) OU jusqu'au prochain logout serveur (mécanisme `sessions.revoked_at` permet rotation manuelle).
- Throttle login (AC-7) contourné par énumération inter-emails (attaquant essaie 1 password sur 1000 emails) → mitigé en V2 par rate-limit Cloudflare 100 req/min/IP (vision Q9) ; phase 1 = acceptable.
- Migration appliquée puis rollback nécessaire (incident) → fichier `migrations/NNNN_users_sessions.down.sql` drope `sessions` puis `users` proprement (FK ON DELETE CASCADE).
- Email contenant des caractères Unicode exotiques (homoglyphes) → normalisation NFKC + lowercase au signup ET au login ; deux emails visuellement identiques mais Unicode-différents sont rejetés `email_taken` (collision sur l'index `LOWER(email)` après NFKC).

## Test oracle

- AC-1 : intégration · `gateway/tests/test_auth.rs::signup_creates_member` — POST valide → 201, body shape, ligne SQL `SELECT tier FROM users WHERE id=$id` = `member`, `password_hash` non vide ne contient pas le password en clair.
- AC-2 : intégration · `signup_email_taken_returns_409` — POST 2x même email, second = 409 `email_taken`. Mesure latence `histogram` sur 50 essais success vs 50 essais collision : médiane delta ≤ 50 ms (test informatif, pas hard-fail CI phase 1, voir Open question).
- AC-3 : intégration property-like · table de cas (`""`, `"a@"`, `"a@b"`, `"a@b.c"` OK, email > 254 chars, password 11 chars, password 129 chars, password contenant `\0`) → assertion 400 `invalid_request` sur les cas invalides, 201 sur les cas valides.
- AC-4 : intégration · `login_success_creates_session_and_cookie` — signup puis login → 200, body shape, ligne `sessions` créée, `Set-Cookie: archiviste_session=...` avec attributs `HttpOnly`, `Secure`, `SameSite=Lax`, `Max-Age=604800`.
- AC-5 : contract · décoder le JWT retourné par AC-4 avec une crate JOSE en mode "headers only" (sans vérif sig) → header contient `alg=EdDSA`, `typ=JWT`, `kid` non vide ; payload contient `sub`, `tier`, `sid`, `iat`, `exp=iat+604800`, `iss`, `aud` aux valeurs attendues.
- AC-6 : intégration · `login_wrong_password_returns_401` + `login_unknown_email_returns_401` — même body `invalid_credentials`. Mesure latence comme AC-2 (informatif).
- AC-7 : intégration · `login_throttled_after_five_failures` — 5 POST avec mauvais password sur même email → 5 × 401 ; 6e POST (même email, password correct ou incorrect) → 429 `login_throttled`, header `Retry-After` présent. Avancer un fake clock de 15 min → POST suivant = 200 (succès) ou 401 (échec) selon password. Login réussi entre deux échecs (4e essai correct) remet le compteur à 0.
- AC-8 : intégration · `logout_revokes_session` — login → logout (avec cookie/JWT du login) → 204 + `Set-Cookie: ...Max-Age=0`. POST `/v1/auth/logout` à nouveau avec le même JWT → 401 `session_revoked`. `SELECT revoked_at FROM sessions WHERE id=$sid` non NULL.
- AC-9 : intégration · 3 sous-cas : (a) requête anonyme sans cookie → 200, `tier=anonymous`, `fingerprint` 64 hex chars, cookie `archiviste_anon` posé en réponse. (b) requête anonyme avec cookie `archiviste_anon` existant → même `user_id` (UUIDv5 déterministe). (c) requête avec JWT member valide → 200, `tier=member`, `fingerprint=null`.
- AC-10 : intégration · `anonymous_user_id_is_deterministic` — 2 requêtes successives même IP+UA+cookie → même `user_id` dans `GET /v1/me`. Changer le User-Agent → `user_id` différent. Vérifier que `user_id` matche `uuid5(NIL, sha256_hex(ip||"|"||ua||"|"||anon_cookie))`.
- AC-11 : contract / grep statique Rust · test `gateway/tests/test_auth.rs::every_route_has_auth_marker` qui inspecte le router (ou un grep AST simple) et assert que chaque handler est associé soit à `AuthUser`, soit à `#[public]`. Phase pragmatique : grep sur `gateway/src/lib.rs` et fichiers `routes/` pour les patterns autorisés.
- AC-12 : intégration / property-like · table de JWT malformés (alg=none, alg=HS256 même clé publique, sig altérée, exp dépassé, iat futur > 60s, iss faux, aud faux, kid inconnu, JWT tronqué) → 401 `invalid_token` pour chaque.
- AC-13 : intégration · `revoked_session_rejected` — login → révoque manuellement en SQL → requête sur route auth = 401 `session_revoked`. Idem `expired_session_rejected` en backdate `expires_at`. ## Performance / SLO

- Vérification `AuthUser` (parse JWT + check session DB) : **< 5 ms p95** sur le hot path (route authentifiée). Atteindre la cible exige une requête `sessions` indexée sur PK (`sid`) — pas de cache phase 1 mais latence Postgres locale < 2 ms typique.
- `POST /v1/auth/login` : ≤ 300 ms p95 (dominé par argon2id avec paramètres AC-1 ; ~150-200 ms typique sur Cloud Run `db-f1-micro`).
- `POST /v1/auth/signup` : ≤ 350 ms p95 (argon2id + INSERT + même latence factice côté `email_taken`).
- `GET /v1/me` (anonyme) : ≤ 10 ms p95 (calcul SHA-256 + UUIDv5, pas de DB).
- Overhead Rust gateway global (vision §92, `< 80 ms p95 à 500 users`) : doit rester respecté ; SEC-001 ajoute ≤ 5 ms p95 par requête authentifiée (cf. cible auth check ci-dessus).

## Security / trust boundary

- Frontière Internet → Gateway. Surface étendue : nouvelles routes auth, nouveaux secrets (clé Ed25519), nouvelle table `users` (PII = email + hash).
- A01 Broken Access Control : chaque route déclare son exigence d'auth (AC-11). IDOR sur `GET /v1/me` impossible — la route renvoie strictement le user dérivé du JWT/fingerprint courant, pas de paramètre d'ID. `POST /v1/auth/logout` ne prend pas non plus de `session_id` en query — le `sid` vient du JWT.
- A02 Cryptographic Failures : JWT alg pin `EdDSA` strict (AC-5, AC-12) ; `alg: none` et tout autre alg rejetés ; argon2id paramètres exacts m=19456 KiB, t=2, p=1 (AC-1) ; aucune utilisation de bcrypt ; `rand::rngs::OsRng` pour UUIDv4 cookie et sels (AC-1, AC-10) ; clé privée typée `secrecy::SecretString`, jamais loggée.
- A03 Injection : SQL via `sqlx::query!` macros uniquement (compile-checked). Email normalisé (NFKC + lowercase) avant `LOWER(email)` lookup — pas de `format!` dans la requête.
- A04 Insecure Design : limites strictes (body 4 KiB sur routes auth, AC-17) ; throttle login server-side (AC-7) ; cookie HttpOnly Secure SameSite=Lax ; pas de refresh token (réduit surface) ; sessions server-side révocables (AC-8) plutôt que JWT-only logout (`security.md` §A07 explicite).
- A05 Misconfiguration : erreurs auth retournent une enveloppe stable `{"error":"<code>","request_id":"<uuid>"}` sans stack trace, sans diagnostic DB, sans distinction "user inconnu vs password faux" (AC-6, AC-12). Endpoints debug d'auth interdits.
- A07 Authentication Failures : login throttle 5 / 15 min (AC-7) ; session tokens 32 bytes random hashés argon2id server-side (AC-4) ; logout invalide server-side (AC-8) ; MFA mandatory `author` = mécanisme tier check livré (AC-16), implementation MFA concrète = ADR ultérieur (non-goal explicite).
- A08 Software & Data Integrity : nouvelles deps (`jsonwebtoken` ou JOSE crate, `argon2`) listées dans ADR, lock file `Cargo.lock` mis à jour, `cargo deny check` clean.
- A09 Logging & Monitoring : email jamais loggé en clair (AC-18, `email_sha256` uniquement), password jamais loggé, token brut jamais loggé. Métriques `auth_failures_total` (AC-19) couvrent failed auth + future IDOR (sink alerting = OBS-001 V2).
- A10 SSRF : non applicable à ce ticket (aucun fetch d'URL utilisateur).
- Conversation persistence (`security.md` §RAG-specific) : `user_id` désormais réel pour members/author ; les conversations existantes anonymes (UUIDv5 fingerprint) restent isolables. Pas de path traversal possible — `conversation_id` reste UUIDv4 client-side (UI-001 AC-9 inchangé).

## Observability

- `tracing` structured events nommés (AC-18) : `auth.signup.ok`, `auth.signup.email_taken`, `auth.login.ok`, `auth.login.invalid_credentials`, `auth.login.throttled`, `auth.logout.ok`, `auth.token.invalid`, `auth.session.revoked`.
- Champs systématiques : `request_id`, `event`, `email_sha256` (ou `null` pour `/v1/me`), `tier`, `latency_ms`, `ip_sha256`. **Interdits** : email clair, password (toute forme), token brut, token_hash, full JWT.
- Métrique counter : `auth_failures_total{event}` (AC-19) prêt pour OBS-001 (sink alerting via Cloud Logging log-based metric ou OTel collector).
- Pas d'alerte câblée phase 1 (reportée OBS-001).
- Pas de tracing Langfuse phase 1 (auth = hors pipeline LLM).
