# Plan — GEN-004 Mode 3 lore-gap : branche retrieval-faible + ticket auto avec dedup cosine

## Goal
Livrer Mode 3 lore-gap end-to-end : sur `intent="in_domain"` + `max_score(retrieve) < 0.45`, le pipeline répond « noté pour archives » via un 2e appel LLM zero-shot et crée/incrémente un ticket lore-gap dédupliqué par cosine ≥ 0.85 sur l'embedding de la question — sans bloquer la réponse 200 sur défaillance ticket/embed.

## Risks / open questions

- **R1 (GEN-003 non mergé)** — précondition spec (« GEN-003 mergé : classifier en tête, `Mode=["canon","off_topic"]`, `query_log.intent` renseigné ») non satisfaite sur `main` au moment du plan : `services/query_log.py:37` bind `intent=NULL` hardcoded, aucun module classifier, `Mode=Literal["canon"]`. **Blocker pour PR2**. Décision attendue humain : (a) attendre GEN-003 merge avant /implement PR2, ou (b) plan GEN-003+GEN-004b fusionné. PR1 (migration + service) reste mergeable indépendamment.
- **R2 (LOC total > 300)** — spec estime 290-360 LOC prod. Split D10 appliqué : PR1 ~85 LOC prod (service + migration ~30 SQL), PR2 ~135 LOC prod (router + prompt + models). Chaque PR ≤ 300 LOC vertical-slice respecté.
- **R3 (HNSW partial index `WHERE` clause)** — supporté pgvector ≥ 0.5.0 (released 2023-08). Docker compose et Cloud SQL pin à vérifier au step 1 PR1. Si version < 0.5, blocker → ADR upgrade pgvector OU index full + filtrage SQL (mais perte d'efficacité). No workaround.
- **R4 (D9 — chunks=[] → lore_gap)** — comportement souhaité spec (`max_score=0.0 < 0.45` → lore_gap). Sémantiquement imparfait vs Mode 4 ACL-blocked (GEN-005 non livré). Acceptable V1.
- **R5 (Chunk model AJOUT `score`)** — modèle `workers/src/archiviste_workers/generate/models.py::Chunk` actuel = `{source_path, ord, text}` (pas de `score`). RET-001 expose `score` dans `RetrievedChunk` mais `RetrieveClient` re-mappe via `generate.models.Chunk` qui drop le score. **Choix** : ajouter `score: float` à `generate.models.Chunk` + propagation dans `RetrieveClient.search`. +2-3 LOC, sans churn ailleurs (le builder `build_messages` canon n'utilise pas score, donc invisible côté canon).
- **R6 (ordre ING-003 fail / ticket skip)** — failure modes spec stipule que sur échec ING-003, `ticket_service` n'est PAS appelé (éviter FK orpheline si conversation pas créée). Log ALERT dédié `ticket_service_skipped_after_conversation_log_failed`. Implémenté dans router.py step 12.
- **R7 (D3 — embedder swap INFRA-002)** — INFRA-002 (ordre #2) swap BGE-M3 → mistral-embed déjà mergé selon vision §161. Si retard, embeddings tickets pré-swap deviendraient incompatibles cross-modèles. Out-of-scope ce ticket ; signalé pour traçabilité (script `scripts/reembed_tickets.py` futur).
