# ING-010 — Sync Google Drive : utilities lib (slugify, state, frontmatter merge, paths, normalize, rename)

## Contexte

La vision §52 fige la phase 3 d'ingestion : un dossier Google Drive auteur sert de source amont, exporté en `*.md` versionnés sous `lore/`. Ce ticket livre **uniquement la couche utilitaire pure** du futur sync (slugification, state file I/O, frontmatter merge, résolution paths, normalisation Markdown, rename local) sous forme de package `scripts/gdrive_export/` testable indépendamment.

L'intégration Drive API (auth service account, listing récursif, export gdoc, download PNG, orchestrateur sync, entrypoint CLI, runbook, `.env.example`) est **déférée à ING-013** qui dépendra de ce ticket mergé.

Ce split garantit ≤ 300 LOC src côté utilities et permet de shipper / tester la lib sans dépendance Drive API.

## Critères d'acceptation

- AC-1 : Un package Python `scripts/gdrive_export/` existe avec `pyproject.toml` isolé (lockfile `uv.lock` propre, distinct de `workers/`). `cd scripts && uv sync` installe les deps. `cd scripts && uv run pytest` passe sans dépendance Drive API.
- AC-2 : Le module `scripts/gdrive_export/slugify.py` expose `slugify(text: str, fallback_id: str) -> str` qui applique : `unicodedata.normalize('NFKD')` → drop combining marks → lower → remplacement de tout caractère hors `[a-z0-9]` par `-` → collapse runs de `-` → strip `-` aux extrémités → cap 80 chars. Si le slug résultant est vide, retourne `file-<fallback_id[:8]>`.
- AC-3 : `slugify` est déterministe et idempotent (`slugify(slugify(s, fid), fid) == slugify(s, fid)`). Sortie matche `^[a-z0-9-]+$` ou `^file-[0-9a-f]{8}$`. Longueur ≤ 80.
- AC-4 : Le module `scripts/gdrive_export/normalize.py` expose `normalize_body(text: str) -> str` qui applique `unicodedata.normalize('NFKC', text)` puis dépouille les caractères de contrôle `\x00`-`\x1f` à l'exception de `\n` et `\t` (cohérence ING-001 AC-4, threat-model W-T-1).
- AC-5 : Le module `scripts/gdrive_export/state.py` expose :
  - dataclass `StateEntry` (`local_path: str`, `drive_md5_checksum: str`, `last_exported_at: str` ISO8601 UTC, `body_hash: str` sha256 hex, `archived_at: str | None`).
  - `load_state(path: Path) -> dict[str, StateEntry]` : lit JSON, raise `StateCorruptedError` si non-JSON ou shape invalide ; retourne `{}` si fichier absent (pas de re-init silencieuse côté écriture, mais lecture absent = first run).
  - `save_state(path: Path, state: dict[str, StateEntry]) -> None` : écrit JSON canonique trié (`sort_keys=True`, `indent=2`, `ensure_ascii=False`).
  - `compute_body_hash(body: str) -> str` : sha256 hex sur les bytes UTF-8 du corps.
- AC-6 : Le module `scripts/gdrive_export/frontmatter_merge.py` expose `merge_frontmatter(existing_yaml: str | None, script_managed: dict, defaults_user: dict) -> str` qui :
  - Si `existing_yaml is None` (premier export) : produit un YAML avec `script_managed` ∪ `defaults_user`.
  - Si `existing_yaml` est fourni et parse correctement : écrase les clés `script_managed` (`title`, `source`, `source_id`, `drive_path`, `exported_at`, `archived`, `archived_at`), préserve les clés `defaults_user` (`tags`, `access_tier`) si déjà présentes (sinon crée avec défaut), préserve toute clé custom inconnue intacte.
  - Si `existing_yaml` invalide YAML : raise `FrontmatterMergeError` (caller décide d'abort vs skip).
- AC-7 : Le module `scripts/gdrive_export/paths.py` expose `resolve_local_path(drive_path_components: list[str], slug: str, ext: str, taken_paths: set[Path], drive_file_id: str, root: Path) -> Path` qui : (a) slugifie chaque composant via AC-2, (b) joint sous `root` (POSIX), (c) si le path résultant `<root>/<slugified_components>/<slug><ext>` est déjà dans `taken_paths`, suffixe le stem avec `-<drive_file_id[:8]>` et réessaye ; (d) garantit que tout path retourné reste sous `root` (assert `path.is_relative_to(root)` après résolution).
- AC-8 : Le module `scripts/gdrive_export/rename.py` expose `rename_local_file(old_path: Path, new_path: Path) -> None` qui : (a) crée les parents de `new_path` si absents, (b) appelle `old_path.rename(new_path)` (Python `pathlib`), (c) raise `FileNotFoundError` si `old_path` absent, raise `FileExistsError` si `new_path` existe déjà. **Aucun `subprocess`, aucun `git mv`, aucune commande shell**. La détection git de rename (similarité ≥ 50 %) se fait au commit humain via `git status` / `git diff`.
- AC-9 : Test intégration de rename : un fichier tracké git est créé, `rename_local_file(old, new)` est invoqué dans un repo git tmp ; `git status --porcelain` post-rename affiche bien une ligne `R` (rename détecté par git via similarity ≥ 50 %), pas de paire `D`/`??`. Oracle Python : `subprocess.run(["git", "status", "--porcelain"], ...)` (lecture seule, argv array, pas de mutation) parse la sortie et asserte `R` présent.
- AC-10 : Le `pyproject.toml` de `scripts/` configure `ruff` (rules cohérentes avec `workers/pyproject.toml` : T20, PT, RET, PL, N, FBT, S, B, UP) et `mypy` strict (`disallow_untyped_defs`, `strict_optional`, `warn_return_any`). `cd scripts && uv run ruff check . && uv run mypy .` passe sans warning.
- AC-11 : Le `.pre-commit-config.yaml` racine est étendu avec un hook `local` qui exécute `cd scripts && uv run ruff check .` et `cd scripts && uv run mypy .` sur tout fichier matchant `^scripts/.*\.py$`, scopant l'exécution au package `scripts/` uniquement.
- AC-12 : Le `.gitignore` racine est étendu avec : `.gcp/` (futur dossier credentials local non-commitable), `scripts/.gdrive_state.json` (state runtime, sera versionné en ING-013 mais tracké manuellement), `scripts/.gdrive_state.json.bak` (backup atomique éventuel).
- AC-13 : ADR `docs/adr/0006-gdrive-api-client.md` créé, status `accepted`, justifie l'isolation lockfile `scripts/` séparé du venv `workers/` et l'acceptation `google-api-python-client` côté `scripts/` uniquement (sera ajoutée en ING-013, pas dans ce ticket).
- AC-14 : Aucun module de ce ticket n'importe `googleapiclient`, `google.auth`, `httplib2`, ni n'effectue d'appel réseau. Vérifié par grep CI : `cd scripts && grep -rE "googleapiclient|google\.auth|httplib2|requests\.|httpx\." src/ → exit non-zéro` (test conftest ou hook simple).
- AC-15 : Tous les modules sont strictement typés (mypy strict) et chaque fonction publique exposée en AC-2/4/5/6/7/8 a une docstring courte décrivant l'invariant. Aucune fonction > 40 lignes (cf `clean-code.md`).

## Failure modes

- `state.json` corrompu / non-JSON → `load_state` raise `StateCorruptedError` (caller décide).
- Frontmatter local YAML invalide → `merge_frontmatter` raise `FrontmatterMergeError`.
- Slug résultant vide → fallback `file-<fallback_id[:8]>`, pas une erreur.
- Path traversal tentative (composant Drive contenant `..` ou `/`) → slugifié, donc `..` devient `-` ou vide → fallback ; assertion finale `is_relative_to(root)` empêche toute évasion.
- `rename_local_file` invoqué avec `old_path` absent → `FileNotFoundError` (pas catché silencieusement).
- `rename_local_file` invoqué avec `new_path` déjà existant → `FileExistsError` (pas d'overwrite silencieux).

## Test oracle

- AC-2, AC-3 : unit + property (hypothesis) sur `slugify`. Cas matriciel : Unicode (NFKD), latin diacritiques, CJK, emojis (drop), espaces, tirets multiples, longueur > 80, chaîne vide, chaîne uniquement ponctuation.
- AC-4 : unit · fixture corps avec zero-width chars + NULL bytes + NFD séquences → corps normalisé NFKC, control chars stripped (sauf `\n`, `\t`).
- AC-5 : unit · round-trip JSON canonique (charge → sauve → charge) ; cas corruption (fichier non-JSON, shape invalide) ; cas absent (premier run = `{}`).
- AC-6 : unit · matrice de cas — premier export (existing_yaml=None) ; ré-export avec `tags`/`access_tier` user-modifiés (préservés) ; ré-export avec clé custom `priority: high` (préservée) ; YAML local invalide (raise).
- AC-7 : unit · 2 file IDs distincts produisant le même slug → second suffixé `-<id[:8]>` ; assertion `is_relative_to(root)` ; tentative path traversal (composant `..`) → slugifié vers fallback safe.
- AC-8 + AC-9 : intégration · `tmp_path` initialisé en repo git (`subprocess git init` dans le test seul, pas dans le code prod) ; créer fichier, `git add` + `git commit` ; appeler `rename_local_file(old, new)` ; lire `git status --porcelain` ; asserter présence ligne `R`.
- AC-10 : CI · `cd scripts && uv run ruff check . && uv run mypy .` exit 0.
- AC-11 : pre-commit · `pre-commit run --all-files` déclenche le hook scripts uniquement si fichier `^scripts/` modifié ; vérifié manuellement par humain au merge.
- AC-12 : grep `.gitignore` post-merge contient les 3 entrées attendues.
- AC-13 : présence `docs/adr/0006-gdrive-api-client.md` avec status `accepted`, date `2026-05-09`.
- AC-14 : test conftest ou pre-commit hook local : `grep -rE "googleapiclient|google\.auth|httplib2|requests\.|httpx\." scripts/gdrive_export/ → exit 1`.

## Decisions résolues (humain, 2026-05-09)

- Convention séparateur path : `--` figé (AC-7 utilise `-<8hex>` pour suffixe collision file-id).
- Frontmatter merge (AC-6) : `title`, `source`, `source_id`, `drive_path`, `exported_at`, `archived`, `archived_at` script-managed (overwrite). `tags`, `access_tier` user-managed (préservés si présents, créés au défaut au premier export). Clés custom préservées intactes.
- Split du périmètre original ING-010 en quatre sous-tickets (humain 2026-05-09) :
  - **ING-010** (ce ticket) = utilities pures (slugify, state, frontmatter merge, paths, normalize, rename) + ADR-0006 + isolation lockfile + pre-commit + .gitignore.
  - **ING-013** = Drive API integration (auth SA, drive_client, sync orchestrator, main entrypoint argparse, intégration tests Drive mocked, runbook, `.env.example`).
  - **ING-011** = converters gsheet multi-onglets (séparateur `--`) + gslide (`## Slide N`).
  - **ING-012** = workflow GHA `gdrive-sync.yml` + auto-PR via `peter-evans/create-pull-request@v6`.

## Décisions résolues (humain, 2026-05-09 — round 2)

- **OQ-A — drift corps `.md` ↔ Drive (ex-AC-20)** : déféré ING-013 (logique de drift implique sync orchestrator). La primitive `compute_body_hash` (AC-5) est livrée ici, son usage est en ING-013.
- **OQ-B — rename Drive d'un PNG (ex-AC-9)** : la primitive `rename_local_file` (AC-8) est livrée ici, identique pour gdoc et PNG. Le câblage detect-rename + log dédié `gdrive_sync.image_renamed` est en ING-013.
- **OQ-C — cross-link image dans corps gdoc** : HORS-SCOPE des deux tickets ING-010 / ING-013. Embedding + résolution cross-fichier = ING-002 (vision phase 2).
- **OQ-N — Rename mechanism** : `Path.rename()` pure, pas de `subprocess` `git mv` (security: zero argv surface, élimine toute injection shell par construction). Git détecte les renames via similarity ≥ 50 % au commit humain (`git status --porcelain` montre `R`). Test oracle AC-9 mis à jour en conséquence.
