# OBS-008 — Image conteneur de l'eval runner (build CI + push Artifact Registry)

## Contexte

Le runner Ragas (`eval/`) tourne aujourd'hui uniquement via `uv run` en CI et en local ; aucune image conteneur n'existe, donc OBS-009 (Cloud Run Job qui exécute l'eval live en prod, fetch golden GCS) n'a rien à déployer. Ce ticket livre une image Docker reproductible du runner — code + deps lockées + `baseline.json` bakés — construite et poussée vers l'Artifact Registry `archiviste` existant à chaque merge sur `main`, sous `eval/${sha}` immutable + `eval/:latest`. C'est la slice intermédiaire de l'EPIC OBS-EVAL entre OBS-007 (auth OIDC code-only, mergé) et OBS-009 (Cloud Run Job aval).

## Acceptance criteria

- AC-1 : `infra/docker/eval.Dockerfile` existe, son contexte de build est `eval/`, et `docker build -f infra/docker/eval.Dockerfile eval/` produit une image dont le `ENTRYPOINT`/`CMD` invoque le runner Ragas (`python -m eval.ragas_runner`) de sorte que `docker run <image> --help` affiche l'aide CLI du runner et sort `0` (décision D-1, D-2).
- AC-2 : L'image bake exactement (a) le code runner du dossier `eval/` (modules `ragas_runner.py`, `clients.py`, `oidc.py`, `persist.py`, `gates.py`, `loader.py`, `metrics.py`, `run_writer.py`, `baseline_skip.py` et leurs dépendances internes), (b) les dépendances installées depuis `eval/uv.lock` en `--frozen`/`--locked`, et (c) le fichier `eval/baseline.json` ; elle ne bake **pas** `specs/golden_qa.jsonl` (le golden set est fetché depuis GCS par OBS-009 au runtime) (décision D-3).
- AC-3 : `--output` et `--persist` restent des arguments fournis au runtime (`docker run`), jamais figés dans l'image ; aucune valeur par défaut de sortie ou de persistence n'est bakée dans le Dockerfile (décision D-3).
- AC-4 : L'image installe le jeu de dépendances complet capable du mode **live** (Ragas + judge OpenAI + appels Mistral generate), mais ne bake **aucun** modèle d'embeddings local `bge-m3` ni `sentence-transformers`/`torch` : le mode live est pur-API (Mistral generate côté workers + OpenAI judge), aucune inférence locale d'embeddings n'a lieu dans le conteneur eval (décision D-5).
- AC-5 : `docker run <image> --help` (ou tout chemin n'effectuant aucun appel réseau) s'exécute sans connectivité réseau (`docker run --network none <image> --help` sort `0`) — l'image est auto-suffisante au démarrage, sans `pip install`/`uv sync` au runtime (décision D-1, oracle offline).
- AC-6 : `.github/workflows/<workflow>` construit l'image eval sur `push` vers `main` et la pousse vers `${REGION}-docker.pkg.dev/${PROJECT}/archiviste/eval` sous **deux** tags : `:${{ github.sha }}` (immutable) et `:latest`, en réutilisant le repo Artifact Registry `archiviste` existant (décision D-1, D-4).
- AC-7 : Le step CI réussit (exit `0`) ssi le build de l'image **et** le push des deux tags réussissent ; toute erreur de build ou de push fait échouer le job (oracle = build exit 0 + push ok, décision D-1).
- AC-8 : Aucune ressource Terraform nouvelle n'est introduite : le repo Artifact Registry `archiviste` (`infra/terraform/artifact_registry.tf`) est réutilisé tel quel, aucun `google_artifact_registry_repository` neuf, aucun binding IAM neuf dans `infra/terraform/` pour ce ticket (décision D-4).
- AC-9 : `specs/openapi/gateway-to-workers.yml` n'est **pas** modifié : packager le runner en image est une préoccupation build/CI, pas un changement de contrat REST ; aucune route ni payload nouveau.
- AC-10 : `migrations/*.sql` n'est **pas** modifié : l'image ne porte aucune migration ; le schéma `eval_runs` (OBS-003) reste géré hors image, appliqué indépendamment.

## Non-goals

- Pas de Cloud Run Job, pas de scheduler, pas d'exécution prod de l'eval — c'est OBS-009 (ce ticket produit seulement l'image que OBS-009 déploiera).
- Pas de fetch du golden set : `specs/golden_qa.jsonl` n'est jamais baké dans l'image (OBS-009 le fetch depuis GCS au runtime, AC-2, D-3).
- Pas de nouvelle ressource Terraform ni binding IAM — réutilisation du repo `archiviste` existant (AC-8, D-4).
- Pas de modèle d'embeddings local (`bge-m3` / `sentence-transformers` / `torch`) baké — live mode pur-API (AC-4, D-5).
- Pas de changement du contrat OpenAPI, des migrations, du schéma `eval_runs`, ni du code runner lui-même (l'image package le code OBS-007/OBS-003 existant à l'identique).
- Pas de modification du workflow `eval.yml` (PR offline / `workflow_dispatch` live restent `uv run` ; l'image est un artefact de build `main`, pas le chemin d'exécution des gates PR) — sauf si l'architect tranche au `/plan` que le build vit dans `deploy.yml` plutôt qu'un workflow dédié (OQ-1).
- Pas de signature d'image (cosign), pas de SBOM, pas de scan de vulnérabilité d'image (Trivy/Grype) phase 1 — durcissable en ticket OPS aval si besoin.
- Pas de tag sémantique `vX.Y.Z` sur l'image eval — seulement `:${sha}` + `:latest` (le versionnage release-please concerne le code, pas cette image utilitaire).
- Pas de runtime non-root / hardening avancé au-delà de la base `python:3.12-slim` héritée des Dockerfiles existants (sauf si trivial à ajouter, non requis ici).

## Pre-conditions

- **OBS-007 mergé** (`ff7b411`) : code runner complet dans `eval/` (`oidc.py`, `clients.py` avec auth OIDC), `eval/pyproject.toml` déclare `google-auth>=2`, `eval/uv.lock` présent et à jour.
- **OBS-003 mergé** : `eval/persist.py` + flag `--persist` présents (l'image les bake, le schéma DB reste hors image, AC-10).
- **INFRA-002 mergé** : repo Artifact Registry `archiviste` provisionné (`infra/terraform/artifact_registry.tf`), WIF + SA `gha-deploy@` avec `roles/artifactregistry.writer` opérationnels (réutilisés pour le push, aucun rôle neuf — AC-8).
- **`eval/uv.lock` est la source des deps lockées** : le build installe en `--frozen`/`--locked` ; un lock obsolète fait échouer le build (comportement voulu, reproductibilité).
- **Dockerfiles existants comme modèle** : `infra/docker/workers.Dockerfile` / `gateway.Dockerfile` (base `python:3.12-slim`, `uv sync --frozen`) servent de patron de style — le nouveau `eval.Dockerfile` les mirror.

## Failure modes

- `eval/uv.lock` obsolète ou incohérent avec `eval/pyproject.toml` → `uv sync --frozen`/`--locked` échoue au build → step CI échoue (AC-7), aucune image poussée (pas de tag partiel).
- Push Artifact Registry refusé (auth WIF KO, `artifactregistry.writer` manquant, repo absent) → step push échoue → job échoue (AC-7) ; le tag `:${sha}` n'est pas créé.
- `docker run <image> --help` sortant non-`0` ou nécessitant le réseau → AC-1/AC-5 en échec (image non auto-suffisante) — bug de build (ENTRYPOINT erroné, dépendance manquante, install runtime résiduel).
- Image baked avec `specs/golden_qa.jsonl` ou un modèle d'embeddings local → violation AC-2/AC-4 (le golden vient de GCS au runtime, embeddings pur-API) — à détecter par inspection des layers.
- Build context erroné (≠ `eval/`) → fichiers attendus absents → build échoue ou image incomplète (AC-1, AC-2).

## Test oracle

- AC-1 : contract / smoke local · `docker build -f infra/docker/eval.Dockerfile -t eval-test eval/` exit `0`, puis `docker run --rm eval-test --help` exit `0` et stdout contient les flags du runner (`--mode`, `--set`, `--baseline`, `--output`, `--persist`).
- AC-2 : contract / inspection · `docker run --rm eval-test python -c "import eval.ragas_runner, eval.oidc, eval.persist, eval.clients"` exit `0` ET `docker run --rm eval-test cat baseline.json` (ou chemin baké) lisible ET `docker run --rm eval-test sh -c "test ! -f specs/golden_qa.jsonl"` (golden absent).
- AC-3 : contract · grep `infra/docker/eval.Dockerfile` → aucun `--output`/`--persist` figé dans `CMD`/`ENTRYPOINT` (args passés au runtime uniquement).
- AC-4 : contract / inspection · `docker run --rm eval-test pip list` (ou `uv pip list`) ne contient ni `sentence-transformers` ni `torch` ; `docker run --rm eval-test sh -c "find / -name '*.safetensors' -o -name '*bge-m3*' 2>/dev/null | head"` → vide (aucun poids de modèle baké).
- AC-5 : contract / offline · `docker run --network none --rm eval-test --help` exit `0` (auto-suffisance, aucun install runtime).
- AC-6 : contract / CI · après merge `main`, le job publie `${REGION}-docker.pkg.dev/${PROJECT}/archiviste/eval:${sha}` ET `:latest` ; vérifiable via `gcloud artifacts docker images list .../archiviste/eval` (2 tags présents) ou grep du workflow (`tags:` contient `eval:${{ github.sha }}` et `eval:latest`).
- AC-7 : contract / CI · run du workflow sur `main` : étapes build puis push observées, job vert ssi les deux réussissent ; test négatif (lock cassé volontaire en branche de test) → job rouge, aucun tag poussé.
- AC-8 : contract · `git diff infra/terraform/` après PR → 0 changement ; `! grep -rE 'google_artifact_registry_repository' <diff>` (aucune ressource neuve).
- AC-9 : contract · `git diff specs/openapi/gateway-to-workers.yml` après PR → 0 changement.
- AC-10 : contract · `git diff migrations/` après PR → 0 changement.

## Security / trust boundary

- Frontière : build CI (`gha-deploy@`, WIF, zéro clé JSON) → Artifact Registry `archiviste`. Aucune surface Internet ajoutée.
- **A06 Composants** : deps installées depuis `eval/uv.lock` en `--frozen`/`--locked` — build reproductible, pas de résolution flottante (`secret-hygiene.md` / `security.md` §A06, §A08).
- **A08 Intégrité** : tag `:${sha}` immutable = ref reproductible ; `:latest` = pointeur mouvant pour OBS-009 (cohérent convention `workers:latest` existante).
- **Secret hygiene** : aucun secret baké dans l'image (pas de `LLM_API_KEY`/`OPENAI_API_KEY`/`DATABASE_URL` — fournis au runtime par OBS-009 via env/Secret Manager) ; pas de `.env` ni credential dans le contexte de build (`.dockerignore` exclut, `secret-hygiene.md`).
- Aucun `verify=False`, aucune dépendance git-URL, registres officiels uniquement.

## Observability

## Open questions

- **OQ-1 — Emplacement du build CI : étendre `deploy.yml` (déjà `push: main` + WIF + buildx + `gcloud auth configure-docker`) vs nouveau workflow dédié `eval-image.yml` ?** `deploy.yml` a déjà l'auth WIF, buildx et le pattern build-push ; y ajouter un 3e build (eval) est l'option de moindre coût et garde un seul chemin de publication d'images sur merge. Un workflow dédié isole le cycle de vie de l'image eval (rebuild même si gateway/workers inchangés) mais duplique l'auth/setup. Reco non figée : étendre `deploy.yml` (réutilise auth + buildx, cohérent avec `workers:latest` déjà publié là). À trancher au `/plan` — n'impacte aucun AC observable (AC-6 nomme `<workflow>` sans le figer).
- **OQ-2 — Layout du package eval avec contexte de build `eval/`.** Le code runner s'importe en `eval.ragas_runner` / `eval.oidc` (préfixe `eval.`), et `eval/pyproject.toml` déclare `[tool.hatch.build.targets.wheel] packages = ["eval"]`. Avec un contexte de build = `eval/`, le dossier copié dans l'image est la racine du contexte ; il faut vérifier au `/plan` que les imports `eval.*` résolvent dans le conteneur (p. ex. `WORKDIR /app`, copier le contenu sous `/app/eval/`, ou ajuster `PYTHONPATH`/install editable via `uv sync`). Détail d'implémentation à valider — aucun AC ne fige la stratégie, AC-1/AC-2 exigent seulement que `python -m eval.ragas_runner` et `import eval.*` fonctionnent dans l'image.
- **OQ-3 — `eval.ragas_runner` importe `ragas`/`datasets`/`langchain-openai`, aujourd'hui en `optional-dependencies.dev` de `eval/pyproject.toml`, pas en `dependencies` core.** Le mode live nécessite ces libs (AC-4) ; or l'image ne doit pas baker les outils de test (`pytest`, `hypothesis`, `ruff`, `mypy`). Question : faut-il (a) déplacer `ragas`/`datasets`/`langchain-openai` de `dev` vers un nouvel extra `live` (ou vers `dependencies` core) pour que l'image les installe **sans** baker les outils de lint/test, ou (b) baker tout `--all-extras` (incluant pytest/ruff/mypy, plus lourd) ? Reco : introduire un extra `live` ciblé et installer `--extra live` (ou équivalent) dans le Dockerfile, gardant l'image minimale et le mode live capable. **Cette OQ touche potentiellement `eval/pyproject.toml`/`eval/uv.lock`** (regroupement des deps) — à confirmer en round 2 si l'humain accepte ce reclassement de deps, sinon AC-4 doit préciser le mécanisme d'installation retenu.
