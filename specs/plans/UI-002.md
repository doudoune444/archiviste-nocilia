# Plan — UI-002a Dashboard auteur backend (tickets list + GCS signed URL)

## Goal
Livrer le backend gateway pour le dashboard auteur : 2 endpoints API JSON (`GET /v1/tickets`, `GET /v1/conversations/{id}/signed-url`) gated `author_only`, lecture directe Postgres pour `tickets`/`conversations`, signature V4 GCS via clé SA et `ring` (TTL 5 min, méthode GET). Slice backend uniquement (spec D-1) ; UI-002b frontend = plan dédié post-merge.

## Acceptance criteria recap
Voir `specs/acceptance/UI-002.md` AC-5 à AC-12, AC-19 à AC-23 (verbatim, non recopié — limite ≤ 100 lignes).
Hors scope de cette PR : AC-1, AC-3, AC-4, AC-13 à AC-18, AC-24 (frontend + manuel) → UI-002b.

## Risks / open questions
- **R1 (LOC)** : prod ≈ 15 (config) + 10 (state) + 10 (lib.rs) + 2 (mod) + 80 (tickets) + 50 (conversations) + 3 (gcs/mod) + 100 (gcs/sign) + 40 (errors) + 2 (Cargo.toml) = **~312 LOC prod**. Tests ~260. **Légèrement au-dessus 300 LOC prod** ; mitigation : si `gcs/sign.rs` > 100 LOC, extraire un helper crate OU accepter waiver justifié (cohérent ADR-0008 précédent). Si dérapage > 350 LOC, sub-split = (a) tickets seul, (b) signed-url seul.
- **R2 (SEC-001 ordering)** : SEC-001 doit être mergé avant cette PR (extractor `AuthUser` requis). Si SEC-001 retarde, UI-002a est bloqué (cohérent `no-workaround.md` — pas de stub auth temporaire).
- **R3 (GCS signing key in CI)** : `GCS_SIGNING_PRIVATE_KEY_PEM` absent en CI → tests intégration handlers `conversations` doivent injecter une fixture key via env var de test (`gateway/tests/fixtures/test-signing-key.pem` mockée dans le test). Pas de réel appel GCS en test (signature locale uniquement).
- **R4 (testcontainer DB)** : si l'infra de tests intégration gateway n'a pas encore de PgPool fixture, ce plan introduit le pattern (`sqlx::test` ou manual `setup_test_db()`). Vérifier compatibilité avec `gateway/tests/chat_test.rs` existant.
- **R5 (ring V4 signing complexity)** : impl V4 manuelle = format canonical request strict (ordre alphabétique query params, encoding URI, datetime ISO8601 basic). Si > 100 LOC, blocker → ADR `docs/adr/NNNN-gcs-signed-url-rust.md` proposant `gcp_auth` crate.
- **R6 (AC-22 test mécanisme)** : assertion "OpenAPI inchangé" mieux servie par CI step bash que test Rust ; à confirmer avec reviewer.
- **R7 (clé fixture commitée)** : `gateway/tests/fixtures/test-signing-key.pem` est une vraie clé RSA jetable. `secret-hygiene.md` interdit commit de clés ; arbitrage : clé marquée `# TEST ONLY — DO NOT USE IN PROD`, hook `guard-secret-paths.sh` doit la whitelister sous `tests/fixtures/`. Si refusé : générer la clé à chaque run de test (CPU coût ~200 ms × N tests).
