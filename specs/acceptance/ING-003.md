# ING-003 — Conversation logger (Markdown GCS append + index Postgres)

## Contexte

Le pipeline de génération (GEN-001 + suivants) doit persister chaque tour user/assistant dans une conversation. La vision §39 fige le format : un `.md` append-only par conversation sur GCS (source de vérité) + une ligne légère dans la table `conversations` (FOUND-003) pour l'indexation et l'enforcement futur de l'IDOR. Sans ce ticket, les tickets lore-gap (qui référencent `conversation_id` via FK) et le dashboard auteur ne peuvent pas exister.

## Critères d'acceptation

- AC-1 : `POST /v1/conversations/{conversation_id}/messages` sur le service workers accepte un body JSON `{role, content, timestamp}` et retourne `201 Created` avec body `{conversation_id, message_count, gcs_uri, generation}`.
- AC-2 : Si `conversation_id` n'a pas de ligne dans la table `conversations`, le handler INSERT une ligne `(id=conversation_id, user_id=<body.user_id>, gcs_uri='gs://archiviste-conversations/{conversation_id}.md', message_count=0, created_at=NOW(), updated_at=NOW())` AVANT le premier append, et initialise l'objet GCS avec le header exact `# Conversation {conversation_id}\n\nCreated: {ISO8601}\nUser: {user_id}\n\n---\n` via une création conditionnelle `ifGenerationMatch=0` (refuse l'overwrite si l'objet existe déjà).
- AC-3 : Chaque appel append au `.md` GCS produit un bloc strictement formaté : `## [{ISO8601_timestamp}] {role}\n{content}\n\n` (ligne vide finale incluse), concaténé après le contenu existant via un cycle read-modify-write avec `ifGenerationMatch=<current_generation>`.
- AC-4 : En cas de conflit `ifGenerationMatch` (HTTP 412 retourné par GCS), le handler retry exactement 3 fois avec backoff 50 ms, 200 ms, 800 ms (re-read avant chaque retry) ; après le 3e échec il retourne `409 Conversation` `{"error":"concurrent_write"}`.
- AC-5 : Après un append GCS réussi, le handler exécute `UPDATE conversations SET message_count = message_count + 1, updated_at = NOW() WHERE id = $1` ; si cette UPDATE échoue, le handler émet un log structuré niveau ALERT `{"event":"inconsistency_db_after_gcs","conversation_id":"...","gcs_generation":G}` et retourne quand même `201 Created` (la source de vérité reste le `.md` GCS).
- AC-6 : Le nom de l'objet GCS est strictement `{conversation_id}.md` où `conversation_id` est validé contre la regex `^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$` AVANT toute interaction GCS ou DB ; un `conversation_id` non conforme retourne `400 Bad Request` `{"error":"invalid_conversation_id"}`.
- AC-7 : Le champ `role` du body est validé contre l'ensemble exact `{user, assistant}` ; toute autre valeur retourne `422 Unprocessable Entity` avec `{"error":"invalid_role"}`.
- AC-8 : Le champ `content` est rejeté si vide (`422` `{"error":"empty_content"}`) ou si la taille UTF-8 dépasse 16 KiB (`413 Payload Too Large` `{"error":"content_too_large"}`).
- AC-9 : Le champ `timestamp` est rejeté si non-ISO8601 (`422` `{"error":"invalid_timestamp"}`), si > NOW() + 5 min (`422` `{"error":"timestamp_in_future"}`), ou si < `conversations.created_at` de la conversation cible (`422` `{"error":"timestamp_before_conversation"}`).
- AC-10 : Lorsque le client GCS lève une erreur non-récupérable après 3 retries (timeout, 5xx, indisponibilité), le handler retourne `503 Service Unavailable` `{"error":"storage_unavailable"}` et n'effectue aucune UPDATE DB.
- AC-11 : Chaque requête traitée émet exactement un log structuré JSON `{"event":"conversation_message","conversation_id":"...","role":"...","bytes":N,"latency_ms":L,"gcs_generation":G}` ; le champ `content` n'apparaît dans aucun log (security.md A09).
- AC-12 : Le client GCS lit la config via `GCS_BUCKET` (obligatoire, fail-fast si absent au boot du worker) et `GCS_EMULATOR_HOST` (optionnel, si défini le client cible l'émulateur, sinon Application Default Credentials) ; aucun fallback hardcodé n'existe dans le code.
- AC-13 : `docker-compose.yml` déclare un service `gcs` (image `fsouza/fake-gcs-server:latest`) sous `profiles: ["tools"]`, non démarré par `docker compose up -d`, démarré explicitement pour les tests d'intégration via `docker compose --profile tools up -d gcs`.
- AC-14 : Un test d'intégration valide la séquence complète : create (premier message → INSERT row + header + premier bloc), append (deuxième message sur même conversation → row inchangée sauf compteurs, header non altéré, deuxième bloc ajouté), restart-then-append (arrêt/redémarrage du worker entre les appels, continuité préservée).
- AC-15 : Un test d'intégration de concurrence lance deux appends simultanés sur la même `conversation_id` ; exactement un retourne `201` (avec `message_count` incrémenté de 1 ou 2 selon le retry), l'autre retourne `409 Conversation` `{"error":"concurrent_write"}` après épuisement des 3 retries OU `201` si un retry réussit.

## Non-goals

- Pas de lecture des conversations exposée sur l'API workers (ticket dashboard auteur ultérieur).
- Pas de génération de signed URL (ticket dashboard auteur dédié).
- Pas de TTL / archivage / lifecycle GCS.
- Pas d'encryption applicative (uniform bucket-level + GCS at-rest par défaut suffisent phase 1).
- Pas de pub/sub ou event streaming sur les messages.
- Pas de réconciliation automatique DB ↔ GCS (ticket OBS-* futur).
- Pas de multi-region ou failover GCS.
- Pas de support du rôle `system` (ou autre) phase 1.
- Pas d'idempotence côté serveur (le caller GEN-* gère ses propres clés d'idempotence).
- Pas d'exposition de cet endpoint sur la gateway publique (internal-only, consommé par GEN-001 et suivants).

## Failure modes

- `conversation_id` non-UUID → `400 Bad Request` `{"error":"invalid_conversation_id"}`, aucune interaction DB ni GCS, log `event=invalid_conversation_id`.
- `role` hors `{user, assistant}` → `422` `{"error":"invalid_role"}`.
- `content` vide → `422` `{"error":"empty_content"}`.
- `content` > 16 KiB UTF-8 → `413` `{"error":"content_too_large"}`.
- `timestamp` non-ISO8601 / futur > 5 min / antérieur à `created_at` → `422` avec code d'erreur dédié (cf AC-9).
- Conflit `ifGenerationMatch` après 3 retries → `409 Conversation` `{"error":"concurrent_write"}`, log `event=concurrent_write_exhausted`.
- GCS injoignable / 5xx persistant → `503 Service Unavailable` `{"error":"storage_unavailable"}`, aucune UPDATE DB.
- DB injoignable AVANT l'écriture GCS (création) → `503` `{"error":"database_unavailable"}`, aucune écriture GCS.
- DB injoignable APRÈS écriture GCS réussie (UPDATE post-append) → `201 Created` retourné, log ALERT `event=inconsistency_db_after_gcs` (cf AC-5), réconciliation hors-scope.
- Tentative de création (premier message) sur un objet GCS qui existe déjà (`ifGenerationMatch=0` rejeté) → `409 Conversation` `{"error":"conversation_already_exists"}`.

## Performance / SLO

- Latence p95 d'un append (header init exclu) < 300 ms en local sur fake-gcs-server (mesure indicative, pas un gate CI phase 1).
- Pas de SLA chiffré sur GCS prod phase 1 (intégré dans le SLO chat round-trip global p95 < 3 s — vision §86).

## Security / trust boundary

- Endpoint internal-only : non monté sur la gateway publique. Frontière de confiance gateway → workers (cf threat-model W-S-1) ; auth HMAC `X-Gateway-Auth` reportée à un ticket SEC-* dédié (ce ticket ne l'introduit pas mais ne doit pas l'empêcher).
- Filename strictement dérivé serveur (validé regex avant usage) → mitigation C-E-1 (path traversal).
- `GCS_BUCKET` lu via env, jamais hardcodé. Credentials GCS via Application Default Credentials en prod (Workload Identity Federation cf C-S-1) — pas de clé SA JSON committée (secret-hygiene.md).
- Body JSON capé à 1 MiB côté FastAPI (cohérent avec security.md A04). `content` capé spécifiquement à 16 KiB (AC-8).
- Aucun log du `content` (security.md A09 : « NEVER log full prompts »).
- Bucket : uniform bucket-level access, pas de signed URL générée par ce ticket, pas d'ACL public (threat-model C-I-1).
- Erreur GCS / DB ne fuit pas la stack trace côté réponse (enveloppe `{"error":"..."}` uniquement, security.md A05).

## Observability

- Log structuré par requête (cf AC-11) : `event=conversation_message`, champs `conversation_id`, `role`, `bytes`, `latency_ms`, `gcs_generation`.
- Log ALERT dédié sur incohérence DB post-GCS (cf AC-5) : `event=inconsistency_db_after_gcs`.
- Log INFO sur création de conversation : `event=conversation_created`, champs `conversation_id`, `user_id`.
- Pas de métrique OpenTelemetry exposée à ce stade (réservé tickets OBS-*).

## Decisions résolues (humain, 2026-05-05)

- Filename : flat `{conversation_id}.md` retenu (simple, repose sur unicité UUIDv4). Écart explicite vs threat-model C-T-1 noté → `threat-model.md` à mettre à jour (note d'écart : « namespacing user_id/ reporté, mitigation = uniform bucket-level + regex strict serveur »).
- Parser Markdown ↔ structure (INV-4) : hors-scope ce ticket. INV-4 reste non couvert jusqu'au ticket dashboard auteur dédié. Ce ticket figeune ment le format d'écriture (AC-3).
- Quota 1000 conversations/user (C-D-1) : reporté à un ticket SEC-* dédié. Pas d'enforcement à AC-2.
- FK `user_id` : on fait confiance au caller GEN-* mais on traduit l'erreur Postgres `23503` (foreign_key_violation) en `422 Unprocessable Entity` `{"error":"unknown_user"}` au niveau du repository — pas de SELECT préalable.
- Format ISO8601 : permissif. Accepter toute variante parsable par `datetime.fromisoformat()` (Python 3.12 supporte `Z` natif). Pas de format strict imposé phase 1.
- Backoff retries (AC-4) : pas de jitter. Délais déterministes 50/200/800 ms.
- Logger : `structlog` (cohérent ING-001).
- Client DB : `asyncpg` (cohérent ING-001).
