# Plan — OPS-005 Ingestion `lore/` auto sur push `main` (Cloud Run Job + workflow)

## Goal
Tout changement `lore/**` mergé sur `main` déclenche un Cloud Run Job `archiviste-ingest` qui réindexe le corpus dans pgvector prod, sérialisé et bloquant côté CI.

## Acceptance criteria recap
- AC-1 : `google_cloud_run_v2_job` `archiviste-ingest`, `europe-west9`, image workers, pattern service workers (`cloud_run.tf:224-248`), Cloud SQL, `MISTRAL_API_KEY` en `secret_key_ref`, SA `archiviste-runtime`.
- AC-2 : `command/args = python -m archiviste_workers.ingest --path lore/` + `CLOUD_SQL_IAM_AUTH=true`, `INSTANCE_CONNECTION_NAME`, volume `cloudsql`, `DATABASE_URL` SA email percent-encodé.
- AC-3 : `.github/workflows/ingest-lore.yml`, `on: push branches:[main] paths:['lore/**']`.
- AC-4 : auth WIF `google-github-actions/auth@v2` + `workload_identity_provider`, aucune clé SA JSON.
- AC-5 : `gcloud run jobs execute archiviste-ingest --wait --region europe-west9` (env `REGION`), step échoue si exécution ≠ `Succeeded`.
- AC-6 : Job exit `0` si `errors==0`, exit `1` si `errors>=1` (couvert ING-001 AC-17).
- AC-7 : exit `0`→`Succeeded`→vert ; exit `1`→`Failed`→rouge ; autre exit→`Failed`→rouge.
- AC-8 : `concurrency: { group: ingest-lore, cancel-in-progress: false }`.
- AC-9 : `ingest.summary` (ING-001 AC-16) récupérable en Cloud Logging par execution ID.
- AC-10 : runbook `§Ingestion DB après sync` (`docs/runbook.md:439`) réécrit (auto + fallback) ; checklist `gdrive-sync.yml:141` ajustée/supprimée.

## Risks / open questions
- (R1) **Écart AC vs réalité** : les AC disent « sidecar cloud-sql-proxy » mais le service workers (`cloud_run.tf:155-160, 250-253`) utilise le volume géré `cloud_sql_instance` + annotation `run.googleapis.com/cloudsql-instances`, PAS un container sidecar `cloud-sql-proxy`. Le plan réutilise le pattern réel (volume). Le wording AC est interprété comme « connexion Cloud SQL IAM authn », pas un container littéral. À confirmer humain.
- (R2) **Décision cloud_run.tf vs cloud_run_job.tf** : tranchée → **nouveau `cloud_run_job.tf`**. Motif : `cloud_run.tf` cible les services (l.1 commentaire « Cloud Run services »), un Job est une ressource de cycle de vie distinct ; séparation = diff plus lisible, pas de couplage `lifecycle`.
- (R3) **Job v2 + Cloud SQL volume** : `google_cloud_run_v2_job` supporte `template.template.volumes.cloud_sql_instance` + annotation comme le service. Si le provider exige un sidecar proxy explicit pour les Jobs, c'est un blocker → `docs/blockers.md` + stop (no-workaround).
- (R4) **Runbook obsolète** : `runbook.md:452` affirme téléchargement `BAAI/bge-m3 ~2.3 GiB`. Faux : `embedder.py:21` = `mistral-embed` via API (DEFAULT_MODEL_NAME). Correction incluse dans step 3 (embedder API, pas de modèle 2.3 GiB).
- (R5) `--max_retries = 0` requis : sinon exit `1` ING-001 (erreurs fichiers) serait retenté et masquerait le signal rouge attendu par AC-7.
