# OBS-001 — Page observabilité publique + nav + widget usage (GET /v1/stats)

## Contexte

La vision EPIC OBS (§79-§97) ouvre une surface d'observabilité publique pour la vitrine portfolio : aujourd'hui le chat (`/`, UI-001) est la seule page et aucun signal d'usage agrégé n'est exposé. Ce ticket livre la première slice « facile » de l'EPIC : un header de navigation `Chat | Observabilité` sur le chat, une nouvelle page publique anonyme `/observability`, et un widget d'usage alimenté par un nouvel endpoint public `GET /v1/stats` retournant le nombre total exact de conversations (décision humaine D-1, cf §Décisions). Santé service (OBS-002), persistence Ragas (OBS-003) et panel qualité IA (OBS-004) sont hors périmètre.

## Acceptance criteria

- AC-1 : `GET /observability` retourne `200 OK`, `Content-Type: text/html; charset=utf-8`, et le body contient le sélecteur stable `<section id="usage-widget"` (point d'ancrage des tests d'intégration et du test manuel).
- AC-2 : `GET /observability` est servi sans authentification (anonyme) ; aucun `401`/`403` n'est jamais retourné sur cette route quel que soit l'état d'authentification du requester (cohérent surface publique vision §92).
- AC-3 : `GET /assets/observability.js` retourne `200 OK` avec un `Content-Type` JavaScript (`application/javascript` ou `text/javascript`, charset UTF-8) ; `GET /assets/observability.css` retourne `200 OK` avec `Content-Type: text/css` (charset UTF-8). Les deux assets sont publics (servis sans auth, cohérent UI-001 AC-4).
- AC-4 : `GET /v1/stats` retourne `200 OK`, `Content-Type: application/json; charset=utf-8`, body JSON `{"conversation_count": <int>}` où `conversation_count` est un entier égal au **count exact** (`SELECT count(*) FROM conversations`), sans arrondi ni bucketisation (décision humaine D-1). Aucun autre champ : aucun champ par-utilisateur, aucun timestamp, aucun identifiant n'est présent dans le body.
- AC-5 : Pour un état DB contenant exactement `n` lignes dans `conversations`, `GET /v1/stats` retourne `{"conversation_count": n}` (égalité stricte) : deux états DB de cardinalités distinctes `n1 ≠ n2` produisent deux valeurs exposées distinctes `n1 ≠ n2` (propriété : le mapping count→valeur exposée est l'identité). Pas de table de buckets, pas de constante d'arrondi.
- AC-6 : `GET /v1/stats` est servi sans authentification (anonyme) ; il ne porte aucun guard `author_only`/`member_only` et n'apparaît pas dans le sous-router auth. Un test contractuel (inspection router) vérifie l'absence de guard d'auth sur ce handler.
- AC-7 : Quand Postgres est indisponible (ou la requête échoue) au moment de `GET /v1/stats`, la réponse est `503 Service Unavailable` avec un body d'erreur **sanitisé** `{"error":"upstream_unavailable","request_id":"<uuid>"}` ; aucun détail d'erreur interne (message SQL, host, nom de table, stack, count partiel) n'est exposé dans le body (`security.md` §A05 : « jamais de stack trace ni d'erreur interne brute »). Décision humaine D-1 : la valeur de count est exacte sur le chemin succès, mais le chemin échec reste générique et non verbeux.
- AC-8 : Le chat (`gateway/static/index.html`) contient un header de navigation avec exactement deux liens : `Chat` pointant vers `/` et `Observabilité` pointant vers `/observability`, dans cet ordre. La page `/observability` contient le même header de navigation avec les deux mêmes liens.
- AC-9 : Au chargement de `/observability`, `observability.js` appelle `GET /v1/stats` (même-origine, `fetch` avec `credentials: 'same-origin'`) et insère la valeur `conversation_count` reçue dans `<section id="usage-widget">` en texte brut (`textContent`, jamais `innerHTML`). Le widget affiche un libellé `Conversations : <N>` (décision D-5 ; le libellé exact n'est pas figé byte-for-byte en AC, seul l'usage `textContent` + présence du nombre sont contractuels). La page porte un `<h1>Observabilité</h1>`.
- AC-10 : Pour toute issue de `GET /v1/stats` autre que `200 OK` (4xx, 5xx, network drop, timeout, JSON malformé), `observability.js` insère dans `#usage-widget` le message littéral `Données indisponibles.` ; aucun branchement fin selon le code d'erreur (cohérent ton UI-001 AC-13).
- AC-11 : Les 5 en-têtes de sécurité globaux posés router-wide (UI-001 AC-6 / SEC-003 AC-1 : `Content-Security-Policy`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, `X-Frame-Options: DENY`, `Strict-Transport-Security`) sont présents byte-for-byte sur `GET /observability`, `GET /assets/observability.{js,css}` et `GET /v1/stats`. La valeur CSP n'est PAS modifiée par ce ticket.
- AC-12 : `gateway/static/observability.html` ne contient aucune balise `<script>` inline, aucune balise `<style>` inline, aucun attribut `style="..."` ni gestionnaire `on*=` inline ; tout JS est chargé via `<script src="/assets/observability.js">`, tout CSS via `<link rel="stylesheet" href="/assets/observability.css">` (cohérent UI-001 AC-7, validable par grep statique).
- AC-13 : Les assets servis sous `/assets/*` après ce ticket incluent `observability.js` et `observability.css` en plus des assets pré-existants ; toute route `/assets/<x>` inexistante retourne 404 et le test path-traversal `/assets/../*` reste rejeté 400/404 (UI-001 AC-5 inchangé).
- AC-14 : `specs/openapi/gateway-to-workers.yml` n'est **pas** modifié par ce ticket : `GET /v1/stats` est un endpoint client↔gateway (lecture directe Postgres côté gateway), aucune route workers nouvelle, aucun appel gateway→workers (cohérent UI-002 AC-22).
- AC-15 : Test manuel auteur (procédure documentée `## Test manuel auteur`) coché par l'humain avant `Status: ready`.

## Non-goals

- Pas d'endpoint `GET /v1/status` (probes santé Mistral/GCS/Postgres/workers) ni de widget disponibilité — OBS-002.
- Pas de persistence des résultats Ragas, pas de table eval, pas de write path depuis EVAL-001 — OBS-003.
- Pas de panel qualité IA (faithfulness / answer_relevancy live) — OBS-004.
- Pas de score « qualité réponse » per-réponse, pas de feedback thumbs-up — ticket futur dédié (vision §94).
- Pas d'observabilité infra backend (uptime checks GCP, log-based metrics, alert policies, OTel → Cloud Logging) — OBS-005.
- Pas d'autre métrique d'usage que `conversation_count` phase 1 (pas de nb requêtes, nb tickets, latences, nb users).
- Pas de graphe / chart / série temporelle — affichage d'une valeur scalaire unique phase 1.
- Pas de rafraîchissement auto / polling / SSE — `GET /v1/stats` appelé une fois au chargement.
- Pas d'arrondi / bucketisation / estimation `reltuples` du count phase 1 — `conversation_count` = `count(*)` exact (décision D-1).
- Pas de cache HTTP applicatif ni cache gateway sur `GET /v1/stats` phase 1 — `count(*)` exact à chaque appel (décision D-3 ; ré-arbitrage si p95 > 80 ms post-deploy).
- Pas de framework JS (React/Vue/htmx/Alpine) — ADR-0005 réaffirmé.
- Pas de build step / bundler / transpiler / source maps — édition fichier → refresh navigateur, ADR-0005.
- Pas de dépendance CDN externe (CSP `script-src 'self'` interdit).
- Pas de i18n / pas de switch FR/EN — UI FR uniquement (cohérent UI-001/UI-002).
- Pas de composant nav partagé extrait en module JS réutilisable phase 1 (décision D-2 : vanilla) — header dupliqué en HTML statique sur 2 pages ; trigger ADR-0005 framework NON franchi, ticket nav-refactor à créer dès qu'une 4e page atterrit.
- Pas de rate-limit applicatif spécifique à `GET /v1/stats` (couverture Cloudflare front V1 + SEC-002 V2).
- Pas de modification de la migration `conversations` (lecture seule du count existant).

## Pre-conditions

- **UI-001 mergé** : `ServeFile`/`ServeDir` opérationnels, répertoire `gateway/static/` + `gateway/static/assets/` existants, middleware d'en-têtes de sécurité globaux posé router-level, `gateway/static/index.html` existant (cible du header nav AC-8).
- **SEC-003 mergé** : 5e en-tête `Strict-Transport-Security` posé router-wide (AC-11 référence les 5 headers).
- **FOUND-003 mergé** : table `conversations` créée (migration `0002_schema.sql:44`), source du count AC-4.
- **ING-003 mergé** : des lignes `conversations` sont effectivement écrites lors d'une conversation (sinon count toujours 0).
- `cargo sqlx prepare` exécuté après ajout de la requête `SELECT count(*) FROM conversations` si `sqlx::query!` macro utilisée (sinon `query_as` runtime-typé, cf pattern `gateway/src/handlers/tickets.rs`) — rule `vertical-slice.md`.

## Failure modes

- Postgres indisponible (ou requête en échec) au moment de `GET /v1/stats` → `503` body sanitisé `{"error":"upstream_unavailable","request_id":"<uuid>"}`, log `warn` ; pas de fallback ni de valeur par défaut ni de count partiel (cohérent `no-workaround.md` + `security.md` §A05).
- Table `conversations` vide (`count = 0`) → `200 OK` `{"conversation_count": 0}` ; `observability.js` affiche `0` en clair.
- `GET /v1/stats` retourne `503` ou network drop → `observability.js` affiche `Données indisponibles.` (AC-10).
- Fichier statique manquant au déploiement (`observability.html` absent du conteneur) → `GET /observability` retourne 404 ; détecté en CI par AC-1.
- Path traversal `/assets/../*` → 400/404 servi par `tower-http` (AC-13, UI-001 AC-5 inchangé).

## Test manuel auteur

Procédure exécutée par l'humain avant `Status: ready` (AC-15).

1. **Stack** : `docker compose up -d`, vérifier gateway + workers + postgres `healthy`.
2. **Seed** : envoyer 2-3 questions via `/` (chat) pour créer des lignes `conversations`. Vérifier `SELECT count(*) FROM conversations` ≥ 2.
3. **Ouvrir** `http://localhost:8080/observability` → `<section id="usage-widget">` affiche `Conversations : <N>` (AC-9, libellé D-5). Network tab : `GET /v1/stats` 200, body `{"conversation_count": <N>}` == `count(*)` exact.
4. **Vérifier les en-têtes** (DevTools Network, `GET /observability` et `GET /v1/stats`) : les 5 headers de sécurité présents (AC-11).
5. **Nav** : cliquer `Chat` → revient sur `/` ; depuis `/`, cliquer `Observabilité` → va sur `/observability` (AC-8).
6. **Anonyme** : `curl -i http://localhost:8080/observability` (sans cookie) → 200 ; `curl -i http://localhost:8080/v1/stats` → 200 (AC-2, AC-6).
7. **Erreur** : `docker compose stop postgres` → recharger `/observability` → `#usage-widget` affiche `Données indisponibles.` ; `curl -i http://localhost:8080/v1/stats` → 503 `upstream_unavailable`, aucun détail SQL (AC-7, AC-10).
8. **Path traversal** : `curl -i http://localhost:8080/assets/../Cargo.toml` → 400/404 (AC-13).
9. **Exactitude** : croiser le `conversation_count` exposé avec `SELECT count(*) FROM conversations` → égalité stricte (count exact exposé par design, D-1, AC-5).

## Performance / SLO

- `GET /observability` (HTML statique) : < 50 ms p95 stack locale (fichier statique, pas de DB).
- `GET /v1/stats` : < 80 ms p95 (1 `SELECT count(*)` exact sur `conversations`). Vision §92 budget gateway respecté. Note : `count(*)` exact peut devenir coûteux à fort volume — ré-arbitrage (estimation `reltuples` ou cache) UNIQUEMENT si p95 > 80 ms post-deploy (décision D-3, pas avant).
- Taille `observability.html + observability.js + observability.css` : < 20 KiB cumulé (cible souple, cohérent UI-001/UI-002).

## Security / trust boundary

- Frontière Internet → Gateway. Surface : 1 nouvelle page HTML publique + 2 assets publics + 1 endpoint API public anonyme.
- **A01 Broken Access Control** : `/v1/stats` est public anonyme par design (AC-6) ; il n'expose aucune donnée par-utilisateur, uniquement un cardinal agrégé total (`conversation_count` exact, AC-4/AC-5). Décision D-1 : exactitude assumée — la valeur est un total agrégé (pas de PII, pas de donnée per-user), faible sensibilité, exposée volontairement sur la vitrine portfolio. Pas d'IDOR (aucun identifiant de ressource en path/query).
- **A03 Injection** : `SELECT count(*)` sans paramètre utilisateur ; via `sqlx` macro/`query_as` (compile/runtime-typé), pas de `format!`.
- **A05 Misconfiguration** : CSP UI-001 strictement préservée (AC-11). Erreur 503 enveloppe sanitisée `{"error":"upstream_unavailable","request_id":"<uuid>"}`, jamais de message SQL ni de host ni de stack (AC-7), cohérent §A05 + vision §92 « pas de stack trace ni d'erreur interne brute ». Décision D-1 : le count est exact côté succès, mais le chemin échec reste générique (asymétrie volontaire).
- **A09 Logging & Monitoring** : log `tracing` structuré sur `GET /v1/stats` autorisé (`request_id`, `latency_ms`, `conversation_count`) ; le compte exact peut figurer dans les logs internes (décision D-4 : non public, rétention 30j, accès opérateur — point moot puisque le body API expose déjà le count exact, D-1). Pas de log applicatif sur `GET /observability` ni `GET /assets/*` (TraceLayer suffit, cohérent UI-001 AC-18).
- **RAG-specific** : aucune donnée de conversation (contenu, `gcs_uri`, `user_id`) exposée — uniquement un cardinal agrégé total.
- Pas de secret dans `gateway/static/observability.{html,js,css}` (rappel `secret-hygiene.md`).

## Décisions

Arbitrages humains figés (v2). Chaque décision résout une OQ de la v1.

- **D-1 (ex-OQ-1) — `conversation_count` = count EXACT, pas d'arrondi/bucketisation.** Le body de `GET /v1/stats` retourne `{"conversation_count": <int>}` égal à `SELECT count(*) FROM conversations`. Décision humaine assumée, override délibéré du wording vision §92 « agrégat sanitisé ».
  - Rationale : la valeur est un total agrégé (aucune donnée per-user, aucune PII), faible sensibilité ; l'auteur veut le vrai nombre affiché sur la vitrine portfolio.
  - Garde-fou maintenu : le chemin **échec** reste sanitisé — `503 {"error":"upstream_unavailable","request_id":"..."}`, jamais de détail SQL/host/stack (AC-7, §A05). Asymétrie volontaire : valeur exacte sur succès, erreur générique sur échec.
- **D-2 (ex-OQ-2) — Nav VANILLA, header dupliqué en HTML statique.** Pas de composant JS partagé phase 1.
  - Rationale : le trigger ADR-0005 (framework / build step) n'est PAS franchi (header = 1 composant, pas >3 réutilisables ; zéro dépendance ajoutée ; slice ≤ 300 LOC tenue).
  - Suite : un ticket **nav-refactor** est justifié dès qu'une **4e page** atterrit (DRY `clean-code.md` « extract on third occurrence » devient pertinent à la 3e duplication).
- **D-3 (ex-OQ-3) — `count(*)` exact, AUCUN cache phase 1.** Ré-arbitrage (estimation `reltuples` ou cache TTL gateway) uniquement si **p95 > 80 ms post-deploy** (SLO §Performance). Pas avant — YAGNI.
- **D-4 (ex-OQ-4) — Logs internes peuvent porter le count exact.** Champ `conversation_count` autorisé dans le log `tracing` de `GET /v1/stats` (surface non publique, rétention 30j, accès opérateur). Point devenu moot puisque le body API expose déjà le count exact (D-1) ; la sanitisation reste obligatoire sur le **chemin erreur** (logs `warn` 503 sans détail SQL brut).
- **D-5 (ex-OQ-5) — Libellé `Conversations : <N>` dans `#usage-widget` + `<h1>Observabilité</h1>`.** Le libellé exact n'est PAS figé byte-for-byte en AC (souplesse implémentation) ; seuls l'insertion via `textContent` et la présence du nombre sont contractuels (AC-9).

## Open questions

- **OQ-6 — Code d'erreur 503 : `upstream_unavailable` vs `internal` ?** La spec utilise `{"error":"upstream_unavailable",...}` (cohérent avec d'autres ACs et le fait que la cause est une dépendance DB). Le rappel humain D-1 cite la forme générique `security.md` §A05 `{"error":"internal","request_id":"..."}`. Les deux sont sanitisées (aucune fuite interne) ; il reste à figer la **chaîne `error` exacte** avant `ready`. Reco : conserver `upstream_unavailable` (sémantiquement plus juste — la DB est une dépendance amont, le gateway lui-même fonctionne) ; à confirmer par l'humain. Faible enjeu, purement nommage de contrat d'erreur.
