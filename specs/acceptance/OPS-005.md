# OPS-005 — Ingestion `lore/` automatisée sur push `main` via Cloud Run Job

## Contexte

Aujourd'hui, après un merge de `lore/**` sur `main` (manuel ou via PR gdrive-sync ING-013), l'ingesteur ING-001 doit être déclenché à la main depuis un poste opérateur (cf. checklist PR gdrive-sync `.github/workflows/gdrive-sync.yml:141` et runbook §Ingestion DB après sync). Cet écart manuel laisse le corpus pgvector désynchronisé du contenu mergé tant qu'un humain n'a pas lancé la commande. Ce ticket ferme la boucle de bout en bout : provisionnement du Cloud Run Job d'ingestion (Terraform) + workflow GitHub Actions qui le déclenche, de sorte que tout changement `lore/**` sur `main` ingère automatiquement contre la base Cloud SQL.

## Critères d'acceptation

- AC-1 : Une ressource Terraform `google_cloud_run_v2_job` nommée `archiviste-ingest` est définie dans `infra/terraform/`, région `europe-west9`, image workers (`${local.ar_base}/workers:latest`), réutilisant le pattern du service workers (`infra/terraform/cloud_run.tf:224-248`) : sidecar `cloud-sql-proxy`, `MISTRAL_API_KEY` injecté en `secret_key_ref` (jamais en clair), exécution sous service account `archiviste-runtime`.
- AC-2 : La définition du Job `archiviste-ingest` fixe son `command`/`args` à `python -m archiviste_workers.ingest --path lore/` et reproduit la connexion Cloud SQL IAM authn du service workers (`INSTANCE_CONNECTION_NAME`, `CLOUD_SQL_IAM_AUTH=true`, volume `cloudsql` monté, `DATABASE_URL` SA email percent-encodé), conforme OPS-003 / SEC-005.
- AC-3 : `.github/workflows/ingest-lore.yml` existe et déclare `on: push:` avec `branches: [main]` et `paths: ['lore/**']`, de sorte qu'un push sur `main` ne touchant aucun fichier sous `lore/` ne déclenche pas le workflow.
- AC-4 : Le workflow s'authentifie à GCP via `google-github-actions/auth@v2` en Workload Identity Federation (`workload_identity_provider: ${{ secrets.GCP_WIF_PROVIDER }}`), sans clé SA JSON commitée ni stockée en secret GitHub, cohérent avec `.github/workflows/deploy.yml:26`.
- AC-5 : Le workflow exécute le Job de manière synchrone via `gcloud run jobs execute archiviste-ingest --wait --region europe-west9` (région lue via `env: REGION: europe-west9`, pattern `deploy.yml:16`) et propage l'issue de l'exécution : le step échoue (et donc le run GitHub Actions échoue) si l'exécution du Job n'atteint pas l'état `Succeeded`.
- AC-6 : Le Job `archiviste-ingest` exécute l'ingesteur ING-001 et termine en exit `0` quand `errors == 0`, exit `1` quand `errors >= 1` (au moins un fichier en erreur, les autres traités), conformément à ING-001 AC-17.
- AC-7 : Le mapping exit code → état Cloud Run Job → verdict workflow est défini ainsi : Job exit `0` → execution `Succeeded` → workflow vert ; Job exit `1` (erreurs fichiers ING-001) → execution `Failed` → **workflow rouge** (signal opérateur qu'au moins un `.md` n'a pas été indexé) ; tout autre exit (init fatal, ex. exit `2` ING-001) → execution `Failed` → workflow rouge.
- AC-8 : Le workflow déclare `concurrency: { group: ingest-lore, cancel-in-progress: false }`, de sorte que deux pushes `lore/**` rapprochés sérialisent leurs exécutions du Job (le second attend le premier, aucun n'est annulé).
- AC-9 : Le log JSON `ingest.summary` (ING-001 AC-16) émis par le Job est récupérable dans Cloud Logging pour l'exécution déclenchée (filtrable par l'execution ID du Job), servant d'oracle d'observabilité du run.
- AC-10 : La section runbook `## §Ingestion DB après sync` (`docs/runbook.md:439`) est réécrite pour décrire le déclenchement automatique sur push `main`, et la checklist d'ingestion manuelle du PR body gdrive-sync (`.github/workflows/gdrive-sync.yml:141`) est mise à jour ou supprimée pour ne plus demander une étape manuelle désormais automatisée.

## Non-goals

- Pas d'ingestion locale one-shot depuis poste opérateur : le chemin `uv run python -m archiviste_workers.ingest` reste documenté en runbook comme procédure de secours, mais sort du périmètre de l'automatisation de ce ticket.
- Pas de modification du pipeline ING-001 lui-même (parse / chunk / embed / upsert) — réutilisé tel quel.
- Pas de schedule cron : déclenchement uniquement événementiel sur push `main` paths `lore/**` (un `workflow_dispatch` manuel est toléré comme commodité mais non requis par les AC).
- Pas de re-ingestion forcée / re-chunking sur changement de paramètres d'embedding (resté ticket futur ING-001 Non-goal).
- Pas de détection/suppression des documents orphelins (fichier `lore/` supprimé → ligne `documents` restante) — hors scope.
- Pas de notification externe (Slack / email) sur échec : le signal est le run GitHub Actions rouge + Cloud Logging.
- Pas de garde-fou concurrence applicatif (verrou DB / advisory lock) au-delà du `concurrency:` GitHub d'AC-8 : la dette TOCTOU ING-001 (`pipeline.py:98-109`) reste ouverte (cf. note /plan), ce ticket la mitige seulement via la sérialisation workflow.
- Pas d'ingestion restreinte au diff du push : full scan `--path lore/` à chaque push, idempotence assurée par le `content_hash` ING-001 (AC-11) qui skippe les fichiers `unchanged`.
- Pas de chaînage `workflow_run` après `deploy.yml` : l'ingestion tourne sur l'image workers courante (les `.md` ne dépendent pas de la version du code) ; le cas limite (push mêlant code chunking + `.md`) est documenté en commentaire du workflow.

## Pre-conditions

- ING-001 mergé : `python -m archiviste_workers.ingest --path lore/` opérationnel, contrat exit code (0 / 1 / 2) et logs `ingest.start` / `ingest.document` / `ingest.summary` figés.
- OPS-003 / SEC-005 en place : pattern cloud-sql-proxy `--auto-iam-authn` + SA `archiviste-runtime` (`roles/cloudsql.instanceUser`) opérationnel ; ressource `google_service_account.archiviste_runtime` et `google_secret_manager_secret.mistral_api_key` déjà déclarées dans `infra/terraform/`.
- `MISTRAL_API_KEY` présent dans GCP Secret Manager et accessible par `archiviste-runtime`.
- Workload Identity Federation configuré pour GitHub Actions (`secrets.GCP_WIF_PROVIDER`, `secrets.GCP_PROJECT_ID`), déjà utilisé par `deploy.yml`.
- Image workers `${local.ar_base}/workers:latest` poussée et contenant le module `archiviste_workers.ingest` (le Job tourne sur l'image courante, sans chaînage avec `deploy.yml` — cf. Non-goals).

## Failure modes

- Job exit `1` (≥ 1 fichier ING-001 en erreur : frontmatter invalide, > 1 MiB, erreur DB partielle) → execution `Failed` → workflow **rouge** ; les fichiers valides ont été ingérés ; opérateur inspecte `ingest.summary` + `ingest.document` en erreur dans Cloud Logging.
- Job exit `2` ou crash init (connexion Postgres KO, modèle embedding non chargeable, secret Mistral absent) → execution `Failed` → workflow rouge ; aucun fichier traité.
- `gcloud run jobs execute --wait` time-out ou Job non trouvé (mauvais nom/région) → step `gcloud` échoue → workflow rouge ; message d'erreur `gcloud` propagé.
- Authentification WIF échoue (provider mal configuré, repo non autorisé) → step `auth` échoue avant `gcloud` → workflow rouge.
- cloud-sql-proxy ne devient pas prêt dans le Job → ING-001 sort en erreur init (exit `2`) → execution `Failed` → workflow rouge.
- Push `main` mêlant code workers (ex. logique chunking) + `lore/**` → l'ingestion tourne sur l'image workers **courante** (pré-promotion `deploy.yml`), pas la nouvelle. Accepté : le corpus `.md` ne dépend pas de la version du code ; cas limite documenté en commentaire du workflow. Si le chunking change réellement, re-ingérer manuellement après promotion (fallback runbook).
- Deux pushes `lore/**` rapprochés → `concurrency: group ingest-lore, cancel-in-progress: false` (AC-8) sérialise : le second run attend le premier. Mitige sans la résoudre la dette TOCTOU ING-001 (`pipeline.py:98-109`) — à signaler au /plan.

## Performance / SLO

- Pas de SLO dur. Le `duration_ms` du log `ingest.summary` reste l'indicateur. Le corpus MVP (~10-50 fichiers) garde le run sous quelques minutes ; le re-traitement complet à chaque push est acceptable car l'UPSERT idempotent skippe les fichiers `unchanged` (ING-001 AC-11).

## Security / trust boundary

- Conforme `secret-hygiene.md` : aucun secret commité ; `MISTRAL_API_KEY` via Secret Manager ; auth GCP via WIF (pas de clé SA JSON longue durée).
- Conforme `.claude/rules/security.md` §A09 : aucun secret loggué ; `ingest.summary` ne contient ni clé ni contenu de document.
- Frontière : le déclencheur est un push `main` (déjà soumis à la protection de branche + review humaine). L'ingesteur traite uniquement des `.md` versionnés, pas d'entrée réseau arbitraire (ING-001 normalise NFKC + cap 1 MiB).
- Le Job tourne en `archiviste-runtime` (moindre privilège : accès Cloud SQL + secret Mistral uniquement).

## Observability

- Cloud Logging : logs JSON ING-001 (`ingest.start`, `ingest.document`, `ingest.summary`) capturés pour chaque execution du Job, corrélables par execution ID.
- GitHub Actions : run vert/rouge = signal de premier niveau ; logs `gcloud run jobs execute --wait` visibles dans le step.
- Pas de métrique OpenTelemetry ni d'alerte GCP dédiée ajoutée par ce ticket.
