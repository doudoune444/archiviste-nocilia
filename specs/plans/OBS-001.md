# Plan — OBS-001 Page observabilité publique + nav + widget usage (GET /v1/stats)

## Goal
Ship a public anonymous `/observability` page with a `Chat | Observabilité` nav and a usage widget fed by a new public `GET /v1/stats` returning the exact `count(*)` of `conversations`.

## Acceptance criteria recap
- AC-1 : `GET /observability` → `200`, `Content-Type: text/html; charset=utf-8`, body contains `<section id="usage-widget"`.
- AC-2 : `GET /observability` served anonymously; never `401`/`403` regardless of auth state.
- AC-3 : `GET /assets/observability.js` → `200` JS (`application/javascript`|`text/javascript`, UTF-8) ; `observability.css` → `200` `text/css` (UTF-8) ; both public.
- AC-4 : `GET /v1/stats` → `200`, `application/json; charset=utf-8`, `{"conversation_count": <int>}` = exact `count(*)`, no rounding/bucketing, no other field.
- AC-5 : for DB with `n` rows → `{"conversation_count": n}` strict; `n1 ≠ n2` → distinct values (mapping = identity).
- AC-6 : `GET /v1/stats` anonymous, no `author_only`/`member_only` guard, absent from auth sub-router; contract test asserts no auth on handler.
- AC-7 : Postgres unavailable/query fails → `503` sanitized `{"error":"upstream_unavailable","request_id":"<uuid>"}`, no SQL/host/table/stack/partial-count leak.
- AC-8 : `index.html` + `observability.html` each carry nav with exactly two links — `Chat`→`/`, `Observabilité`→`/observability`, in order.
- AC-9 : on load, `observability.js` calls `GET /v1/stats` (`fetch`, `credentials: 'same-origin'`), inserts `conversation_count` via `textContent` (never `innerHTML`) as `Conversations : <N>`; page has `<h1>Observabilité</h1>`.
- AC-10 : any non-200 (4xx/5xx/network/timeout/malformed JSON) → `#usage-widget` shows literal `Données indisponibles.`; no per-code branching.
- AC-11 : 5 global security headers present byte-for-byte on `/observability`, `/assets/observability.{js,css}`, `/v1/stats`; CSP NOT modified.
- AC-12 : `observability.html` has no inline `<script>`/`<style>`/`style="..."`/`on*=`; JS via `<script src>`, CSS via `<link>`.
- AC-13 : `/assets/observability.{js,css}` served; unknown `/assets/<x>` → 404; `/assets/../*` → 400/404 (UI-001 AC-5 unchanged).
- AC-14 : `specs/openapi/gateway-to-workers.yml` NOT modified (client↔gateway endpoint, no workers call).
- AC-15 : manual author test checklist done before `Status: ready`.

## Risks / open questions
- AC-9 widget label `Conversations : <N>` is D-5 (not byte-frozen); tests assert `textContent` usage + number presence only, never exact bytes — keep test loose to avoid over-constraining.
- AC-7 error `request_id` is a fresh UUID from `ApiError::into_response`, not the request-scoped `RequestId`. Consistent with existing dashboard handlers; acceptable per AC-7 (only "a uuid" required). Flag for reviewer in case request-id correlation is later desired.
- `make_app_state` builds `db_pool=None`; confirm via `state.rs` it is `None` and not a lazy pool, else AC-7 no-pool path won't fire deterministically. (tickets.rs:113-116 relies on identical `None` semantics — high confidence.)
- AC-5 fixture: `conversations.user_id` is NOT NULL FK → must insert sentinel `users` row first; `gcs_uri` NOT NULL UNIQUE → distinct per row (precedent test_dashboard_backend.rs:358-368).
