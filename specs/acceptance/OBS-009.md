# OBS-009 — Cloud Run Job d'eval live one-shot vers Cloud SQL prod (slice réduite)

## Contexte

L'image eval (OBS-008) existe dans l'Artifact Registry mais rien ne l'exécute en prod : le dashboard qualité public (`GET /v1/quality`, OBS-004) n'a aucun snapshot Ragas `live` réel à afficher. Ce ticket provisionne un Cloud Run Job qui, exécuté **une fois** manuellement par l'opérateur, lance le runner Ragas en mode live et persiste un snapshot dans `eval_runs`, de sorte que le dashboard montre une mesure réelle au lieu de `no_data`. Slice réduite et délibérément non récurrente : pas de scheduler, pas de re-bake baseline, zéro nouveau rôle IAM.

## Acceptance criteria

- AC-1 : Une exécution du Job `archiviste-eval` via `gcloud run jobs execute archiviste-eval --region=<region>` se termine avec un exit code `0` (le run Ragas live complet réussit jusqu'au persist inclus).
- AC-2 : Après cette exécution, `SELECT count(*) FROM eval_runs WHERE runner_mode='live'` a augmenté d'**exactement 1** par rapport à sa valeur avant exécution (une seule ligne `live` insérée, pas zéro, pas plusieurs).
- AC-3 : La ligne insérée porte `runner_mode='live'` et un `finished_at` non nul renseigné par le run qui vient de s'exécuter.
- AC-4 : Après l'exécution, `GET /v1/quality` en prod retourne `200` avec un corps **non** `no_data`, dont le `finished_at` est égal au `finished_at` de la ligne `eval_runs` nouvellement insérée (le dashboard reflète bien ce run-là).
- AC-5 : Le Job lit le golden set depuis GCS au runtime à `gs://archiviste-lore-corpus/golden/golden_qa.jsonl` ; le sha256 de cet objet est égal au sha256 du fichier local `specs/golden_qa.jsonl` de l'opérateur (source canonique, non versionnée) au moment de sa publication dans le bucket privé.
- AC-6 : Le Job se connecte à Cloud SQL prod (`archiviste-db`) en IAM authn sous l'identité `archiviste-runtime` (volume `cloudsql` + `DATABASE_URL` IAM-auth + `INSTANCE_CONNECTION_NAME`), miroir du Job d'ingestion, sans mot de passe en clair.
- AC-7 : Le Job est défini en Terraform sous `infra/terraform/` avec `containers.command` figeant l'invocation `python -m eval.ragas_runner --mode live --persist --set <golden_path> --output <tmp_path>` (commande figée dans le Job, miroir du Job d'ingestion ; aucune exécution déclenchée par GHA).
- AC-8 : Aucun nouveau rôle ni binding IAM n'est introduit : la lecture du golden réutilise le grant `roles/storage.objectViewer` existant de `archiviste-runtime` sur `archiviste-lore-corpus`, et l'écriture DB réutilise `roles/cloudsql.client` + `roles/cloudsql.instanceUser` existants.

## Non-goals

- Pas de Cloud Scheduler ni de cron : l'exécution est one-shot manuelle (`gcloud run jobs execute`). Terraform provisionne le Job, ne le déclenche pas.
- Pas d'exécution automatique par GHA / CI : aucun workflow ne lance le Job sur merge ou sur un événement.
- Pas de recalibration de Gate A, pas de re-bake de `baseline.json`.
- Pas de nouveau rôle ni binding IAM (AC-8). Si la review découvre que la SA manque réellement d'une permission, c'est une surprise à **remonter**, pas à supposer ni à patcher.
- Pas de modification du code runner (`eval/ragas_runner.py`, `eval/persist.py`, `eval/oidc.py`, etc.) : le ticket déploie le code OBS-007/OBS-008 existant tel quel.
- Pas de modification du contrat `specs/openapi/gateway-to-workers.yml` ni du schéma `eval_runs` (`migrations/0005_eval_runs.sql`).
- Pas de répétition / runs multiples ni de logique d'idempotence du Job : il insère une ligne par exécution (AC-2) ; ré-exécuter en insère une autre — c'est attendu, le dashboard prend la plus récente.

## Pre-conditions

- **OBS-008 mergé (#129)** : image eval `eval:latest` présente dans l'Artifact Registry `archiviste`, runner live-capable (Ragas + judge Mistral + Mistral generate, pur-API), `ENTRYPOINT` = `python -m eval.ragas_runner`.
- **OBS-007 mergé** : `eval/oidc.py` prêt pour l'auth OIDC des appels workers authentifiés.
- **OBS-003 / `migrations/0005_eval_runs.sql` appliqué** : table `eval_runs` (13 colonnes) existe en prod ; `eval/persist.py` lit `DATABASE_URL` (sans fallback) et fait un INSERT paramétré.
- **OBS-004 mergé (#127)** : `GET /v1/quality` (`gateway/src/handlers/quality.rs`) déployé en prod, lit `eval_runs WHERE runner_mode='live' ORDER BY finished_at DESC LIMIT 1`.
- **Cloud SQL prod `archiviste-db`** : IAM authn activée, user `archiviste-runtime` provisionné ; Job d'ingestion (`infra/terraform/cloud_run_job.tf`) sert de patron miroir (volume `cloudsql` + `DATABASE_URL` IAM-auth + `INSTANCE_CONNECTION_NAME`).
- **Grants IAM existants suffisants** : `archiviste-runtime` a `roles/storage.objectViewer` sur `archiviste-lore-corpus` (`storage.tf:34-37`) + `roles/cloudsql.client` & `roles/cloudsql.instanceUser` (`iam.tf:40-44`).
- **Secret Manager** : clé API Mistral (judge + generate) disponible via `secret_key_ref`, comme pour le Job d'ingestion (`MISTRAL_API_KEY`).
- **Golden set publié dans GCS** : le golden set doit exister à `gs://archiviste-lore-corpus/golden/golden_qa.jsonl` au moment de l'exécution. Source canonique = le fichier local `specs/golden_qa.jsonl` de l'opérateur. Publication = upload one-time documenté en runbook (commande manuelle opérateur), pas un step CI.
- **Golden gitignored & humain-only** : `specs/golden_qa.jsonl` est gitignored (`.gitignore:105`), jamais tracké en git ni public ; sa copie canonique est le fichier local de l'opérateur, distribué en privé via le bucket GCS `archiviste-lore-corpus` (`public_access_prevention=enforced`).

## Failure modes

- Golden absent ou illisible à `gs://archiviste-lore-corpus/golden/golden_qa.jsonl` → fetch échoue → run échoue, exit ≠ `0`, **aucune** ligne insérée (AC-1/AC-2 en échec, état observable propre : count inchangé).
- Connexion Cloud SQL refusée (IAM authn KO, user `archiviste-runtime` absent, `cloudsql.instanceUser` manquant) → `eval/persist.py` ne peut écrire → exit ≠ `0`, count inchangé.
- Clé API Mistral absente/invalide (judge ou generate) → le run Ragas échoue avant persist → exit ≠ `0`, count inchangé.
- Image `eval:latest` introuvable / tag non poussé → Job ne démarre pas (échec de pull) → aucune ligne.
- `eval_runs` absente (migration non appliquée en prod) → INSERT échoue → exit ≠ `0`.
- Run live réussit mais `GET /v1/quality` renvoie encore `no_data` ou un `finished_at` ne matchant pas → AC-4 en échec (incohérence dashboard ↔ ligne ; à investiguer : tri `finished_at`, fuseau, mode lu).

## Security / trust boundary

- Frontière : Cloud Run Job (`archiviste-runtime`) → Cloud SQL prod (IAM authn, zéro mot de passe) + lecture golden GCS (objectViewer existant) + appels Mistral (clé via Secret Manager).
- **A02 / Secret hygiene** : `MISTRAL_API_KEY` via `secret_key_ref` (`security.md` §A09, `secret-hygiene.md`) ; `DATABASE_URL` IAM-auth sans mot de passe (SA email percent-encodé `%40`, miroir ingest). Aucun secret figé dans le Job ni l'image.
- **A01 least-privilege** : zéro nouveau rôle (AC-8) ; objectViewer scoped bucket + `cloudsql.client`/`instanceUser` couvrent exactement read-golden + write-DB.
- Golden lu depuis bucket privé `archiviste-lore-corpus` (`public_access_prevention=enforced`, uniform access). Aucune surface Internet ajoutée.

## Observability

- Logs d'exécution = logs Cloud Run Job standard + logging structlog du runner (OBS-003/007). Pas d'observabilité custom phase 1.
- Le snapshot persisté est lui-même le signal observable (ligne `eval_runs` + dashboard `/v1/quality`).
