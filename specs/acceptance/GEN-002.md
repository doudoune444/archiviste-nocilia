# GEN-002 — Forwarder gateway `POST /v1/chat` vers workers `/v1/generate`

## Contexte

Le walking skeleton MVP (vision §107) requiert une porte d'entrée publique côté gateway Rust qui transporte une question utilisateur vers le pipeline RAG hébergé côté workers Python. En phase 1 sans auth (vision §73), le gateway hardcode `user_tier=anonymous` et la sentinel `user_id=00000000-0000-0000-0000-000000000000` (FOUND-003 AC-2). Ce ticket ferme la boucle end-to-end client → gateway → workers en posant un forwarder minimaliste, traçable par `request_id` et tolérant aux pannes upstream.

## Critères d'acceptation

- AC-1 : `POST /v1/chat` accepte un body JSON `{"query": string, "conversation_id": string|null}`, retourne 200 quand workers répond 200 et passe le body workers tel quel au client.
- AC-2 : Chaque requête entrante reçoit un `request_id` UUIDv4 généré par le gateway ; ce `request_id` est présent dans le header `X-Request-Id` de la réponse client ET dans le body forwardé aux workers.
- AC-3 : Le body envoyé à `POST {WORKERS_BASE_URL}/v1/generate` contient exactement les champs `{query, conversation_id, user_id, user_tier, request_id}` avec `user_id="00000000-0000-0000-0000-000000000000"` et `user_tier="anonymous"` (valeurs littérales, jamais lues d'un header ou body client).
- AC-4 : Le gateway ajoute le header `X-Request-Id: <request_id>` à la requête sortante vers workers.
- AC-5 : `query` absent, vide, non-string, ou de taille > 4096 octets UTF-8 produit une réponse 400 `{"error":"invalid_request","request_id":"<uuid>"}` sans contact workers.
- AC-6 : `conversation_id` non-null et non-UUID valide produit une réponse 400 `{"error":"invalid_request","request_id":"<uuid>"}` sans contact workers.
- AC-7 : Body non-JSON ou taille body > 1 MiB produit une réponse 400 `{"error":"invalid_request","request_id":"<uuid>"}` sans contact workers.
- AC-8 : Si l'appel workers dépasse 35 secondes (read timeout), gateway répond 504 `{"error":"upstream_timeout","request_id":"<uuid>"}`.
- AC-9 : Si la connexion workers échoue (DNS, connection refused, reset) avant 5 secondes (connect timeout), gateway répond 503 `{"error":"upstream_unavailable","request_id":"<uuid>"}`.
- AC-10 : Si workers répond avec un statut 4xx (autre que 400) ou 5xx, gateway répond 502 `{"error":"upstream_error","request_id":"<uuid>"}` ; le body workers n'est PAS forwardé au client.
- AC-11 : Si workers répond 400, gateway répond 502 `{"error":"upstream_error","request_id":"<uuid>"}` (un 400 workers signale une violation de contrat interne, pas une erreur client).
- AC-12 : Aucune réponse d'erreur du gateway ne contient de stack trace, message d'erreur upstream brut, chemin de fichier, query SQL, ou nom de host workers ; uniquement `{"error":"<code>","request_id":"<uuid>"}` (security A05 / G-I-1).
- AC-13 : Pour chaque requête traitée, le gateway émet exactement un log JSON structuré `{"event":"chat","request_id":"<uuid>","query_len":<int>,"upstream_status":<int|null>,"status":<int>,"latency_ms":<int>}` ; le champ `query` brut n'apparaît jamais dans les logs (security A09).
- AC-14 : Le `reqwest::Client` utilisé pour appeler workers est instancié une seule fois au boot (pool keep-alive partagé) et réutilisé pour toutes les requêtes `/v1/chat`.
- AC-15 : Le body workers est lu via `response.bytes()` capé strictement à **256 KiB** ; au-delà, gateway répond 502 `{"error":"upstream_error","request_id":"<uuid>"}` + log warn `event=upstream_body_too_large`. Le passthrough byte-pour-byte est conservé sous cette limite (cf AC-1).
- AC-16 : Si le client envoie un header `X-Request-Id`, il est **ignoré** ; gateway génère toujours son propre UUIDv4 (AC-2). Justification : phase 1 sans auth, header client non-trusté ; corrélation distribuée reportée à un ticket SEC-* / OBS-* dédié.

## Non-goals

- Pas d'auth (JWT, sessions, OAuth) — ticket SEC-* dédié, vision §73.
- Pas de rate-limiting (Redis sliding window, `tower_governor`) — ticket SEC-* dédié.
- Pas de cost-guard / circuit breaker LLM — ticket SEC-* dédié.
- Pas de cache de réponses gateway — ticket SEC-* dédié.
- Pas de prompt-injection filter côté gateway (déjà côté workers via GEN-001 / W-T-1).
- Pas de PII scrub côté gateway — ticket SEC-* dédié.
- Pas de circuit breaker custom (peut être posé incrémentalement plus tard).
- Pas de streaming SSE / chunked — réponse JSON complète uniquement.
- Pas de versioning multi-API (`/v2/`) — phase 1.
- Pas d'enregistrement de la query dans `query_log` côté gateway — fait côté workers (GEN-001 ou OBS-* dédié).
- Pas d'ajout de l'endpoint au contrat `specs/openapi/gateway-to-workers.yml` (ce yml décrit gateway↔workers ; le contrat client↔gateway est documenté ici uniquement, pas généré phase 1).
- Pas de modification du contrat workers `/v1/generate` (assumé existant et figé par GEN-001).
- Pas de vérification HMAC sur l'appel workers (W-S-1) — couvert par un ticket SEC-* quand workers exposé hors localhost docker.

## Failure modes

- Body invalide / `query` manquante / `query` > 4 KiB / `conversation_id` non-UUID / body > 1 MiB → 400 `invalid_request`, log structuré, aucun appel workers.
- Workers ne répond pas dans 35s → 504 `upstream_timeout`, log avec `upstream_status: null`.
- Workers injoignable (DNS / connection refused / reset, < 5s) → 503 `upstream_unavailable`, log avec `upstream_status: null`.
- Workers répond 4xx/5xx → 502 `upstream_error`, log avec `upstream_status: <code workers>`.
- Workers répond 200 avec body non-JSON ou body > taille raisonnable (à figer dans le plan) → 502 `upstream_error`, log warning.
- Panic/erreur interne handler (non attendu, deny par lints) → 500 `{"error":"internal","request_id":"<uuid>"}` (filet de sécurité).

## Performance / SLO

- Overhead gateway p95 < 80 ms à 500 users concurrents (vision §91), exclusif de la latence workers/LLM.
- Round-trip total chat p95 < 3 s (vision §89), validé downstream par GEN-001 + load test EVAL-* / OBS-*.
- Le keep-alive du `reqwest::Client` doit éviter le coût de handshake TCP/TLS répété (AC-14).

## Security / trust boundary

- Frontière Internet → Gateway (G-* du threat model).
- A04 (Insecure Design) : body cap 1 MiB explicite, query length cap 4 KiB explicite.
- A05 (Misconfiguration) : enveloppe d'erreur uniforme, jamais de fuite (AC-12 + G-I-1).
- A09 (Logging & Monitoring) : log structuré, `query` brute jamais loggée (AC-13).
- W-S-1 (Workers spoofing) : non couvert par ce ticket — assume gateway et workers communiquent sur le réseau Docker interne (`workers://workers:8000`). Quand workers est exposé en Cloud Run (production), un ticket SEC-* dédié ajoutera HMAC `X-Gateway-Auth`.
- Hardcoding `user_id` sentinel et `user_tier="anonymous"` est volontaire phase 1 (vision §73). Aucune entrée client ne peut influencer ces valeurs (AC-3).
- Aucun `unwrap()` / `expect()` sur input client (lints `gateway/Cargo.toml`).

## Observability

- Log JSON par requête : event `chat` (AC-13).
- Pas de métrique OpenTelemetry / trace Langfuse exposée à ce stade — réservé OBS-*.
- Pas d'alerte spécifique au-delà des codes HTTP émis.

## Decisions résolues (humain, 2026-05-05)

- A1 — Contrat workers : **workers orchestre retrieve→generate en interne** (cf GEN-001 AC-3 qui appelle `/v1/retrieve`). Body forwardé par gateway = `{query, conversation_id, user_id, user_tier, request_id}` (pas de `contexts`). Le fichier `specs/openapi/gateway-to-workers.yml` sera **régénéré** par RET-001/GEN-001 (initialisation squelette OpenAPI) et amendé pour refléter le contrat réel.
- A2 — `user_tier` sémantique : valeur littérale `"anonymous"` envoyée à workers représente le **scope ACL** (mappé directement sur `documents.access_tier ∈ {public}`), pas l'identité utilisateur. En phase 1 sans auth, gateway hardcode toujours `anonymous`. Le mapping `users.tier → acl_scope` (ex. `member → members`, `author → author_only`) sera introduit par un ticket SEC-* d'auth.
- OQ-3 — Variable env : **`WORKERS_URL`** conservé (cohérence repo, évite churn FOUND-001).
- OQ-4 — Réponse 200 : **passthrough byte-pour-byte** sous le cap 256 KiB (cf AC-15). Tolère ajouts de champs workers sans casser gateway.
- OQ-5 — Header client `X-Request-Id` : **ignoré**, gateway toujours génère (AC-16). Phase 1 sans auth = client non-trusté.
- OQ-6 — Cap body workers : **256 KiB** (cf AC-15).
- OQ-7 — `conversation_id: null` : **accepté**, workers gère la création (cf GEN-001 AC-2).
