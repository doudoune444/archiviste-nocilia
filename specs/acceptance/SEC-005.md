# SEC-005 — Cloud SQL IAM token injection in sqlx + asyncpg pools

## Contexte

INFRA-002 PR-f a basculé les services applicatifs sur Cloud SQL IAM database auth (`DATABASE_URL=postgres://<sa>@localhost/archiviste?host=/cloudsql/...`, password vide), partant de l'hypothèse que le proxy Cloud Run v2 intégré injecterait côté client le jeton OAuth IAM dans le slot password. La vérif post-déploiement (`docs/blockers.md` 2026-05-29 §INFRA-002 PR-f) prouve l'hypothèse fausse : Cloud Run v2 expose le socket Unix mais n'injecte aucun token, et Cloud SQL rejette le handshake (`password authentication failed for user "archiviste-runtime@<project>.iam"`). Les deux pools (gateway sqlx, workers asyncpg) doivent fournir eux-mêmes un access token IAM frais (TTL 1 h) à chaque acquisition de connexion. SEC-005 réutilise le `TokenProvider` livré par SEC-004 (déplacé en module `auth_metadata` partagé avec GCS) et l'instancie deux fois avec des scopes OAuth distincts : `sqlservice.admin` pour Cloud SQL, scope GCS par défaut pour signBlob.

## Critères d'acceptation

- AC-1 : Le module `gateway/src/gcs/token.rs` (livré par SEC-004) est renommé en `gateway/src/auth_metadata/token.rs`, et un nouveau module `auth_metadata` est créé (`gateway/src/auth_metadata/mod.rs`). La struct `TokenProvider` est exposée via `crate::auth_metadata::TokenProvider`. Le sous-module `gateway/src/gcs/` réexporte `pub use crate::auth_metadata::token::TokenProvider;` pour préserver les call sites SEC-004 sans churn. Un `grep -RIn 'gcs::token::TokenProvider' gateway/src` retourne 0 résultat hors le réexport ; tous les usages internes pointent sur `auth_metadata::TokenProvider`.
- AC-2 : `TokenProvider::with_base_url` accepte un argument supplémentaire `scope: &str` (ou `scopes: &[&str]`) qui est concaténé à l'URL metadata du fetch token sous forme `?scopes=<url-encoded>` (cf. doc metadata server `/computeMetadata/v1/instance/service-accounts/default/token?scopes=...`). Le constructeur `TokenProvider::new()` existant (sans scope) est retiré ou conserve la sémantique « scope par défaut du SA Cloud Run » ; deux constructeurs nommés sont fournis : `TokenProvider::for_gcs_signing()` (scope par défaut, équivaut à l'ancien comportement) et `TokenProvider::for_cloud_sql()` (scope `https://www.googleapis.com/auth/sqlservice.admin`). Aucun cache n'est partagé entre les deux instances : chaque `TokenProvider` détient son propre `RwLock<Option<CachedToken>>`.
- AC-3 : `gateway/src/state.rs` `AppState` expose deux champs : `gcs_token_provider: Arc<TokenProvider>` (renommage du `token_provider` SEC-004 existant) et `sql_token_provider: Arc<TokenProvider>`. Le call site `sign_get` (`gateway/src/handlers/conversations.rs`) référence désormais `state.gcs_token_provider`. Le grep `rg 'state\.token_provider\b' gateway/src` retourne 0 résultat post-refactor.
- AC-4 : `gateway/src/lib.rs:495` (`PgPoolOptions::new()...connect(&config.database_url)`) est remplacé par une construction `PgPoolOptions::new().max_connections(5).max_lifetime(Duration::from_secs(45 * 60)).before_acquire(|conn, _meta| { ... })` où le hook `before_acquire` exécute, par connexion : (a) `sql_token_provider.get_or_refresh()` pour obtenir le bearer IAM courant, (b) `conn.execute(format!("SET SESSION AUTHORIZATION ...").as_str())` n'est PAS utilisé — à la place, le token frais est injecté **au moment du `connect`** via `PgConnectOptions::password(...)` reconstruit à partir du `DATABASE_URL`. La forme exacte retenue : `PgPoolOptions::connect_with(PgConnectOptions::from_str(&config.database_url)?.password(token.expose_secret()))` n'étant pas compatible avec `before_acquire` (qui agit post-connect), le pattern utilisé est `PgPoolOptions::connect_lazy_with(opts)` + un hook `after_connect` (équivalent sqlx `connect_with` capable d'injecter le password à chaque nouveau connect physique du pool). Le ticket retient `after_connect` comme hook nominal et `before_acquire` uniquement comme garde-fou anti-token-expired sur acquisition (vérifie `expires_at`, ferme la connexion pour forcer un reconnect si le token ayant servi à l'établir est dans la fenêtre `< exp - 60s`). Cache-hit (token frais déjà en cache) ≤ 1 ms wall-clock par acquisition (mesuré via test).
- AC-5 : `max_lifetime = Duration::from_secs(45 * 60)` (45 min) est positionné sur `PgPoolOptions` côté gateway : les connexions sont recyclées avant l'expiration du token IAM (TTL 1 h Cloud SQL IAM). La constante est nommée (`SQL_CONNECTION_MAX_LIFETIME_SECS = 45 * 60`) dans `gateway/src/lib.rs` ou un module dédié, pas un magic number.
- AC-6 : Côté workers Python, `workers/src/archiviste_workers/db.py` `create_pool` est étendu avec un paramètre `token_provider: SqlTokenProvider` (nouvelle classe livrée par ce ticket, fichier `workers/src/archiviste_workers/auth_metadata/token.py`, ~80 LOC). Le pool `asyncpg.create_pool` reçoit un kwarg `password=<callable>` (asyncpg accepte un callable async retournant `str` — vérifier au plan ; sinon usage de `setup=` callback qui ré-authentifie). Pattern retenu si callable non supporté : `connection_class=` custom héritant de `asyncpg.Connection` et override de la phase auth, OU pre-fetch du token avant `create_pool` + `max_inactive_connection_lifetime=45*60` pour garantir le recyclage. Le choix exact entre les deux patterns est tranché au `/plan` ; l'AC contractuel = « workers s'authentifient à Cloud SQL via token IAM frais récupéré sur le metadata server, sans password statique en env ».
- AC-7 : `SqlTokenProvider` Python implémente la même sémantique que la version Rust : (a) fetch sur `GET http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token?scopes=https://www.googleapis.com/auth/sqlservice.admin` avec header `Metadata-Flavor: Google`, (b) cache en mémoire avec refresh-ahead (`now >= expires_at - 60s`), (c) lock `asyncio.Lock` autour du refresh (équivalent `RwLock` Rust), (d) timeouts `connect=2s` `total=5s` via une instance `httpx.AsyncClient(timeout=httpx.Timeout(5.0, connect=2.0))` **dédiée au `SqlTokenProvider`** (pas de réutilisation du client `services/http_client.py` workers — cohérence avec le pattern SEC-004 Rust qui instancie son propre `reqwest::Client` dans `TokenProvider`). Le client est créé dans le constructeur du provider et fermé proprement au shutdown du `lifespan`. Le bearer est encapsulé `pydantic.SecretStr` pour garantir la redaction en log/repr (`security.md §A09`).
- AC-8 : Boot gateway : si le premier fetch token IAM (au premier `connect` du pool) échoue (metadata server indisponible, `403` quota, timeout), `run()` retourne `Err` et le process exit non-zéro. Aucun fallback password statique n'est tenté. Le log d'exit contient `event="boot.sql_pool_init_failed"` + `reason_code` ∈ `{"metadata_token_failed","cloud_sql_auth_failed","timeout","network"}`. Le grep `rg 'env::var.*PGPASSWORD\|password.*env' gateway/src` retourne 0 ligne (aucune lecture de password depuis l'env).
- AC-9 : Boot workers : si `create_pool` lève une exception (metadata fetch fail, Cloud SQL auth fail), le `lifespan` FastAPI propage l'exception et le process exit non-zéro. Aucun catch silencieux. Le log d'exit contient `event="boot.sql_pool_init_failed"` + `reason_code` parmi le même ensemble qu'AC-8. Le `lifespan` actuel (`workers/src/archiviste_workers/main.py:39`) `pool = await create_pool(settings.database_url)` est remplacé par `pool = await create_pool(settings.database_url, token_provider=SqlTokenProvider())` et la création du `SqlTokenProvider` se fait en amont du `create_pool` afin que toute erreur d'instanciation soit fail-fast.
- AC-10 : Chaque échec du chemin de connexion DB (gateway et workers) émet un événement `tracing` (gateway) / `structlog` (workers) niveau `warn`, event name `sql_pool.connection_failed`, avec champs `reason_code` ∈ exactement `{"metadata_token_failed","cloud_sql_auth_failed","timeout","network"}` + `latency_ms` + `request_id` (si en contexte requête, sinon omis au boot). Aucun champ ne contient le token bearer, ni le `DATABASE_URL` complet (host autorisé, password jamais loggé). Le mapping : metadata 4xx/5xx → `metadata_token_failed` ; Cloud SQL rejette le handshake (`28P01` / `28000`) → `cloud_sql_auth_failed` ; dépassement timeout (2s connect / 5s metadata, 30s Cloud SQL connect) → `timeout` ; erreur transport (DNS, TCP RST) → `network`.
- AC-11 : Tests d'intégration gateway (`gateway/tests/test_sql_pool.rs` nouveau OU extension de `test_dashboard_backend.rs`) couvrent via mockito du metadata server : (a) cas nominal → 200 OK metadata (token `test-sql-token`, expires_in 3600) → pool acquire réussit contre le **service Postgres `docker-compose.yml`** (port 5432, base `archiviste`) — ce même service est démarré en CI via `docker compose up -d postgres` avant l'étape `cargo test` (pas de skip CI) ; (b) metadata 500 → boot exit `Err` + log `reason_code=metadata_token_failed` ; (c) cache-hit token : deux `pool.acquire()` consécutifs déclenchent **un seul** fetch metadata (mockito `expect(1)` sur token endpoint) ; (d) refresh-ahead : token avec `expires_in=61`, deuxième acquire après fake clock +2s → `expect(2)` fetchs metadata (le second a basculé dans la fenêtre `< exp - 60s`).
- AC-12 : Tests d'intégration workers (`workers/tests/test_sql_pool.py` nouveau) couvrent les mêmes 4 cas (a) à (d) via `pytest-httpx` ou `respx` pour mocker le metadata server. Le test (a) cas nominal s'exécute contre le **service Postgres `docker-compose.yml`** (réutilisation `conftest.py` existant), démarré en CI via `docker compose up -d postgres` avant `pytest` — aucun skip CI conditionnel.
- AC-13 : `.env.example` (racine + `workers/.env.example` si présent) ne mentionne plus `PGPASSWORD` ni `DATABASE_PASSWORD`. Un commentaire ajouté au-dessus de `DATABASE_URL` indique : « Cloud SQL IAM auth : le slot password de l'URL doit rester vide ; le token IAM est injecté runtime par SEC-005 (gateway sqlx hook + workers asyncpg hook). »
- AC-14 : `docs/runbook.md` §8 (ou section Cloud SQL IAM) est mise à jour avec : (a) note explicite que SEC-005 remplace l'hypothèse « proxy injecte token » de PR-f, (b) commande de vérif post-déploiement `gcloud run services logs ... | grep sql_pool.connection_failed` qui doit retourner 0 ligne, (c) procédure local dev : `gcloud auth application-default login` + binding `roles/cloudsql.instanceUser` sur le dev (déjà documenté PR-e, référence croisée). Aucune procédure manuelle de rotation token n'est requise (refresh-ahead lazy AC-7).
- AC-15 : `CHANGELOG.md` entrée `### Security` sous `## [Unreleased]` : `SEC-005: gateway sqlx + workers asyncpg pools now fetch a fresh Cloud SQL IAM access token from the metadata server on each new physical connection (45-min max_lifetime, refresh-ahead cache); resolves INFRA-002 PR-f deploy blocker (empty PG password). Reuses SEC-004 TokenProvider relocated to auth_metadata module.`

## Non-goals

- Pas de retour à un PG password statique en Secret Manager (Option A explicitement rejetée par l'opérateur, `docs/blockers.md` 2026-05-29 §INFRA-002 PR-f).
- Pas de sidecar `cloud-sql-proxy --auto-iam-authn` (incompatible Cloud Run v2 integrated proxy, contourne le motif initial de PR-f).
- Pas de cache partagé entre `gcs_token_provider` et `sql_token_provider` (scopes différents, sémantique différente : un token GCS ne s'authentifie pas auprès de Cloud SQL et inversement).
- Pas de circuit breaker autour du metadata server phase 1 (volume très faible : ~1 fetch / 45 min / instance × N instances).
- Pas de métrique counter dédiée `sql_pool_connection_failures_total{reason_code}` phase 1 (instrumentation = log `tracing`/`structlog` AC-10, métrique → OBS-001 V2).
- Pas de support multi-projet / cross-project Cloud SQL phase 1.
- Pas de live test `--ignored` contre Cloud SQL réel : la vérif se fait via runbook curl/logs post-déploiement (AC-14).
- Pas de retry exponential backoff sur metadata token fetch (un seul try ; échec = exit / 503 selon contexte). Backoff complet = SEC-* V2.
- Pas de modification du `DATABASE_URL` format (path Unix socket `?host=/cloudsql/...` conservée tel quel ; seul le mécanisme d'injection password change).
- Pas de propagation du `reason_code` vers le client HTTP côté workers (les handlers continuent de retourner `500` opaque sur défaillance DB).
- Pas de migration vers une crate / lib Cloud SQL officielle (`google-cloud-sql-rs` ou équivalent) : refactor minimal réutilisant `TokenProvider` existant.

## Failure modes

- Metadata server indisponible au boot → process exit non-zéro + log `event=boot.sql_pool_init_failed reason_code=metadata_token_failed`. Aucun fallback.
- Cloud SQL IAM auth rejette le token (token valide mais SA pas binder `roles/cloudsql.instanceUser`) → process exit non-zéro + log `reason_code=cloud_sql_auth_failed`. Mitigation = re-vérifier bindings INFRA-002 PR-e.
- Metadata server `403` (quota IAM access token dépassé, très rare : ~1000 req/s/projet limite, trafic attendu ~1 fetch/45 min) → exit + log `reason_code=metadata_token_failed`.
- Timeout fetch token (> 5s total) → exit ou requête échoue selon contexte (boot vs runtime) + log `reason_code=timeout`.
- Race condition cache token : deux acquisitions simultanées trouvent le token expiré, deux fetchs metadata parallèles → acceptable (idempotent, coût négligeable, sémantique `RwLock` / `asyncio.Lock` = pas de blocage longue durée). Documenté comme acceptable.
- Token cache hit mais Cloud SQL rejette `28P01` (token expiré côté serveur, clock skew client/serveur > 60s buffer) → la connexion échoue, sqlx ferme et retente via `before_acquire` garde-fou (AC-4) qui force un reconnect avec un token frais. Si l'échec persiste après reconnect → requête échoue `500` + log `reason_code=cloud_sql_auth_failed`. Phase 1 : pas de retry au-delà du reconnect implicite.
- Token frais obtenu mais connexion Cloud SQL timeout (> 30s socket connect) → log `reason_code=timeout`. Mitigation : vérifier socket Cloud Run `/cloudsql/<instance>` monté.
- Local dev sans `gcloud auth application-default login` ET sans `roles/cloudsql.instanceUser` sur le dev → boot fail + log `reason_code=metadata_token_failed` (sur Linux dev hors GCP) OU `cloud_sql_auth_failed` (si auth ADC OK mais pas de binding DB). Runbook §8 documente le setup.

## Performance / SLO

- `before_acquire` cache-hit ≤ 1 ms wall-clock (AC-4). Acquisition sqlx pool baseline ~0.1 ms ; le hook ajoute la lecture `RwLock<Option<CachedToken>>` (~10 µs) + comparaison `Instant`.
- Cache-miss (premier connect ou refresh-ahead) : fetch metadata typique 5-20 ms sur Cloud Run + auth Cloud SQL 50-200 ms = ~250 ms worst case par connexion physique.
- Refresh metadata par instance : ~1 fetch toutes les 45 min × N instances. Pour N=10 instances : ~13 fetchs/h. Quota IAM access token = 1000 req/s/projet → marge x100k.
- p95 acquire pool (cache hit) ≤ 5 ms (overhead négligeable vs baseline sqlx).
- p95 handler gateway (qui acquire une connexion) ≤ 200 ms inchangé (le coût SEC-005 est sub-millisecond cache-hit).
- Worst-case boot : ~5 s (metadata fetch + Cloud SQL connect + first migration check), sous le timeout Cloud Run startup 240 s.

## Security / trust boundary

- A02 Cryptographic Failures : suppression définitive de tout password DB statique en env / Secret Manager. La gateway/workers n'ont jamais accès qu'à un token court (TTL 1 h max) obtenu via workload identity Cloud Run. Conforme `security.md` §A02 « no SA JSON keys » étendu aux credentials DB.
- A07 Authentication Failures : auth Cloud SQL utilise le mécanisme natif IAM (token bearer émis par Google sur preuve d'identité du SA Cloud Run). Pas de credential applicatif géré.
- A09 Logging : token bearer JAMAIS loggé (AC-10), `DATABASE_URL` ne contient jamais de password donc safe en log host. Conforme §A09 (sensitive types : `SecretString` Rust / `SecretStr` Python AC-7).
- A10 SSRF : URLs fetchées par `TokenProvider` / `SqlTokenProvider` = constantes (`http://metadata.google.internal/...`). Aucun input user ne contrôle l'URL.
- A05 Misconfiguration : disparition d'une variable d'env password réduit la surface secret-management. Fail-closed au boot (AC-8, AC-9) évite tout démarrage en mode dégradé silencieux.
- IAM principle of least privilege : aucun nouveau binding ajouté par SEC-005 (les bindings `roles/cloudsql.client` + `roles/cloudsql.instanceUser` sont déjà en place via INFRA-002 PR-e). Scope OAuth `sqlservice.admin` strictement requis pour Cloud SQL IAM auth — pas de scope sur-privilégié.
- Trust boundary : nouvelle dépendance runtime sur metadata server Cloud Run (déjà critique pour SEC-004, donc pas d'élargissement). Indisponibilité = boot fail explicite (AC-8/9) ou requête fail au reconnect (failure modes), jamais fail-open.

- Event boot `event="boot.sql_pool_init_failed"` même structure (AC-8, AC-9), différencié par absence de `request_id` et présence d'un champ `phase="boot"`.
