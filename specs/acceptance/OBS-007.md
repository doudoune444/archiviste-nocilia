# OBS-007 — Runner OIDC auth pour appels workers authentifiés (code-only)

## Contexte

Le runner Ragas (`eval/`) ne peut pas appeler les workers prod : les workers exigent IAM `run.invoker` (aucun binding `allUsers`), donc tout appel HTTP sans en-tête `Authorization` reçoit `403`. Aujourd'hui `eval/clients.py` ne pose que `{"X-Request-Id": ...}` et `eval/ragas_runner.py` interroge `/healthz` sans authentification — le runner n'a aucun code de token OIDC. Ce ticket (première slice de l'EPIC OBS-EVAL) ajoute, **code uniquement (aucun changement infra)**, la récupération d'un ID token Google signé depuis le metadata server (audience = URL workers) et son attachement en `Authorization: Bearer <id_token>` sur les appels workers, en restant un no-op pour local/CI.

## Acceptance criteria

- AC-1 : Quand l'URL workers cible a un scheme `https` ET un host **hors** de `{localhost, 127.0.0.1, ::1}`, chaque appel sortant du runner vers les workers (`GET /healthz`, `POST /v1/retrieve`, `POST /v1/generate`) porte un en-tête `Authorization: Bearer <id_token>` en plus de `X-Request-Id` (décision D-1).
- AC-2 : Quand l'URL workers cible est `http://localhost:8000` (ou tout host ∈ `{localhost, 127.0.0.1, ::1}`, **ou** scheme `http`), aucun de ces appels ne porte d'en-tête `Authorization` ; le comportement réseau du runner est identique à avant ce ticket (les en-têtes restent `{"X-Request-Id": ...}`) (décision D-1, no-op local/CI).
- AC-3 : La décision « attacher un token / ne pas attacher » dépend **uniquement** de l'URL workers (scheme + host), sans flag CLI ni variable d'environnement nouvelle ; il n'existe aucune nouvelle option de configuration introduite par ce ticket (décision D-1).
- AC-4 : L'audience de l'ID token est dérivée de `--workers-url` réduite à `scheme://host[:port]` — le path, la query et un éventuel slash de fin sont retirés (normalisation) ; pour `--workers-url https://workers.example.run.app/` ou `.../v1/retrieve`, l'audience est `https://workers.example.run.app` (décision D-2).
- AC-5 : La récupération du token passe par un seam injecté `IdTokenProvider` (contrat : `audience: str -> token: str`) ; les clients et la sonde `/healthz` reçoivent ce provider par injection, et aucun appel réseau réel au metadata server n'a lieu dans les tests (décision D-4).
- AC-6 : En cible authentifiée (AC-1), si le provider échoue à produire un token, le runner **échoue vite, avant d'exécuter la moindre entrée** du golden set (aucun appel `retrieve`/`generate` n'est tenté), termine avec exit code `5`, et n'effectue **aucun** fallback non authentifié (décision D-3).
- AC-7 : L'échec de récupération de token (AC-6) émet un log `event=oidc_token_failed` qui ne contient **aucun octet du token** ni détail de credential brut (audience workers loggable, mais pas de bytes de token, pas de clé) (décision D-3, `security.md` §A09).
- AC-8 : L'exit code `5` est **distinct** des codes existants du runner (`0` succès, `1` gate/error-rate, `2` schéma/CLI, `3` workers unreachable, `4` persist failure) ; le docstring d'en-tête de `eval/ragas_runner.py` documente `5 — OIDC token fetch failure (authenticated target)` (décision D-3).
- AC-9 : `eval/pyproject.toml` déclare `google-auth>=2` dans les `dependencies` du projet (core, pas `dev`) et ajoute un override mypy `ignore_missing_imports` pour `google.auth` / `google.auth.*` (miroir de l'override `psycopg2` existant) ; aucun ADR n'est requis (décision D-5).
- AC-10 : En cible non authentifiée (AC-2), le provider `IdTokenProvider` n'est **jamais** invoqué (aucune tentative de récupération de token) — vérifiable via un provider espion qui assert zéro appel sur le chemin localhost.
- AC-11 : Aucun octet de token n'apparaît jamais dans le fichier de run JSON, stdout, stderr, ni dans un message de log — y compris sur le chemin d'échec (AC-7) (`security.md` §A09, cohérent OBS-003 AC-10 pour les secrets).
- AC-12 : `specs/openapi/gateway-to-workers.yml` n'est **pas** modifié : l'ajout de l'en-tête `Authorization` est une préoccupation transport runner→workers, pas un changement de contrat REST ; aucune route nouvelle, aucun champ de payload nouveau (décision D-1).

## Non-goals

- Pas de changement infra : aucun fichier Terraform, aucun binding IAM, aucune image, aucun Cloud Run Job (l'eval Job tourne déjà comme SA `archiviste_runtime` qui détient `run.invoker` sur workers — VÉRIFIÉ, `cloud_run.tf:287-293`). OBS-008 (image eval) et OBS-009 (Cloud Run Job + fetch golden GCS) sont des tickets aval séparés.
- Pas de fallback non authentifié sur cible authentifiée — échec de token = exit `5`, jamais un appel `Authorization`-less « au cas où » (AC-6, D-3).
- Pas de flag CLI ni d'env var nouvelle pour activer/désactiver l'auth — la décision est dérivée de l'URL (AC-3, D-1).
- Pas de cache / refresh / rotation de token applicatif — le provider fournit un token à la demande ; la durée de vie / le refresh sont la responsabilité de la bibliothèque sous-jacente, hors périmètre code de ce ticket.
- Pas de support d'autres schémas d'auth (clé API statique, JWT custom, mTLS) — uniquement l'ID token Google OIDC du metadata server.
- Pas de changement du contrat OpenAPI, des payloads `retrieve`/`generate`, ni du schéma du fichier de run (AC-12).
- Pas de modification du write path Postgres / `--persist` (OBS-003) ni des gates EVAL-001.
- Pas de retry/backoff sur la récupération de token — un échec = exit `5` direct (D-3).

## Pre-conditions

- **EVAL-001 mergé** : `eval/ragas_runner.py` + `eval/clients.py` opérationnels, exit codes `0..3` en place ; `--workers-url` (défaut `http://localhost:8000`) déjà parsé (`ragas_runner.py:60`).
- **OBS-003 mergé** : exit code `4` (persist failure) déjà réservé dans le docstring du runner — `5` est le prochain libre, sans collision (AC-8).
- **Cible prod existante (VÉRIFIÉ, hors périmètre de ce ticket)** : workers déployés `ingress=INGRESS_TRAFFIC_ALL`, endpoint public sans VPC connector ; l'eval Job tourne comme SA `archiviste_runtime` qui détient déjà `run.invoker` sur workers (`cloud_run.tf:287-293`) — **aucun changement IAM n'est requis ni inclus**.
- **Environnement metadata-aware en prod** : au runtime du Job Cloud Run, le metadata server Google est joignable et signe les ID tokens pour l'audience demandée (hypothèse d'exécution, non testée en CI ; les tests injectent un fake — AC-5).

## Failure modes

- Cible authentifiée + provider échoue à produire un token → aucune entrée exécutée, log `event=oidc_token_failed` sans bytes de token, exit code `5`, aucun appel workers non authentifié (AC-6, AC-7, AC-8).
- Cible non authentifiée (`http`/localhost) → provider jamais appelé, en-têtes inchangés vs avant ticket, aucun exit `5` possible sur ce chemin (AC-2, AC-10).
- `--workers-url` avec path/slash de fin sur cible authentifiée → audience normalisée à `scheme://host[:port]` ; le token est demandé pour l'origine, pas pour le path (AC-4).
- Workers répondent `403` malgré token attaché (IAM mal configuré côté infra, hors périmètre) → traité par le chemin d'erreur EXISTANT du runner (workers unreachable / `EntryError(upstream_error)`), **pas** par exit `5` (lequel est strictement réservé à l'échec de *récupération* de token, pas à un rejet workers).

## Touch points (informatif, non contraignant pour l'architect)

- `eval/clients.py` — `RetrieveClient` / `GenerateClient` reçoivent l'`IdTokenProvider` (seam, AC-5) ; sur cible authentifiée (AC-1) ajoutent `Authorization: Bearer <id_token>` aux en-têtes (aujourd'hui `{"X-Request-Id": ...}`, lignes 62 et 106), sinon header inchangé (AC-2).
- `eval/ragas_runner.py` — `_check_workers_reachable` (`/healthz`, ligne 86) attache le token sur cible authentifiée (AC-1) ; le runner construit le provider + décide authed/non-authed à partir de `--workers-url`, fail-fast exit `5` avant `_run_all_entries` si le token échoue (AC-6) ; docstring exit codes complété par `5` (AC-8).
- `eval/oidc.py` (ou module dédié) — **nouveau** : `IdTokenProvider` (contrat audience→token, AC-5), normalisation d'audience (AC-4), prédicat « cible authentifiée » (scheme https + host hors localhost, AC-1/AC-2).
- `eval/pyproject.toml` — `google-auth>=2` en `dependencies`, override mypy `google.auth`/`google.auth.*` (AC-9).
- `eval/tests/` (ou `tests/`) — unit tests : header présent (authed) / absent (localhost) avec fake provider (AC-1/AC-2/AC-5/AC-10), normalisation audience (AC-4), fail-fast exit `5` + log redacted (AC-6/AC-7/AC-11).
- `CHANGELOG.md` — entrée `## [Unreleased]` section OBS.

## Test oracle

- AC-1 : unit · `RetrieveClient`/`GenerateClient` + fake provider, cible `https://workers.example.run.app` → en-têtes capturés contiennent `Authorization: Bearer <fake>` ET `X-Request-Id`.
- AC-2 : unit · mêmes clients, cible `http://localhost:8000` → en-têtes capturés == `{"X-Request-Id": ...}` exactement, pas de clé `Authorization`.
- AC-3 : unit · aucune nouvelle option dans `_parse_args` ni nouvelle lecture d'env pour piloter l'auth ; le switch dépend seulement de l'URL (assertion : 2 runs même URL ⇒ même décision, indépendamment de l'environnement).
- AC-4 : unit/property · `audience("https://h.run.app/")` == `audience("https://h.run.app/v1/retrieve")` == `https://h.run.app` ; port préservé si présent (`https://h:8000/x` → `https://h:8000`).
- AC-5 : unit · injecter un fake `IdTokenProvider` ; assert qu'aucun appel réseau metadata réel n'est émis (le fake est l'unique source de token) sur tous les chemins testés.
- AC-6 : unit · provider lève sur cible authentifiée → `main()` retourne `5`, et un espion sur `RetrieveClient.search`/`GenerateClient.generate` assert **zéro** appel (fail-fast avant `_run_all_entries`).
- AC-7 : unit · provider lève → log capturé contient `event=oidc_token_failed` et **aucune** sous-chaîne du token fake injecté.
- AC-8 : contract · `main()` mappe l'échec token → `5` ; grep du docstring d'en-tête `ragas_runner.py` contient la ligne `5 — ...` ; `5 ∉ {0,1,2,3,4}` (non-collision).
- AC-9 : contract · `eval/pyproject.toml` `dependencies` contient `google-auth>=2` ; `[[tool.mypy.overrides]]` couvre `google.auth`/`google.auth.*` ; `mypy --strict` passe sur `eval/`.
- AC-10 : unit · provider espion (compteur d'appels) + cible localhost → compteur == 0 sur tous les appels du run.
- AC-11 : property · token fake = chaîne sentinelle haute-entropie ; après un run (authed ET chemin d'échec) → la sentinelle n'apparaît dans aucun de : run-file, stdout, stderr, logs capturés.
- AC-12 : contract · `git diff specs/openapi/gateway-to-workers.yml` après PR → 0 changement.

## Security / trust boundary

- Frontière : Runner (Job Cloud Run, SA `archiviste_runtime`) → workers Cloud Run authentifiés (`run.invoker`). Aucune surface Internet publique ajoutée (le runner est un client, pas un serveur).
- **A07 / A02 Auth** : ID token Google OIDC obtenu via le metadata server, audience = origine workers (AC-4) ; pas de credential statique en code, pas de clé en repo (`secret-hygiene.md`). Aucun `verify=False`.
- **A09 Logging** : le token est un secret — jamais sérialisé dans run-file/stdout/stderr/logs, y compris sur le chemin d'échec (AC-7, AC-11) ; le type porteur du token doit redacter (cf `secret-hygiene.md` §A09, équivalent `SecretStr`). Le log `oidc_token_failed` ne porte que l'audience, pas de bytes.
- **A05 Misconfiguration** : pas de fallback non authentifié sur cible authentifiée (AC-6) — un échec de token est un échec dur (exit `5`), pas une dégradation silencieuse vers un appel `403`.
- **A10 SSRF (note)** : l'audience est dérivée de `--workers-url` fournie par l'opérateur (non par un utilisateur final) ; pas de surface SSRF utilisateur introduite. (Le metadata server `169.254.169.254` est ici une dépendance légitime intentionnelle, non une cible SSRF.)

## Observability

- Log runner structuré (structlog, cohérent EVAL-001/OBS-003) : `event=oidc_token_failed` (champ `audience`, **sans** token) sur le chemin d'échec (AC-7). Un éventuel `event=oidc_token_attached` (audience seule, sans token) est autorisé mais non requis.
- Aucune métrique OpenTelemetry nouvelle, aucun tracing Langfuse (hors pipeline LLM) phase 1.

## Décisions

Arbitrages humains figés (round-1, 5 recos acceptées telles quelles).

- **D-1 — Switch dérivé de l'URL, pas de flag/env.** Token attaché ssi scheme `https` ET host ∉ `{localhost, 127.0.0.1, ::1}` ; sinon aucun en-tête `Authorization`. Aucune nouvelle option de config. Pas de changement OpenAPI — préoccupation transport (AC-1, AC-2, AC-3, AC-12).
- **D-2 — Audience = `--workers-url` normalisée à `scheme://host[:port]`.** Path/query/slash de fin retirés ; règle de normalisation contractuelle (AC-4).
- **D-3 — Échec token sur cible authentifiée = fail-fast, exit `5` dédié, log redacted, aucun fallback.** Échec *avant* d'exécuter la moindre entrée ; exit `5` distinct de `1/2/3/4` ; log `event=oidc_token_failed` sans bytes de token ; jamais d'appel non authentifié de repli (AC-6, AC-7, AC-8).
- **D-4 — Seam étroit `IdTokenProvider` (audience→token) injecté.** Injecté dans clients + sonde `/healthz` ; les tests injectent un fake et assertent header présent (authed) / absent (localhost), sans appel metadata réel (AC-5, AC-10).
- **D-5 — `google-auth>=2` en dépendances core + override mypy `google.auth.*`, pas d'ADR.** Miroir de l'override `psycopg2` existant (AC-9).

## Open questions

- **OQ-1 — Emplacement du module OIDC.** `eval/oidc.py` (proposé) vs intégration dans `eval/clients.py`. Sans incidence sur les ACs (toutes formulées en comportement observable) ; trancher au `/plan`.
- **OQ-2 — Type porteur du token côté Python.** `security.md` §A09 demande un type redacted (`pydantic.SecretStr`) pour tout token. `pydantic>=2.9` est déjà une dépendance core (`eval/pyproject.toml:10`). Le wrapping du token en `SecretStr` est-il **exigé contractuellement** (durcir AC-11 en imposant le type) ou suffit-il de garantir la non-sérialisation observable ? Reco : exiger `SecretStr` (défense en profondeur, coût nul). À confirmer par l'humain.
- **OQ-3 — API exacte de `google-auth` pour le metadata fetch.** Le contrat de seam (`audience→token`) est figé ; la fonction sous-jacente concrète (p. ex. `google.oauth2.id_token.fetch_id_token`) est un détail d'implémentation laissé à l'architect/implementer (AC ne nomme aucune fonction). Vérifier au `/plan` que la lib expose bien un fetch metadata par audience sans dépendance lourde supplémentaire (sinon `no-workaround.md` → blocker).
