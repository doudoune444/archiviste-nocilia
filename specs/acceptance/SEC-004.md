# SEC-004 — GCS V4 signing via IAM signBlob (drop SA private key)

## Contexte

UI-002 a livré le signage V4 GCS pour `GET /v1/conversations/{id}/signed-url` en RSA-PKCS1-SHA256 hors-ligne (`gateway/src/gcs/sign.rs`), avec la clé privée du service account injectée via `GCS_SIGNING_PRIVATE_KEY_PEM`. `.claude/rules/security.md` §A02 interdit la détention de clés SA en clair (mode opérationnel "no SA JSON keys"), et `docs/blockers.md` 2026-05-28 §INFRA-002 a entériné qu'INFRA-002 PR-e ne sert qu'à débloquer le boot avec un placeholder ; le refactor sécurité est porté ici. SEC-004 bascule le signage vers l'API HTTP IAM `signBlob` (auto-impersonation du SA runtime), supprime le champ secret du `Config`, et ajoute le binding IAM minimal requis.

## Critères d'acceptation

- AC-1 : `gateway/src/config.rs` ne contient plus le champ `gcs_signing_private_key_pem` ni la lecture de la variable d'environnement `GCS_SIGNING_PRIVATE_KEY_PEM`. Un `grep -RIn 'GCS_SIGNING_PRIVATE_KEY_PEM\|gcs_signing_private_key_pem' gateway/ infra/ .env.example` retourne zéro résultat hors fichiers de documentation historique (CHANGELOG, runbook §changelog). Le champ `gcs_signing_sa_email` est conservé.
- AC-2 : La fonction `sign_get` (renommée ou conservée) de `gateway/src/gcs/sign.rs` produit une URL signée V4 GCS valide en appelant l'endpoint HTTP IAM `POST https://iamcredentials.googleapis.com/v1/projects/-/serviceAccounts/{sa_email}:signBlob` avec body `{"payload": "<base64(string_to_sign)>"}` et header `Authorization: Bearer <token>` ; la valeur `signedBlob` (base64) retournée est décodée puis hex-encodée et insérée comme `X-Goog-Signature` dans la query string. L'URL finale conserve exactement le format `https://{bucket}.storage.googleapis.com/{object}?X-Goog-Algorithm=GOOG4-RSA-SHA256&X-Goog-Credential=...&X-Goog-Date=...&X-Goog-Expires=300&X-Goog-SignedHeaders=host&X-Goog-Signature=<hex>` (TTL `SIGNED_URL_TTL_SECONDS=300` inchangé). La valeur `X-Goog-Algorithm=GOOG4-RSA-SHA256` est conservée : IAM `signBlob` signe avec la clé Google-managée par défaut du SA (RSA-2048 + SHA-256, ref `iamcredentials.googleapis.com` v1) — compatible GCS V4 signing.
- AC-3 : Le token OAuth bearer utilisé pour appeler `signBlob` est obtenu auprès du metadata server Cloud Run `GET http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token` avec header `Metadata-Flavor: Google`. Le token et son `expires_in` sont encapsulés dans un struct dédié `TokenProvider` (fichier `gateway/src/gcs/token.rs`, ~50 LOC) exposé via `AppState` par référence (Arc). Le `TokenProvider` détient en interne un `tokio::sync::RwLock<Option<CachedToken>>` ; aucune `RwLock` brute n'est ajoutée à `AppState`. Premier appel = fetch lazy ; appels suivants réutilisent le token cache. Refresh-ahead : le token est rafraîchi quand `now >= expires_at - 60s`. Tolérance race : sur réponse `401` de `signBlob` avec un token cache, la cache est invalidée et un seul retry est effectué avec un token frais. Le struct dédié est injectable en test (mock HTTP) pour découpler la logique cache/refresh de `AppState`.
- AC-4 : Toute défaillance du chemin de signage (token fetch metadata échoué, `signBlob` 4xx/5xx, timeout réseau, IO) est convertie en `ApiError::UpstreamUnavailable` (HTTP 503, enveloppe `{"error":"upstream_unavailable","request_id":"<uuid>"}` réutilisée — contrat SEC-001 inchangé). Aucune retry policy applicative au-delà du retry-on-401-after-cache-invalidation d'AC-3 (phase 1).
- AC-5 : Chaque échec d'AC-4 émet un événement `tracing` structuré niveau `warn` `event="dashboard.signing_failed"` avec champ `reason_code` ∈ exactement `{"signblob_403","signblob_429","signblob_5xx","metadata_token_failed","network","timeout"}` (mapping : `403 Forbidden` du signBlob → `signblob_403` ; `429` → `signblob_429` ; `5xx` → `signblob_5xx` ; échec fetch token metadata → `metadata_token_failed` ; erreur transport ou DNS → `network` ; dépassement timeout dédié → `timeout`). Aucun champ ne contient le token bearer, le `signedBlob` brut, ni la `string_to_sign` complète. Champs systématiques additionnels : `request_id`, `user_id`, `latency_ms`.
- AC-6 : Le client HTTP utilisé pour `signBlob` ET pour le fetch metadata token utilise des timeouts dédiés hardcodés dans `gateway/src/gcs/sign.rs` : `connect_timeout = 2s`, `total_timeout = 5s`. Aucune valeur n'est lue depuis `Config`. La worst-case latence du handler `GET /v1/conversations/{id}/signed-url` reste ≤ 10s (fetch token + signBlob, dans le pire cas cache miss), sous le timeout workers global 35s.
- AC-7 : `infra/terraform/iam.tf` (ou équivalent module Terraform existant) contient une ressource `google_service_account_iam_member` (ou `google_service_account_iam_binding`) qui accorde `roles/iam.serviceAccountTokenCreator` à `serviceAccount:archiviste-runtime@<project>.iam.gserviceaccount.com` SUR le service account lui-même `archiviste-runtime@<project>.iam.gserviceaccount.com` (self-impersonation). Un `terraform plan` post-merge sur l'environnement cible montre cette binding existante (idempotent) ou son ajout, sans aucune modification d'autres bindings IAM hors-scope.
- AC-8 : `infra/terraform/cloud_run.tf` ne déclare plus de bloc `env` injectant `GCS_SIGNING_PRIVATE_KEY_PEM` sur le container `archiviste-gateway`. Le secret Manager `gcs_signing_private_key` (créé par INFRA-002 PR-e comme placeholder) est supprimé via un bloc `removed` Terraform (Terraform >= 1.7) déclaré dans `infra/terraform/secrets.tf`, garantissant la suppression reproductible audit-traçable en repo (aligné ADR-0003). Aucune suppression manuelle `gcloud secrets delete` n'est acceptée comme alternative. Le secret `GCS_SIGNING_SA_EMAIL` (valeur = identifiant SA runtime) reste injecté tel quel.
- AC-9 : Les tests d'intégration `gateway/tests/test_dashboard_backend.rs` et `gateway/tests/test_signed_url.rs` (et tout autre test qui construit un `Config`) ne référencent plus `gcs_signing_private_key_pem` ni la fixture RSA `rsa_fixture`. Les scenarios qui exerçaient `sign_get` utilisent désormais `mockito` pour stub l'endpoint IAM `signBlob` ET l'endpoint metadata token : (a) cas nominal → 200 OK metadata + 200 OK signBlob → URL signée conforme AC-2 ; (b) signBlob 403 → 503 + log `reason_code=signblob_403` ; (c) signBlob 429 → 503 + `reason_code=signblob_429` ; (d) signBlob 500 → 503 + `reason_code=signblob_5xx` ; (e) metadata token 500 → 503 + `reason_code=metadata_token_failed` ; (f) signBlob timeout (mockito delay > 5s ou close) → 503 + `reason_code=timeout` ; (g) cache hit : deux appels successifs → un seul fetch metadata (mockito `expect(1)` sur l'endpoint token) ; (h) retry-on-401 : signBlob répond 401 puis 200 → URL retournée, mockito vérifie 2 appels signBlob + 2 appels token.
- AC-10 : `gateway/tests/common/jwt_helpers.rs` ne contient plus le champ `gcs_signing_private_key_pem` dans le helper de construction de `Config` pour les tests. `gateway/Cargo.toml` n'a plus de dépendance dev sur `rsa` ni `pkcs8` si elles servaient uniquement à `rsa_fixture` ; `ring` reste si utilisé par d'autres modules.
- AC-11 : `.env.example` ne mentionne plus `GCS_SIGNING_PRIVATE_KEY_PEM`. Une ligne de commentaire indique que le signage GCS se fait désormais via IAM signBlob auto-impersonation (renvoi au runbook §5).
- AC-12 : Le call site unique `gateway/src/handlers/conversations.rs:86` est mis à jour pour appeler la nouvelle signature de `sign_get` (qui devient `async` puisqu'elle effectue des appels HTTP). La fonction prend en argument un handle `&TokenProvider` (ou `&AppState`) au lieu du `&str` PEM. Aucun autre call site n'apparaît au `grep sign_get gateway/src`.

## Non-goals

- Pas de migration vers une crate GCS officielle (`google-cloud-storage`, `cloud-storage-rs`) — refactor minimal, dep ajoutée seulement si nécessaire pour le client HTTP IAM (le `reqwest` déjà transitif via `axum` / workers client est suffisant ; aucune nouvelle dep lourde sans ADR).
- Pas de mécanisme de retry exponential backoff sur `signBlob` (un seul retry conditionnel au cas 401-after-cache, AC-3). Backoff complet = SEC-* V2 si observé en prod.
- Pas de circuit breaker autour de l'API IAM phase 1.
- Pas de signage des URLs en local dev sans `gcloud auth application-default login` ET `roles/iam.serviceAccountTokenCreator` sur `archiviste-runtime` — aucun fallback `cfg(debug_assertions)` (rule `secret-hygiene.md` "no fallback default in code"). L'endpoint retourne 503 si non configuré ; runbook documente la procédure d'activation locale.
- Pas de support multi-projet / cross-project SA impersonation phase 1 (le SA cible reste `archiviste-runtime` du projet courant).
- Pas de rotation de token côté serveur initiée par signal externe — refresh-ahead lazy (AC-3) suffit.
- Pas de live test `--ignored` contre l'API IAM réelle phase 1 — vérification post-déploiement via curl runbook (cf Observability).
- Pas de propagation du `reason_code` vers le client HTTP (le client reçoit toujours `upstream_unavailable` opaque, AC-4).
- Pas de métrique counter dédiée `signblob_failures_total{reason_code}` dans ce ticket — l'instrumentation observabilité reste limitée au log `tracing` (AC-5) ; passage en métrique = OBS-001 V2.
- Pas de modification du contrat REST `GET /v1/conversations/{id}/signed-url` (path, query, response body shape, status codes inchangés — UI-002 reste source de vérité).

## Failure modes

- Metadata server indisponible (fetch token fail) → `503 upstream_unavailable` + log `reason_code=metadata_token_failed`. Aucun fallback. Sur Cloud Run, condition très rare.
- `signBlob` retourne `403 Forbidden` (binding `serviceAccountTokenCreator` manquante ou révoquée) → `503 upstream_unavailable` + log `reason_code=signblob_403`. Mitigation = AC-7 (Terraform binding). Détection en prod via le log warn.
- `signBlob` retourne `429 Too Many Requests` (quota IAM dépassé) → `503 upstream_unavailable` + log `reason_code=signblob_429`. Quota IAM signBlob défaut = 60 000 req/min/projet ; volume attendu ~10 req/s peak (`docs/blockers.md` 2026-05-28) → marge x100. Si atteint, ADR caching plus agressif.
- `signBlob` retourne `5xx` (indisponibilité Google) → `503 upstream_unavailable` + log `reason_code=signblob_5xx`. Pas de retry phase 1.
- Timeout réseau (> 5s total) → `503 upstream_unavailable` + log `reason_code=timeout`.
- Race condition cache token : deux requêtes simultanées trouvent le token expiré, deux fetchs metadata parallèles → acceptable (idempotent, coût négligeable, sémantique RwLock = pas de blocage longue durée). Documenté comme acceptable.
- Token cache hit mais token rejeté `401` par signBlob (rotation côté Google, clock skew) → invalidation cache + retry unique (AC-3). Si le retry échoue aussi 401 → `503` + log `reason_code=signblob_403` (le 401 final est traité comme 403 côté reason_code pour ne pas créer un 7e code).
- Local dev sans `gcloud auth application-default login` valide → endpoint `/v1/conversations/{id}/signed-url` retourne `503` systématiquement. Runbook documente le setup attendu.
- Placeholder PEM (legacy INFRA-002 PR-e) encore présent en env Cloud Run au déploiement SEC-004 → AC-1 garantit que le champ n'est plus lu ; le placeholder devient mort code, supprimé par AC-8.

## Performance / SLO

- `sign_get` p95 (cache hit) ≤ 250 ms : 1 appel HTTP `signBlob` (datacenter-local sur Cloud Run) typique 50-150 ms.
- `sign_get` p95 (cache miss, premier appel ou refresh) ≤ 400 ms : metadata fetch (~5-20 ms) + signBlob (~50-150 ms) + marge.
- Handler `GET /v1/conversations/{id}/signed-url` p95 ≤ 500 ms (overhead DB lookup conservation_id ≤ 50 ms + sign_get + sérialisation).
- Volume attendu (`docs/blockers.md` 2026-05-28) : ~0,01 req/s moyen, ~10 req/s peak. Quota IAM signBlob projet = 60 000 req/min = 1 000 req/s — marge x100, aucune contention attendue.
- Réduction d'appels metadata grâce au cache (AC-3) : sans cache ~864 fetchs/jour (1/req à 0,01 req/s 24h) ; avec cache ~24 fetchs/jour (token TTL 1h = 24 refresh/jour). Réduction ~36x, cohérent avec brief.

## Security / trust boundary

- A02 Cryptographic Failures : suppression de la clé privée SA en clair côté gateway. La gateway n'a plus la matière cryptographique permettant de signer arbitrairement au nom du SA — seule l'API IAM (côté Google) détient la clé. Conforme `.claude/rules/security.md` §A02 mode "no SA JSON keys".
- A05 Misconfiguration : la disparition du secret Manager `gcs_signing_private_key` réduit la surface secret-management ; un seul placeholder à nettoyer (AC-8).
- A07 Authentication Failures : l'auto-impersonation via metadata token utilise le mécanisme natif Cloud Run (workload identity) — pas de credential applicatif géré par la gateway.
- A09 Logging : token bearer, `signedBlob` brut, `string_to_sign` complète JAMAIS loggés (AC-5). Conforme §A09 (sensitive types). Aucun nouveau champ PII.
- A10 SSRF : l'appel `signBlob` cible une URL FIXE `https://iamcredentials.googleapis.com/...{sa_email}:signBlob` ; le `sa_email` provient de la config gateway (variable env trusted, pas de input user) → pas de risque SSRF. Le metadata server URL est également constant. Aucun input utilisateur ne contrôle l'URL d'aucune des deux requêtes.
- Trust boundary : nouveau dépendance runtime sur 2 endpoints Google (`metadata.google.internal`, `iamcredentials.googleapis.com`). Indisponibilité de l'un ou l'autre = endpoint `/signed-url` retourne 503 (AC-4) — failure mode connu, pas de fail-open.
- IAM principle of least privilege : binding ajoutée (AC-7) = `roles/iam.serviceAccountTokenCreator` strictement sur `archiviste-runtime` self → permet uniquement `signBlob`/`signJwt`/`generateAccessToken` du SA sur lui-même. Aucun autre SA impacté.
