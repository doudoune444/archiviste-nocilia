# GEN-003 — Mode 2 off-topic : intent classifier + branche refus poli

## Contexte

La vision (`docs/vision.md` §20-25, ordre d'attaque #4) impose 4 modes de réponse routés par un classifier d'intent. GEN-001/GEN-002 ont livré Mode 1 canon end-to-end mais le handler `/v1/generate` hardcode `mode="canon"` (cf `workers/src/archiviste_workers/generate/router.py` ligne 207 et `generate/models.py` `Mode = Literal["canon"]`). Sans étape `intent` en tête de pipeline, toute question hors-univers (recettes de cuisine, météo, code Python, sport — cf golden_qa `GQA-035..038`) traverse retrieve+LLM, gaspille tokens, et produit une réponse incohérente avec la persona Archiviste. Ce ticket pose la première étape du pipeline `intent → ACL → retrieve → coherence → answer` (vision §32) en livrant uniquement la branche `off_topic` qui court-circuite après `intent`, retourne un refus poli in-world + suggestions in-domain, et trace `mode="off_topic"` dans `query_log`. Les branches `lore_gap` (GEN-004) et `acl_blocked` (GEN-005) restent hors-scope.

## Critères d'acceptation

- AC-1 : `POST /v1/generate` accepte le même body qu'en GEN-001 (`{query, conversation_id, user_id, user_tier, request_id}`) et retourne `200 OK` avec body `{answer, citations: [], mode, conversation_id, request_id, usage, retrieve_ms, llm_ms}` où `mode ∈ {"canon", "off_topic"}` ; aucun autre champ n'est ajouté ou retiré par rapport à GEN-001.
- AC-2 : Avant tout appel à `/v1/retrieve`, le handler appelle un classifier d'intent qui retourne exactement une valeur de l'ensemble `{"in_domain", "off_topic"}` ; la valeur est écrite dans `query_log.intent` (colonne déjà présente, cf `migrations/0002_schema.sql:71`).
- AC-3 : Si le classifier retourne `off_topic`, le pipeline court-circuite : aucun appel à `/v1/retrieve`, aucun appel LLM de génération canon, aucun appel ING-003 `messages` n'est exécuté entre la décision intent et la génération du refus.
- AC-4 : Le classifier d'intent est implémenté via un second appel LLM passant par le wrapper `workers/src/archiviste_workers/services/llm.py` existant (provider config-driven, MVP `mistral-small-latest` conforme vision §69) ; le code applicatif `generate/router.py` n'importe aucun nom de provider.
- AC-5 : Le system prompt du classifier d'intent est figé byte-for-byte (testable via constante exportée) et instruit le LLM à retourner exclusivement le mot `in_domain` ou le mot `off_topic`, sans guillemets, sans ponctuation, sans préambule. Le user prompt contient la question brute préfixée par `[user query]: ` ; aucun chunk lore n'est injecté (le classifier est zero-shot, le contexte de l'univers Nocilia est dans le system prompt sous forme de description courte ≤ 500 caractères).
- AC-6 : La sortie LLM du classifier est normalisée (lowercase + trim) puis matchée contre `{"in_domain", "off_topic"}` ; toute autre sortie est traitée comme `in_domain` (fail-open vers canon) et un log warn `{"event":"intent_unparseable","request_id":"...","raw":"<sortie tronquée 100 chars>"}` est émis ; aucune erreur n'est retournée au client.
- AC-7 : Quand `intent="off_topic"`, le handler génère la réponse via un **second appel LLM** avec un prompt dédié (system + user) qui produit (a) un refus court in-world ton Archiviste (≤ 80 mots), (b) exactement 3 suggestions de questions in-domain plausibles formulées comme des questions complètes ; le format de sortie attendu est texte libre (le LLM intègre suggestions dans la prose, pas de JSON structuré).
- AC-8 : Le system prompt de génération du refus off_topic est figé byte-for-byte (constante exportée, testable) et contient au moins : identité Archiviste, instruction « refus poli court », instruction « propose 3 questions in-domain plausibles », instruction « ne romps jamais le character », instruction « réponds dans la langue de la question » (cohérence avec GEN-001 OQ-5).
- AC-9 : Le champ `citations` de la réponse vaut toujours `[]` quand `mode="off_topic"` (aucun chunk consommé) ; le champ `retrieve_ms` vaut `0` ; le champ `llm_ms` agrège la durée des **deux appels LLM** (classifier + refus) en millisecondes.
- AC-10 : Le champ `usage` de la réponse agrège `prompt_tokens` et `completion_tokens` des **deux appels LLM** (somme arithmétique des deux `Usage` extraits via `extract_usage`) ; `cost_eur` est la somme des coûts des deux appels (chacun calculé via `pricing.compute_cost_eur`).
- AC-11 : La conversation est persistée via ING-003 quand `mode="off_topic"` : exactement deux POST séquentiels `/v1/conversations/{id}/messages` (rôle `user` = `query`, rôle `assistant` = `answer` refus) avec le même contrat qu'en GEN-001 AC-14 ; un échec ING-003 émet un log ALERT `conversation_log_failed` sans bloquer la réponse 200.
- AC-12 : Exactement une ligne `query_log` est insérée avec `(mode='off_topic', intent='off_topic', status_code=200, latency_ms=<total wall clock>, prompt_tokens=<somme deux appels>, completion_tokens=<somme deux appels>, cost_eur=<somme deux appels>)` ; un échec INSERT émet un log ALERT `query_log_failed` sans bloquer la réponse 200.
- AC-13 : Si le classifier dépasse le timeout dur de **5 secondes** (timeout dédié, inférieur au 30 s du LLM de génération canon), le handler traite le résultat comme `in_domain` (fail-open vers canon) et émet un log warn `{"event":"intent_timeout","request_id":"..."}` ; aucune erreur n'est retournée au client.
- AC-14 : Si le classifier retourne une erreur 4xx ou 5xx du provider LLM (auth, quota, rate-limit, 5xx upstream), le handler traite le résultat comme `in_domain` (fail-open vers canon) et émet un log warn `{"event":"intent_upstream_error","request_id":"...","status":<code>}` ; aucune erreur n'est retournée au client.
- AC-15 : Quand `mode="off_topic"`, si le **second** appel LLM (génération du refus) timeout > 30 s → `504 Gateway Timeout` `{"error":"llm_timeout"}`, ligne `query_log` `status_code=504`, `mode='off_topic'`, `intent='off_topic'`, `prompt_tokens` et `completion_tokens` reflètent uniquement le classifier (seul appel comptabilisé), `cost_eur` idem.
- AC-16 : Quand `mode="off_topic"`, si le **second** appel LLM retourne 4xx/5xx provider → `502 Bad Gateway` `{"error":"llm_upstream"}`, ligne `query_log` `status_code=502`, mêmes règles de comptage usage que AC-15.
- AC-17 : Le pattern de prompt-injection détecté en GEN-001 AC-20 (`(ignore (all )?(prior|previous) instructions|new instructions|system\s*:|disregard (the )?above)`) reste appliqué sur la `query` AVANT classifier ; le user prompt envoyé au classifier ET au LLM de refus reçoit le préfixe `[user query, suspected injection]: ` au lieu de `[user query]: `, et le log warn `prompt_injection_suspected` est émis une seule fois par requête.
- AC-18 : Les validations input GEN-001 sont inchangées et s'appliquent AVANT le classifier : `query` vide ou > 4 KiB → `400 invalid_query`, `user_id`/`request_id` non-UUID → `400`, `user_tier` hors ensemble → `422`. Aucune ligne `query_log` n'est écrite, aucun appel LLM (classifier inclus) n'est fait sur ces cas.
- AC-19 : La branche `in_domain` du classifier déclenche le pipeline canon GEN-001 intégralement inchangé (retrieve → prompt build → LLM → parse citations → log) ; la seule différence observable côté client par rapport à GEN-001 est que `query_log.intent` vaut désormais `'in_domain'` au lieu de `NULL`.
- AC-20 : Le log INFO de fin de requête est étendu pour inclure le champ `intent` : `{"event":"generate","request_id":"...","conversation_id":"...","intent":"in_domain|off_topic","mode":"canon|off_topic","status":200,"retrieve_ms":R,"llm_ms":L,"chunks":C,"citations":K}` ; `chunks` et `citations` valent `0` quand `mode="off_topic"`.

- Pas de mise à jour de `specs/golden_qa.jsonl` (les 4 entrées `mode:off_topic` GQA-035..038 sont suffisantes pour EVAL-001 ; raffinage post-eval out-of-scope).

## Failure modes

- `query` vide / > 4 KiB / `user_id` ou `request_id` non-UUID / `user_tier` hors ensemble → mêmes codes que GEN-001 (400/422), aucun appel classifier, aucune ligne `query_log` (AC-18).
- Classifier timeout > 5 s → fail-open `in_domain` → pipeline canon ; log warn `intent_timeout` (AC-13).
- Classifier retourne 4xx/5xx provider → fail-open `in_domain` → pipeline canon ; log warn `intent_upstream_error` (AC-14).
- Classifier retourne une chaîne hors `{in_domain, off_topic}` → fail-open `in_domain` ; log warn `intent_unparseable` (AC-6).
- LLM de refus (mode off_topic) timeout > 30 s → `504 llm_timeout`, ligne `query_log` partielle (AC-15).
- LLM de refus retourne 4xx/5xx → `502 llm_upstream`, ligne `query_log` partielle (AC-16).
- ING-003 4xx/5xx en mode off_topic → réponse `200 OK`, log ALERT `conversation_log_failed` (AC-11).
- INSERT `query_log` échoue → réponse `200 OK`, log ALERT `query_log_failed` (AC-12).
- `query` matchant prompt-injection → préfixe `[user query, suspected injection]: ` propagé classifier + LLM refus (AC-17).

## Performance / SLO

- Latence classifier intent : **p95 < 500 ms** (vision §89 budget global p95 < 3 s ; classifier ajoute un appel LLM court ~50-100 tokens output, timeout dur 5 s déclenche fail-open AC-13).
- Latence totale mode off_topic round-trip : **p95 < 3 s** (vision §89) — classifier 500 ms + LLM refus 1-2 s + ING-003 × 2 < 200 ms + overhead handler.
- Aucune gate CI bloquante phase 1 ; mesure indicative via `query_log.latency_ms` filtrée sur `mode='off_topic'`.

## Security / trust boundary

- Endpoint internal-only inchangé (cohérent GEN-001) ; le classifier est un appel LLM interne au worker, pas une surface réseau nouvelle.
- Prompt injection (security.md « RAG-specific ») : pattern détecté par `injection_filter.detect_injection` (déjà livré GEN-001) propagé en préfixe sur le classifier ET le LLM de refus (AC-17). Le classifier reste robuste car son system prompt instruit une sortie binaire stricte ; une sortie hors `{in_domain, off_topic}` (potentiellement injectée) déclenche fail-open canon (AC-6) sans bypass.
- Aucun chunk lore n'est envoyé au classifier (AC-5) — la surface d'injection « poisoned doc » (W-I-1) est nulle sur cette branche.
- Détail provider LLM jamais exposé au client (cohérent GEN-001 AC-21) — uniquement `{"error":"llm_upstream"}` ou `{"error":"llm_timeout"}` sur la branche refus.
- Aucun log de la `query` brute, du contenu du refus, ou de la sortie LLM classifier (security.md A09) ; seuls metadata (intent, latencies, request_id, model, prompt_tokens) loggés. La sortie classifier raw n'apparaît qu'en cas `intent_unparseable` tronquée à 100 chars (AC-6) — acceptable car sortie courte, pas de PII attendue.
- `LLM_API_KEY` reste `pydantic.SecretStr` (security.md A09), partagée entre classifier et générateur (1 secret, 1 provider conforme vision Q1).

## Observability

- Log INFO `event=generate` étendu avec champ `intent` (AC-20).
- Log warn dédiés (nouveaux) : `intent_timeout`, `intent_upstream_error`, `intent_unparseable`.
- Log ALERT inchangés (réutilisation GEN-001) : `conversation_log_failed`, `query_log_failed`.
- `query_log.intent` désormais renseigné systématiquement (`'in_domain'` ou `'off_topic'`) — permet requêtes analytiques mode-mix post-deploy sans modification schéma.
- Pas d'intégration Langfuse ni OpenTelemetry à ce stade (cohérent GEN-001, réservé OBS-*) ; le wrapper `services/llm.py` doit conserver son point d'extension pour brancher Langfuse plus tard sans toucher `intent.py` ni `router.py`.

## Drifts détectés vs vision.md (à valider par l'humain au plan)

- **Drift D1 — Timeout classifier 5 s** : non spécifié par la vision (qui n'évoque qu'un budget p95 < 3 s global). Le ticket fige 5 s comme valeur cohérente avec le besoin « classifier rapide » (vision §69 « single provider phase 1 ») et inférieure au timeout 30 s du LLM de génération (GEN-001 AC-11). À confirmer au `/plan`.
- **Drift D2 — Fail-open vers canon** sur erreur/timeout classifier (AC-13/14) : choix conservateur (jamais refuser à tort une question légitime) mais introduit un coût LLM canon inutile sur des erreurs classifier répétées. Alternative possible : fail-closed avec réponse générique « réessayez ». La vision ne tranche pas. Choix proposé = fail-open, à confirmer.
- **Drift D3 — Génération du refus via second appel LLM** : la vision §20 dit « Refus court + suggestion de questions in-domain » sans préciser si la réponse est templatée (constante) ou générée. Le ticket choisit génération LLM (AC-7) pour cohérence ton in-world Archiviste et personnalisation des 3 suggestions. Coût marginal acceptable (~80 mots × prix Mistral Small). Alternative templatée possible si l'humain préfère économies coût/latence.
- **Drift D4 — Pas d'ACL avant retrieve** : la vision §32 décrit le pipeline `intent → ACL → retrieve → coherence → answer`. Ce ticket implémente uniquement `intent → retrieve (canon) | refus (off_topic)`. L'étape `ACL` (filtrage `access_tier`) est reportée à GEN-005 Mode 4 mystère (cohérent ordre d'attaque #8 vision §167). À confirmer que ce séquencement est acceptable.
- **Drift D5 — Pas de coherence step** : la vision §32 mentionne `coherence` entre `retrieve` et `answer`. Aucun ticket existant ne le couvre, vision plan n'en parle pas dans l'ordre d'attaque #1-9. Hors-scope GEN-003, signalé pour traçabilité.
